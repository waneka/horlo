# Debug: Phase 61 404 + React #419 (soft-nav only)

## ⚠⚠ REFRAME 2026-05-26 (recurrence ~8) — SHARED, NON-DETERMINISTIC cause; per-route fixes are whack-a-mole

After deploy 719fef9 (which changed ONLY /watch/new gap#9 + the filmstrip tile — touched NEITHER the profile route NOR /w/[ref] #419 code), the user reports the symptom FLIPPED: **profile /u/[username]/[tab] now 404s on soft-nav, and /w/[ref] does NOT.** The exact opposite of the post-8a49a19 split. User's own diagnosis (compelling): "it's like each release is inconsistent" + "the cache takes a little longer to fill up on those [profile] pages."

CONCLUSION — the working model changes:
- This is NOT "route X has a structural defect, route Y doesn't." It is a SHARED PPR/Cache-Components soft-nav fragility that surfaces on WHICHEVER route's edge cache fills into the bad state first. Each deploy wipes the edge cache → a freshly-deployed route behaves like a hard load (full dynamic SSR = always works) until its cached PPR shell fills, then #419 reappears. WHICH route you catch broken is a function of cache-fill timing, not the latest code.
- Therefore the per-route fixes this session were treating symptoms: admin-client signing (8a49a19) and the /w/[ref] Suspense wrap (f9c7a4b) may each be correct/necessary, but the SHARED root cause persists and keeps resurfacing.
- KEY INVARIANT to exploit: **hard refresh ALWAYS works** on every affected route (full dynamic SSR). The fix that most directly kills the whole family is to make soft-nav behave like a hard load — i.e., remove the prefetched PPR STATIC SHELL from these inherently per-viewer-dynamic routes (force fully dynamic) so there is no partial-prerender resume to abort. Candidate: `await connection()` (next/server) at the top of the affected page(s), or otherwise opting them out of the PPR static shell while keeping `'use cache'` data segments. Routes are per-user/cookie-dependent anyway, so the lost PPR optimization is marginal vs. the recurring prod breakage.
- DO NOT guess-deploy another per-route structural patch. Either (a) decisively force the affected dynamic routes out of the PPR static-shell path (bigger hammer, principled — soft-nav == hard-load), or (b) research the SHARED cause (root layout/Header cookie access? a shared 'use cache' fn with a latent dynamic-API access — getCatalogById/getLikesForTargetCached/getBatchedWatchCountsCached/ProfileShellResolver/signCoverUrls?) before any further change.
- VERIFICATION REMAINS PROD-ONLY and is now known to be NON-DETERMINISTIC per cache-fill timing — a single post-deploy "looks fixed" read is NOT sufficient; must observe each affected route AFTER its cache fills.

> ⚠️ **SUPERSEDED — the block below was an interim (admin-client-only) resolution; it was NECESSARY but NOT SUFFICIENT (the connection() static-shell opt-out above was the decisive part). Retained for history.**

- **root_cause:** Phase 61 added direct `createSupabaseServerClient()` calls (which invoke `cookies()`) for photo URL signing into RSCs that also call `'use cache'` functions. Unlike `getCurrentUser()` which is wrapped in `React.cache()` (memoized per render, does not re-execute `cookies()` in subsequent calls within the same render), the direct `createSupabaseServerClient()` calls re-execute `cookies()` fresh in every render pass. With Cache Components / PPR, Next.js runs a prospective + final runtime prerender pair (two separate render passes) to populate cache entries and produce the prefetch RSC payload. The unmemoized `cookies()` call in the photo signing code creates interference between these two passes, corrupting the PPR prerender boundary and causing React #419 on soft-nav. Call ordering (before or after 'use cache' calls) does NOT resolve this — the issue is that the call is unmemoized and re-executed in the secondary prerender pass.
- **fix:** Replaced `createSupabaseServerClient()` with `createSupabaseAdminClient()` (service role, synchronous, zero `cookies()` calls) in all photo signing code: `src/lib/storage/signCoverUrls.ts` (used by `[tab]/page.tsx` for cover URL signing) and `src/app/w/[ref]/page.tsx` (direct photo signing in Branch 1 + D-06). The admin client is safe for storage URL signing because paths are user-scoped by construction (IDOR fix) and signing creates time-limited access tokens — not data queries.
- **verified:** `npm run build` exits 0; all affected routes remain `◐ Partial Prerender`. Prod soft-nav re-test PENDING (requires `vercel deploy --prod` + user verification).
- **durable_rule:** In PPR routes, ONLY use `createSupabaseServerClient()` (or any `cookies()`-reading API) through a `React.cache()`-memoized wrapper OR within a component/function that is isolated in its own `<Suspense>` boundary (so it never re-executes in secondary prerender passes). For storage URL signing specifically: prefer `createSupabaseAdminClient()` — it is synchronous, cookie-free, and safe for generating time-limited signed URLs for user-scoped storage paths.
