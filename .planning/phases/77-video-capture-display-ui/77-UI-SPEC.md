# Phase 77 — UI Design Contract

> Visual and interaction contract for Phase 77: Video Capture + Display UI.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> This phase is additive-only. The existing photo capture and display paths
> must remain byte-identical before and after all changes (VID-15 regression
> gate). Every new element specifies parallel-or-absent behavior to the photo
> path it extends.

- Progress ring SVG: `size-16` (64px) wrapper, centered over the Record button. Ring stroke is positioned at the exact button edge — no gap between button and ring.

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px (text-sm) | 400 (font-normal) | 1.5 | Helper text ("Photo optional", "Video optional", error copy) |
| Label | 12px (text-xs) | 600 (font-semibold) | 1.5 | Button labels, tile overlay text, "Video unavailable" label |
| Heading | 16px (text-base) | 600 (font-semibold) | 1.25 | Dialog title "Log a wear" (existing; not changed) |
| Counter / status | 12px (text-xs) | 400 (font-normal) | 1 | Character counter, muted-foreground status labels |

Font-weight guardrail: use `font-semibold` (600) for button text and tile
overlays. DO NOT use `font-medium` (500) — prior phases have recurrently
shipped `font-semibold` where `font-medium` was spec'd, so this spec pins
`font-semibold` to match the shipped convention. Only `font-normal` (400) is
used for body text and helper text.

## Color

| Role | Light value | Dark value | Usage |
|------|-------------|-----------|-------|
| Dominant (60%) | oklch(0.985 0.003 75) = `bg-background` | oklch(0.14 0.005 75) | Page background, dialog background, pre-capture chooser bg-muted/30 surface |
| Secondary (30%) | oklch(0.95 0.005 75) = `bg-muted` | oklch(0.24 0.005 75) | Camera preview bg-black (recording surface), WearVideoClient container bg-muted, pre-capture chooser bg-muted/30 |
| Accent (10%) | oklch(0.76 0.12 75) = `bg-accent` | oklch(0.78 0.13 75) | Active/selected states only — "Discard" and "Retake"-equivalent text links |
| Destructive | oklch(0.55 0.22 27) light / oklch(0.65 0.2 27) dark | — | Recording indicator red dot only; video upload error inline text |
| White overlays | white / white/80 | white / white/80 | VideoPlayBadge icon, progress ring sweep stroke, recording red dot, tile bottom gradient text |
| Black overlays | black/70 | black/70 | Semi-transparent backdrop on VideoPlayBadge, tile bottom gradient scrim |

Accent (`bg-accent text-accent-foreground`) is reserved for:
1. The "Discard" text button on the video post-capture preview (mirrors existing
   "Retake" / "Choose another" link styling — `text-accent underline` pattern)
2. Active ring state on the chooser zone when recording is in progress (NOT applied; ring is red per D-11)

Accent is NOT used for:
- The Record 3s button (uses `variant="default"` = `bg-primary`)
- The progress ring stroke (white for visibility on the dark camera preview)
- The VideoPlayBadge backdrop (black/50 semi-transparent)

Dark mode pairing rule (durable guardrail): any `bg-X` utility applied to an
outline Button must be paired with `dark:bg-X`. The outline Button cva already
injects `dark:bg-input/30 dark:hover:bg-input/50`. If any new class overrides
`bg-` on an outline Button, the `dark:` paired override must accompany it.

Recording red dot color: `bg-red-500` (or `bg-destructive`). Must use
`bg-destructive dark:bg-destructive` pair — destructive tokens shift between
light and dark mode.

## Copywriting Contract

### Pre-Capture Chooser (3-button zone)

| Element | Copy |
|---------|------|
| Button 1 — camera photo | "Take wrist shot" |
| Button 2 — video record | "Record video" |
| Button 3 — upload | "Upload photo" |
| Helper text (existing) | "Photo optional" |
| Helper text when video button present | "Photo optional" — do NOT change to "Media optional"; keep the existing text unchanged (VID-15) |

The "Record video" button renders only when `supportsVideoCapture === true`.
When the button is hidden, the chooser falls back to the existing 2-button
layout — the helper text "Photo optional" stays unchanged.

### Live Recording Surface (VideoCaptureView)

| Element | Copy |
|---------|------|
| Primary action (before recording) | "Record 3s" |
| Primary action (while recording) | "Recording…" — disabled, ring animates |
| Primary action (poster extracting) | "Processing…" — disabled |
| Cancel button | "Cancel" |
| Recording announcement (sr-only for a11y) | aria-live="polite": "Recording started" on start; "Recording complete" on stop |
| Red dot aria-label | aria-label="Recording in progress" |

### Post-Capture Preview (video path)

| Element | Copy |
|---------|------|
| Discard link | "Discard" |
| No separate "Retake" — Discard returns to live video camera |

Rationale: the video path always discards the whole clip and returns to the
live VideoCaptureView (identical MediaStream). There is no "re-record from
upload" branch, so only one action is needed (vs. the photo path's
Retake / Choose another split per D-07).

### WearVideoClient — Detail Page Player

| Element | Copy |
|---------|------|
| Error fallback label | "Video unavailable" |
| Pause state aria-label | "Video paused — tap to resume" |
| Playing state aria-label | "Video playing — tap to pause" |

### Error States (in ComposeStep inline `role="alert"`)

| Error scenario | Copy |
|----------------|------|
| Camera unavailable for video | "Camera unavailable — use Upload photo instead." |
| Camera denied for video | "Camera access denied — use Upload photo instead." |
| Recording failed mid-way | "Recording failed — please try again." |
| Poster extraction failed | "Could not process the clip — please re-record." |
| Video upload failed | "Video upload failed — please try again." |
| Poster upload failed | "Poster upload failed — please try again." |
| Server Action: video too large | "Video too large — maximum 5 MB." (mirrors Phase 76 VID-09 server message) |
| Server Action: other failure | "Could not log that wear. Please try again." |

Error copy pattern: all errors follow the existing ComposeStep pattern —
plain `text-sm text-destructive` in a `role="alert"` paragraph. No toast.
No modal. No retry button. Error clears on next user interaction.

### VideoPlayBadge (tile overlay — no copy)

The badge is icon-only (`<Play fill="white" />`). No text label, no tooltip.
Duration label "0:03" is explicitly excluded per D-15.

### Destructive Actions

No destructive actions requiring a confirmation dialog in this phase.
"Discard" resets local state only (no DB mutation, no server call) — no
confirmation needed.

## Component Inventory

### New Components

#### 1. `VideoPlayBadge` — `src/components/wear/VideoPlayBadge.tsx`

Purpose: shared centered play icon overlay for tile surfaces. Used by
WywtTile, WearCard (in bottom-sheet/tile contexts), and any future video tile.

Visual contract:
- Container: `absolute inset-0 flex items-center justify-center pointer-events-none`
- Backdrop circle: `rounded-full bg-black/50 flex items-center justify-center`
  - Size scales with tile: `clamp(32px, 24%, 56px)` square — ~24% of the tile's
    min dimension, capped at 56px. Implementation: `style={{ width: 'clamp(32px, 24%, 56px)', height: 'clamp(32px, 24%, 56px)' }}`
  - The tile must be `relative overflow-hidden` for the absolute badge to clip correctly
- Icon: `<Play className="fill-white text-white" />` from lucide-react (filled triangle)
  - Icon size: same as backdrop minus 8px padding on each side; use `size-[calc(100%-16px)]` or hardcode `size-5` for the capped case
  - `fill="white"` is a `style` prop or the `fill-white` Tailwind class — confirm lucide's Play icon accepts `fill` class
  - The `<Play>` icon in lucide-react uses the stroke-based path by default; for a filled appearance apply `fill="white" stroke="none"` via `className="fill-white stroke-none size-5"` or `style={{ fill: 'white', stroke: 'none' }}`
- Dark mode: `bg-black/50` — works in both modes (on top of the poster image)
- Visibility: rendered ONLY when `mediaType === 'video'` in the parent
- NOT rendered in the composer's post-capture preview (D-16)

#### 2. `VideoCaptureView` — `src/components/wywt/VideoCaptureView.tsx`

Purpose: parallel to `CameraCaptureView`. Takes a pre-acquired `MediaStream`
as a prop and produces `{ videoBlob: Blob; posterBlob: Blob }`. Never calls
`getUserMedia` itself (iOS gesture context discipline: stream-as-prop pattern).

Visual contract (live camera preview region):
- Wrapper: `relative w-full aspect-square overflow-hidden rounded-md bg-black`
  — matches `CameraCaptureView` wrapper exactly (same aspect ratio, same rounding)
- Camera `<video>`: `block h-full w-full object-cover` with `autoPlay playsInline muted`
  — matches `CameraCaptureView` exactly
- `WristOverlaySvg`: `pointer-events-none absolute inset-0` — reuse identically
- Recording indicator (D-11): pulsing red dot
  - Position: `absolute top-3 left-3 z-20`
  - Size: `size-3` (12px) filled circle
  - Color: `bg-red-500 dark:bg-red-400` (or `bg-destructive dark:bg-destructive`)
  - Animation: `animate-pulse` (Tailwind built-in; respects `prefers-reduced-motion`)
  - Visibility: rendered only when `recording === true`
- Progress ring (D-09): SVG ring around the Record button that fills clockwise
  - Position: absolute overlay over the button center in the button strip below the video
  - Implementation: SVG `<circle>` with `strokeDasharray` / `strokeDashoffset` animated
    via CSS `animation: ring-fill 3s linear forwards` started at `recording === true`
  - Stroke color: `white` for contrast; transparent background
  - Stroke width: `3px`
  - Ring diameter: matches the "Record 3s" button's `min-h-11` outer height (44px)
  - The ring wrapper sits as a `relative` container around the Record button; the SVG
    is `absolute inset-0 pointer-events-none`

Button strip (below camera preview):
- Container: `mt-3 flex justify-center gap-2` — matches `CameraCaptureView` exactly
- Cancel button: `<Button variant="outline" className="min-h-11">Cancel</Button>`
  - Active during recording (D-12): NOT disabled while recording
  - Tapping while recording: aborts clip, returns to live camera, no blob produced
- Record button (before recording): `<Button className="min-h-11">Record 3s</Button>`
  - Uses `variant="default"` (bg-primary) — NOT outline
  - Changes to `<Button disabled className="min-h-11">Recording…</Button>` during recording
  - Changes to `<Button disabled className="min-h-11">Processing…</Button>` during poster extraction
  - Progress ring SVG wraps the outer layout, not the button element itself

Full component prop interface:
```typescript
interface VideoCaptureViewProps {
  stream: MediaStream
  preferredMimeType: string          // from useMediaCapability()
  onVideoReady: (result: { videoBlob: Blob; posterBlob: Blob }) => void
  onError: (message: string) => void
  onCancel: () => void
  disabled?: boolean
}
```

#### 3. `WearVideoClient` — `src/components/wear/WearVideoClient.tsx`

Purpose: parallel to `WearPhotoClient`. Renders `<video autoPlay muted loop playsInline>` with poster, tap-to-pause/resume (D-06), and error fallback (D-08).

Visual contract:
- Container class: `w-full aspect-[4/5] overflow-hidden bg-muted md:rounded-lg md:max-w-[600px] md:mx-auto md:max-h-[70vh] relative`
  — identical to `WearPhotoClient`'s happy-path container (VID-15 visual parity)
- `<video>` element:
  - `className="w-full h-full object-cover"`
  - Required attributes: `autoPlay muted loop playsInline`
  - `onError={() => setFailed(true)}`
  - `onClick` on the container div toggles `videoRef.current.pause()` / `.play()`
- `WearPhotoOverlays` rendered above the video (same absolute overlay pattern as `WearPhotoClient`)
- Error fallback (D-08): when `failed === true` or `signedVideoUrl === null`:
  - Show poster `<img>` (same classes: `w-full h-full object-cover`) if `signedPosterUrl` exists
  - Show `bg-muted` fill if no poster
  - Overlay: `<span className="absolute bottom-2 left-2 text-xs font-semibold text-white/70">Video unavailable</span>`
  - `WearPhotoOverlays` rendered as normal
  - Container must be `relative` in all error states (existing pattern from `WearPhotoClient` Phase 56 fix)

No retry state machine (unlike `WearPhotoClient`). Videos do not have the
CDN propagation window that photos have — the Server Action HEAD-probes before
returning success, and the page is only opened after successful `router.push`.

Prop interface:
```typescript
interface WearVideoClientProps {
  signedVideoUrl: string | null
  signedPosterUrl: string | null
  altText: string
  watchImageUrl: string | null
  brand: string
  model: string
  username: string | null
  displayName: string | null
  avatarUrl: string | null
  createdAt: Date
  watchId: string
}
```

#### 4. `useMediaCapability` — `src/hooks/useMediaCapability.ts`

Purpose: capability probe hook. Returns `{ supportsVideoCapture: boolean, preferredMimeType: string | null }`.
Returns `{ supportsVideoCapture: false, preferredMimeType: null }` on first render
(SSR + pre-hydration) to prevent hydration mismatches.

Probe chain (first supported wins):
1. `'video/mp4;codecs=avc1'`
2. `'video/webm;codecs=vp9'`
3. `'video/webm;codecs=vp8'`
4. `'video/webm'`

No UI rendering — pure logic hook.

## Surface-by-Surface Interaction Contracts

### Surface 1: ComposeStep pre-capture chooser (3-button)

State: `mediaState.kind === 'none' && cameraStream === null`

Layout: dashed border zone, `flex flex-wrap justify-center gap-2`, `py-8 bg-muted/30`

Interactions:
- "Take wrist shot": same as existing `handleTapCamera` (no change)
- "Record video": calls `handleTapVideoCamera()` — `getUserMedia` is the FIRST `await` (iOS gesture discipline enforced)
- "Upload photo": same as existing proxy through `photoUploaderRef` (no change)

"Record video" visibility: `{supportsVideoCapture && <Button ...>Record video</Button>}` — completely absent (not `opacity-0` or `disabled`) when unsupported.

### Surface 2: VideoCaptureView (live recording)

State: `cameraStream !== null && mediaSource === 'video'`

Pre-recording state:
- Live camera preview with `WristOverlaySvg`
- "Record 3s" button enabled, "Cancel" button enabled
- No red dot visible
- No ring visible

Recording in-progress state (after "Record 3s" tap):
- Red dot: `absolute top-3 left-3 z-20 size-3 rounded-full bg-destructive animate-pulse` — visible
- Progress ring: SVG ring `absolute inset-0 pointer-events-none` around the Record button area, clockwise fill animation over exactly 3.0s
  - CSS keyframes: `@keyframes ring-fill { from { stroke-dashoffset: CIRCUMFERENCE } to { stroke-dashoffset: 0 } }`
  - Animation: `animation: ring-fill 3.0s linear forwards`
  - Color: stroke `white`; must be in a `relative` wrapper div around the Record button
- "Recording…" button: disabled, no click effect
- "Cancel" button: enabled — tapping calls `handleCancelRecording()` which calls `recorder.stop()` with a `cancelled` guard flag; stream stays alive for re-record

Post-recording (poster extraction in progress):
- Red dot: hidden
- Ring: completed (full fill stays)
- "Processing…" button: disabled
- "Cancel" button: ALSO disabled during extraction (brief state, ~100ms; user shouldn't need to cancel here)

### Surface 3: ComposeStep post-capture preview (video)

State: `mediaState.kind === 'video'`

Layout: matches photo preview layout exactly (same `relative` wrapper)

Visual:
- `<video autoPlay muted loop playsInline src={objectUrl} className="w-full rounded-md object-cover" />`
- No X button (unlike photo preview) — only "Discard" link below
- NO `VideoPlayBadge` (D-16)
- "Discard" link: `<button className="mt-2 text-xs font-semibold text-accent underline">Discard</button>`
  - Tapping stops cameraStream, clears `mediaState`, returns to live `VideoCaptureView` (a fresh camera stream is NOT re-acquired automatically — `mediaSource` resets to show the pre-capture chooser or re-enters the live video state)

### Surface 4: WywtTile (home rail tile) — video branch

State: `tile.mediaType === 'video'`

Visual changes vs photo tile:
- Poster image: `<Image src={signedPosterUrl} fill className="object-cover" unoptimized />`
- `<VideoPlayBadge />` overlaid: `absolute inset-0 flex items-center justify-center pointer-events-none`
  - Tile size: `size-24 md:size-28` (96px / 112px square) — badge backdrop: `clamp(22px, 24%, 28px)` ≈ `size-6` on mobile, scaling to ~27px on desktop
  - Use `size-6` (24px) backdrop with `size-3` (12px) icon for the 96px tile context
- Bottom gradient + username/time text: UNCHANGED
- Ring/viewed state: UNCHANGED (ring is on the tile button, not inside the image area)

Tap behavior: UNCHANGED — opens `WywtOverlay` at the tile's index, which shows the `WearCard`. `WearCard` then handles the video branch.

### Surface 5: WearVideoClient — /wear/{id} detail page

State: `mediaType === 'video' && signedVideoUrl !== null`

Visual contract (matches `WearPhotoClient` happy-path layout):
- Container: `relative w-full aspect-[4/5] overflow-hidden bg-muted md:rounded-lg md:max-w-[600px] md:mx-auto md:max-h-[70vh]`
- `<video>`: `w-full h-full object-cover` with `autoPlay muted loop playsInline`
- `WearPhotoOverlays`: rendered above the video (same z-10 top + bottom gradient scrims)
- Tap-to-pause: `onClick` on the container div — toggling `videoRef.current.pause()` / `.play()`
  - NO visual pause indicator is required (D-06 specifies tappable to pause/resume — no UI affordance beyond the video itself pausing)

Error fallback (D-08):
- Poster `<img>` with `w-full h-full object-cover`
- `<span className="absolute bottom-[60px] left-3 z-10 text-xs font-semibold text-white/70">Video unavailable</span>` (above bottom overlay scrim)
- `WearPhotoOverlays` rendered as normal
- Entire container still `relative` (required for absolute overlays)

### Surface 6: WearsLane video slide

State: `slide.mediaType === 'video'`

`WearCard` receives `mediaType="video" signedVideoUrl={...} signedPosterUrl={...}` via spread from the slide data. `WearCard` delegates to `WearVideoClient`. The `WearVideoClient` inside the stories lane autoplays-loops exactly as on `/wear/{id}` (D-07).

Segmented progress indicator: NOT driven by `video.onEnded`. The existing embla
`'select'` event is the only driver. Video loops indefinitely per slide until the
user swipes. No `onEnded` listener may be added to any video element.

## Implementation Guardrails (executor reminders)

1. `playsInline` is MANDATORY on every `<video>` element — in `WearVideoClient`,
   in `VideoCaptureView`'s live preview `<video>`, and in the post-capture preview
   `<video>` in ComposeStep. Missing it causes iOS Safari fullscreen takeover.

2. `getUserMedia` MUST be the FIRST `await` in `handleTapVideoCamera` in ComposeStep.
   No `setState`, `setError(null)`, `analytics`, or any other `await` may precede it.
   `cameraOpeningRef.current = true` is synchronous (ref write, not setState) and is
   safe to place before `getUserMedia`.

3. `WearPhotoClient` and `WearDetailHero` must have ZERO edits in Phase 77. Any
   change to these files is a VID-15 violation.

4. The progress ring CSS animation must respect `prefers-reduced-motion`. When reduced
   motion is preferred, the ring should either: (a) be replaced by a simple fill that
   appears instantly when recording starts, or (b) skip animation and show as full at
   3s. Apply `@media (prefers-reduced-motion: reduce) { .ring-fill-animation { animation: none } }`.

5. `VideoPlayBadge` must be used via the shared component — never inline the badge JSX
   into individual tile components. Single source of truth for sizing math (D-14).

6. Outline Button dark mode pairing rule: the Record button uses `variant="default"` —
   not outline. The Cancel button uses `variant="outline"`. No additional `bg-` override
   is needed on the Cancel button, so no dark pairing issue arises. If any future class
   overrides `bg-` on the Cancel button, add the `dark:` pair.

7. Do NOT add `video.onEnded` listeners anywhere. The `loop` attribute handles repeat;
   the stories lane progress bar is driven by embla `'select'`, not video events.

8. Object URL lifecycle: `URL.createObjectURL(videoBlob)` in the ComposeStep post-capture
   preview must be `URL.revokeObjectURL`-ed in a `useEffect` cleanup when the blob changes
   or the component unmounts (same pattern as `photoPreviewUrl` in ComposeStep).

9. `src/app/spike-mr-capture/` must be deleted in Wave 1 Task 1 — it is an unauthenticated
   production route from the spike that must not ship with Phase 77.
