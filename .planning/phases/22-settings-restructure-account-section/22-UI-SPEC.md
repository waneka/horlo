## Spacing Scale

Tailwind 4 default scale (multiples of 4). Phase 22 uses these tokens explicitly:

| Token | Tailwind Class | Value | Phase 22 Usage |
|-------|----------------|-------|---------------|
| 1 | `gap-1`, `p-1`, `mb-1` | 4px | Icon gaps inside `TabsTrigger` |
| 2 | `gap-2`, `p-2`, `mb-2` | 8px | Inline form-field row gap, dialog action gap |
| 3 | `gap-3`, `py-3` | 12px | Auth-card footer button stack gap |
| 4 | `gap-4`, `p-4`, `mb-4` | 16px | Card interior padding, vertical form-field rhythm, section row padding |
| 6 | `gap-6`, `mt-6`, `mb-6` | 24px | Tab list ↔ panel gap, section-to-section spacing inside a panel |
| 8 | `py-8`, `space-y-8` | 32px | Page top/bottom padding (mobile), Account section group separation (Email card → Password card) |
| 12 | `lg:py-12` | 48px | Page top/bottom padding (≥`lg` breakpoint) |

**Exceptions (do NOT add new tokens):**
- Tab list fixed width: `w-44` (176px) — between `w-40` (160) and `w-48` (192). This is a deliberate tabs-sidebar width and matches Tailwind's existing `w-44` token; it is NOT a custom value. Justification: keeps "Notifications" + icon visible on one line without forcing the panel narrower than 320px on a 768px breakpoint viewport.
- Toggle switch geometry inside `PrivacyToggleRow` is a documented existing exception (`h-6 w-10`, knob `size-5`, `translate-x-[2px]` / `translate-x-[18px]`). **Do NOT respec; reuse the file as-is.**

## Layout

### Page Frame (Server Component `src/app/settings/page.tsx`)

```html
<main className="mx-auto max-w-4xl px-4 py-8 lg:px-8 lg:py-12">
  <h1 className="text-xl font-semibold">Settings</h1>
  <SettingsTabsShell ... />
</main>
```

**Decision: change `max-w-2xl` (today) → `max-w-4xl`** [Claude's Discretion — rationale]: vertical-tabs layout needs ~176px sidebar + ≥320px panel + 24px gap = ~520px minimum content width. `max-w-2xl` (672px) leaves only 472px for the panel after the sidebar; `max-w-4xl` (896px) gives 696px panel width which matches existing single-column form surfaces (`/login`, `/reset-password` use `max-w-sm` for their inner Card; that Card sits inside this 696px area with breathing room). The page-frame `mx-auto` centering remains unchanged.

The `<h1>` text "Settings" sits above the Tabs shell with no descriptive subtitle (today's "Manage your privacy controls. Other sections coming soon." copy is removed — section labels themselves convey scope).

### Vertical Tabs Shell (`SettingsTabsShell.tsx`, Client Component)

```html
<Tabs orientation="vertical" value={activeTab} onValueChange={...} className="mt-6 gap-6">
  <TabsList variant="line" className="w-44 self-start sticky top-4">
    <TabsTrigger value="account">      <User /> Account       </TabsTrigger>
    <TabsTrigger value="profile">      <IdCard /> Profile      </TabsTrigger>
    <TabsTrigger value="preferences">  <Sliders /> Preferences  </TabsTrigger>
    <TabsTrigger value="privacy">      <Lock /> Privacy        </TabsTrigger>
    <TabsTrigger value="notifications"><Bell /> Notifications  </TabsTrigger>
    <TabsTrigger value="appearance">   <Palette /> Appearance   </TabsTrigger>
  </TabsList>
  <div className="min-w-0 flex-1">
    <TabsContent value="account">      <AccountSection ... />     </TabsContent>
    <TabsContent value="profile">      <ProfileSection ... />     </TabsContent>
    <TabsContent value="preferences">  <PreferencesSection ... /> </TabsContent>
    <TabsContent value="privacy">      <PrivacySection ... />     </TabsContent>
    <TabsContent value="notifications"><NotificationsSection ... /></TabsContent>
    <TabsContent value="appearance">   <AppearanceSection />      </TabsContent>
  </div>
</Tabs>
```

**Layout proportions:**
- Tab list: fixed `w-44` (176px), `variant="line"` (transparent background, accent left/right indicator via existing `after:` rules in `tabs.tsx:64`).
- Tab list sticks to top via `self-start sticky top-4` so it stays visible while scrolling long panels (e.g., the Privacy tab with 3 toggle rows + future restyle).
- Panel area: `flex-1 min-w-0` — `min-w-0` prevents flex-shrink overflow when a panel contains a wide form input.
- Gap between list and panels: `gap-6` (24px) — matches the page top spacing rhythm.

**Active tab indicator:**
- The existing `tabs.tsx` `variant="line"` ships a vertical accent strip via `after:` (`group-data-vertical/tabs:after:-right-1 group-data-vertical/tabs:after:w-0.5`) using `bg-foreground` at full opacity when `data-active`.
- Active label color shifts from `text-foreground/60` → `text-foreground`.
- On hover (non-active), label color shifts to `text-foreground` (same as active text color).
- Reuse the primitive — DO NOT respec custom indicator styles.

**Tab list typography (inherited from `tabs.tsx` `TabsTrigger`):**
- Resting: `text-sm font-medium text-foreground/60`
- Hover: `hover:text-foreground` (no background change in `line` variant)
- Active: `data-active:text-foreground` + the `after:opacity-100` accent strip
- Focus: `focus-visible:border-ring focus-visible:ring-[3px] focus-visible:ring-ring/50` (existing keyboard ring)

Each `TabsTrigger` includes a 16px lucide icon + label, justified to start (`group-data-vertical/tabs:justify-start` already in primitive). Icon mapping:

| Tab | Icon | Lucide name |
|-----|------|-------------|
| Account | `User` | `lucide-react/User` |
| Profile | `IdCard` | `lucide-react/IdCard` |
| Preferences | `SlidersHorizontal` | `lucide-react/SlidersHorizontal` |
| Privacy | `Lock` | `lucide-react/Lock` |
| Notifications | `Bell` | `lucide-react/Bell` |
| Appearance | `Palette` | `lucide-react/Palette` |

### Mobile Responsive (<`lg`, i.e. <1024px)

[Claude's Discretion — D-CHOICE-MOBILE-LAYOUT]

**At `<sm` (<640px):** Vertical tabs collapse to a horizontal scroll-snap tab bar above the panel. Apply via `flex-col sm:flex-row` on the `Tabs` root + `flex-row sm:flex-col` + `overflow-x-auto sm:overflow-visible w-full sm:w-44` on the `TabsList`. The `after:` indicator from `tabs.tsx` already handles both orientations because the existing CSS uses `group-data-horizontal` and `group-data-vertical` selectors.

```html
<Tabs orientation="vertical" className="mt-6 flex-col gap-4 sm:flex-row sm:gap-6">
  <TabsList variant="line"
    className="w-full overflow-x-auto sm:w-44 sm:self-start sm:sticky sm:top-4">
    ...
  </TabsList>
  ...
</Tabs>
```

**Edge case:** Even though `<TabsList>` renders horizontally on mobile, `Tabs.Root` retains `orientation="vertical"` so keyboard arrow keys still match the developer mental model (`ArrowDown/Up` between tabs). This is acceptable per WAI-ARIA — orientation is a navigation hint, not a layout dictate. If a future audit flags the discrepancy, swap to a media-query-driven `orientation` prop.

**At `≥sm` and `<lg` (640–1023px):** The vertical tabs layout applies (sidebar + panel). The page-frame `max-w-4xl` shrinks to fit viewport with `px-4`.

### Visual spec — `<EmailChangePendingBanner>`

```html
<div
  role="status"
  aria-live="polite"
  className="mb-4 rounded-lg border border-l-2 border-border border-l-accent bg-muted/40 p-3 text-sm"
>
  <p className="text-foreground">
    Confirmation sent to <strong>{oldEmail}</strong> and <strong>{newEmail}</strong>.
    Click both links to complete the change.
  </p>
  <div className="mt-2">
    <Button
      type="button"
      variant="outline"
      size="sm"
      onClick={handleResend}
      disabled={resending}
    >
      {resending ? 'Resending…' : 'Resend confirmation'}
    </Button>
  </div>
</div>
```

**Rationale:**
- `border-l-accent` is the **ONLY** new accent surface introduced by Phase 22 (1 of 4 reserved-for items above). Left-border accent gives the banner a "status" affordance that distinguishes it from a regular card without using a colored fill (which would clash with toggle ON state across light + dark).
- `bg-muted/40` is a soft tint — uses an existing token at 40% opacity. Resolves cleanly in both light and dark mode.
- `border-l-2` (2px) accent strip + `border` (default 1px) on the other three sides for the standard card rhythm.
- `p-3` (12px) padding is tighter than `<SettingsSection>`'s `p-4` (16px) — the banner is subordinate UI; tighter padding signals "alert" without screaming.
- "Resend confirmation" is `variant="outline" size="sm"` (h-7 px-2.5, 12.8px text). NOT `variant="default"` because it's a secondary affordance — primary action remains "Update email".
- Resend button text shows `Resending…` (Unicode ellipsis) during the `useTransition` to mirror the `reset-password-form.tsx` "Updating…" pattern.

## Interaction Patterns

### Tabs

| Action | Behavior |
|--------|----------|
| Click tab trigger | `onValueChange` fires → `setActiveTab(value)` → `window.history.pushState(null, '', '#${value}')`. NOT `router.push`. |
| Mount with `#preferences` (or any valid hash) | `useEffect` reads `window.location.hash`, parses via D-16 parser, `setActiveTab(parsed)`. |
| Mount with no hash | `window.history.replaceState(null, '', '/settings#account')` and stay on Account. |
| Mount with `#unknown-tab` | Parser falls through to `'account'` (D-17 default). |
| Browser back/forward | `hashchange` event listener re-parses hash and calls `setActiveTab`. |
| Keyboard `ArrowDown` / `ArrowUp` on focused tab | base-ui handles roving focus; moves between tabs (vertical orientation). |
| Keyboard `Tab` from focused tab | Moves into panel content (base-ui default for `Tabs`). |

### Email Change

| Action | Behavior |
|--------|----------|
| Submit valid new email | `setSubmitting(true)` → `supabase.auth.updateUser({ email: newEmail })` → on success: `toast.success('Confirmation sent. Check your inbox.')` (intermediate toast), refresh server data via `router.refresh()` so the page re-reads `user.new_email` and renders the banner. |
| Banner renders | When `pendingNewEmail` (= `user.new_email`) is non-null. Persists across reloads (D-05). |
| Click "Resend confirmation" | `supabase.auth.updateUser({ email: pendingNewEmail })` (D-06 — same call, NOT `auth.resend()`). On success: `toast.success('Confirmation resent.')`. |
| User clicks confirm link in old email | Routed via `/auth/callback` → `verifyOtp({type:'email_change'})` → `redirect('/settings#account?status=email_changed')` per D-12. `<StatusToastHandler />` fires the success toast and strips the `?status=` param while preserving `#account`. |

### Password Change

| Action | Behavior |
|--------|----------|
| Submit (fresh session, < 24h since `last_sign_in_at`) | Direct `updateUser({ password })` (D-10). On 200: success toast + clear form. On 401 (defense-in-depth): open re-auth dialog with copy "Please confirm your password to continue." |
| Submit (stale session, > 24h since `last_sign_in_at`) | Open re-auth dialog (no toast). Hold the new password in dialog state for the post-reauth `updateUser` call. |
| Submit re-auth dialog | `signInWithPassword({email: currentEmail, password: currentPassword})` → on success: `updateUser({ password: pendingNewPassword })` → on success: close dialog + success toast. On either 401: keep dialog open, show inline `text-destructive` error with neutral copy. |
| Cancel re-auth dialog | Close dialog, leave form populated. User can edit + resubmit. |

### Status Toast Strip (D-13/D-14)

| Action | Behavior |
|--------|----------|
| Mount on `/settings#account?status=email_changed` | `useSearchParams()` reads `status` → `toast.success('Email changed successfully')` → `router.replace(pathname + window.location.hash)` to strip `?status=` while preserving `#account`. The replace is fire-once per status value (use a ref guard to prevent re-fire on Strict Mode double-mount). |

### Privacy / Notifications Toggles (migrated, no behavior change)

`PrivacyToggleRow` continues to use `useOptimistic` + `useTransition` + `updateProfileSettings` Server Action. **No interaction changes** in Phase 22.

### Preferences Embed

Click any control inside `<PreferencesClient>` → existing `savePreferences` Server Action fires → `revalidatePath('/preferences')` runs server-side. **Note:** `revalidatePath('/preferences')` revalidates the OLD route, but since `/preferences` now `redirect()`s to `/settings#preferences`, the fresh data lands when the user navigates to the Preferences tab on a future page load. **Consider adding `revalidatePath('/settings')` to `savePreferences`** in implementation — flagged for planner.

### `/preferences` Redirect

Visiting `/preferences` returns a 307 with `Location: /settings#preferences`. The browser honors the fragment, lands on `/settings`, and the SettingsTabsShell's mount-time hash parser activates the Preferences tab.

## Anti-Patterns Flagged

These are explicitly forbidden by this contract. The checker should fail any deviation.

1. **DO NOT fork `tabs.tsx`.** Reuse the existing `<Tabs>`, `<TabsList variant="line">`, `<TabsTrigger>`, `<TabsContent>` primitives. Vertical orientation, roving focus, ARIA, and the active-strip indicator already work — there is no scenario in Phase 22 where forking is required.
2. **DO NOT introduce custom Tailwind tokens.** Use only existing CSS-variable-aliased utilities (`bg-background`, `bg-card`, `bg-muted`, `bg-accent`, `text-foreground`, `text-muted-foreground`, `text-destructive`, `border-border`, `ring-ring`, etc.). The Tailwind 4 config is locked.
3. **DO NOT respec typography beyond the 3-size / 2-weight contract above.** The page uses IBM Plex Sans throughout; Geist Mono is for code; Instrument Serif (`font-serif`) is reserved for auth-card titles only and is NOT used in the in-tab Account section.
4. **DO NOT use `Button variant="primary"`** — the design system uses `variant="default"` for the primary action style. `primary` is not a valid CVA variant in `button.tsx`.
5. **DO NOT use `router.push` for tab changes** (Pitfall 1). `pushState` only.
6. **DO NOT use `router.replace(pathname)` to strip `?status=`** (Pitfall 2). Always reconstruct: `router.replace(pathname + window.location.hash)` per D-14.
7. **DO NOT preemptively patch the dialog with `stopPropagation`** (Pitfall 5). Verify with a smoke test; only mitigate if observed.
8. **DO NOT show `user.new_email` in the "current email" disabled `Input`** (SET-04 explicit). Show `user.email` until confirmation completes.
9. **DO NOT render a Cancel button on the pending banner** (D-06). Resend-only.
10. **DO NOT accent-fill the pending banner background** (color contract). The accent appears ONLY as a 2px left-border. Background is `bg-muted/40`.
11. **DO NOT add a `variant="destructive"` button anywhere in Phase 22.** No destructive primary actions are in scope; the deleted Delete Account dialog is gone for good (D-03).
12. **DO NOT change `<PrivacyToggleRow>` visual or interaction.** Migrate verbatim. Phase 23 (SET-09 / SET-11) handles the restyle pass.

| ID | Description | Mitigation |
|----|-------------|------------|
| FG-1 | `getUser()` returns stale state for 1–2s after first email-confirmation click | Banner copy "Click both links to complete the change" is forgiving; eventual consistency lands on next render. DO NOT poll. |

| FG-6 | iOS Safari zooms the page on `<Input>` focus when font-size is <16px | The existing `input.tsx` rule already sets `text-base md:text-sm` for this exact reason — do not override to `text-sm` on mobile. |
