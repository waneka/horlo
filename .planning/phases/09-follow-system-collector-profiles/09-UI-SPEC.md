**Accent (`--accent`) reserved for вҖ” Phase 9 additions:**
- **Primary Follow CTA** fill (ProfileHeader + LockedProfileState placements only вҖ” solid `bg-accent text-accent-foreground`)
- **Shared taste-tag pills** in Common Ground (via existing `TasteTagPill` вҖ” already accent by Phase 8 rule)
- **Overlap label strong state** (`"Strong overlap"` text rendered as `text-accent-foreground` on `bg-accent` pill)
- **Owner-side HorizontalBarChart fill** in Common Ground dual bars (owner fill = `bg-accent`; viewer fill = `bg-muted` with accent ring for distinction)

**Accent NOT used on вҖ” Phase 9 negative constraints:**
- Inline Follow button in list rows вҖ” use **outline variant** (`border border-border text-foreground`), never accent fill, to avoid lighting up every list row
- "Following" (already-following) state вҖ” muted `bg-muted text-muted-foreground`; accent returns only on hover to reveal the Unfollow intent (D-09)
- Follower/following count numerals in ProfileHeader вҖ” remain `text-muted-foreground` per Phase 8 rule
- Common Ground hero band background вҖ” remains `bg-card`, not `bg-accent` (band is visually quiet; content inside the band uses accent sparingly)

**Destructive (`--destructive`) reserved for вҖ” Phase 9:**
- Unfollow hover/tap state text color only: `Following` (muted) вҶ’ hover/tap вҶ’ `Unfollow` (`text-destructive`). The button background stays `bg-muted` вҖ” no destructive fill. This matches the intentionally low-friction Twitter/X convention locked in D-09.
- Error toast on follow/unfollow failure (reuses Phase 8 error-toast pattern; `text-destructive` accent only)

All new components MUST reference tokens via Tailwind classes (`bg-accent`, `text-muted-foreground`, `text-destructive`). Never hardcode hex or oklch values.

## Interaction Contracts

### Follow button вҖ” primary variant (ProfileHeader, non-owner view)

**Anatomy:** solid `bg-accent text-accent-foreground` when in `not-following` state; flips to `bg-muted text-muted-foreground` with label "Following" in `following` state.

**States (must be explicit):**
| State | Label | Fill | Text | Trigger |
|-------|-------|------|------|---------|
| not-following | "Follow" | `bg-accent` | `text-accent-foreground` | Default for non-owner, viewer authenticated, no follow row |
| following (rest) | "Following" | `bg-muted` | `text-muted-foreground` | After successful follow, or if follow row exists on load |
| following (hover/focus desktop) | "Unfollow" | `bg-muted` | `text-destructive` | Hover or keyboard focus on "Following" button |
| following (tap 1 mobile) | "Unfollow" | `bg-muted` | `text-destructive` | First tap on "Following" вҖ” reveals intent without firing |
| pending | Current label with spinner | `opacity-70 cursor-wait` | Unchanged | While `useTransition` is pending |
| disabled (self-profile) | N/A вҖ” button is not rendered | вҖ” | вҖ” | `viewerId === ownerId` |
| disabled (unauth) | "Follow" (opens sign-in) | `bg-accent` | `text-accent-foreground` | Unauthenticated viewer вҖ” click triggers sign-in redirect (behavior in Claude's discretion per RESEARCH; default: redirect to `/login?next=/u/[username]`) |
| error (rolled back) | Rolled back to pre-click label | Pre-click fill | Pre-click text | On Server Action failure вҖ” local state rolls back, error toast shown |

**Mobile two-tap flow (D-09):**
1. Tap 1 on "Following" вҶ’ label flips to "Unfollow" (`text-destructive`), no network call. Tap-away or scroll-away reverts to "Following".
2. Tap 2 on "Unfollow" вҶ’ fires Server Action, optimistic state change.

**Accessibility:**
- `aria-pressed` reflects current following state on the button
- `aria-label` reads "Follow {displayName ?? username}" on not-following state; "Unfollow {displayName ?? username}" on hover/focus/mobile-revealed state
- Keyboard focus ring: default `--ring` accent glow
- Spinner icon has `aria-hidden="true"`; pending state announced via `aria-busy="true"` on the button

### Follow button вҖ” locked variant (LockedProfileState)

- Identical visual to primary variant (solid accent fill in not-following state)
- Positioned below the "This profile is private." copy, centered
- Follows same optimistic/rollback + hover-swap rules
- Clicking succeeds вҖ” the follow row writes immediately even though content stays locked (D-08). No confirmation dialog. No "request sent" messaging (not a request-based flow).

### Follow button вҖ” inline variant (follower/following list rows)

- **Outline-only style** (no accent fill) вҖ” `border border-border text-foreground` at rest
- Label: "Follow" (not-following) / "Following" (following) / "Unfollow" (hover-swap)
- Size: height 32px desktop, 40px mobile; horizontal padding 12px
- `stopPropagation` on click so clicking Follow does not trigger the row's `Link` navigation (D-14)
- **Hidden entirely** on the row that represents the viewer themselves (you don't see a Follow button on your own row in your own followers list)

### Count reconciliation interaction (D-06, D-22)

Expected sequence on Follow:

1. Click вҶ’ local `isFollowing` flips to true, local `followerCount++` immediately (0ms perceived)
2. Spinner appears on button (`useTransition` pending)
3. Server Action writes `follows` row + calls `revalidatePath('/u/[username]', 'layout')`
4. Server Action returns `{ success: true }` (typically 100вҖ“400ms)
5. Client calls `router.refresh()` вҖ” layout re-renders with server-fresh counts
6. Spinner clears; button settles on "Following"

Error path: Server Action returns `{ success: false, error }` вҶ’ local state rolls back (button flips back to "Follow", count decrements back) вҶ’ error toast: "Couldn't follow. Try again." (copy below).

## Privacy & State Visibility Rules

These UI-level rules enforce Phase 8 privacy behavior during non-owner viewing. They are visual-contract rules вҖ” the DAL and RLS layers enforce the actual access; the UI must not leak absence-vs-presence information.

| Scenario | UI surface |
|----------|-----------|
| Profile is fully private (`profile_public = false`) + viewer is not owner | `LockedProfileState` (unchanged from Phase 8; Follow button now functional) вҖ” no tab row, no hero band |
| Profile is public + viewer is not owner + tab X is private | Tab row shows all 5 tabs (D-15); tab X content = `LockedTabCard`; other tabs load normally |
| Profile is public + viewer is not owner + zero overlap | No Common Ground hero band, no 6th tab вҖ” tab row shows 5 tabs only |
| Profile is public + viewer is not owner + non-zero overlap | Hero band + 6th tab both render; tab row shows 6 tabs |
| Viewer is owner (any setting combination) | No FollowButton, no hero band, no 6th tab, no `LockedTabCard` вҖ” owner sees everything per Phase 8 contract |
| Viewer is unauthenticated | Profile public: tabs render as read-only; Follow button is visible and routes to sign-in on click. Profile private: `LockedProfileState` renders with Follow button that also routes to sign-in. |

**Stable-UI guarantee:** the tab row is structurally identical (5 tabs) for all non-owner views regardless of private-tab count. Only the 6th Common Ground tab varies conditionally (on/off). This prevents layout shift and avoids signaling "this user hides N tabs" to the viewer.

**Owner-only UI removed on other profiles (inherited from Phase 8, Phase 9 must preserve вҖ” D-19):**
- "+ Add Watch" card on Collection tab
- "Log Today's Wear" CTA on Worn tab
- Per-note visibility pill + 3-dot "Remove Note" menu on Notes tab
- Inline "Edit Profile" affordance on ProfileHeader
- "Collection Observations" card on Stats tab (exposes aggregates the owner wouldn't want public)

## Implementation Notes for Executor

1. **`FollowButton` hook pattern (D-06):** Client Component using `useState` (for `isFollowing` + `followerCount`) + `useTransition` + `router.refresh()` from `next/navigation`. Do NOT use `useOptimistic` вҖ” RESEARCH.md pattern 2 locks the manual-state approach because of the compound state (label + count). The parent ProfileHeader passes `initialIsFollowing` + `initialFollowerCount` from server-rendered data.

2. **`router.refresh()` + `revalidatePath` pair:** On the Server Action side (`src/app/actions/follows.ts`), call `revalidatePath('/u/[username]', 'layout')` after every successful follow/unfollow. On the Client side in `FollowButton`, call `router.refresh()` after the Server Action resolves. Both are required: `revalidatePath` for cache correctness on future navigations, `router.refresh()` for immediate on-screen reconciliation.

3. **Hover-swap implementation (D-09):** Pure CSS-and-state вҖ” no JS required for the desktop hover. Use `group` pattern:
   ```tsx
   <button className="group bg-muted text-muted-foreground hover:text-destructive focus:text-destructive">
     <span className="group-hover:hidden group-focus:hidden">Following</span>
     <span className="hidden group-hover:inline group-focus:inline">Unfollow</span>
   </button>
   ```
   Mobile two-tap requires local `revealed` state in the button that clears on blur/scroll.

4. **Common Ground computation (server-only):** The `getTasteOverlap(viewerId, ownerId)` DAL must be called from the `layout.tsx` Server Component. Never invoke it from a Client Component вҖ” it reads both users' full watch lists. `src/lib/tasteOverlap.ts` is pure-function server-safe code; no `'use server'` directive needed (it's imported, not an action).

5. **Common Ground thresholds:** The overlap label derives from average similarity scores (viewer's watches vs. owner's collection). Specific thresholds are the planner's call вҖ” calibrate against `GOAL_THRESHOLDS` in `src/lib/similarity.ts`. The label enum shipped to the client is bounded to three literals: `'Strong overlap' | 'Some overlap' | 'Different taste'`. Do not ship raw scores to the client.

6. **Empty-state vs. locked-state distinction:** An empty tab (e.g. owner has public collection but hasn't added any watches) is NOT the same as a locked tab. Empty = Phase 8 empty-state copy ("Nothing here yet."). Locked = `LockedTabCard` with the "keeps their X private" copy. The `[tab]/page.tsx` must decide based on the `profile_settings.{tab}_public` flag, NOT the data row count.

7. **Avatar fallback:** Re-use existing `AvatarDisplay` logic вҖ” initials on `bg-accent text-accent-foreground` if URL fails or is empty. Follower/following rows use `size={40}`.

8. **Keyboard traversal:** Tab-row to hero band link to 6th-tab content must be fully keyboard-navigable. The "See full comparison вҶ’" link is a real `<a>` / `<Link>`, not a styled `<span>` вҖ” no custom focus traps.

9. **Layout-shift avoidance:** `CommonGroundHeroBand` renders server-side on every non-owner render with data already fetched вҖ” no client-side loading skeleton. The band is either present (fully rendered) or absent (empty overlap path rendering the compact one-line message). Never flash a placeholder.

10. **Follow-button on unauth view:** The spec leaves the sign-in redirect flow to Claude's discretion (per CONTEXT D-10). Default behavior: render the Follow button with its standard visual; clicking calls a client-side guard that detects no auth session and navigates to `/login?next=/u/[username]` (using `next/navigation` `useRouter`). No Server Action is fired. Same behavior on the LockedProfileState Follow button.

11. **List-card link target:** `/u/{username}/collection` вҖ” explicit, not just `/u/{username}`. Matches D-14 and skips the layout-level redirect hop for navigation perf.

12. **No schema work in this phase:** All tables (`follows`, `profiles`, `profile_settings`) already land in Phase 7. Phase 9 verifies RLS policies on `follows` match D-21 (INSERT/DELETE: `auth.uid() = follower_id`; SELECT: public) вҖ” if missing, add them in a plain-SQL migration file. No Drizzle schema changes.

13. **Font:** `font-sans` Tailwind class вҖ” Geist Sans via `--font-sans`. No new font imports.

14. **Dark mode:** inherited from Phase 8's CSS-variable system. No `dark:` overrides for base tokens.
