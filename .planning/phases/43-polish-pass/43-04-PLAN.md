<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Supabase Storage | user-selected file bytes cross into the `avatars` bucket |
| browser → `{userId}/` folder path | the upload path embeds the user id |
| client → `updateProfile` Server Action | the stored avatar URL crosses into the profiles table |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-43-05 | Tampering | Storage path `{userId}/avatar.jpg` | mitigate | RLS policies `avatars_insert_own_folder` / `avatars_update_own_folder` enforce `(storage.foldername(name))[1] = (SELECT auth.uid())::text` — a user cannot write into another user's folder even if `buildAvatarPath` is called with a forged id. Folder enforcement is at the database layer (Task 1 migration). |
| T-43-06 | Tampering | malicious / polyglot image upload | mitigate | `stripAndResize` re-encodes the image through a canvas before upload — raw file bytes never reach Storage; only the canvas-produced JPEG. The bucket `allowed_mime_types` allow-list (jpeg/png/webp) is a second layer. |
| T-43-07 | Information Disclosure | EXIF metadata (GPS, device, timestamp) in avatar | mitigate | `stripAndResize`'s canvas re-encode drops all EXIF unconditionally — no metadata survives to the stored avatar. |
| T-43-08 | Denial of Service | oversized file upload | mitigate | Client-side 8 MB guard in `AvatarUploader.handleChange` + a 4 MB `file_size_limit` on the bucket itself (Task 1 migration). |
| T-43-09 | Spoofing | content-type spoofing on upload | mitigate | Upload sets `contentType: 'image/jpeg'` explicitly and the bytes are a canvas-produced JPEG; bucket `allowed_mime_types` rejects anything outside the image allow-list. |
| T-43-10 | Information Disclosure | public bucket exposes avatars | accept | Avatars are profile photos shown to all viewers by design — public read is the intended behavior. No private data in the bucket. |

No `high`-severity threats remain after mitigation — this plan does not block.
</threat_model>
