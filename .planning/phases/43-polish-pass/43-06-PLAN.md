<objective>
Close GAP-43-03 (PLSH-06): avatar upload fails with Supabase Storage `403 "new row violates
row-level security policy"`.

**Root cause (confirmed, see 43-HUMAN-UAT.md GAP-43-03):** the `avatars` bucket has
INSERT/UPDATE/DELETE RLS policies but NO `SELECT` policy. `uploadAvatarPhoto` uses
`upsert: true`. Supabase Storage upsert uploads require a `SELECT` policy because the
storage-api looks up the existing object row on `storage.objects` to decide insert-vs-update.
Without SELECT, the lookup fails and the request is rejected.

**Fix:** a NEW Supabase migration adding an `authenticated`, folder-scoped `SELECT` policy
on the `avatars` bucket — `avatars_select_own_folder` — mirroring the
`avatars_insert_own_folder` predicate. The migration must be applied (`supabase db push`)
before verification: a `[BLOCKING]` checkpoint task handles the push.

This is a **DB-only fix.** No application code changes — `uploadAvatarPhoto` keeps
`upsert: true`, the uploader, `AvatarUploader.tsx`, and `updateProfile` all stay as-is.

Purpose: avatar upload (PLSH-06) was marked Complete in REQUIREMENTS.md but UAT showed it
is non-functional in practice. This restores the shipped feature.

Output: one new SQL migration file + the migration applied to the linked Supabase project.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-43-G06-01 | Information disclosure | `avatars_select_own_folder` SELECT policy on `storage.objects` | mitigate | The SELECT policy `USING` clause keeps the folder predicate `(storage.foldername(name))[1] = (SELECT auth.uid())::text` so an authenticated user can SELECT only object rows in their own avatar folder — it does NOT expose other users' avatar object metadata. The policy is `TO authenticated` (never `TO public`). Public CDN avatar reads already bypass RLS on a public bucket; this policy governs only the authenticated storage-api write path, so it does not widen public exposure. |
| T-43-G06-02 | Tampering | avatar upload write path (`avatars_insert_own_folder` / `avatars_update_own_folder`) | accept | Unchanged by this plan — INSERT/UPDATE folder enforcement from `20260516000000_phase43_avatar_bucket.sql` (T-43-05) stays intact. Adding a folder-scoped SELECT policy does not grant any new write capability; upsert still requires the existing INSERT/UPDATE policies to pass. |
| T-43-G06-03 | Elevation of privilege | over-permissive SELECT policy | mitigate | Explicitly prohibits a bare `USING (bucket_id = 'avatars')` and `TO public` — a plan acceptance criterion and the Task 1 bash verify gate (`! grep -qF 'TO public'`) enforce the folder-scoped, authenticated-only form. |
</threat_model>
