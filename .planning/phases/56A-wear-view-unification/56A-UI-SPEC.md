## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding (gap-1, p-1) |
| sm | 8px | Compact element spacing (gap-2, p-2) |
| md | 16px | Default element spacing, overlay padding (p-3=12px, p-4=16px) |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:
- **Touch targets**: minimum 44px height/width on all interactive controls (LikeButton, comment trigger, overflow "…" button, close button). Source: existing `min-h-[44px] min-w-[44px]` in `LikeButton.tsx`.
- **Bottom nav height**: `h-[calc(80px+env(safe-area-inset-bottom))]` — existing inline calc form in `BottomNav.tsx`, untouched on routes where nav renders.
- **Safe-area inset (stories engagement row)**: The engagement row on the stories lane must clear the iOS home indicator when the bottom nav is hidden. The row uses `pt-3 pb-[calc(0.75rem+env(safe-area-inset-bottom))]` — inline calc matching the project's existing `BottomNav.tsx` pattern (`pb-[env(safe-area-inset-bottom)]` with explicit base padding). No `pb-safe` utility exists in `globals.css`; the inline calc form is mandatory.
- **Overlay inset**: photo overlays use `p-3` (12px) per existing `WearPhotoOverlays` convention — not changing.

## Color

Color follows a 60/30/10 distribution: dominant `--background` (60%), `--card`/`--muted` surfaces (30%), `--accent` reserved for active nav only (10%).

All color values are CSS custom properties from `src/app/globals.css` (shadcn base-nova, neutral hue). Light and dark modes both defined.

| Role | CSS Variable | Light Value (approx) | Usage |
|------|-------------|----------------------|-------|
| Dominant (60%) | `--background` | oklch(0.985 0.003 75) ≈ near-white warm | Route background for `/wear/[id]` detail page body, `<main>` |
| Secondary (30%) | `--card` / `--muted` | oklch(1 0 0) / oklch(0.95 0.005 75) | Sheet panel background (`bg-popover`), engagement row background, photo placeholder (`bg-muted`) |
| Accent (10%) | `--accent` | oklch(0.76 0.12 75) ≈ warm amber | Reserved for: active nav tab icons/labels only; NOT used on stories overlay controls |
| Destructive | `--destructive` | oklch(0.55 0.22 27) ≈ warm red | Liked Heart icon fill only (existing `LikeButton` contract) |

Accent reserved for: active BottomNav tab (Home/Search/Explore/Profile icons + labels). No new accent usage introduced in Phase 56A.

**Stories lane color contract** (full-screen overlay on photo):
- Photo gradient scrims: `rgba(0,0,0,0.45)` top-to-bottom (top overlay), `rgba(0,0,0,0.45)` bottom-to-top (bottom overlay). Exact values from existing `WearPhotoOverlays` — carried forward.
- Overlay text on photo (`hasPhoto: true`): `text-white` (CSS white, not a CSS variable).
- Overlay text on muted fallback (`hasPhoto: false`): `text-foreground` (CSS variable).
- Engagement row on the stories lane: rendered over the photo — use `bg-transparent` with controls in `text-white` (matching overlay text, since they sit over the scrimmed bottom area). This is a NEW specification for the stories engagement row (no prior precedent).
- Overflow "…" button and close "X" on the stories lane: `text-white` (over photo, top-right corner).
- Bottom sheet panel: `bg-popover text-popover-foreground` per existing `SheetContent` — no change.

## Component Inventory

This section is the primary new content for this phase. Every shared and route-specific component is fully specified here.

### 1. Shared WearCard

Extracted from `WearPhotoOverlays` + `WearPhotoClient` + `WearDetailHero`.

**Container**: `relative w-full aspect-[4/5] overflow-hidden bg-muted`
- Mobile: full-bleed, no rounding (edge-to-edge). Class: no `rounded-lg`.
- Desktop (`md:`): `md:rounded-lg md:max-w-[600px] md:mx-auto` — preserves existing convention.
- Both routes share this exact container class. The stories route wraps it in an embla slide; the detail route wraps it in an `<article>`.

**Photo layer**: Native `<img>` with `w-full h-full object-cover`. Never `next/image` (Pitfall F-2).

**Overlays** (both on photo and on muted fallback):
- Top overlay: gradient scrim top→bottom, `rgba(0,0,0,0.45)` at 0% → transparent at 40%. Contains `p-3` flex row: 40px avatar + username/displayName link + dot separator + relative timestamp. All `text-sm font-semibold` (username) or `text-sm opacity-70` (dot + timestamp). Text is `text-white` when `hasPhoto: true`, `text-foreground` when `hasPhoto: false`.
- Bottom overlay: gradient scrim bottom→top, `rgba(0,0,0,0.45)` at 0% → transparent at 40%. Contains `p-3` flex column: brand (`text-sm font-semibold`) + model (`text-sm`). Same `hasPhoto` text-color switch.
- Overlays are suppressed (`display: none` or conditional render) while photo is in 'pending' CDN-retry state — matches existing `WearPhotoClient` behavior (do not paint text over the PhotoSkeleton shimmer).

**EN-2/3/4 folds** (from CONTEXT.md):
- EN-2: top overlay renders on both photo and no-photo fallback paths — already implemented; ensure shared `WearCard` preserves this.
- EN-3: bottom overlay (brand/model) renders on all three paths (signed-url, watchImageUrl fallback, no-photo placeholder) — already implemented; preserve.
- EN-4: `hasPhoto` toggle switches text from `text-white` to `text-foreground` on muted fallback — already implemented; preserve.

### 2. Engagement Row (D-11)

Rendered at the bottom of the `WearCard` on BOTH routes. This row is part of the shared card contract.

**Stories lane** (over the photo, outside the `aspect-[4/5]` container, pinned to bottom of slide):
- Positioned immediately below the card's aspect container within the slide, OR rendered as a footer within the full-height slide area below the card. Default: footer below the card, not overlapping the photo. The photo's bottom gradient scrim provides visual context but the engagement row itself is NOT overlaid on top of the photo.
- Layout: `flex items-center px-4 pt-3 pb-[calc(0.75rem+env(safe-area-inset-bottom))]`. Left slot: comment icon button (MessageCircle, 20px, `text-white`). Right slot: `LikeButton` (existing component, unchanged). Between: `flex-1` spacer.
- The `pb-[calc(0.75rem+env(safe-area-inset-bottom))]` bottom padding clears the iOS home indicator (env(safe-area-inset-bottom) > 0 on notched devices) when the bottom nav is hidden on this route. On non-notched devices the calc resolves to `0.75rem` = 12px, matching the `pt-3` top value. No `pb-safe` utility exists in `globals.css`; the inline calc is the only safe-area mechanism in this project.
- Background: `bg-transparent` on the stories lane (the body below the photo is the page background, which is `bg-background`). No explicit background needed — it sits below the full-screen photo.
- Touch targets: all buttons `min-h-[44px] min-w-[44px]`.

**Detail page** (`/wear/[id]`) — existing row extended:
- Layout: existing `flex items-center px-4 py-3 border-t border-border md:max-w-[600px] md:mx-auto`.
- Left slot: comment icon button (MessageCircle, 20px, `text-muted-foreground`). Tapping opens the inline comment section (scroll down; no sheet).
- Right slot: `LikeButton` (existing, unchanged).
- The existing `<div className="flex-1 min-h-[44px]" aria-hidden />` spacer is replaced by the actual comment trigger button.

### 3. Overflow Menu ("…") (D-01 / D-08)

Present on BOTH routes, top-right corner of the photo area (not the card container — the actual top-right of the screen or card). Shares the corner with the close/back affordance on the stories lane.

- Icon: `MoreHorizontal` (lucide-react, 20px).
- Stories lane position: `absolute top-3 right-3 z-20` over the photo (above both gradient scrims at `z-10`).
- Detail page position: `absolute top-3 right-3 z-20` within the card container.
- Touch target: `min-h-[44px] min-w-[44px]`.
- Text color: `text-white` on stories (photo background), `text-white` on detail when `hasPhoto: true`, `text-foreground` when `hasPhoto: false`.
- Dropdown: `@base-ui/react` Dropdown or shadcn DropdownMenu. Menu items:
  - "Share / Copy link" — copies `/wear/[wearEventId]` permalink to clipboard (D-01). Always visible.
  - "Add to wishlist" — calls `addToWishlistFromWearEvent`. Conditionally visible per D-09: hidden when viewing own wear, when watch is already owned, or when watch is already on wishlist. On success: `sonner` toast ("Added to wishlist"). On error: `sonner` toast ("Could not add to wishlist").

### 4. Close Affordance (Stories Lane Only)

- Icon: `X` (lucide-react, 20px).
- Position: `absolute top-3 left-3 z-20` over the photo (not top-right, which is occupied by "…").
- Touch target: `min-h-[44px] min-w-[44px]`.
- Text color: `text-white`.
- Action: `router.back()` (or `router.push('/')` if no history entry exists).

**Layout: top-left (X close) + top-right (… overflow) sharing the top band:**
Both sit at `z-20`, above the gradient scrims at `z-10`. The top overlay's avatar row sits at `z-10` (pointer-events-auto). The close and overflow icons must not collide with the avatar area — position them at `top-3` (12px), while the avatar row starts at `p-3` (12px from edge). They co-exist because the close (left) and overflow (right) are outside the avatar row's flex container. If needed, add `pl-12` to the avatar row to clear the close icon — specify this explicitly: avatar row gets `pl-[52px]` left padding to clear the 44px touch target of the X.

### 5. Comment Host — Bottom-Sheet Variant (Stories Lane, D-10)

The `WearCommentHost` component with `variant="bottom-sheet"`.

- Trigger: comment icon button in the engagement row (tap opens the sheet).
- Sheet: `<Sheet open={commentOpen} onOpenChange={setCommentOpen}>`.
- Content: `<SheetContent side="bottom" className="max-h-[60vh] overflow-y-auto">`.
- Header: `<SheetHeader><SheetTitle>Comments</SheetTitle></SheetHeader>`.
- Body (Phase 56A placeholder): `<p className="text-sm text-muted-foreground px-4 py-6 text-center">No comments yet.</p>`.
- Close: existing `SheetContent showCloseButton={true}` (X icon top-right of the sheet panel).
- Swipe-pause: while `commentOpen === true`, `emblaApi.reInit({ watchDrag: false })`. Restored when sheet closes.
- Keyboard (mobile): the sheet's `bottom-0` position and `overflow-y-auto` combined with `max-h-[60vh]` allow the sheet to scroll internally. No explicit `resize` behavior needed — the `@base-ui/react/dialog` primitive handles viewport resizing.
- Sheet animation: slide-up from `translate-y-[2.5rem]` to `translate-y-0`, duration 200ms ease-in-out. Fade in/out via `data-starting-style:opacity-0` / `data-ending-style:opacity-0`. Source: existing `sheet.tsx`.

### 6. Comment Host — Inline Variant (Detail Page, D-10)

The `WearCommentHost` component with `variant="inline"`.

- Container: `<section>` with `px-4 pt-4 pb-6 md:max-w-[600px] md:mx-auto` — matches existing `md:max-w-[600px] md:mx-auto` convention.
- Header: `<h2 className="text-sm font-semibold text-foreground mb-3">Comments</h2>`.
- Body (Phase 56A placeholder): `<p className="text-sm text-muted-foreground text-center py-4">No comments yet.</p>`.
- Separator: `border-t border-border` above the section (same pattern as the engagement row separator).
- The comment trigger button in the engagement row on the detail page scrolls to this section (smooth scroll via `scrollIntoView({ behavior: 'smooth' })`).

### 7. WearsLane (Stories Container, `/wears/[username]`)

Client Component wrapping embla-carousel-react.

- Container: `h-dvh overflow-hidden` — fills the dynamic viewport height without page scroll. On the stories route, `bg-background` body is the fallback behind the embla slides.
- Slides: `flex` (embla container) with each slide at `flex-none w-full`. Each slide displays one `WearCard` (full-width, `aspect-[4/5]`), the engagement row below it, and potentially a note snippet if `wear.note` is non-null (see §Note Copy).
- The carousel's visual area: the photo fills the horizontal width, with the engagement row below. On mobile, the combined photo + engagement row + (optional note) fits within `h-dvh`. If note is present, truncate to 2 lines (`line-clamp-2 text-sm text-muted-foreground px-4 pt-2`).
- Embla options: `{ startIndex: initialSlideIndex, align: 'start', containScroll: false, duration: getEmblaDuration() }`.
- Swipe is paused when comment sheet is open: `emblaApi.reInit({ watchDrag: !commentOpen })`.
- No visible scroll indicators, no dot pagination — swipe-only navigation. (Progress bars deferred — see §Deferred.)

## Route-Specific Layout Contracts

### `/wears/[username]` — Stories Lane

**Mobile (`< 768px`):**
- Full-screen, no nav chrome. BottomNav and SlimTopNav are hidden for this route.
- Nav hiding approach: add `pathname.startsWith('/wears/')` check to `BottomNav` client-side visibility predicate (Option B from RESEARCH.md A2 assumption). Also hide `SlimTopNav` via the same pathname check. This follows the existing `isPublicPath` pattern and avoids route-group restructuring.
- Container: `h-dvh overflow-hidden fixed inset-0 bg-background z-0`.
- Top-left: X close button. Top-right: "…" overflow. Both `absolute z-20`.
- Photo card: full-width `aspect-[4/5]`. Below photo: engagement row (`flex items-center px-4 pt-3 pb-[calc(0.75rem+env(safe-area-inset-bottom))]`), optional note (2-line truncated).
- Bottom-sheet comment host renders portaled over the photo via `SheetPortal`.

**Desktop (`md+`, 768px and above):**
- Default choice: **centered 600px column** — consistent with the `md:max-w-[600px] md:mx-auto md:rounded-lg` convention already established by `WearDetailHero` and `WearPhotoClient`.
- Rationale: full-viewport immersive on desktop creates aspect-ratio problems (the 4:5 photo would stretch or require complex centering). The centered column is the established Horlo convention for wear photos on desktop.
- The `SlimTopNav` and `DesktopTopNav` ARE visible on desktop. Only bottom nav is hidden (it's `md:hidden` by default already — no change needed on desktop).
- Container wraps the embla slides: `max-w-[600px] mx-auto`. The `fixed inset-0` full-screen class applies only on mobile (use `md:relative md:inset-auto md:h-auto md:overflow-visible` to unset on desktop).
- On desktop the page scrolls normally (the single centered slide is taller than the viewport if photo + engagement row together exceed viewport height — allow natural scroll).
- The `h-dvh overflow-hidden fixed inset-0` full-screen constraint is mobile-only. Apply via responsive class: `fixed inset-0 h-dvh overflow-hidden md:static md:inset-auto md:h-auto md:overflow-visible`.

### `/wear/[id]` — Detail Permalink

**All breakpoints:**
- Nav chrome retained (existing behavior). `<article className="flex flex-col gap-4 pt-4">`.
- Photo card: `aspect-[4/5] w-full bg-muted md:rounded-lg md:max-w-[600px] md:mx-auto` — unchanged.
- Engagement row: `flex items-center px-4 py-3 border-t border-border md:max-w-[600px] md:mx-auto`.
- Inline comment section below engagement row: `px-4 pt-4 pb-6 md:max-w-[600px] md:mx-auto`, separated by `border-t border-border`.

## Deferred Items (resolved here, not in 56A scope)

### Progress Bars / Ring Progress

**Decision: Explicitly deferred out of Phase 56A.**

Classic Instagram-style progress segments require an auto-advance timer — without a timer, progress bars have no meaningful meaning (there is nothing to "progress" through at a defined rate). Phase 56A's stories lane is **swipe-only with no auto-advance**. Building progress bars without auto-advance would be misleading UI.

Slide position indicator (e.g., N of M text or dots) is also out of scope for 56A. If slide position context is needed, it can be added in a future polish phase.

This decision is documented here so the planner and executor do not build progress UI in Phase 56A.
