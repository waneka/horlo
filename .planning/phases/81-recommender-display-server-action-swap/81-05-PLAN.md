# Plan 81-05: Same-Family + Lineage Rail Canonical Display

<objective>
Extend the read-time canonical-JOIN pattern (Plan 02's fix for `topUpFromCatalogPopularity`)
to the two watch-detail-page rails that CONTEXT.md § Deferred Ideas explicitly flagged
as revisit-if-visible surfaces: `getSameFamilyForCatalog` and `getLineageForReference`.
Both currently read `watches_catalog.brand` / `watches_catalog.model` denorm columns
directly, so drift catalog rows (e.g., `Hamilton Watch / DriftTest Chrono` with canonical
`Hamilton` / `Khaki Field Mechanical`) surface with drift denorm strings on the detail
page's SameFamilyRail and LineageRail.
</objective>

</truths>

<threat_model>
- **T-81-P05-01: Denorm regression via missed JOIN.** If either function loses the INNER JOIN in a future refactor, drift returns to the detail page. Mitigation: acceptance criterion asserts `innerJoin(brands` and `innerJoin(watchFamilies` (or equivalent SQL) counts stay ≥ 1 per function.
- **T-81-P05-02: INNER JOIN loses rows.** Post-Phase-80 all `watches_catalog` rows have NOT NULL `brand_id` + `family_id`. INNER JOIN cannot lose rows. Same guarantee Plan 02 relied on. Verify via post-fix grep of the schema: `brandId: uuid().notNull()` and `familyId: uuid().notNull()`.
- **T-81-P05-03: GROUP BY drift.** `getSameFamilyForCatalog` uses GROUP BY. When adding `brands.name` / `watchFamilies.name` to the SELECT, they MUST also appear in GROUP BY or aggregate over — otherwise Postgres will reject the query. Acceptance criterion: `npm run build` + a live query hit against local Supabase both succeed.
- **T-81-P05-04: Lineage CTE arm asymmetry.** `getLineageForReference` uses a recursive CTE with seed + recursive arms — both must carry the JOIN and project canonical strings, or the recursive arm returns drift while the seed returns canonical (or vice versa). Acceptance criterion: both arms use `b.name` / `f.name`.
</threat_model>
