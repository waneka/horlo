**Create `src/app/actions/watches.ts` (per D-11, D-12, D-13, D-14):**

Start with `'use server'` directive at the top of the file.

Import `revalidatePath` from `next/cache`, DAL functions from `@/data/watches`, `ActionResult` from `@/lib/actionTypes`, and Watch type from `@/lib/types`.

**Zod validation schemas (per D-14):**
Try using `createInsertSchema` from `drizzle-orm/zod` to auto-generate schemas from the Drizzle table. If this works:
```typescript
import { createInsertSchema } from 'drizzle-orm/zod'
import { watches } from '@/db/schema'

const insertWatchSchema = createInsertSchema(watches).omit({
  id: true,
  userId: true,
  createdAt: true,
  updatedAt: true,
})
```

If `drizzle-orm/zod` does NOT work (Zod version incompatibility — see RESEARCH.md Open Question 1), fall back to hand-written Zod schemas that validate the same fields as `Omit<Watch, 'id'>`. Import zod and define:
```typescript
import { z } from 'zod'
// or import { z } from 'zod/v3' if zod@4 is installed but drizzle needs v3 compat

const insertWatchSchema = z.object({
  brand: z.string().min(1),
  model: z.string().min(1),
  reference: z.string().optional(),
  status: z.enum(['owned', 'wishlist', 'sold', 'grail']),
  // ... all Watch fields except id
})
```

**Export these Server Actions:**

1. `addWatch(userId: string, data: unknown): Promise<ActionResult<Watch>>` (per D-12)
   - Validate `data` with `insertWatchSchema.safeParse(data)`
   - If invalid: return `{ success: false, error: 'Invalid watch data: ...' }` with a summary of field errors
   - If valid: call `watchDAL.createWatch(userId, parsed.data)`
   - On success: call `revalidatePath('/')` (per D-13), return `{ success: true, data: watch }`
   - On catch: return `{ success: false, error: 'Failed to create watch' }` (per D-08/D-12 — never throw)

2. `editWatch(userId: string, watchId: string, data: unknown): Promise<ActionResult<Watch>>`
   - Validate with a partial schema (all fields optional): `insertWatchSchema.partial().safeParse(data)`
   - Delegate to `watchDAL.updateWatch(userId, watchId, parsed.data)`
   - revalidatePath('/') on success

3. `removeWatch(userId: string, watchId: string): Promise<ActionResult<void>>`
   - No input validation needed (just IDs)
   - Delegate to `watchDAL.deleteWatch(userId, watchId)`
   - revalidatePath('/') on success

**Note on userId parameter:** Phase 3 accepts userId as an explicit parameter for testability (per D-07). Phase 4 will replace this with session-derived userId. Do NOT read userId from cookies or headers in this phase. Add a comment: `// TODO(Phase 4): Replace userId parameter with session-derived userId`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> Server Action | Untrusted input crosses network boundary |
| Server Action -> DAL | Validated input passed to data layer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Tampering | Unvalidated input to Server Actions | mitigate | Zod safeParse on every action before DAL delegation (D-14); unknown fields stripped |
| T-03-08 | Information Disclosure | Error stack traces in action responses | mitigate | Catch blocks return generic error strings, never raw Error objects or stack traces (D-12) |
| T-03-09 | Elevation of Privilege | Client-supplied userId | accept | Phase 3 accepts userId for testability (D-07); Phase 4 replaces with session-derived userId. Documented with TODO comment. |
| T-03-10 | Spoofing | Direct POST to Server Action endpoint | accept | Next.js encrypts action IDs preventing enumeration; auth enforcement deferred to Phase 4 |
</threat_model>
