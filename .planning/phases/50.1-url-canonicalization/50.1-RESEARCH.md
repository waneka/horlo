### `findViewerWatchByCatalogId` simplification (SC#2)

Current shape (`src/app/catalog/[catalogId]/page.tsx:282-308`):

```typescript
async function findViewerWatchByCatalogId(
  userId: string,
  catalogId: string,
): Promise<{ id: string; acquisitionDate: string | null } | null> {
  const rows = await db
    .select({
      id: watchesTable.id,
      acquisitionDate: watchesTable.acquisitionDate,
      createdAt: watchesTable.createdAt,
    })
    .from(watchesTable)
    .where(and(
      eq(watchesTable.userId, userId),
      eq(watchesTable.catalogId, catalogId),
      eq(watchesTable.status, 'owned'),  // BUG-01 fix (D-02): only 'owned' rows are "truly owned"
    ))
    .limit(1)
  if (rows.length === 0) return null
  const row = rows[0]
  const iso = row.acquisitionDate !== null
    ? row.acquisitionDate
    : new Date(row.createdAt).toISOString()
  return { id: row.id, acquisitionDate: iso }
}
```

After the redirect, only `row.id` is consumed. SC#2 names two correct options:

#### Option A: Drop the `status='owned'` filter, keep `acquisitionDate` selection (minimal diff)

- Pros: Smallest diff; one-line removal at line 296.
- Cons: Changes the semantic meaning of the function — wishlist/grail/sold rows now match, and the redirect fires on them. **This is wrong** per SC#3 and SC#6: "wishlist-, sold-, and unrelated-status rows do NOT trigger the redirect (the Phase 48 BUG-01 edge cases stay green)." The status filter is what guarantees BUG-01 stays fixed.
- **Verdict: REJECT.** SC#2 says "the function either drops the status filter or is simplified accordingly (planner's call — both are correct once the flip is gone)" — but SC#2's "drops the status filter" framing was written assuming the flip's removal eliminates the *need* for the filter as a *callout-gating* mechanism. The filter still serves a redirect-gating purpose post-flip: it's what stops wishlist watches from redirecting away from the catalog page. SC#3 + SC#6 are the binding constraints; SC#2 must yield to them.

#### Option B: Keep the `status='owned'` filter; simplify the return shape and acquisitionDate plumbing

- The function now only needs to return the `id` (caller just passes it to `redirect()`).
- Drop `acquisitionDate` and `createdAt` from the SELECT; drop the `iso` computation; return `{ id: string } | null` (or simplify further to `string | null`).
- Update the JSDoc to reflect the new contract.
- The `T-20-06-01` security comment about `userId AND catalogId` scoping remains accurate; adding the third `status='owned'` filter still narrows, not widens.
- **Verdict: RECOMMEND.** This is the "simplified accordingly" reading of SC#2. It preserves BUG-01 semantics, removes dead fields from the SELECT, and produces cleaner code.

#### Option C: Keep current shape verbatim

- The unused `acquisitionDate` / `createdAt` return fields become dead code (TS will not flag because the type is structural — the consumer just stops reading them).
- **Verdict: REJECT.** Leaves dead code; conflicts with the spirit of SC#2 ("simplified accordingly").

**Planner's recommended choice: Option B.** Keep the `status='owned'` filter (required by SC#3 + SC#6), simplify the return type to `{ id: string } | null`, drop the `acquisitionDate` + `createdAt` SELECT fields, drop the ISO computation, update the JSDoc. ### Three representative patterns

**Pattern A — Simple unconditional redirect from a Server Component (`src/app/u/[username]/page.tsx:1-11`):**

```typescript
import { redirect } from 'next/navigation'

export default async function ProfileIndexPage({
  params,
}: {
  params: Promise<{ username: string }>
}) {
  const { username } = await params
  // OUTSIDE try/catch — redirect() throws NEXT_REDIRECT and must propagate.
  redirect(`/u/${username}/collection`)
}
```

Shape highlights: named import; called *after* `await params` (request-time resolution); the function never returns — `redirect()` throws. The inline comment explicitly notes the try/catch caveat (mirrors the Next 16 docs warning).

**Pattern B — Conditional redirect mid-function (`src/app/insights/page.tsx:21-25`):**

```typescript
export default async function InsightsRetirementPage() {
  const user = await getCurrentUser()
  const profile = await getProfileById(user.id)
  redirect(profile?.username ? `/u/${profile.username}/insights` : '/')
}
```

Shape highlights: data fetch first, then a single `redirect()` call with the target URL computed inline. This is the closest pattern to what Phase 50.1 needs — `redirect()` fires after a conditional data check.

**Pattern C — Redirect inside a catch handler (`src/app/watch/new/page.tsx:45-53`):**

```typescript
export default async function NewWatchPage({ searchParams }: NewWatchPageProps) {
  let user: { id: string; email: string }
  try {
    user = await getCurrentUser()
  } catch (err) {
    if (err instanceof UnauthorizedError) {
      redirect('/signin')
    }
    throw err
  }
  // ... rest of function ...
}
```

Shape highlights: the `redirect()` is **outside** the inner `getCurrentUser()` try block — the docs warning is honored. The catch handler narrows on the auth error class, then redirects (which throws `NEXT_REDIRECT`). The catch's outer `throw err` re-raises any non-auth error.

### Two existing redirect-assertion idioms in the codebase

The codebase has **two distinct patterns** for asserting `redirect()` behavior; the planner picks one. Both are battle-tested.

#### Idiom 1: Mock `next/navigation` so `redirect` throws a NEXT_REDIRECT digest, then assert the mock was called

Source: `tests/app/insights-retirement.test.tsx` (lines 1-53). This is the cleaner pattern for asserting both the *fact* of the redirect AND the *target URL*.

```typescript
// tests/app/insights-retirement.test.tsx:3-9
vi.mock('next/navigation', () => ({
  redirect: vi.fn((url: string) => {
    const err: any = new Error('NEXT_REDIRECT')
    err.digest = `NEXT_REDIRECT;push;${url};307`
    throw err
  }),
}))

// ... (other mocks) ...

import { redirect } from 'next/navigation'

it('redirects to /u/{username}/insights when profile is present', async () => {
  vi.mocked(getCurrentUser).mockResolvedValue({ id: 'u-1', email: 'a@b.co' })
  vi.mocked(getProfileById).mockResolvedValue({ id: 'u-1', username: 'alice' } as any)
  await expect(InsightsRetirementPage()).rejects.toThrow('NEXT_REDIRECT')
  expect(redirect).toHaveBeenCalledWith('/u/alice/insights')
})
```

Shape: `vi.mock('next/navigation', ...)` returns a mock `redirect` that throws an Error with a `digest` matching the real Next.js `NEXT_REDIRECT;push;{url};{statusCode}` shape. The test then asserts (a) the call rejects with `'NEXT_REDIRECT'`, and (b) `redirect` was called with the expected URL. This is the canonical pattern in this codebase.

**SC#3 mapping:** "GET `/catalog/[catalogId]` for an owned catalog ref returns a 307 with `Location: /watch/[id]`" → in this mock pattern, the 307 status is encoded in the digest (`;307` suffix); asserting `redirect` was called with `/watch/per-user-uuid` AND that the function throws `'NEXT_REDIRECT'` is the vitest equivalent of asserting a real 307 + Location header. The digest format is the Next.js internal convention for surfacing redirect data through a thrown Error.

`[CITED: tests/app/insights-retirement.test.tsx:1-53]`

#### Idiom 2: Don't mock `next/navigation`; assert the real digest string the framework throws

Source: `tests/app/preferences-redirect.test.ts` (lines 1-43).

```typescript
it('redirects with hash preserved — Location header = /settings#preferences', async () => {
  const PreferencesPage = (await import('@/app/preferences/page')).default

  let captured: unknown = null
  try {
    ;(PreferencesPage as () => never)()
  } catch (e) {
    captured = e
  }

  expect(captured).not.toBeNull()
  const digest = (captured as { digest?: string; message?: string })
  const stringForm = String(digest?.digest ?? digest?.message ?? captured)
  expect(stringForm).toContain('NEXT_REDIRECT')
  expect(stringForm).toContain('/settings#preferences')

```

Shape: no `vi.mock('next/navigation', ...)` — the real Next.js `redirect()` is invoked, which throws an internal Error with a `digest` field. The test catches the throw and asserts the digest string contains both `'NEXT_REDIRECT'` and the target URL.

**When to use:** Idiom 2 is preferred for cases where there's no other complex DAL/auth mocking needed (the preferences page is a simple redirect-only file). Idiom 1 is preferred when the page does data fetching that needs mocking — which is the catalog page's case.

## 8. Risks, Landmines, and Anti-Patterns to Avoid

### Risk 1: Putting the redirect in `proxy.ts` — DO NOT

- **Memory:** `feedback_proxy_router_cache_poisoning` (2026-05-19) — a `NextResponse.redirect` (any 3xx) issued from `proxy.ts` on an RSC prefetch is stored in Next 16's in-memory Router Cache; subsequent soft-nav clicks serve the cached redirect → browser renders 404. A hard refresh works (full-doc nav bypasses Router Cache). Signature: cold-cache-works → warm-cache-breaks → refresh-works. `[VERIFIED: memory file read]`
- **Live precedent:** The `/u/*` profile route took three attempts to fix; final solution was `isProfilePath()` ungate so the proxy *never* emits a redirect for `/u/*`. See `.planning/debug/profile-page-404-top-nav.md`.
- **`src/proxy.ts` current state (37 lines, verified):** Contains exactly one `NextResponse.redirect` call at line 22 — `loginUrl` for the unauthenticated-non-public-non-profile path. **NO `/catalog/` handling.** SC#4 is already satisfied; the test just needs to confirm this stays true.
- **SC#4 grep guard for tests:**
  ```bash
  grep -E "/catalog/|catalog/\[" src/proxy.ts
  # Expected: no output
  ```
  Promote this to a test that reads `src/proxy.ts` and asserts the substring `/catalog/` is absent — mirrors the `tests/app/insights-retirement.test.tsx:47-52` "module does NOT declare 'use cache'" pattern.

### Risk 2: `redirect()` inside a try/catch

- **Source:** `node_modules/next/dist/docs/01-app/02-guides/redirecting.md` lines 81-87: "`redirect` throws an error so it should be called **outside** the `try` block when using `try/catch` statements." `[CITED: Next 16 docs]`
- **Why it matters:** The catalog page does NOT currently have a try/catch around the conditional block at lines 107-115. **Adding `redirect()` there is safe.** But if a future refactor wraps the area in a try/catch (for error-boundary purposes, for example), the `NEXT_REDIRECT` throw would be caught and the redirect would silently fail. The existing inline comment in `src/app/u/[username]/page.tsx:9` makes this explicit ("OUTSIDE try/catch — redirect() throws NEXT_REDIRECT and must propagate.") — the planner should add the same comment at the new redirect site.
- **Verification:** `src/app/catalog/[catalogId]/page.tsx` has no `try {` keywords anywhere — verified by inspection. The redirect lives in clean control flow.

### Risk 3: `redirect()` called from a Client Component

- The Next 16 docs allow `redirect()` in Client Components during the rendering process but not in event handlers. The catalog page is a Server Component (no `'use client'` directive — verified by grep). No risk here.

### Risk 4: BUG-01 regression — dropping the `status='owned'` filter

- **Anti-pattern:** Reading SC#2 ("no longer needs the `status='owned'` scope ... drops the status filter") as license to remove the filter.
- **Why it's an anti-pattern:** SC#2 reads as if dropping the filter is correct, but SC#3 and SC#6 require that wishlist/grail/sold rows do **not** trigger the redirect. The filter is what enforces this at the query level. Drop it, and the redirect fires on wishlist watches → BUG-01 reappears in a new form (now the user is *redirected away* from the catalog page instead of being shown a misleading "You own this" callout — even worse UX). **§2 Option B** keeps the filter and is the correct read. The WR-01 predicate-AST guard at `tests/app/catalog-page.test.ts:305-338` already protects this invariant.
- **Verification:** Run `npx vitest run tests/app/catalog-page.test.ts` post-edit. The "BUG-01 — query is scoped to status='owned' (predicate guard)" test must remain green. If it fails, the filter was removed; revert.

### Risk 5: Test-infra-induced phase bloat

- **Not a risk in this codebase.** Vitest is fully set up (`vitest.config.ts`); the redirect-mock pattern is established (`tests/app/insights-retirement.test.tsx`); the catalog page already has a comprehensive test file (`tests/app/catalog-page.test.ts`) with all DAL mocks in place. The planner adds ~30 LOC of test changes to an existing file, not a new framework. **No Playwright recommendation; no manual smoke test required.**

### Risk 6: `self-via-cross-user` framing has type-system consumers that survive

- **What:** Grep reveals 8 references to `self-via-cross-user` and `VerdictBundleSelfOwned` across the codebase (catalog page, CollectionFitCard, CollectionFitCard test, WishlistRationalePanel, CatalogPageActions comment, composer.ts, verdict/types.ts). `[VERIFIED: grep]`

  - **Option X (minimal — recommended for phase scope):** Leave dead code in place; tag with a `TODO: cleanup after Phase 50.1` comment somewhere or open a follow-up quick task. The dead branches don't execute and don't break anything.
  - **Option Y (full cleanup):** Remove `VerdictBundleSelfOwned` type, the `'self-via-cross-user'` framing literal, the CollectionFitCard `YouOwnThisCallout` branch (lines 30-31 + 124+), and the corresponding CollectionFitCard test.
