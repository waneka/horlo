# Phase 41: Account Danger Zone + Branded Auth Emails - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase delivers two fully independent deliverables, both off the catalog
hierarchy serial spine:

1. **Account Danger Zone (SET-13)** — two destructive account actions in
   `/settings#account`: "Wipe Collection" (deletes all `watches` + `wear_events`
   + `wear-photos/{userId}/` storage files; preserves account, profile, follows)
   and "Delete Account" (full hard delete via service-role
   `supabase.auth.admin.deleteUser()`). Both gated by type-to-confirm input +
   password re-auth + multi-step modal. No grace period (single-user scale).

2. **Branded Auth Emails (SET-14)** — three rebranded Supabase Auth email
   templates (Confirm signup, Reset Password, Change Email): 600px single-column
   HTML, brand color, single CTA button, built with `react-email` 6.1.1 +
   `@react-email/components`. HTML output is pasted into the Supabase Auth
   dashboard — **no Next.js app code ships for the emails themselves.**

Discussion clarified HOW to implement these. New capabilities are out of scope.

</domain>

<decisions>
## Implementation Decisions

### Danger Zone — Modal Architecture
- **D-01:** Two separate modal components — `WipeCollectionModal` and
  `DeleteAccountModal`. Not a shared parametrized component. Each can diverge
  freely in copy, severity, and warnings.
- **D-02:** Step flow for both modals: **Warn → (type-to-confirm input +
  password field on one combined step) → execute.** Step 1 is a warning screen
  listing exactly what gets destroyed. Step 2 combines the type-to-confirm input
  and the current-password field on a single final step. Then execute.
- **D-03:** The password re-auth is a **new inline step inside each Danger Zone
  modal**, reusing the Phase 22 `PasswordReauthDialog` *pattern* (the
  `signInWithPassword({ email, password })` session-refresh logic) — NOT
  reusing the `PasswordReauthDialog` component itself. `PasswordReauthDialog` is
  currently coupled to the password-change use case (`pendingNewPassword` prop).
  Keep each danger flow self-contained as one modal.

### Danger Zone — Type-to-Confirm
- **D-04:** Fixed keyword per action: user types `WIPE` to confirm Wipe
  Collection, `DELETE` to confirm Delete Account. Distinct keywords prevent
  muscle-memory carryover between the two flows.
- **D-05:** The execute button stays **disabled until the typed text matches the
  keyword exactly** — the disabled button is the gate; no inline mismatch error
  state. Mirrors the existing `PasswordReauthDialog` disabled-until-filled
  pattern (`disabled={pending || !password}`).

### Danger Zone — Post-Action Behavior
- **D-06:** After Wipe Collection succeeds: close the modal, fire a Sonner
  success toast (e.g. "Collection wiped"), stay on the `/settings` Account tab.
  Consistent with existing settings save patterns.
- **D-07:** After Delete Account succeeds: sign the user out and redirect to the
  marketing landing page `/`. No new route. (Login page rejected as jarring;
  goodbye page rejected as unneeded code.)
- **D-08:** `notifications.actor_id` cascade — Account Delete cascades
  nulling/removal of `notifications.actor_id` rows in OTHER users' inboxes. At
  single-user scale this is moot (no other inboxes exist). **No mention in the
  modal UI.** Documented here per success criterion 3 — see UX Note below.

### Branded Auth Emails
- **D-09:** react-email source files live in a **top-level `emails/` directory
  at repo root**, build-excluded (NOT under `src/`, so it never enters the
  Next.js build). The react-email preview server runs against it during dev.
  This keeps a committed source of truth for future template edits while
  honoring "no Next.js code change ships."
- **D-10:** Email header uses a **styled "Horlo" text wordmark** in the brand
  color — NOT a hosted `<img>` logo. Reason: no Horlo logo asset exists in the
  repo (only Next.js default SVGs in `public/`). A text wordmark is bulletproof
  across clients (no blocked-image risk, dark-mode friendly). Revisit a hosted
  image logo if/when a logo is designed (see Deferred Ideas).
- **D-11:** Brand color matches the app's existing accent token from
  `src/app/globals.css` (`--accent: oklch(0.76 0.12 75)`). NOTE for planner:
  email HTML cannot rely on `oklch()` — convert the accent to a hex/rgb fallback
  for Outlook MSO and older clients.
- **D-12:** CTA button copy is action-specific per template: "Confirm your
  email" / "Reset your password" / "Confirm email change".

### UX Note — Account Delete Cascade (success criterion 3)
Deleting an account cascades to `notifications.actor_id` rows in other users'
inboxes: notifications generated BY the deleted user lose their actor reference
(nulled or row-removed, depending on the FK `ON DELETE` rule). This is
schema-correct behavior. At Horlo's current single-user scale (one user,
twwaneka@gmail.com) there are no other inboxes, so it is observationally
inert. It is intentionally NOT surfaced in the Delete Account modal copy. When a
second user signs up, revisit whether the warning copy should mention it.

### Claude's Discretion
- Exact warning-screen copy listing what gets destroyed (within D-02 structure).
- Exact toast wording for the Wipe success path (D-06).
- Email body copy and visual polish details (within 600px single-column,
  wordmark header, single CTA constraints).
- Service-role Supabase client construction approach — no service-role/admin
  client exists in `src/lib/supabase/` today; planner/researcher decides how to
  create one safely server-side for `auth.admin.deleteUser()`.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/components/settings/PasswordReauthDialog.tsx` — re-auth *pattern* source.
  Copies `signInWithPassword({ email: currentEmail, password })` session-refresh
  logic into each Danger Zone modal step (D-03). Do NOT reuse the component
  directly — it is coupled to `pendingNewPassword`.
- `src/components/settings/AccountSection.tsx` — Account tab content composer
  (`space-y-8`). The Danger Zone section is appended here, below
  `EmailChangeForm` + `PasswordChangeForm`.
- `src/lib/hooks/useFormFeedback.ts` — `useFormFeedback({ dialogMode: true })`
  hook used by `PasswordReauthDialog` for pending/error/toast handling; reuse
  for the danger modals.
- `src/components/ui/dialog`, `button`, `input`, `label` — shadcn primitives
  already used by the reauth dialog.
- Sonner toasts — established success-feedback pattern (D-06).

### Established Patterns
- Supabase clients live in `src/lib/supabase/`: `client.ts` (browser, anon),
  `server.ts` (server, anon), `proxy.ts`. **No service-role/admin client
  exists** — one must be created for `auth.admin.deleteUser()` (D-13 discretion).
- Server actions live in `src/app/actions/` (`watches.ts`, `wearEvents.ts`,
  etc.) — Wipe Collection's DB+storage purge belongs as a new server action.
- `wear-photos` storage bucket: paths are `{userId}/{wearEventId}.jpg` in a
  private bucket (see `src/lib/storage/wearPhotos.ts`, `src/lib/wywtTypes.ts`).
  Wipe + Delete must purge `wear-photos/{userId}/` BEFORE the DB delete.
- App accent token: `--accent: oklch(0.76 0.12 75)` in `src/app/globals.css`.

### Integration Points
- Danger Zone UI mounts inside `AccountSection.tsx` on `/settings#account`.
- Wipe Collection → new server action deleting `watches` + `wear_events` rows +
  `wear-photos/{userId}/` storage objects.
- Delete Account → service-role server action: purge storage → DB hard delete →
  `auth.admin.deleteUser()` → client signs out → redirect `/`.
- Emails: `emails/` dir → react-email render → HTML pasted into Supabase Auth
  dashboard. No app-code wiring.

</code_context>

<specifics>
## Specific Ideas

- Type-to-confirm keywords are literal: `WIPE` and `DELETE` (uppercase, D-04).
- Email header is a text wordmark reading "Horlo" (D-10).
- CTA labels are fixed strings (D-12): "Confirm your email", "Reset your
  password", "Confirm email change".
- Email layout constraints (from SET-14): 600px single-column HTML, single CTA
  button; cross-client verified on Apple Mail iOS dark mode, Outlook MSO
  conditional, Gmail web.

</specifics>
