## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-12 | Tampering | `{{ .ConfirmationURL }}` token | mitigate | The token is emitted as a literal string and verified post-export via grep — a corrupted/escaped token would break the auth link. Task 3 acceptance criterion checks the literal token survives export un-escaped. |
| T-41-13 | Information Disclosure | email body copy | accept | Templates contain no PII and no account-specific data beyond the Supabase-substituted confirmation URL. The footer copy is generic. No exposure surface. |
| T-41-14 | Spoofing | email branding / phishing resemblance | accept | The branded templates make legitimate Horlo emails recognizable; they do not weaken DKIM/SPF (RESEARCH — template content does not affect signing; Phase 21 Resend SMTP at mail.horlo.app is unaffected). Anti-spoofing remains at the SMTP/DKIM layer, out of scope here. |
| T-41-15 | Tampering | `emails/` leaking into the app build | mitigate | Task 1 adds `"emails"` to `tsconfig.json` `exclude`; `npm run build` exit-0 confirms the build graph is unaffected. react-email packages are devDependencies only (plan 41-01). |
</threat_model>
