<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input -> modal state | Type-to-confirm keyword and password are user-entered client state |
| modal -> browser Supabase client | Re-auth `signInWithPassword` runs on the browser client before the destructive action |
| modal -> server action | The modal invokes `wipeCollection` / `deleteAccount` â€” server-side identity is re-verified by the action (plan 41-02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-08 | Spoofing | re-auth bypass in the modal | mitigate | The execute handler ALWAYS runs `signInWithPassword` first and short-circuits with `Password incorrect.` on failure â€” the server action is unreachable without a successful password re-auth. The disabled-until-keyword-match gate (D-05) is a UX guard, not the security boundary; the real boundary is the server action's `getCurrentUser()` session check (plan 41-02 T-41-05). Even if a user bypasses the client gate via devtools, the server action only ever touches the verified session's own `user.id`. |
| T-41-09 | Tampering | accidental destructive action | mitigate | Two-step modal (warn -> confirm), distinct type-to-confirm keywords (`WIPE` vs `DELETE`, D-04 â€” prevents muscle-memory carryover), and the disabled execute button (D-05) collectively make an accidental destructive trigger require deliberate, action-specific input. |
| T-41-10 | Information Disclosure | error copy | accept | Wrong-password error is the neutral `Password incorrect.` (no account-existence or rate-limit detail leaked). Server-failure copy is generic. No sensitive data surfaced. Low risk. |
| T-41-11 | Repudiation | post-delete session state | mitigate | After `deleteAccount` succeeds, the browser client `signOut()` clears the local session and `router.push('/')` leaves the authenticated area â€” preventing a stale post-delete session from appearing usable. |
</threat_model>
