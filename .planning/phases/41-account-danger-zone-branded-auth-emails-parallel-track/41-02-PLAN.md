<objective>
Track A backend. Build the server-only service-role Supabase client and the two
Danger Zone server actions (`wipeCollection`, `deleteAccount`) that the modals
in plan 41-03 will call.

Purpose: SET-13 requires two destructive account operations. The critical
correctness finding (RESEARCH Pitfall 1): `public.users` has NO FK to
`auth.users`, so `auth.admin.deleteUser()` alone leaves `public.users` and its 9
cascade children orphaned. `deleteAccount` MUST explicitly delete the
`public.users` row. Storage must be purged before any DB delete (SET-13 success
criterion 2 / RESEARCH Pitfall 2).
Output: `src/lib/supabase/admin.ts` (service-role client factory) and
`src/app/actions/account.ts` (`wipeCollection` + `deleteAccount`).

Decision coverage (CONTEXT.md): this plan supports D-06: (wipeCollection returns
a clean ActionResult so the 41-03 modal can fire the post-Wipe toast — the toast
itself is UI, owned by 41-03), D-07: (deleteAccount does the hard delete and
deliberately does NOT sign out or redirect — the post-Delete sign-out + redirect
to `/` is owned by the 41-03 modal), and D-08: (this plan emits NO
`notifications.actor_id` cascade copy — the cascade is schema-correct behavior
documented in CONTEXT.md only).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> server action | The modals (plan 41-03) invoke these actions; the action must re-derive identity server-side, never trust a client-supplied user id |
| server action -> service-role client | The service-role key bypasses RLS — any logic flaw scopes a destructive op to the wrong user |
| server action -> Supabase Storage | Storage purge runs against `wear-photos/{userId}/` — the prefix must be the authenticated user's own id |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-03 | Elevation of Privilege | `deleteAccount` / `wipeCollection` | mitigate | Both actions derive the user id ONLY from `getCurrentUser()` (server-side `auth.getUser()`); no user id is accepted as a parameter. Every delete is scoped `eq(...userId, user.id)` / `eq(users.id, user.id)`. The service-role client is used ONLY for `auth.admin.deleteUser(user.id)` where the id is the already-authenticated caller's id. |
| T-41-04 | Information Disclosure | `src/lib/supabase/admin.ts` | mitigate | `import 'server-only'` hard-fails any build that imports the admin client into a client bundle. Client created per-call so the key is not held at module scope. Confined to `src/app/actions/account.ts` — never imported by a Server Component that streams HTML. |
| T-41-05 | Spoofing | re-auth bypass | mitigate | The destructive server action does NOT itself enforce re-auth — re-auth runs client-side in the modal (plan 41-03, D-03). The server action's defense is the session check via `getCurrentUser()`: a request without a valid Supabase session is rejected with `Not authenticated`. A caller cannot bypass re-auth to delete a DIFFERENT account because the action only ever touches `user.id` from the verified session. Re-auth is a confirmation gate, not the authZ boundary. |
| T-41-06 | Tampering | storage purge prefix | mitigate | The purge prefix is `${user.id}/` derived from the authenticated session — never from client input. The `wear-photos` DELETE RLS policy independently enforces `(storage.foldername(name))[1] = auth.uid()` as defence-in-depth for the session-scoped client. |
| T-41-07 | Denial of Service | orphaned data after partial failure | mitigate | Storage purge runs first; if it throws, no DB delete occurs (fail-closed). If `db.delete(users)` succeeds but `auth.admin.deleteUser` fails, the action returns an error — the orphaned `auth.users` row is recoverable by re-running; documented as accepted residual in SUMMARY. |
</threat_model>
