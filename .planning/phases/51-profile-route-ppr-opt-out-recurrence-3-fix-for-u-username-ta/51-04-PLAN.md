**Defensive note on session validity:** `getSession()` returns a session whose JWT may be expired. The `@supabase/ssr` package's cookies adapter SHOULD silently refresh expired sessions on read (via the `setAll` cookie writer), but does so without an explicit network call — it uses the refresh token cookie. If the session is invalid (decryption error or refresh failure), `data.session` is null, `user` becomes null, and the proxy treats the request as anon. This is the correct fail-closed behavior for an optimistic check.

**The trade-off explicitly accepted by this change:** a forged or stolen session JWT will be accepted by the proxy gate (because no server verification happens). Sensitive operations on pages and Server Actions MUST continue to use `getCurrentUser()` (which calls `getUser()` — server-verified). The proxy gate is optimistic; page/action gates are authoritative. This is the documented Supabase + Next.js pattern (`authentication.md:1031` + Supabase SSR docs).
