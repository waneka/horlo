<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| viewer browser → LikeButton client island | viewer-controlled clicks; the only client input forwarded to the server is `{ type, id }` (already Zod `.strict()`-validated in the Phase 55 action) |
| Next.js cache layer → per-viewer hydration read | `getLikesForTargetCached` cache entries are keyed by `viewerId`; a shared key would leak one viewer's liked-state to another |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-56-01 | Information Disclosure | `getLikesForTargetCached` 'use cache' entry | mitigate | SEC-05: pass `viewerId` as an explicit function argument so Next.js serializes it into the cache key — viewer A's `viewerHasLiked` cannot be served to viewer B. Auth is resolved in the page (uncached), never inside the cache scope. |
| T-56-02 | Tampering | LikeButton → toggleLikeAction payload | accept | Client sends only `{ type, id }`; the Phase 55 action re-derives actor from `getCurrentUser()` and validates with Zod `.strict()` (mass-assignment-proof). No new attack surface — UI consumes a secured backend. |
| T-56-03 | Spoofing / Repudiation | optimistic count in client state | accept | The optimistic increment is cosmetic; the authoritative count is reconciled from `result.data.count` (server-confirmed) and the DB UNIQUE constraint backstops idempotency (LIKE-05). No trust placed in client-computed counts. |
| T-56-04 | Denial of Service | rapid like/unlike churn | accept | `disabled={pending}` blocks double-fire per tab; Phase 55 dedup (NOTIF-14) and `onConflictDoNothing` handle concurrent churn server-side. No new surface. |
</threat_model>
