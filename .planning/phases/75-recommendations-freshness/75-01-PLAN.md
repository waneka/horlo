<objective>
DISC-RECS-CACHE — Close the cache-invalidation gap on the home "From Collectors Like You" rail so it responds read-your-own-write to viewer collection changes.

Purpose: `CollectorsLikeYou.tsx` declares `'use cache' + cacheLife('minutes')` but does NOT register a cacheTag. Per Next 16 Cache Components, a `'use cache'` segment is invalidated ONLY by `revalidateTag()` matching a tag it itself registered — `revalidatePath('/')` invalidates the route-segment cache but NOT nested `'use cache'` components. The four watch mutations therefore leave the rail stale for up to 1hr (`cacheLife('minutes')` expire) after every collection change. Fix is the per-viewer tag pattern v7.0 Phase 63 established (`viewer:${id}:counts` at src/app/actions/comments.ts:167), with the inverse semantic (default-not-'max') because this requires read-your-own-write, not cross-user fan-out.

Output:
- `cacheTag(`viewer:${viewerId}:recs`)` registered in CollectorsLikeYou.tsx
- 4 × `revalidateTag(`viewer:${user.id}:recs`)` calls (default semantics) wired into watches.ts mutations
- `src/app/actions/__tests__/watches-recs-invalidation.test.ts` regression test (4 cases)
- Short header comment in watches.ts documenting the invariant for future actions
</objective>

Severity: LOW. |
| T-75-01-03 | Tampering | mutation invariant | accept | A future mutation action could be added to watches.ts WITHOUT the new `revalidateTag(`viewer:${user.id}:recs`)` call, silently re-introducing the stale-rec bug.
