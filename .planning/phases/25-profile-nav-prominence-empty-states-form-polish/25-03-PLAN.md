<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server (Header) → Client (UserMenu) | `avatarUrl: string \| null` flows from server-loaded `getProfileById` (DAL) into a client-rendered `<AvatarDisplay>` Image. The DAL is the canonical source — avatarUrl is NEVER user-controlled in-route (no `searchParams`-derived avatar). |
| Client → external image host | `<AvatarDisplay>` renders avatarUrl via Next.js `<Image>`. URL goes through `getSafeImageUrl(avatarUrl)` which validates and constrains to the project's allowed image origins (existing helper, src/lib/images.ts). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-03-01 | I (XSS via avatarUrl) | UserMenu → AvatarDisplay → Next/Image | mitigate | Per orchestrator security note: "XSS via avatarUrl rendering — must use `<img src>` with no innerHTML; URL must come from server-loaded profile only (never user-controlled in-route)". `<AvatarDisplay>` uses Next.js `<Image>` (which only sets `src` attribute — no innerHTML), and the URL flows through `getSafeImageUrl()` (existing helper) for origin allow-listing. avatarUrl is read in `Header.tsx` from `getProfileById(user.id)` — never from `searchParams` or any client-controlled source. No new attack surface introduced; existing controls cover it. |
| T-25-03-02 | T (open redirect via Avatar Link href) | UserMenu Avatar Link `/u/${username}/collection` | mitigate | The href template uses server-loaded `username` only. `username` originates from the same `getProfileById(user.id)` row in Header.tsx — server trust source.
