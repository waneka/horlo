| 12 | `p-12` | 48px | Empty-state Card interior padding (inherits existing `CollectionTabContent.tsx:62` `p-12`) — DO NOT respec |

**Hit target tokens (avatar dual-affordance — D-01 / D-CHOICE-AVATAR-GEOMETRY):**

| Element | Tailwind class | Value | Rationale |
|---------|----------------|-------|-----------|
| Avatar circle hit target | `size-11` (44×44) | 44px | Matches WCAG 2.5.5 minimum touch target. Avatar fills the box; the `<Link>`'s clickable area = the 44px box. |
| Avatar visual diameter | `size-9` (36×36) inside 44px hit area | 36px | Slightly smaller than the hit target so the chrome doesn't feel "heavy" on desktop; the 4px ring of empty hit area is invisible padding. ALT: `size-10` (40×40) is acceptable if planner prefers visual parity with the existing UserMenu initials button (`Button size="sm"` ≈ 32px tall — neither matches; pick `size-9` to hit the "avatar dominates" feel from CONTEXT §specifics). |
| Chevron Button hit target | `size-6` (24×24) | 24px | Per CONTEXT §specifics: "chevron should be small enough that it visually feels like a sub-affordance, not a co-equal button". `size-6` is the smallest `Button size="icon"` variant and keeps the chevron a clear hint, not a co-equal hit. |
| Chevron icon size | `size-3.5` (14×14) inside the 24px button | 14px | Tight visual fit inside the 24px hit; `size-4` (16) would feel cramped. |
| Internal gap between avatar and chevron | `gap-1` (4px) | 4px | Per CONTEXT §specifics, chevron is a "more" hint hugging the avatar — NOT `gap-2` which would visually separate them into two distinct buttons. |
| Outer container shape | `rounded-full` (matches avatar circle) — see §Layout | n/a | The two adjacent hit targets are wrapped in a `flex items-center gap-1` container with NO outer border or background. The avatar's own `rounded-full` provides the visual anchor; the chevron Button has its own `rounded-md` (default `Button` shape). The dual-affordance reads as an avatar + adjacent chevron, NOT as a pill-shaped combo button. |

**Exceptions (do NOT introduce new tokens):**
- The 44px avatar hit target uses `size-11` which IS a Tailwind default token (44px = 11 × 4px). NOT a custom value.
- The 24px chevron Button uses `size-6` which IS a Tailwind default token (24px). The shadcn `Button size="icon-xs"` variant is `size-6 px-1` per `button.tsx` CVA — reuse the variant, do not respec.
- The 14px chevron icon uses `size-3.5` which IS a Tailwind default token (14px). NOT a custom value.

**Mobile slot replacement (D-03):**
- Slim top nav each interactive child uses `inline-flex h-11 w-11 items-center justify-center` (44×44 hit area) — see `SlimTopNav.tsx:42-44` and `:48-51`. The dual-affordance container (avatar + chevron) replaces the existing 44×44 Settings cog slot. Total replaced width = 44 (avatar) + 4 (gap) + 24 (chevron) = 72px, slightly wider than the 44px Settings cog. Right-edge layout still fits at all mobile breakpoints because the cog was alone in its slot; adjacent siblings (Bell at 44px) keep `gap-2` (8px) between siblings.

## Layout

### Avatar Dual-Affordance Container (NEW — D-01 / D-02 / D-04)

**Location:** Replaces `<UserMenu>`'s current `<DropdownMenuTrigger render={<Button variant="ghost" size="sm">{initials}</Button>} />` in `UserMenu.tsx:36-42`. The Phase 25 dual-affordance is a sibling pair (Link + DropdownMenu trigger Button) wrapped in a flex container.

```tsx
// New component: <UserMenuTrigger> (or inlined inside UserMenu)
<div className="flex items-center gap-1">
  {/* LEFT: Avatar circle is a Link to /u/{username} */}
  <Link
    href={`/u/${username}/collection`}
    aria-label={`Go to ${username}'s profile`}
    className="inline-flex size-11 items-center justify-center rounded-full focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring/50"
  >
    <AvatarDisplay
      avatarUrl={avatarUrl}
      displayName={null}
      username={username}
      size={40}              // size-10 visual diameter (NOT a default in AvatarDisplay; planner adds 40 to the size union, OR uses size={40} via a new prop variant)
      className="pointer-events-none"   // hit handled by the wrapping Link's 44px box
    />
  </Link>

  {/* RIGHT: Chevron Button is the DropdownMenu trigger */}
  <DropdownMenu>
    <DropdownMenuTrigger
      render={
        <Button
          variant="ghost"
          size="icon-xs"           // size-6 (24×24)
          aria-label="Open account menu"
        >
          <ChevronDown className="size-3.5" aria-hidden />
        </Button>
      }
    />
    <DropdownMenuContent align="end" className="w-64">
      {/* existing dropdown content unchanged — D-04 */}
    </DropdownMenuContent>
  </DropdownMenu>
</div>
```

**Layout details:**
- The Link uses `inline-flex size-11 items-center justify-center` to make the 44×44 box the click target. The inner `<AvatarDisplay size={40}>` (or `size={36}` if planner prefers — both acceptable; D-CHOICE-AVATAR-GEOMETRY) renders smaller and is centered inside that box.
- **`AvatarDisplay` size extension:** The component currently accepts `size: 40 | 64 | 96`. `size={40}` (`size-10` = 40px) is already supported. The planner uses `size={40}` directly — no component change needed. The 4px difference between the 40px avatar visual and the 44px Link hit area is invisible (matches the hit-target padding pattern Apple uses).
- **Alternative (smaller avatar):** If the planner finds 40px too dominant against a 24px chevron, swap to `size={40}` and shrink visually via `className="scale-90"` (no — that's a hack); cleaner: keep `size={40}`. CONTEXT §specifics says "about 24px wide vs ~44px avatar" — `size={40}` is close enough that the 5:3 ratio feels right.
- **Gap of `gap-1` (4px):** Visually couples the avatar and chevron as one affordance pair. NOT `gap-2` (would feel too separate) and NOT `gap-0` (chevron would touch avatar — looks broken on the rounded edge).
- **No outer border or background on the wrapper.** The two children retain their independent shapes (avatar = circle, chevron Button = rounded-md from Button primitive). This is per CONTEXT §specifics: "two adjacent hit targets in one rounded container" interpreted as "two visually-grouped hit targets" — NOT a single pill-shaped combo. Adding an outer `bg-muted rounded-full` background would make it look like one button.
- **Focus rings (NEW for Phase 25):**
  - Avatar Link: `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring/50` directly on the Link (since AvatarDisplay's inner circle doesn't paint its own ring). Ring sits on the Link's 44px box.
  - Chevron Button: inherits Button primitive's `focus-visible:border-ring focus-visible:ring-[3px] focus-visible:ring-ring/50` (already in `button.tsx`).
  - Two separate focus rings — keyboard users tab through Link first, then chevron Button. This is correct: each is an independent affordance.

### `useFormFeedback()` Hook Contract (NEW — D-17 / D-19)

**Location:** `src/lib/hooks/useFormFeedback.ts`.

**Hook signature:**
```ts
interface UseFormFeedbackOptions {
  /** When true (dialog forms — D-19), suppress the inline banner; toast only. */
  dialogMode?: boolean
}

interface UseFormFeedbackReturn<T> {
  pending: boolean
  state: 'idle' | 'pending' | 'success' | 'error'
  message: string | null
  run: (
    action: () => Promise<ActionResult<T>>,
    opts?: { successMessage?: string; errorMessage?: string },
  ) => Promise<void>
  reset: () => void
}

export function useFormFeedback<T = unknown>(
  options?: UseFormFeedbackOptions,
): UseFormFeedbackReturn<T>
```

**Hybrid behavior (D-16 + D-19):**

| Mode | Toast | Banner |
|------|-------|--------|
| `dialogMode: false` (default — inline-page forms) | `toast.success(message)` 3s auto-dismiss | inline `<FormStatusBanner state="success" />` 5s auto-dismiss |
| `dialogMode: true` (ProfileEditForm, PasswordReauthDialog, EmailChangeForm) | `toast.success(message)` 3s | banner suppressed (dialog dismounts) |
| Error (any mode) | `toast.error(message)` 3s | inline `<FormStatusBanner state="error" />` persistent until next `run()` (errors don't auto-clear — user needs to read them) |

The hybrid hook should NOT be applied to optimistic-update components on the success path — the optimistic flip IS the success signal. The hook fires only on revert/error for these components (D-18 calls this out explicitly).

## Anti-Patterns Flagged

These are explicitly forbidden by this contract. The checker should fail any deviation.

1. **DO NOT make the avatar+chevron a single `<Button>` with a click handler that branches on click target.** The two affordances MUST be two separate elements (Link + Button) with independent focus states and ARIA semantics. Single-button-with-branching breaks keyboard navigation and screen-reader expectations.
2. **DO NOT use `gap-2` between avatar and chevron.** `gap-1` (4px) keeps them visually coupled. `gap-2` (8px) would make them feel like two separate buttons.
3. **DO NOT add an outer `bg-muted rounded-full` background to the avatar+chevron container.** They share visual proximity, not a single button shell. CONTEXT §specifics interprets "rounded container" as "rounded children grouped together", not "one pill-shaped combo".
4. **DO NOT style the disabled "Add by URL" button with `variant="outline"` or `variant="ghost"`.** Per CONTEXT §specifics: "neither is outline — both should be primary-weighted". Use `variant="default"` + `disabled` + `opacity-60`.
5. **DO NOT use destructive coloring on the ExtractErrorCard icon.** Use `text-muted-foreground`. The destructive left-border alone signals error; doubling up over-signals.
6. **DO NOT respec spacing tokens.** Empty-state Card uses existing `p-12` from `CollectionTabContent.tsx:62`. FormStatusBanner uses Phase 22's `p-3 + border-l-2`. ExtractErrorCard uses `p-4 + gap-3`. All inherited.
7. **DO NOT change the Sonner toast copy for `Mark all read`.** It MUST be `Notifications cleared` per CONTEXT §specifics — NOT "Marked all read".
8. **DO NOT auto-dismiss the FormStatusBanner error state.** Errors persist until next `run()` (D-16 says success "5s auto-dismiss" — error must NOT auto-clear; users need to read errors).
9. **DO NOT introduce new icons beyond the locked D-14 set.** Phase 25's icons are: `ChevronDown` (avatar dual-affordance), `Lock`, `FileQuestion`, `Clock`, `Gauge`, `WifiOff` (ExtractErrorCard). Empty-state Cards use NO new icons (text-only Card). The `<AddWatchCard>` `Plus` icon is existing.
10. **DO NOT change `<PrivacyToggleRow>` to use the hybrid hook on success path.** Optimistic-update pattern is the visual feedback; suppressing it would be redundant. Hook fires only on error revert.
11. **DO NOT show empty-state CTAs to non-owners.** D-10: every empty-state Card MUST gate the CTA wrapper on `isOwner`. Non-owners see read-only owner-aware copy.
12. **DO NOT introduce new accent surfaces beyond the FormStatusBanner left-border.** Phase 25's accent budget is one new generalization of the existing email-change banner pattern.
13. **DO NOT introduce new destructive surfaces beyond the ExtractErrorCard + FormStatusBanner-error left-borders.** Both reuse the same `border-l-2 border-l-destructive` technique. The destructive token is otherwise reserved for inline form error copy and the existing Sign-out button text.
14. **DO NOT skip wrapping the disabled tooltip trigger with a `<span>`.** Disabled buttons don't fire pointer events on Safari; the `<span>` wrapper is required for the tooltip to trigger on hover.
15. **DO NOT introduce custom Tailwind tokens.** Use only existing CSS-variable-aliased utilities. The Tailwind 4 config is locked.
16. **DO NOT default the FormStatusBanner success copy to "Successfully saved" or "Updated successfully".** Default is `Saved` per CONTEXT §specifics ("default to 'Saved' (no fluff)"). Per-form overrides like "Profile updated" are explicit, not generic.
