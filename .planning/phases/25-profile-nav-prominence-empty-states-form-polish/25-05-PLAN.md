## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client URL → page Server Component | `searchParams.manual` and `searchParams.status` are user-controlled. Page Server Component whitelists with literal-match (`=== '1'` and `=== 'wishlist'`); only the resolved boolean / typed-literal flows to the Client. |
| Server Component → Client tab components | `hasUrlExtract`, `collectionCount`, `username`, `viewerId` are all server-derived; no user input passes through. |
| Client (NotesEmptyOwnerActions) → router.push | `router.push('/watch/${watchId}/edit#notes')` uses watchId from `<WatchPickerDialog>` `onWatchSelected` callback. The dialog's watch list is server-fetched (filtered to status='owned' belonging to the viewer); watchId originates from a server-trusted set. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-05-01 | T (open redirect via ?manual / ?status) | /watch/new searchParams whitelisting | mitigate | Per orchestrator security note: "?manual=1 and ?status=wishlist query params: open redirect — params are consumed locally by AddWatchFlow; verify no URL-construction sinks". Page applies LITERAL-MATCH whitelisting (`sp.manual === '1'` and `sp.status === 'wishlist'`) before passing to AddWatchFlow. The resulting boolean / typed-literal is local FlowState only — no URL is constructed from these params. AddWatchFlow does NOT call `router.push` with these values. Defense-in-depth: server whitelist + client typing both reject anything else. |
