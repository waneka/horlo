    Create `src/lib/hooks/useFormFeedback.ts` per UI-SPEC §"useFormFeedback Hook Contract" (D-17). Implementation contract (locked signature):

    ```ts
    'use client'
    import { useCallback, useEffect, useRef, useState, useTransition } from 'react'
    import { toast } from 'sonner'
    import type { ActionResult } from '@/lib/actionTypes'

    export interface UseFormFeedbackOptions {
      /** When true (dialog forms — D-19), suppress the inline banner; toast only. Hook still tracks state; consumer chooses not to render FormStatusBanner. */
      dialogMode?: boolean
    }

    export interface UseFormFeedbackReturn<T> {
      pending: boolean
      state: 'idle' | 'pending' | 'success' | 'error'
      message: string | null
      dialogMode: boolean
      run: (
        action: () => Promise<ActionResult<T>>,
        opts?: { successMessage?: string; errorMessage?: string },
      ) => Promise<void>
      reset: () => void
    }

    export function useFormFeedback<T = unknown>(
      options?: UseFormFeedbackOptions,
    ): UseFormFeedbackReturn<T>
    ```

    Implementation specifics (LOCKED per UI-SPEC §Hybrid Behavior + D-16/D-19):

    1. Use React's `useTransition()` internally. Hook OWNS the transition; consumer forms MUST NOT wrap submit in their own `useTransition` (FG-8 in UI-SPEC).
    2. Internal state: `state: 'idle' | 'pending' | 'success' | 'error'`, `message: string | null`. Use a single `useState` object OR two `useState` calls — both fine.
    3. Use a `useRef<NodeJS.Timeout | null>(null)` to track the auto-dismiss timeout for cleanup.
    4. `run(action, opts)` flow:
       a. Call `reset()` first (clears any prior timeout + sets state to 'pending'). This implements D-16 "Both clear on next form interaction".
       b. Set state to 'pending' (via `startTransition` so React batches with the action's awaited setState calls).
       c. `await action()` inside `startTransition`.
       d. If `result.success === true`: set state to 'success', `message = opts?.successMessage ?? 'Saved'`, fire `toast.success(message)` (3s — Sonner default). Schedule `setTimeout(() => setState('idle'), 5000)` and store the handle in the ref.
       e. If `result.success === false`: set state to 'error', `message = opts?.errorMessage ?? result.error`, fire `toast.error(message)`. DO NOT schedule auto-clear — error persists until next `run()` (D-16; UI-SPEC Anti-Pattern #8).
    5. `reset()` clears any pending timeout via the ref AND sets state to 'idle', message to null.
    6. `useEffect` cleanup: on unmount, clear the timeout ref to prevent setState-on-unmount.
    7. Return `pending` from `useTransition` (NOT a derived `state === 'pending'` — they can briefly differ; `useTransition`'s `isPending` is the canonical source).
    8. Return `dialogMode: options?.dialogMode ?? false` so consumers can branch on it without re-passing the option.

    Sonner toast notes: `toast.success(message)` and `toast.error(message)` — Sonner's API. The toaster is already mounted in root layout (`src/components/ui/ThemedToaster.tsx`).

    Optimistic-update carve-out (D-18; UI-SPEC §Hybrid Behavior table footnote): the hook does NOT special-case PrivacyToggleRow. It is the CALLER's job (in 25-06) to opt-out of the success path for optimistic components. Document this in a hook-file JSDoc block.

    NO timer logic for the FormStatusBanner — the banner is purely presentational; auto-dismiss is the hook's responsibility (achieved by clearing state to 'idle', which makes the banner return null).

    Anti-patterns to avoid (D-16; UI-SPEC §Anti-Pattern #8):
    - DO NOT auto-clear the error state. Errors persist until the next `run()` call.
    - DO NOT call `startTransition` in addition to the hook's internal `useTransition` from consumer forms.
    - DO NOT use `useFormStatus` here — that's for inline `<form action>` (D-20 / 25-06 MarkAllReadButton); this hook composes around `useTransition` for the manual-submit pattern that 8+ Phase 25 forms use.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server Action client → server | `run()` invokes user-supplied `action: () => Promise<ActionResult<T>>`. The action is a Server Action (RPC across the boundary). The hook does not validate inputs — that is the action's responsibility. |
| Hook → Sonner | `toast.success/error(message)` renders user-supplied `message`. Sonner escapes text content (no HTML injection); `message` originates from server (`result.error`) or developer (`opts.successMessage`). |
| Hook → React lifecycle | `setTimeout(5000)` requires cleanup-on-unmount to avoid setState-on-unmounted-component warnings. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01-01 | I (Information disclosure) | useFormFeedback `result.error` → `toast.error(message)` + banner | mitigate | Server Actions follow ActionResult contract — `error` is a user-facing string set explicitly by the action author. Hook does NOT log raw errors to console or surface stack traces. The hook is a passive forwarder — server-side errors that include sensitive info are an action-author problem (out-of-scope; see 25-06 audit). |
| T-25-01-02 | T (Tampering) | client-controlled `successMessage` / `errorMessage` props | accept | These are developer-supplied at compile time, not user-supplied. No XSS risk because Sonner escapes text. No tampering surface. |
| T-25-01-03 | D (Denial of service) | rapid-fire `run()` calls leak `setTimeout` handles | mitigate | `run()` calls `reset()` first, which `clearTimeout(ref.current)`. Cleanup useEffect on unmount also clears any pending timeout. No leak. |
| T-25-01-04 | E (Elevation of privilege) | client-side hook executes user actions | accept | Hook does not touch auth tokens, cookies, or any privileged surface. Server Actions enforce their own auth via getCurrentUser; hook is presentation-only. |
| T-25-01-05 | I (stale closure) | useFormFeedback success/pending state cleared on next form interaction; no orphaned timers on unmount (per orchestrator security note) | mitigate | Test 15 (unmount mid-pending) verifies no setState-on-unmounted warning. `useEffect` cleanup function clears `timeoutRef.current` if non-null. Confirmed by RTL `unmount()` test + `console.error` spy. |
</threat_model>
