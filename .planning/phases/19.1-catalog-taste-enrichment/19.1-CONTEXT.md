# Phase 19.1: Catalog Taste Enrichment - Context

**Gathered:** 2026-04-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Add LLM-derived structured taste attributes to `watches_catalog`, computed at catalog-row time and cached. The enrichment runs as a **second-pass call** after spec extraction (URL path) or as the only LLM call (manual-entry path). Vision is supported on manual entry via an optional photo upload. `WatchForm` hides the existing `styleTags` / `roleTags` / `designTraits` pickers because their values now come from the catalog. **No verdict copy, no badge UI, no `analyzeSimilarity()` changes** in this phase — those all land in Phase 20 (verdict copy) and v5.0 (engine rewire).

The phase delivers taste *substrate* for Phase 20 to consume. By itself it is intentionally invisible to the end user except for: (a) tag pickers disappearing from the form, (b) an optional photo field appearing on manual entry.

**Out of scope for this phase:**
- Verdict copy / contextual phrasings (Phase 20)
- `<CollectionFitCard>` / badge UI / read-only chip surfaces (Phase 20)
- `analyzeSimilarity()` reading taste fields (v5.0 engine rewire)
- Owner override path for incorrect enrichments (deferred to v4.x)
- Tag taxonomy audit / `styleTags` / `roleTags` / `designTraits` cleanup (v5.0 prerequisite)
- Pending UI ("Analyzing taste…") on WatchDetail (Claude's discretion at plan time)
- TOS / acceptable-use policy covering user-uploaded photos that become canonical catalog images (deferred — flagged by user)

<decisions>
## Implementation Decisions

### Schema Delta (vs locked architecture in memory)

- **D-01: `crowd_signals` is DROPPED from the schema entirely.** User decision during discussion: "the juice is not worth the squeeze." The locked architecture in `memory/project_taste_enrichment_arch_2026_04_29.md` listed this field; this phase supersedes that memory. Final schema fields on `watches_catalog`:
  - `formality` numeric(3,2) — 0..1
  - `sportiness` numeric(3,2) — 0..1
  - `heritage_score` numeric(3,2) — 0..1
  - `primary_archetype` text — closed vocab (`dress | dive | field | pilot | chrono | gmt | racing | sport | tool | hybrid`)
  - `era_signal` text — closed vocab (`vintage-leaning | modern | contemporary`)
  - `design_motifs` text[] — closed vocab, 0..N values, vocab pinned at plan-time
  - `confidence` numeric(3,2) — self-rated by LLM
  - `extracted_from_photo` boolean DEFAULT false — true when a vision call produced this row's taste

- **D-02: Categorical fields use CHECK constraints + TS unions, not pgEnum.** Mirrors Phase 17 D-04 (avoids Phase 24-style rename+recreate when vocab evolves). `primary_archetype`, `era_signal` ship with `CHECK (col IN (…))`. `design_motifs` is `text[]` with array-element CHECK if the planner deems it cheap, else TS-only validation (see D-06).

- **D-03: Existing Phase 17 D-06 image columns are reused.** `image_url`, `image_source_url`, `image_source_quality` already exist. Phase 19.1 adds `'user_uploaded'` as an allowed value to whichever guard Phase 17 shipped (CHECK constraint or free-text). The manual-entry photo path writes through these existing columns; no new image columns.

### Vocabularies

- **D-04: `design_motifs` vocabulary is deferred to plan-time research.** The phase researcher proposes a 25-30 motif draft list (visual/aesthetic descriptors that don't overlap `primary_archetype` or `complications` — examples: `sandwich-dial`, `california-dial`, `guilloché`, `enamel-dial`, `meteorite-dial`, `skeletonized`, `integrated-bracelet`, `exhibition-caseback`, `cushion-case`, `tonneau`, `bauhaus`, `domed-crystal`, `patina-friendly`, `gilt-dial`, `fume-dial`, etc.). User reviews and locks the list during the plan-checker round before code lands.

- **D-05: Vocabulary violations are dropped + warned, not coerced.** When the LLM emits a `design_motifs` value (or `primary_archetype` / `era_signal` value) outside the closed vocab, the validator filters it out AND emits a `console.warn` server-side (or, planner discretion: insert into a small `taste_vocab_drift_log` table) so vocab drift is auditable. Mirrors `validateAndCleanData` shape in `src/lib/extractors/llm.ts` but adds the warning side-effect.

- **D-06: Vocabularies live in TS constants, not the database.** New module `src/lib/taste/vocab.ts` exports readonly arrays + Zod enums. Column type stays `text[]` for `design_motifs`; categoricals stay `text` with optional CHECK constraint. Adding a value = code deploy, no migration. Mirrors how `STYLE_TAGS` / `DESIGN_TRAITS` already live in `src/lib/constants.ts`.

### Enrichment Call Mechanics

- **D-07: URL extract path uses two-pass sequential calls.** Existing `extractWithLlm()` (`src/lib/extractors/llm.ts`) is unchanged — spec extraction prompt and validation stay byte-identical to avoid regressing Phase 17 / Phase 19 contracts. After the row commits, a separate `enrichTasteAttributes()` call fires using the catalog row's spec fields + `image_url` (if set) as input. Cost: ~2x the existing LLM cost per URL paste, but keeps prompts focused and tunable independently. First-write-wins (D-13) avoids re-running on the same `catalog_id`.

- **D-08: Manual-entry path runs the taste call with optional vision.** When the WatchForm submits without a photo, `enrichTasteAttributes()` runs text-only with the typed brand/model/reference/specs as input. When the user uploaded a photo, the same function runs in vision mode (multi-modal Sonnet call). Both run after the catalog upsert commits (mirrors D-07).

- **D-09: Both paths are fire-and-forget after row insert.** Server Action / route handler does NOT await the taste promise. Mirrors the `logActivity` / `logNotification` / Phase 17 CAT-08 catalog-upsert pattern: row commits successfully even if the taste call fails, and the user navigates immediately. Taste fields = NULL on first render; appear after page refresh ~3-5s later (text) or ~5-8s later (vision).

- **D-10: Failure handling is log + null + manual re-enrichment, no auto-retry.** Timeouts, 429s, 5xx, JSON parse errors, Zod validation failures all fall through to the same handler: row keeps taste fields = NULL, structured event logged. Operator runs `npm run db:reenrich-taste` to retry failed rows. **Trade-off accepted:** transient blips leave NULL gaps until manual intervention. Acceptable at v4.0 personal-MVP scale; revisit if it bites at multi-user scale.

- **D-11: Model is the latest Sonnet (`claude-sonnet-4-6`) with tool-use structured output.** User said "Sonnet 4.5" picking from the offered options, but per the session knowledge update the latest Sonnet ID is `claude-sonnet-4-6` — this CONTEXT locks the latest. Structured output uses Anthropic tool-use (`tools` array with `input_schema` mirroring Zod) rather than the regex-JSON-extraction pattern in `extractWithLlm()`. Tool-use is the canonical Anthropic mechanism for structured output and avoids the brittle `\{[\s\S]*\}` parse step. Vision is enabled when a photo is supplied (manual-entry-with-photo path).

- **D-12: Service module lives at `src/lib/taste/`.** Sibling to `src/lib/extractors/`. Files: `vocab.ts` (closed vocabularies + Zod schemas), `prompt.ts` (text-only and vision prompt builders), `enricher.ts` (the single `enrichTasteAttributes()` entry point with `text` and `vision` modes), `types.ts` (`CatalogTasteAttributes` + `EnrichmentInput` types). Planner discretion to consolidate or split further.

### Confidence + Re-enrichment

- **D-13: Re-enrichment is force-only via script — never auto in the live flow.** Once a catalog row has any taste data (any confidence value), the live enrichment paths skip it. Phase 17 D-13 first-non-null-wins extends to taste. Updates exclusively flow through `npm run db:reenrich-taste` with explicit flags:
  - `--force` — override the skip and overwrite existing taste
  - `--confidence-below=0.5` — only re-run rows whose stored confidence is below the threshold
  - `--catalog-id=<uuid>` — re-run a specific row (debugging / one-off corrections)
  - `--dry-run` — report what would re-enrich and estimated cost without API calls

- **D-14: Two-tier confidence gating drives Phase 20 verdict copy.** Stored as `numeric(3,2)`; downstream copy logic:
  - `confidence >= 0.7` → full contextual copy fires (Phase 20 FIT-02 phrasings)
  - `0.5 <= confidence < 0.7` → hedged copy ("likely a heritage piece", "appears to lean dressy") — Phase 20 implements
  - `confidence < 0.5` → fall back to the existing 6 fixed `SimilarityLabel` values
  - **Phase 20 owns the actual copy generation; this phase only commits to storing the right confidence value and exposing it to callers.** Phase 19.1 does NOT add gate enforcement code.

### Backfill

- **D-15: Backfill is a standalone TS script with dry-run.** `scripts/backfill-taste.ts` mirroring Phase 17 D-14 `scripts/backfill-catalog.ts`. CLI flags:
  - `--batch-size=20` (default; small enough to stay under Anthropic rate limits and let the script resume cleanly)
  - `--dry-run` — count rows without taste, estimate total cost (`<rowCount> * $0.005` text-only, `+ <photoRowCount> * $0.013` vision), exit without API calls
  - `--resume` — short-circuit `WHERE taste_confidence IS NULL` so re-runs skip already-tasted rows (idempotent)
  - Final assertion: `SELECT COUNT(*) FROM watches_catalog WHERE confidence IS NULL` after the run, log the residual

- **D-16: Backfill is a documented post-deploy step in `docs/deploy-db-setup.md`, NOT auto-run.** Same posture as `db:backfill-catalog` — manual gate so the operator confirms costs and timing. Cron is NOT used; one-time bulk operation followed by ongoing per-write enrichment in the live flow.

- **D-17: ANTHROPIC_API_KEY missing → skip + structured event.** Same shape as `logActivity`: enricher detects missing key, emits a structured log event (event name like `taste_enrichment_skipped:no_api_key` with `{catalog_id, source}`), returns a no-op promise. Local dev without the key works seamlessly; CI tests pass without the key; production misconfig surfaces in observability instead of silently degrading.

### WatchForm UX Surgery

- **D-18: WatchForm hides the `styleTags` / `roleTags` / `designTraits` picker UIs entirely.** No replacement input. The form fields stay in the type system (per-user `watches.styleTags` etc. remain in the DB and `analyzeSimilarity()` still reads them — Phase 17 D-12 silent infrastructure honored). Existing rows keep their values; new rows get `[]`. This is a UI-only hide.

- **D-19: WatchForm gains an optional photo upload field on the manual-entry path.** Position and copy at planner discretion; encouragement copy should communicate "strongly encouraged but not required" (memory: vintage pieces especially benefit from photos). Non-blocking — user can submit without a photo. The URL-import path does NOT add a photo field (the URL extract already populates `image_url`).

- **D-20: Photos write to a NEW dedicated `catalog-source-photos` Supabase Storage bucket.** Not the existing `wear-photos` bucket. Same RLS pattern (three-tier; folder-enforcement INSERT/UPDATE/DELETE), same HEIC→JPEG worker, same EXIF strip, same canvas reencoded JPEG ≤1080px. Folder convention `{userId}/{catalogId-or-pending}/{filename}.jpg`. Provisioning migration mirrors `supabase/migrations/.../phase11_storage_rls_secdef_fix.sql` and the Phase 15 wear-photos bucket migration.

- **D-21: The uploaded photo becomes the catalog's canonical `image_url`.** Phase 17 D-06 columns are reused: `image_url` = signed Supabase URL, `image_source_url` = bucket path, `image_source_quality` = `'user_uploaded'` (NEW value to add to allowed set). Cross-user implication: if Owner B types the same brand/model/reference Owner A uploaded a photo for, Owner B sees Owner A's photo on the catalog row. This is consistent with how URL-extracted images already work (CAT-07 COALESCE first-non-null-wins). **User flagged: a TOS / acceptable-use policy covering this is needed before scaling beyond personal-MVP** — captured in Deferred Ideas, not blocking v4.0.

- **D-22: `extracted_from_photo` boolean is set true on rows enriched from a photo, false otherwise.** Lives on the catalog row (D-01). Lets future verdict copy say things like "based on the photo you provided" vs generic phrasings. Phase 19.1 wires the field; Phase 20 may consume it.

- **"Fire-and-forget with some sort of messaging on the watch detail page that some fields are pending? idk maybe that's not worth it."** User leaning toward fire-and-forget without a pending UI. Captured as Claude's Discretion — the planner can ship a small dimmed "Analyzing taste…" indicator on WatchDetail if it's cheap, or omit entirely.

- **Latest Sonnet model (`claude-sonnet-4-6`) preferred over current `claude-sonnet-4-20250514` in `extractWithLlm()`.** User said "Sonnet 4.5" picking from the offered options; per the session knowledge update (2026-02-27) the latest Sonnet ID is `claude-sonnet-4-6`. The new taste enricher uses the latest model; the existing `extractWithLlm()` is NOT migrated (out of scope; risk of regressing Phase 17 / Phase 19 spec extraction). Plan-time: confirm the model ID is current at execution time.
