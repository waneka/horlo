## Summary

Phase 19.1 adds 8 taste columns to `watches_catalog`, a `catalog-source-photos` Storage bucket with RLS, a server-only taste enricher (`enrichTasteAttributes` via Anthropic tool-use), DAL helpers (`updateCatalogTaste`, `applyUserUploadedPhoto`), fire-and-forget wiring in `addWatch` and `/api/extract-watch`, WatchForm photo upload, and operator backfill/reenrich scripts.

The implementation is structurally sound. Key invariants (D-07 extract lock, D-09 fire-and-forget, D-13 first-write-wins, T-19.1-05-05 ownership check) are correctly implemented. RLS folder-enforcement follows the Phase 11 pattern. All SQL uses parameterized `sql` template tags — no raw string concatenation.

Three warnings were found: a safety-guard false-positive in `reenrich-taste.ts` (the `--force` full-catalog guard mischecks `confidenceBelow` with a falsy test), a stored signed URL that will become a dead link after 7 days, and a lax UUID regex in the path builder that accepts syntactically invalid UUIDs. Four info items are noted: zod v4 installed vs the SUMMARY stating v3, the `server-only` tsconfig shim bypassing tsc enforcement, a missing `tsx` devDependency in package.json, and a note on the prompt's complications field not being injection-guarded.

## Warnings

### WR-01: `reenrich-taste.ts` full-catalog safety guard uses falsy check on `confidenceBelow`, misfires when value is `0`

**File:** `scripts/reenrich-taste.ts:69`

**Issue:** The guard that prevents force re-enriching the entire catalog uses `!args.confidenceBelow` as its predicate:

```typescript
if (!args.confidenceBelow && !args.catalogId && !args.dryRun) {
  // refuses to run
}
```

`confidenceBelow` is typed as `number | null`. When the operator passes `--confidence-below=0`, `parseFloat('0')` returns `0`, which is falsy. `!0 === true`, so the entire guard expression becomes `true` and the script incorrectly exits with "would re-enrich the ENTIRE catalog. Refusing." even though a valid predicate (`confidence < 0`) was provided. This is a false-positive refusal rather than a false-negative pass — so it is not a safety hole — but it is a correctness bug that will confuse operators.

**Fix:**

```typescript
if (args.confidenceBelow === null && !args.catalogId && !args.dryRun) {
  console.error(`[reenrich-taste] --force without --confidence-below or --catalog-id ...`)
  process.exit(1)
}
```

### WR-02: 7-day signed URL stored as permanent `image_url` in catalog

**File:** `src/app/actions/watches.ts:126`

**Issue:** `addWatch` calls `getCatalogSourcePhotoSignedUrl(photoPath, 60 * 60 * 24 * 7)` (604 800-second TTL) and stores the resulting URL as `image_url` in `watches_catalog` via `applyUserUploadedPhoto`. Supabase signed URLs expire. After 7 days any catalog row whose `image_url` came from this path will have a dead image URL. Because `applyUserUploadedPhoto` uses `COALESCE` first-non-null semantics (D-21), the expired URL cannot be overwritten by the live enrichment path without an explicit operator re-enrichment.

This issue is documented as "deferred polish" in the Phase 19.1 SUMMARY. The risk is real: users who add a watch with a reference photo will see the catalog image work for 7 days and then silently break. Raising as a warning rather than info because the damage is permanent (stuck bad `image_url`) unless an operator runs `db:reenrich-taste --force`.

**Fix options:**

Option A (recommended for v4.0): store the **bucket path** as `image_url` instead of the signed URL, and generate signed URLs on demand at render time via the enricher or a Server Component helper. Requires a render-time signing step.

Option B (simpler): extend the TTL to match the product's data retention horizon (e.g., 10 years = `315_360_000` seconds). Supabase supports up to 1 year TTLs; for longer lifetimes use the public-bucket pattern.

Option C (minimal): add a periodic cron job (pg_cron or a scheduled script) that refreshes expired `user_uploaded` image URLs by re-signing them from the bucket path stored in `image_source_url`.

Regardless of option chosen, document the current expiry behavior prominently in the runbook so operators know to use `db:reenrich-taste --force --catalog-id=<id>` when a user reports a broken image.

### WR-03: Path builder UUID regex accepts syntactically invalid strings (all-hyphen, no version/variant bits)

**File:** `src/lib/storage/catalogSourcePhotos.ts:22`

**Issue:** The UUID validation regex is:

```typescript
const UUID_RE = /^[0-9a-f-]{36}$/i
```

This accepts strings that are 36 characters from the set `[0-9a-f-]` regardless of UUID structure. Examples that pass: `------------------------------------` (all hyphens), `aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa` (no hyphens), `aaaaaa-aaaaaa-aaaaaa-aaaaaa-aaaaaaaaaa` (wrong hyphen positions). These strings are valid per the regex but are not valid UUIDs.

The practical risk is low because: (1) `uploadCatalogSourcePhoto` is called from `WatchForm.handleSubmit` which always passes `user.id` (a real UUID from Supabase Auth) and `'pending'`; (2) the server-side Zod regex in `insertWatchSchema` accepts the same character set but also validates the first segment against `user.id` (ownership check). However, a future caller of `buildCatalogSourcePhotoPath` passing an arbitrary string in `middle` could produce structurally invalid paths that still pass validation.

**Fix:** Tighten to RFC 4122 UUID structure:

```typescript
const UUID_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i
```

The same tightening should be applied to the Zod regex in `insertWatchSchema` for the `userId` segment, currently `[0-9a-f-]{36}`.

### IN-02: `server-only` resolved to no-op shim in `tsconfig.json` `paths`

**File:** `tsconfig.json:24`

**Issue:** The tsconfig `paths` entry `"server-only": ["./tests/shims/server-only.ts"]` causes TypeScript (tsc) to resolve the `server-only` guard import to a no-op export. This silences `tsc --noEmit` for files that import `server-only` but are otherwise callable from a client context. The real enforcement is Next.js's webpack/turbopack bundler analysis (not tsc), so production builds are protected. However, if any future server-only module is accidentally imported into a client path, the error will surface only at Next.js build time rather than at type-check time.

This was an intentional decision per the Plan 06 SUMMARY (needed to allow `tsx` to execute enricher.ts in scripts), and it is the same tradeoff accepted by the vitest shim. The risk is understood and bounded to the backfill scripts.

**Fix:** No code change required. Document the limitation explicitly in `tsconfig.json` comments so future contributors do not assume `tsc --noEmit` provides `server-only` enforcement.

### IN-03: `tsx` is used in `db:backfill-taste` / `db:reenrich-taste` npm scripts but is not listed in `devDependencies`

**File:** `package.json:13-15`

**Issue:** The scripts `db:backfill-taste` and `db:reenrich-taste` invoke `tsx` directly:

```json
"db:backfill-taste": "tsx --env-file=.env.local scripts/backfill-taste.ts"
```

`tsx` is not listed in `package.json` `devDependencies`. It appears to be available globally in the development environment (or implicitly via another package), but a fresh `npm install` on a new machine will not install it, causing the scripts to fail with `tsx: command not found`.

**Fix:** Add `tsx` to `devDependencies`:

```json
"tsx": "^4"
```

### IN-04: `complications` field in prompt builder skips `safeField` — injected raw via `join`

**File:** `src/lib/taste/prompt.ts:53`

**Issue:** The `buildSpecBlock` function applies `safeField` (which calls `JSON.stringify` for injection safety) to string fields like brand, model, crystalType, dialColor. However, complications are joined directly without `safeField`:

```typescript
spec.complications.length ? `Complications: ${spec.complications.join(', ')}` : null,
```

If a complication string contained LLM prompt-injection content (e.g., `"Ignore all previous instructions..."`) it would be inserted literally into the prompt text.

In practice, the risk is low: complications are selected from a closed list (COMPLICATIONS constant) via the form UI and stored in a text array — they are not free-text from external sources. The LLM tool-use `strict: true` and Zod validation on the output side also contain any damage. Still, the inconsistency is worth resolving for defense-in-depth.

**Fix:** Apply `safeField` to each complication:

```typescript
spec.complications.length
  ? `Complications: ${spec.complications.map((c) => safeField(c)).join(', ')}`
  : null,
```
