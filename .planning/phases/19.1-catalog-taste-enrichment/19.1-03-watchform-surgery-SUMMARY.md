## CatalogPhotoUploader.tsx

- 258 lines
- Remove-callback contract: **Option A** — separate `onClear?: () => void` prop. `onPhotoReady` always receives a valid `Blob`; never `null`. Parent's `setPhotoBlob(null)` lives in `onClear`. Rationale: single-responsibility — `onPhotoReady` is the "success" signal, `onClear` is the "cancel" signal. No type widening needed.

## Photo Blob Handoff Path

```
<CatalogPhotoUploader
  onPhotoReady={(blob) => { setPhotoBlob(blob); setPhotoError(null) }}
  onClear={() => setPhotoBlob(null)}
  onError={(message) => setPhotoError(message)}
  disabled={isPending}
/>
       ↓
  photoBlob state in WatchForm (Blob | null)
       ↓
  handleSubmit (Plan 05 wires upload to catalog-source-photos bucket)
```

## Threat Surface Scan

No new network endpoints, auth paths, file-access patterns, or schema changes introduced in this plan. The file-selection → canvas re-encode boundary (T-19.1-03-01 through T-19.1-03-03) was pre-registered in the plan's threat model and all mitigations implemented:
- T-19.1-03-01: Every file path routes through `stripAndResize` before `onPhotoReady` — EXIF dropped unconditionally
- T-19.1-03-02: `accept="image/*,.heic,.heif"` on file input + canvas re-encode rejects non-decodable bytes
- T-19.1-03-03: `MAX_FILE_BYTES = 8MB` client-side gate before any processing
