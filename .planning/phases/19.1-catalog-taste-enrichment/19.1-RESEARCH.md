## Design Motifs Vocabulary Draft

**Goal:** A 25-30 motif vocabulary of visual/aesthetic descriptors that does NOT overlap with `primary_archetype` (functional category: dress/dive/etc.) or `complications` (date/chrono/gmt/moon-phase/etc.). The vocab must give the LLM enough surface area to flag visually distinctive watches (vintage Submariners, gilt-dial Speedmasters, integrated-bracelet Royal Oaks) without becoming a tag-soup that defeats the structuring exercise.

Existing `STYLE_TAGS` (8 values): `diver`, `dress`, `field`, `pilot`, `chronograph`, `gmt`, `sport`, `tool`. **All overlap with `primary_archetype`** — design_motifs must NOT include these.

### Rationale

- **No overlap with `primary_archetype`:** none of the 28 are functional-category words.
- **No overlap with `era_signal`:** none of the 28 are time-period words; `patina-friendly` and `bauhaus` are *visual aesthetics* that *correlate* with vintage-leaning but don't *define* era.
- **No overlap with `complications`:** none of the 28 are functions (date, chrono, gmt, moon, power-reserve, world-time).
- **Concrete, recognizable, finite:** every motif points to a feature a knowledgeable LLM can identify from text + spec data alone (vision boosts confidence but isn't required).
- **Roughly equal granularity:** `sandwich-dial` and `breguet-hands` are about the same level of specificity; the vocab doesn't mix "vintage" (era-level) with "applied indices" (feature-level).
- **Vintage/heritage signals carried by aesthetic motifs**, not era words — `gilt-dial`, `domed-crystal`, `patina-friendly`, `tropic-strap` all read "vintage-leaning" without the LLM having to overload `era_signal`.

## Anthropic Tool-Use Reference

### Why tool-use over regex-JSON

The existing `extractWithLlm()` parses with `textContent.text.match(/\{[\s\S]*\}/)` then `JSON.parse(jsonMatch[0])`. This works but is brittle: any preamble like "Sure, here's the JSON: {...}" inside leading text breaks if the regex is wrong, escape sequences need handling, and the LLM can drift outside the schema (extra fields, missing fields, wrong types).

Anthropic's tool-use mechanism is the canonical structured-output path: define a tool with an `input_schema` (JSON-Schema), force `tool_choice = { type: 'tool', name: 'record_taste_attributes' }`, and the API returns a `tool_use` content block with a typed `input` object that already conforms to the schema (with `strict: true` available for hard guarantees). [CITED: platform.claude.com/docs/en/build-with-claude/structured-outputs] [CITED: platform.claude.com/docs/en/agents-and-tools/tool-use/overview]

### Minimal call shape (text-only path, D-08 manual entry without photo)

```typescript
// src/lib/taste/enricher.ts (sketch)
import Anthropic from '@anthropic-ai/sdk'
import { z } from 'zod'
import { PRIMARY_ARCHETYPES, ERA_SIGNALS, DESIGN_MOTIFS } from './vocab'

const TasteSchema = z.object({
  formality:       z.number().min(0).max(1),
  sportiness:      z.number().min(0).max(1),
  heritage_score:  z.number().min(0).max(1),
  primary_archetype: z.enum(PRIMARY_ARCHETYPES),
  era_signal:        z.enum(ERA_SIGNALS),
  design_motifs:     z.array(z.enum(DESIGN_MOTIFS)).max(8),
  confidence:        z.number().min(0).max(1),
})
type TasteAttributes = z.infer<typeof TasteSchema>

const TASTE_TOOL: Anthropic.Messages.Tool = {
  name: 'record_taste_attributes',
  description: 'Record structured taste attributes for a watch.',
  input_schema: {
    type: 'object',
    properties: {
      formality:      { type: 'number', minimum: 0, maximum: 1, description: 'How formal/dressy 0..1' },
      sportiness:     { type: 'number', minimum: 0, maximum: 1 },
      heritage_score: { type: 'number', minimum: 0, maximum: 1, description: 'How storied/historically-significant 0..1' },
      primary_archetype: { type: 'string', enum: [...PRIMARY_ARCHETYPES] },
      era_signal:        { type: 'string', enum: [...ERA_SIGNALS] },
      design_motifs:     {
        type: 'array',
        items: { type: 'string', enum: [...DESIGN_MOTIFS] },
        description: 'Zero or more visual/aesthetic motifs from the closed vocabulary.',
        maxItems: 8,
      },
      confidence: {
        type: 'number', minimum: 0, maximum: 1,
        description: 'Self-rated confidence in this assessment (0..1). Use <0.5 for ambiguous cases.',
      },
    },
    required: ['formality','sportiness','heritage_score','primary_archetype','era_signal','design_motifs','confidence'],
  },
}

export async function enrichTasteAttributesText(
  catalogRow: CatalogSpecForEnrichment,
): Promise<TasteAttributes | null> {
  const apiKey = process.env.ANTHROPIC_API_KEY
  if (!apiKey) {
    console.log(JSON.stringify({
      event: 'taste_enrichment_skipped:no_api_key',
      catalog_id: catalogRow.id,
      source: 'text',
    }))
    return null
  }

  const client = new Anthropic({ apiKey })
  const response = await client.messages.create({
    model: 'claude-sonnet-4-6',
    max_tokens: 1024,
    tools: [TASTE_TOOL],
    tool_choice: { type: 'tool', name: 'record_taste_attributes' },
    messages: [{ role: 'user', content: buildTextPrompt(catalogRow) }],
  })

  const toolUse = response.content.find(
    (c): c is Anthropic.Messages.ToolUseBlock => c.type === 'tool_use',
  )
  if (!toolUse) return null

  const parsed = TasteSchema.safeParse(toolUse.input)
  if (!parsed.success) return null
  return validateAndCleanTaste(parsed.data)  // D-05 vocab filter + warn
}
```

### `tool_choice` forcing the tool

Per [CITED: platform.claude.com/docs/en/agents-and-tools/tool-use/overview], `tool_choice: { type: 'tool', name: 'record_taste_attributes' }` forces the model to call exactly that tool, eliminating the possibility of a plain text response. Without forcing, the model can return text instead of a tool call. For Phase 19.1, ALWAYS force the tool — there is no path where a free-text response is useful.

### `strict: true` consideration

Anthropic supports `strict: true` on tools, which guarantees parameters match the schema. [CITED: platform.claude.com/docs/en/build-with-claude/structured-outputs]. Recommend setting `strict: true` on `TASTE_TOOL` — the 8-field schema is small enough that strict-mode latency overhead is negligible, and it reduces the surface area where the validator has to drop bad values. The Zod parse + D-05 vocab filter remain as defense-in-depth.

### Vision + tool-use combined

The call shape above (image content block + text content block + tools array + tool_choice forcing) is the canonical pattern. Verified against [CITED: platform.claude.com/docs/en/build-with-claude/vision] and [CITED: anthropic-cookbook tool_use/extracting_structured_json.ipynb] — vision and tool-use compose without special handling. The model receives the image, considers it alongside the text spec context, and emits the taste attributes via the forced tool call.

### Photo fetch shape

Two options for getting the image bytes inside the enricher:

1. **Pass blob through call chain:** WatchForm → server action → enricher receives `{ photoBlob }` directly. Cleaner (no second fetch), but the enricher then can't be re-invoked from `db:reenrich-taste` script without the original blob.
2. **Re-fetch via signed URL (recommended):** Server-side action persists the photo to bucket and writes `image_source_url = bucket-path` to catalog. Enricher reads `catalogRow.image_source_url`, calls `supabase.storage.from('catalog-source-photos').createSignedUrl(path, 60)` (60s TTL), `fetch()`es the URL, base64-encodes. Re-runnable from script.

Recommend option 2 — the re-enrichment script flow (D-13) needs to work without the original blob, and the signed URL pattern matches Phase 15 wear-photo signed-URL conventions.

### D-13: COALESCE first-non-null-wins upsert pattern [VERIFIED: src/data/catalog.ts:163-227 `upsertCatalogFromExtractedUrl`]

The exact COALESCE shape is:

```sql
ON CONFLICT ON CONSTRAINT watches_catalog_natural_key DO UPDATE SET
  movement = COALESCE(watches_catalog.movement, EXCLUDED.movement),
  -- ... per scalar column
  -- Tag arrays use a different shape:
  style_tags = CASE WHEN array_length(watches_catalog.style_tags, 1) IS NULL

```

For Phase 19.1, the taste enrichment writes happen as a SEPARATE UPDATE (not part of an upsert), because:
- The catalog row already exists (taste runs AFTER catalog upsert commits — D-07/D-08)
- D-13 says: skip-if-any-taste, force-only re-run via script
- So the live enrichment path is `UPDATE watches_catalog SET taste_fields = ... WHERE id = $1 AND confidence IS NULL` (the `AND confidence IS NULL` enforces D-13 first-write-wins)

### Bucket migration delta

Phase 11's `wear-photos` migration uses three-tier RLS (owner / public-visibility / followers-visibility) because wear-photos are user-private content with conditional sharing. **Phase 19.1's `catalog-source-photos` is structurally simpler**: any authenticated user can SELECT (because the photos *become* canonical catalog images — public reach is the point), but only the owner can INSERT/UPDATE/DELETE (folder-enforced).

Recommended SELECT policy:

```sql
DROP POLICY IF EXISTS catalog_source_photos_select_authenticated ON storage.objects;
CREATE POLICY catalog_source_photos_select_authenticated ON storage.objects
  FOR SELECT
  TO authenticated
  USING (bucket_id = 'catalog-source-photos');
```

INSERT/UPDATE/DELETE: copy verbatim from Phase 11 wear-photos (folder enforcement on `(storage.foldername(name))[1] = (SELECT auth.uid())::text`).

Bucket creation: `public: false`, `file_size_limit: 8388608` (8 MB raw upload — matches Phase 15 ≤8MB before client-side compression), `allowed_mime_types: ARRAY['image/jpeg', 'image/png', 'image/heic', 'image/heif', 'image/webp']`. Note Phase 11 used `5242880` (5MB) and didn't allow HEIC at the bucket level — but that's fine because the HEIC conversion happens client-side BEFORE upload, so the uploaded bytes are always JPEG. Following the same convention (bucket only sees JPEG/PNG/WEBP) is simpler.

### Cross-user visibility asymmetry to document

Per CONTEXT.md `<code_context>` "Two-layer privacy on user-content buckets":

> Owner can see/replace own uploads (RLS-gated), but the *resulting catalog `image_url`* is public via the existing CAT-02 public-read policy. Document the asymmetry; it's intentional and parallels the wear-photos vs wear-events visibility model.

The bucket SELECT policy is `TO authenticated` (any signed-in user can fetch any catalog photo via signed URL), but the *enriched* `image_url` on the catalog row is read by `searchCatalogWatches` etc. and surfaces in the Server-Component-rendered UI. Photos uploaded to this bucket ARE understood by the user to become public via the catalog. The TOS deferral in CONTEXT.md `<deferred>` covers the policy work.

### Path convention precision (D-20)

`{userId}/{catalogId-or-pending}/{filename}.jpg` — the inner segment is either a UUID (when catalog row exists) or the literal string `pending` (when uploaded before catalog row creation). Filename should be a UUID-stamped JPEG (e.g., `${crypto.randomUUID()}.jpg`) to avoid collisions, mirror Phase 15 wear-photos' `{wearEventId}.jpg` shape but allow multiple uploads per catalog row over time. Phase 19.1 only writes once per catalog (D-21 first-write-wins), so collision is theoretical only.

Existing log events in the codebase (per the `logActivity` / `logNotification` pattern) use:
- Structured fields: `{ event, ... }` JSON
- Snake-case event names
- A single `console.log` line per event (parseable by log aggregators)

Recommended event vocabulary for Phase 19.1:

| Event | When | Payload |
|-------|------|---------|
| `taste_enrichment_skipped:no_api_key` | D-17 missing API key | `{ catalog_id, source, timestamp }` |
| `taste_enrichment_skipped:already_enriched` | D-13 row has confidence | `{ catalog_id, existing_confidence, timestamp }` |
| `taste_enrichment_started` | Enricher invoked | `{ catalog_id, source, mode: 'text'\|'vision', timestamp }` |
| `taste_enrichment_succeeded` | Tool-use parse + Zod parse + vocab filter all pass | `{ catalog_id, source, mode, confidence, latency_ms, timestamp }` |
| `taste_enrichment_failed` | Any error in the chain (timeout, 429, 5xx, parse, Zod) | `{ catalog_id, source, mode, error_class, latency_ms, timestamp }` |

Use `console.log(JSON.stringify({...}))` (NOT `console.error`) — these are signal events, not errors. Failure events at `:failed` level can use `console.error` for log-aggregator severity routing.

The `event` key uses `:` to namespace sub-types (e.g., `taste_enrichment_skipped:no_api_key`), which matches the `:` convention seen in `revalidateTag('viewer:${userId}')` and similar patterns in the codebase.

## Open Questions For The Planner

1. **Photo-before-row vs row-before-photo ordering on manual entry path.** WatchForm submit flow needs to decide: upload photo → call addWatch → catalog upsert → enricher (with photo path), OR call addWatch → catalog upsert → upload photo → enricher (with photo path). The first ordering creates orphan-photo risk if addWatch fails; the second blocks the form submission on photo upload. Recommendation: **first ordering with `pending` folder convention** (D-20) — photo uploads first to `{userId}/pending/{uuid}.jpg`, addWatch returns the catalogId, server-side post-mutation step renames the storage path to `{userId}/{catalogId}/{uuid}.jpg` (or just stores the pending path on the catalog row — simpler). The planner should pick a path; both work. Phase 15's `logWearWithPhoto` uses path-renames internally.

2. **Should the enricher receive the photo bytes inline (manual-entry-with-photo path) or always re-fetch via signed URL?** Inline is faster (one less network round-trip), but breaks symmetry with the backfill/re-enrichment script which can only re-fetch. Recommendation: **always re-fetch** — symmetric, simpler, only ~50ms slower in fire-and-forget mode (irrelevant). See "Photo fetch shape" above.

3. **Should `tool` parameter use `strict: true`?** Strict mode guarantees schema conformance with slight latency overhead. Recommend YES — tiny schema, robust output. But verify Anthropic SDK 0.88 supports `strict` on the Tool type at execution time.

4. **Should `taste_enrichment_audit` table exist?** Per CONTEXT.md Claude's Discretion. Argument for: durable audit trail of all enrichment attempts (succeeded + failed). Argument against: stdout structured events are sufficient for v4.0 personal-MVP scale. Recommendation: **stdout only** for now; create the table in v4.x if observability proves insufficient.

5. **Pending UI on WatchDetail when taste fields are NULL?** User said "idk maybe that's not worth it." Recommendation: **skip** in initial plan; small follow-up if a user UAT reports the 3-5s gap is jarring. Cost: ~30 LOC for a `<TastePendingHint />` component. Drop into the deferred-polish bucket.

6. **`design_motifs` array max length?** Recommended 8 (per the Zod sketch). User-flag at plan-check round.

7. **What happens if the LLM returns `confidence: null` or omits it entirely?** Schema marks `confidence` as required, so a strict tool call should fail. Belt-and-suspenders: if Zod parse succeeds but `confidence` is undefined/NaN, treat as enrichment failure. Document in the validator.

8. **Edit-mode photo upload?** D-19 says "manual-entry path." Currently undefined for edit mode (existing watches whose user wants to add a photo retroactively). Recommendation: **defer** — edit-mode photo upload becomes part of an owner-override path (deferred per CONTEXT.md). Existing rows can get photos via re-running URL extract or direct admin tooling (also deferred).
