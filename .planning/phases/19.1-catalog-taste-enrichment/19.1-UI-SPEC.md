# Phase 19.1 — UI Design Contract

> Visual and interaction contract for the WatchForm picker hide (D-18) plus the new optional reference-photo upload (D-19/D-20/D-21). Phase 19.1 is intentionally a thin UI surface: only `WatchForm.tsx` changes user-visibly. Verdict copy, badges, and `<CollectionFitCard>` are Phase 20 — explicitly out of scope here.

**Distinction copy (existing wear-photos vs new reference photo):** the new field is labeled "Reference Photo" and its description explicitly says "Used as the catalog's reference image." The existing wrist-shot upload lives only inside the WYWT post dialog (`/log-wear`) and is labeled "Take wrist shot" / "Upload photo" inside the wear flow. Users encountering the WatchForm field will not see the WYWT field on the same screen. **No "wear photo vs reference photo" disambiguation banner is needed.** If user testing surfaces confusion, fall back to renaming the WatchForm field to "Catalog Reference Photo" — this is the explicit fallback and the planner can flip it without a re-spec.

**No "your photo will be used as the canonical catalog image visible to other users" disclosure copy is shown in v4.0.** Rationale: D-21 cross-user visibility is correct architecturally but the v4.0 personal-MVP posture has effectively one user, so the disclosure would create concern out of proportion to the actual audience. The user has explicitly flagged the TOS / acceptable-use policy as a deferred follow-up — that work, not in-form disclosure copy, is the right surface for it. Re-evaluate at multi-user launch.

**The phase reuses the existing `@/components/wywt/PhotoUploader` pipeline** (HEIC worker + `stripAndResize`) but does NOT import the WYWT `PhotoUploader` component as-is, because it is wired to the WYWT bucket and lifecycle. Instead the planner will create `@/components/watch/CatalogPhotoUploader` (or planner-named equivalent under `src/components/watch/`) that mirrors the PhotoUploader signature (`onPhotoReady(jpeg: Blob)` + `onError(message: string)`) but writes to the new `catalog-source-photos` bucket via a new `@/lib/storage/catalogPhotos` helper. The visual contract is identical — only the bucket path and copy strings differ.

**Removals (from `WatchForm.tsx`):**
- Three `<Card>` blocks at lines 511–538 (Style), 540–567 (Role), 569–593 (Design Traits) — DELETE
- Two validation rules at lines 113–118 (`styleTags` length-required, `roleTags` length-required) — DELETE
- Asterisks on the Style and Role labels become irrelevant (lines 514, 543) — deleted with the cards
- The `STYLE_TAGS`, `DESIGN_TRAITS`, `ROLE_TAGS` imports (lines 22, 23, 24) become unused — DELETE the imports too (TS strict + ESLint will flag otherwise)

**Form-flow re-flow after removal:** the existing form sequence is `[URL Import (create only)] → [Basic Info] → [Specifications] → [Complications] → [Style] → [Role] → [Design Traits] → [Notes]`. After removal it becomes `[URL Import] → [Basic Info] → [Specifications] → [Complications] → [Reference Photo (create only, manual-entry only)] → [Notes]`. The `<form className="space-y-8">` parent provides 32px gaps between cards uniformly, so removing three cards collapses the layout cleanly without any awkward gap or visible artifact. **No re-flow code is needed.** No grid/section layout adjustment.

**Reference Photo card position rationale:**
- Placed AFTER Complications and BEFORE Notes (i.e. takes the slot vacated by the bottom of the three deleted cards)
- This keeps "describe the watch" sections (Basic Info → Specifications → Complications → Reference Photo) grouped, with Notes (free-text) at the bottom as the natural last-step
- It does NOT live at the top of the form, because the URL-import card is conditional (`mode === 'create'`) and adding the photo card adjacent to it would visually compete for the user's attention as a second "alternative entry point" — which it isn't (the photo augments manual entry, it doesn't replace it)
- It is gated to `mode === 'create'` AND the URL-import path being unused. **The simplest gate is `mode === 'create'`** — both the URL-import card and the Reference Photo card show on the create page; the user can use either, both, or neither (per D-19 the URL-import path also doesn't need a photo because URL extraction already sets `image_url`, but if both are present, the explicit upload wins per first-non-null-wins). On `mode === 'edit'` the field is hidden — re-enriching an existing watch's reference photo is the deferred owner-override path, not in 19.1.

## Photo Upload Field — Interaction States

The "Reference Photo" card has four interaction states, all rendered inline (no modal). Adapted from Phase 15 PhotoUploader contract.

### State 1 — Empty (default, no photo chosen)

Visual:
- Card with title "Reference Photo" and description "Optional, but strongly encouraged — vintage pieces especially benefit. Used as the catalog's reference image."
- Centered "Choose photo" outline button with leading lucide `<Image />` icon (16px)
- Below the button, muted-foreground 12px caption: "JPG, PNG, or HEIC up to 8 MB"

Interaction:
- Button click → opens native file picker (`accept="image/*,.heic,.heif"`)
- Selecting a file transitions to State 2

### State 2 — Processing (HEIC convert / EXIF strip / canvas re-encode)

Visual:
- Same card chrome as State 1
- The button changes its label to "Processing…" and `disabled={true}`; lucide `<Loader2 className="animate-spin" />` replaces `<Image />`
- The 8-MB caption stays visible

Interaction:
- File input is disabled
- No interaction available until pipeline resolves (~50–500ms HEIC, ~50–150ms strip+resize for typical files)
- On success → State 3; on failure → State 4

### State 3 — Preview ready (photo encoded, awaiting form submit)

Visual:
- 96×96 `rounded-md` thumbnail showing the resized JPEG (object-fit cover, generated via `URL.createObjectURL(blob)`)
- 8px gap to the right: small muted-foreground 12px caption "Photo ready"
- Top-right corner of the thumbnail: floating ghost button with lucide `<X />` (size 16px), `aria-label="Remove photo"`, hover state `bg-muted` (touch target 44×44 by way of `size="icon"` Button absolute-positioned with `top-1 right-1`)
- The "Choose photo" button is replaced by an inline text link "Choose another" (Tailwind `text-sm text-muted-foreground hover:text-foreground underline-offset-4 hover:underline`) below the thumbnail row — matches Phase 15 ComposeStep "Choose another" pattern

Interaction:
- Click X → revoke the object URL, clear local blob state, return to State 1
- Click "Choose another" → opens native file picker without removing the current preview (re-replaces on selection); during reselection processing, transitions through State 2

### State 4 — Error

Visual:
- Same card chrome as State 1 (button reverts to "Choose photo" enabled)
- Below the upload button, in place of the 8-MB caption: a `text-sm text-destructive` line with one of the error copy strings from the Copywriting Contract above
- Error string clears automatically on the next file-picker open

Interaction:
- Identical to State 1
- Error does NOT block form submission — the user can submit without a photo per D-19

### Submit-time photo upload behavior

This is NOT a separate user-visible state — it happens during `isPending` after `<Button type="submit">` is clicked. Visual shape:

- The form-level submit button shows "Adding..." (existing copy, unchanged)
- The Reference Photo card's "Choose another" link and X button become disabled (`disabled={isPending}`)
- The photo card does NOT show its own progress indicator; the form-level pending state covers it (single source of truth, avoids double-spinner anxiety)
- If `uploadCatalogPhoto()` rejects mid-submit, the form silently saves the watch WITHOUT the photo per D-09 fire-and-forget posture, then surfaces the "Photo upload failed" copy as a Sonner toast on the destination page (recipient: `/u/{username}/collection/{watchId}` after redirect). The watch row exists; the catalog row exists; the taste enrichment fires text-only because `extracted_from_photo=false`.

**Critical:** the "photo upload failed" toast is emitted by the destination page's mount effect (reading a flash query param or `sessionStorage` flag), NOT by the form. The form has already navigated away. This pattern matches the Phase 999.1 MR-01 toast hybrid. Planner can simplify to a single `setSubmitError` inline banner in the form if the cross-page toast wiring is too heavy for a thin phase — but the toast pattern is preferred.

## Pending UI on WatchDetail (Claude's Discretion — Recommendation)

**Recommendation: do NOT ship the "Analyzing taste…" pending indicator on WatchDetail in Phase 19.1.**

Rationale:
- Phase 19.1 ships taste fields as silent infrastructure (Phase 17 D-12 catalog-as-silent-infrastructure pattern extends to taste). No verdict copy reads these fields in this phase — Phase 20 owns that.
- An "Analyzing taste…" indicator would imply user-visible consequence ("when this finishes, X will appear") but X (verdict copy variation) does not yet exist as of Phase 19.1's ship date. Showing the indicator without the eventual visible outcome creates a phantom-progress UX.
- The taste fields converge in 3–8 seconds (text) or 5–12 seconds (vision) per RESEARCH.md. By the time the user navigates to WatchDetail, taste is typically already populated. A pending indicator would flash briefly for a small minority of refresh races.
- The user said "idk maybe that's not worth it" — explicit lean toward not shipping.
- **Trade-off accepted:** if Phase 20 verdict copy launches and a small percentage of just-added watches show generic copy (taste-confidence < 0.5 fallback) during the convergence window, that's acceptable degradation. The 6 fixed `SimilarityLabel` values still render. No broken UI, just less rich phrasing for ~5–10 seconds.

**If Phase 20 surfaces a UX gap during testing**, the indicator can be added in Phase 20 (where it is contextually meaningful because verdict copy is on-screen). In 19.1 it adds visual noise without payoff. **Do not implement.**
