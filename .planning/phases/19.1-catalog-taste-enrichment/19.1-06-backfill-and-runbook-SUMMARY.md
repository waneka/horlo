## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Added server-only path alias to tsconfig.json**
- **Found during:** Task 1 — verifying `tsx --env-file=.env.local scripts/backfill-taste.ts --dry-run`
- **Issue:** `scripts/backfill-taste.ts` imports from `../src/lib/taste/enricher` which has `import 'server-only'` at the top. The `server-only` package is not in `node_modules` (it lives only inside `node_modules/next/dist/compiled/server-only/`) — so tsx would fail to resolve it at script startup, even on the dry-run path that never calls the enricher.
- **Fix:** Added `"server-only": ["./tests/shims/server-only.ts"]` to `tsconfig.json` `paths`. tsx respects tsconfig path aliases at runtime. The shim is a no-op export `{}` — already present from Plan 02's vitest configuration. Next.js's actual `server-only` enforcement is in the webpack/turbopack bundler (import-graph analysis), not in tsconfig — so the alias does not bypass any real guard in production builds.
- **Files modified:** `tsconfig.json`
- **Commit:** `5cf779e`
