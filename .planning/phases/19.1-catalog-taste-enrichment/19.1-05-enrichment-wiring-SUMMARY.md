# Phase 19.1 Plan 05: Enrichment Wiring Summary

**One-liner:** Wired enrichTasteAttributes fire-and-forget into addWatch + /api/extract-watch, and upload-before-submit photo handoff in WatchForm with D-21/D-22 integration tests passing against local DB.

## What Was Built

### Task 1: addWatch Server Action extended

`src/app/actions/watches.ts` grows three concerns:

1. **Zod schema extension** — `photoSourcePath` optional field with regex `^[0-9a-f-]{36}/(pending|[0-9a-f-]{36})/[0-9a-f]+\.jpg$` and max 256 chars.

2. **Ownership check (T-19.1-05-05)** — immediately after Zod parse, before watch insert: `photoSourcePath.split('/')[0] !== user.id` → `{ success: false, error: 'Photo path does not match authenticated user' }`.

3. **Fire-and-forget dispatch** — after catalog upsert + linkWatchToCatalog, a second `if (catalogId)` block:
   - D-21 photo write-through: `getCatalogSourcePhotoSignedUrl(photoPath, 7-day TTL)` → `applyUserUploadedPhoto(catalogId, {imageUrl, imageSourceUrl})`
   - D-08 enrichment: `enrichTasteAttributes({catalogId, source: 'manual', spec, photoSourcePath})` → `updateCatalogTaste(catalogId, taste)` (first-write-wins per D-13)

Await semantics: **Option A** — synchronous await inside addWatch. Adds ~1-3s latency but guarantees writes complete before Next.js tears down the request lifecycle. Documented in SUMMARY and decisions.

### Task 2: /api/extract-watch route extended

Route now captures the catalogId returned by `upsertCatalogFromExtractedUrl` (`let catalogId: string | null = null`), then after the existing catalog block dispatches:
```typescript
const { enrichTasteAttributes } = await import('@/lib/taste/enricher')
const { updateCatalogTaste } = await import('@/data/catalog')
const taste = await enrichTasteAttributes({ catalogId, source: 'url-extract', ..., photoSourcePath: null })
if (taste) await updateCatalogTaste(catalogId, taste)
```

D-07 lock confirmed: `git diff src/lib/extractors/llm.ts` produces zero output.

### Task 3: WatchForm.handleSubmit extended

`handleSubmit` gains a photo upload block before the `addWatch` call (mode=create only):
1. Get `user.id` via `createSupabaseBrowserClient().auth.getUser()`
2. Call `uploadCatalogSourcePhoto(user.id, 'pending', photoBlob)` 
3. On success: `photoSourcePath = result.path`
4. On failure: log non-fatal, proceed without path
5. Pass `{ ...formData, photoSourcePath }` (when path present) to `addWatch`

Plan 03's `photoBlob` state and `<CatalogPhotoUploader>` JSX are untouched — only new lines added.

## D-20 Path Convention

Photos remain at `{userId}/pending/{uuid}.jpg` permanently. No rename to `{userId}/{catalogId}/{file}.jpg` after mutation. Rationale: `getCatalogSourcePhotoSignedUrl` regenerates the signed URL per-fetch; the bucket path is operational scaffolding only. Rename would add Storage API failure modes for no observable user benefit. `pending/` is a permanent valid middle segment per the Zod regex `(pending|[0-9a-f-]{36})`.

## Await Semantics (Option A)

Both `addWatch` and the `/api/extract-watch` route use **Option A**: synchronously await enrichTasteAttributes inside the handler. This adds ~1-3s latency to the user's action but guarantees the write completes. Option B (detach via `void Promise.resolve().then(...)`) would risk the Anthropic call being killed mid-flight if Next.js tears down the request. Acceptable at v4.0 personal-MVP scale where the user spends seconds filling the form.
