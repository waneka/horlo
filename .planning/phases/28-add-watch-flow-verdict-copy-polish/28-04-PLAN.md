<objective>
Ship the inline-commit half of UX-09: rewire the two inline 3-CTA Wishlist
commit sites (`/search` row + `/catalog/[id]`) so their `toast.success` calls
include Sonner's built-in action slot pointing to `/u/{viewerUsername}/wishlist`.
The user clicks "Add to Wishlist" inline, sees a "Saved to your wishlist"
toast with a "View" button, and clicks "View" to land on their wishlist tab.

Per D-05 (table row 5/6 in UI-SPEC), these two sites NEVER suppress the toast
— after commit the user stays on /search or /catalog/[id], which never
matches the `/u/{viewerUsername}/wishlist` destination. So the suppress
comparison is unnecessary at these two sites. The toast ALWAYS fires; only
the action-slot presence depends on viewerUsername (non-null → action slot
attached; null → bare action-less toast).

Plan 04 also threads `viewerUsername` through the `/search` and
`/catalog/[id]` Server Components into their respective Client Components.
This mirrors the Plan 03 thread-through at `/watch/new` but for two more
read-side surfaces.

Output: 2 Client Component edits (action-slot toast wiring) + 2 Server
Component edits (viewerUsername resolution + thread-through). Plan 05 owns
the AddWatchFlow + WatchForm commit-site toasts (those need the suppress
logic + nav-on-commit, both tightly coupled to ADD-08 returnTo).

This plan does NOT touch FormStatusBanner (D-07 — banner CTA-link variant is
forbidden indefinitely).

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-04-01 | XSS via toast body | Sonner toast.success body string | mitigate | Body is a literal string `'Saved to your wishlist'` — no template interpolation, no user data. React/Sonner default escaping applies. |
| T-28-04-02 | XSS via action label | Sonner action.label | mitigate | Label is the literal string `'View'`. No user input. |
| T-28-04-03 | Open redirect via action onClick | router.push(`/u/${viewerUsername}/wishlist`) | accept | viewerUsername is server-resolved from `getProfileById(user.id)` — comes from the authenticated user's own profile row in the DB. No untrusted-input pathway. The signup trigger validates username format at insert (lowercased, regex-constrained per memory ref). The constructed href is always same-origin path-only. |
| T-28-04-04 | Information disclosure | Toast surfaces watch-add to other tabs | accept | Sonner toast is portal-mounted at the layout root; visible only in the user's own session. No cross-user data leakage. |
</threat_model>
