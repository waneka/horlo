# Phase 20 — UI Design Contract

> Visual and interaction contract for the Collection Fit verdict surface across three rendering contexts (`/watch/[id]` cross-user, new `/catalog/[catalogId]`, and `/search?tab=watches` inline-expand). The card itself is a pure renderer (D-04). All design tokens are inherited from the existing `base-nova` shadcn preset and the OKLCH theme already declared in `src/app/globals.css` — Phase 20 introduces zero new tokens. Generated by `gsd-ui-researcher`, verified by `gsd-ui-checker`.

## Typography

| Role | Size | Weight | Line Height | Where it appears |
|------|------|--------|-------------|------------------|
| Body | 14px (`text-sm`) | 400 (`font-normal`) | 1.43 (Tailwind default for `text-sm`) | Contextual phrasings; most-similar list rows; "You own this" callout body; role-overlap warning copy |
| Label | 12px (`text-xs`) | 500 (`font-medium`) — inherits from `<Badge>` cva | 1 | Verdict label inside `<Badge variant="outline">`; "Owned"/"Wishlist" status pills (already specified Phase 19) |
| Heading | 16px (`text-base`) | 600 (`font-semibold`) — `<CardTitle>` default | 1.5 (`leading-none` per shadcn — overridden to 1.5 via title's containing flex row) | "Collection Fit" card title (the visible name in the header) |
| Display | not used | — | — | (no display-tier text in this phase — verdicts are inline cards, not page-level heroes) |

**Sizes (3 declared):** 12px / 14px / 16px. Inside the GSD UI-checker 3-4 size cap.
**Weights (2 declared):** 400 (body) + 500/600 (label/heading). Treat the 500-vs-600 distinction as a semantic continuation, not a third weight, per Phase 19.1 precedent.

**Headline phrasing typography:** the "headline" verdict phrasing (e.g. *"Fills a hole in your collection — your first dive watch."*) is rendered as `text-sm font-medium text-foreground` — body-sized so it visually parallels the contextual phrasings, but `font-medium` (weight 500) and full-`foreground` color so it reads as the lead sentence of the card body. **Do not enlarge it to `text-base` or `text-lg`** — the verdict is not a hero on a detail page; the page already has its own H1 (the watch name).

**Contextual phrasings typography:** `text-sm text-muted-foreground` — same shape as the existing "Analysis" bullets in `SimilarityBadge.tsx:32`. Each phrasing renders on its own line with a 4px gap between them (`space-y-1`).

**Hedged copy typography (confidence 0.5–0.7):** identical to full-confidence phrasings — `text-sm text-muted-foreground`. The hedge is a copy-level signal ("possibly aligns…"), not a typographic one. **Do not italicize, lower opacity, or add icons** — the words carry the signal.

**Most-similar list typography:** brand+model is `text-sm text-muted-foreground`; the inline percentage (`67% similar`) is `text-muted-foreground/70` — slightly dimmer to recede behind the brand name. Mirrors `SimilarityBadge.tsx:54` exactly.

**Confidence-fallback fixed-label typography:** when `confidence < 0.5` the card renders only the 6-fixed-label description (`getSimilarityLabelDisplay(label).description`). Same `text-sm text-muted-foreground` styling — no visual cue distinguishes "fixed label fallback" from "rich contextual" copy. The user sees a simpler verdict; the fact that we hedged is invisible.

**Accent reserved for** (explicit list — not "every interactive element"):
- The role-overlap warning row icon + copy (`text-accent` on `<AlertTriangle />` + warning text)
- The Accordion.Trigger keyboard focus ring (`focus-visible:ring-2 focus-visible:ring-ring`) — already inherited from `WatchSearchRow.tsx:49`
- The "You own this" callout's owner-link affordance (Phase 20.1 will likely render an "Add to wishlist/collection" button on the catalog surface; that button uses the form-level primary CTA convention from Phase 19.1 and is the only intentional accent-use addition. **Do not paint anything inside `<CollectionFitCard>` accent in Phase 20** — keep accent for the role-overlap row only.)
- Hover/focus state of the catalog page's "Visit owner's collection" link (rare; only renders when D-08 self-via-cross-user fires)

**No verdict-label-specific color mapping is introduced.** The 6 `SimilarityLabel` values (`core-fit`, `familiar-territory`, `role-duplicate`, `taste-expansion`, `outlier`, `hard-mismatch`) all render as `<Badge variant="outline">` — same styling for all six. Rationale:
- The legacy `getSimilarityLabelDisplay()` at `src/lib/similarity.ts:343` shipped non-theme `bg-green-100`, `bg-yellow-100`, etc. — these ignored the existing OKLCH theme and broke in dark mode (verified — `bg-green-100 text-green-800` is not theme-aware). Phase 20 deliberately does not perpetuate that mistake.
- A label-to-color mapping forces six new color decisions that would expand the palette beyond the 60/30/10 split. The verdict label is a chip, not a status indicator; the contextual phrasing carries the tonal weight (e.g. *"fills a hole"* is positive; *"competes with your Speedmaster"* is cautionary). Color cannot match copy nuance reliably.
- If user testing surfaces "I can't tell positive verdicts from negative ones at a glance" the followup is a tonal icon prefix (e.g. `<CheckCircle />` for core-fit, `<AlertTriangle />` for role-duplicate), NOT a colored chip. That's deferred to Phase 20.1 / v5+ if needed.

**Cross-user framing visual cue:** `framing === 'cross-user'` does NOT visually differ from `framing === 'same-user'`. Both render the same `<Card>` chrome with the same "Collection Fit" title. Rationale: the surrounding page already establishes "viewing someone else's collection" via the page header, breadcrumb (`/u/{username}/...`), and the absence of edit buttons. Adding a "viewing someone else" lens indicator inside the card would be redundant and visually noisy. The card's content (the verdict itself, computed against the *viewer's* collection) is contextually correct without a chrome-level cue.

**`framing === 'self-via-cross-user'` (D-08) IS visually distinct:** the entire card body is replaced by a small callout — not a normal verdict. See **Component Inventory § "You own this" callout** below.

**Loading state color:** `<Skeleton>` uses the existing `bg-muted` shimmer (`tw-animate-css` `animate-pulse` variant). No accent overlay during loading.

**Error state color:** Server Action failures show a Sonner toast in the existing `text-destructive` styling — see Copywriting Contract § Error.

## Component Inventory

### `<CollectionFitCard>` (NEW — pure renderer per D-04)

Path: `src/components/insights/CollectionFitCard.tsx`
Replaces: `src/components/insights/SimilarityBadge.tsx` (delete after WatchDetail migrates — single consumer)
Props: `{ verdict: VerdictBundle }` (zero other props; zero imports of `analyzeSimilarity` or `composeVerdictCopy`)

Layout grid (top-to-bottom inside `<CardContent>`):
1. **Header row** (`<CardHeader>`): "Collection Fit" title + `<Badge variant="outline">` with the `SimilarityLabel.text` value, separated by `gap-3`. Mirrors `SimilarityBadge.tsx:20-25` exactly.
2. **Headline phrasing** (`<p className="text-sm font-medium text-foreground">`): one sentence, the lead phrasing chosen by the composer. **Always present** (composer guarantees a non-empty headline; falls back to fixed-label description if no template fires).
3. **Contextual phrasings** (`<ul className="text-sm text-muted-foreground space-y-1">`): zero or more supporting phrasings, bulleted with the existing `<span>•</span>` glyph from `SimilarityBadge.tsx:35`. Each `<li>` is a single line; no nested structure. Hidden entirely when `contextualPhrasings.length === 0` (the headline carries the verdict alone).
4. **Most-similar list** (`<div>` with `<h4>` heading "Most Similar in Collection" + `<ul>`): renders only when `verdict.mostSimilar.length > 0`. Each row is `{brand} {model}` left-aligned, `{percent}% similar` right-aligned (`flex items-center justify-between`). Maximum 3 rows shown — `analyzeSimilarity()` already caps at 3 per `src/lib/similarity.ts` (verified). **No thumbnails** — verdict-text-only is the legacy posture; adding thumbnails would expand the card vertical footprint by ~48px and require image-loading state per row, both unjustified at this surface. RESEARCH.md Open Q5 explicitly resolved this.
5. **Role-overlap warning** (`<p className="text-sm text-accent flex items-center gap-2">`): renders only when `verdict.roleOverlap === true`. Leading icon: `lucide-react` `<AlertTriangle className="size-4" aria-hidden />` (replaces the inline SVG at `SimilarityBadge.tsx:65-77` — use the lucide icon for consistency with the rest of the codebase). Copy: "May compete for wrist time with similar watches".

Visual hierarchy decision (open question 1 from prompt): **the verdict label badge is the smallest, highest-density element; the headline phrasing is the primary visual element.** Rationale:
- The label is a categorical chip — it's there for users who already know the taxonomy.
- The headline is the natural-language sentence that explains the relationship — that's what new and returning users actually read.
- The most-similar list is supporting evidence, not the lead.
- The role-overlap warning is a footnote — present only when the data warrants it.

This means the eye's first stop after the title row should be the headline sentence, not the badge. Achieved by placing the headline immediately under the header (no other element between them) and giving it `font-medium` while the contextual lines below are `font-normal`.

### Verdict label visual treatment (open question 2 from prompt)

**Decision: outline badge, no icon, no per-label color.** Locked above in § Color. This is a deliberate departure from the legacy `getSimilarityLabelDisplay()` color scheme — see Color rationale.

If the user later requests visual differentiation, the followup ladder is (in order):
1. Add a leading lucide icon to the badge: `<CheckCircle />` for `core-fit` + `taste-expansion`; `<AlertCircle />` for `role-duplicate` + `outlier`; `<XCircle />` for `hard-mismatch`; nothing for `familiar-territory`. Icon colored by `text-accent` (warning) or `text-muted-foreground` (neutral) — never red/green saturated.
2. Then if still insufficient: add a tonal background tint via theme tokens (e.g. `bg-accent/10` for warnings) — never raw `bg-yellow-100`.
3. Then if still insufficient: a separate Tonal Indicator pill alongside the label (similar to how Phase 19.1 considered a separate confidence pill — and rejected it).

In Phase 20: ship plain outline badge.

### Confidence hedging visual cue (open question 3 from prompt)

**Decision: no visual cue — copy-only signal.** Rationale:
- The hedge ("Possibly", "Likely") is the most economical signal: a single word that the user reads anyway.
- A visible "uncertain" icon or italic styling shifts cognitive load: now the user has to decode TWO signals (the icon AND the copy) to understand "this is a guess."
- The copy hedge is also semantically richer: "Possibly aligns with your heritage-driven taste" is more honest than an opaque ⚠️ icon next to a confident sentence.
- A11y bonus: screen readers naturally read the hedged word; an icon-only signal requires aria-label gymnastics.

If the user later requests a visible cue, the followup is a small `<HelpCircle />` icon at the right edge of the headline row, with a hover tooltip "Based on a partial taste signal" — opt-in disclosure, not always-visible decoration. **Do not implement that in Phase 20.**

### Most-similar list presentation (open question 4 from prompt)

**Decision: inline list, brand+model + score percentage, no thumbnails.** Mirrors existing `SimilarityBadge.tsx:48-58` rhythm exactly.
- Brand+model: `text-sm text-muted-foreground` left-aligned, `truncate`
- Percentage: `text-muted-foreground/70` right-aligned, fixed-width
- Row height: implicit (single line of `text-sm` ≈ 20px) — no explicit `h-` utility
- Max 3 rows (engine already caps)
- No thumbnail column

Rationale: the most-similar list is supporting evidence for the verdict, not a navigation surface. Users don't click these rows in Phase 20 (the rows are static text). Adding thumbnails would invite the misread "these are clickable" — and clicking them is Phase 20.1 (compare-with-watch-I-own drill-down, FIT-05 deferred).

### `<Accordion>` (FIT-04 inline-expand on `/search?tab=watches`)

Library: `@base-ui/react/accordion` (verified — see RESEARCH.md Open Q1)
Props pattern:
- `<Accordion.Root multiple={false}>` — one-at-a-time per D-05 (default)
- `<Accordion.Item value={catalogId}>` — keyed by catalog UUID, NOT array index
- `<Accordion.Header>` wraps `<Accordion.Trigger render={<button className="absolute inset-0">}>` — see Pitfall 6 in RESEARCH.md
- `<Accordion.Panel>` with `data-[state=open]:animate-in data-[state=closed]:animate-out data-[state=closed]:fade-out-0 data-[state=open]:fade-in-0 data-[state=closed]:slide-out-to-top-2 data-[state=open]:slide-in-from-top-2` — uses `tw-animate-css` utilities already in the project

Animation duration: **150ms** (Tailwind `duration-150`) — fast enough that opening feels immediate, slow enough to register as a transition. base-ui Accordion does not animate by default; we attach Tailwind utilities that respond to the `data-state` attributes the primitive emits.

Keyboard contract (provided by base-ui Accordion out of the box; do not hand-roll):
- `Tab` — moves focus between row triggers
- `Enter` / `Space` — toggle the focused row
- `ArrowDown` / `ArrowUp` — moves focus between row triggers (loops by default; do NOT pass `loopFocus={false}`)
- `Home` / `End` — jump to first/last row trigger
- `Escape` — collapses the currently-open row (verified in base-ui docs; if any Phase 20 implementer finds it doesn't, fall back to a `useEffect`-attached `keydown` listener on the Accordion.Root)

Focus management:
- When a row collapses (programmatically or via Escape), focus stays on the row trigger that was most recently interacted with
- When a row expands, focus stays on the trigger; pressing Tab afterward moves focus into the panel content (specifically: into the most-similar list rows, which are static text and thus not focusable in Phase 20 — so Tab continues past the panel to the next row trigger). This is acceptable; the panel content is informational, not interactive.

### Accordion expand affordance — what does the trigger look like? (open question 5 from prompt)

**Decision: the entire row is the trigger. The visible "Evaluate" button stays on the right edge as visual affirmation but does NOT have a separate click handler.**

The whole-row click affordance is provided by `<Accordion.Trigger render={<button className="absolute inset-0">}>` per Pitfall 6 — same DOM shape as the legacy `<Link className="absolute inset-0">`, just a button instead of an anchor.

### "You own this" callout (D-08 — `framing === 'self-via-cross-user'`)

### Mobile breakpoints (open question 12 from prompt)

| Breakpoint | Behavior |
|------------|----------|
| `< 640px` (default mobile) | `<CollectionFitCard>` is full-width inside its parent container; the search-row inline-expand panel is full-width below the row; no thumbnail column changes (the row already uses `size-10` thumbnail mobile / `size-12` desktop per Phase 19) |
| `640px–768px` | No layout change — card stays full-width within the page container's `max-w-4xl` cap |
| `≥ 768px` (desktop) | Same — card is full-width within `max-w-4xl`. The `min-h-20 md:min-h-20` on the search row provides slightly more breathing room on desktop. Most-similar list and contextual phrasings stay single-column at all breakpoints — no two-column desktop variant. |

**Rationale:** the verdict card is dense, vertically rhythmic content. A two-column desktop layout (e.g. headline left / most-similar right) would force responsive logic without improving scanability. Single-column at all breakpoints is the simplest legible layout.

**One mobile-specific consideration:** the role-overlap warning row (`<p className="text-sm text-accent flex items-center gap-2">`) wraps to two lines on small viewports (the copy is 6 words). That's acceptable — the warning is the lowest-priority element on the card.

## Cross-Surface Integration Notes

### `/watch/[id]/page.tsx` (existing, MODIFIED)

- Server Component, already exists. Adds `verdict` computation when `collection.length > 0` and threads it as a prop to `WatchDetail`.
- Both same-user (`isOwner === true`) and cross-user (`isOwner === false`) branches compute verdict — the only branch difference is `framing: 'same-user'` vs `'cross-user'`.
- Self-via-cross-user (D-08) is NOT triggered by this route (per RESEARCH.md Q4 resolution — the route is keyed by `watches.id`, not catalog id; viewer can't reach their own watch via a "cross-user" path here without already being the owner).

### `/catalog/[catalogId]/page.tsx` (NEW per D-10 lock)

- Server Component (matches D-03 static-surface compute).
- Looks up by `watches_catalog.id` (catalog UUID).
- Computes whether viewer owns this catalog ref (`SELECT FROM watches WHERE userId=viewer AND catalogId=catalogId LIMIT 1`).
- If owned → `framing: 'self-via-cross-user'`, render the "You own this" callout (D-08).
- If not owned and `collection.length === 0` → render the page WITHOUT the card (D-07 hide).
- If not owned and `collection.length > 0` → `framing: 'cross-user'`, render `<CollectionFitCard>` with full verdict.

Page chrome: watch image (using `getCatalogPhotoSignedUrl` from Phase 19.1 for catalog-source-photos, else `imageUrl` field), brand+model+reference heading, specs sublabel, then `<CollectionFitCard>`. The page does not have its own H1 — the watch name (`{brand} {model}`) is the H1 (use `<h1 className="text-2xl font-semibold">`). This is the only place in this phase where `text-2xl` font-size appears — and it's a page-heading concern outside the card itself, so the typography tier is governed by the page-level type system, not the card's 3-size budget.

### `/search?tab=watches` row inline-expand (FIT-04, NEW)

- Client Component (lazy compute via Server Action per D-06).
- The `WatchSearchRow` rendering is wrapped in a new `WatchSearchRowsAccordion` shell that owns Accordion state + cache.
- Each row's expand fires `getVerdictForCatalogWatch(catalogId)` Server Action; cached by `catalogId` keyed on collection-revision.
- The "Evaluate" → "Hide" copy toggle and the chevron rotation are driven by the Accordion's `data-state` attribute on the `<button>` trigger.

### `DiscoveryWatchCard` (Phase 18, MODIFIED per D-10)

- Currently non-clickable (`src/components/explore/DiscoveryWatchCard.tsx:14` notes Phase 20 will light it up).
- Phase 20 wraps the card in `<Link href={`/catalog/${watch.catalogId}`}>` (entire card body becomes the click target). No new visual shapes — just a Link wrapper.
- `WatchSearchRow`'s `/evaluate?catalogId=` href is removed (FIT-04 above); does NOT redirect to `/catalog/[catalogId]`. Search rows expand inline; only DiscoveryWatchCard navigates.
