## Spacing Scale

This phase reuses the existing right-column `space-y-6` (24px) rhythm established by Phase 64. No new spacing values are introduced.

| Token | Value | Usage in This Phase |
|-------|-------|---------------------|
| xs | 4px | Reserved (not used) |
| sm | 8px | Vertical gap between chip rows (`space-y-2`); gap between header and chip list (`space-y-2` on the wrapping section) |
| md | 16px | Reserved (not used) |
| lg | 24px | Outer rhythm ŌĆö the module slots into the right column's existing `space-y-6` stack as another sibling between the LikeButton/jump row and the Last-Worn line |
| xl | 32px | Reserved (not used) |
| 2xl | 48px | Reserved (not used) |
| 3xl | 64px | Reserved (not used) |

Horizontal chip layout: `gap-3` (12px) between avatar and the username/displayName text column. **12px exception to the 8-point scale** ŌĆö matches the existing comfortable avatar-to-text gap convention used by avatar-bearing rows across the app (e.g., comment author rows, profile primitives). 12px is the standard Tailwind `gap-3` which is one of the most-used utility values in the codebase. Confirmed by orchestrator as a valid existing convention.

Exceptions:
- **Touch target**: Each chip row enforces `min-h-[44px]` to guarantee a finger-friendly tap target on mobile (WCAG 2.5.5 AAA, project-wide convention from Phase 64). The 40px `AvatarDisplay` plus 4px of vertical padding hits 44px naturally; explicit `min-h-[44px]` on the `<li>` is required as a safeguard regardless of text height.
- **Avatar size**: 40px (`size={40}` ŌĆö the smallest legal value in the `AvatarDisplay` literal union, matching `OtherOwnersRoster`). Not on the 8-point scale; constrained by the primitive's three-size API (40 / 64 / 96). Documented exception per `AvatarDisplay.tsx:10`.

## Typography

All sizes are Tailwind utility classes; the underlying scale is Tailwind 4 + shadcn `base-nova` preset. Confirmed from `OtherOwnersRoster.tsx` and `WatchDetailHero.tsx` source reads.

| Role | Tailwind Class | Effective Size | Weight | Line Height |
|------|---------------|----------------|--------|-------------|
| Module header ("From your circle") | `text-sm font-medium` | 14px | 500 (medium) | 1.5 |
| Chip username (`@username`) | `text-sm font-medium` | 14px | 500 (medium) | 1.5 |
| Chip displayName (when present) | `text-xs` | 12px | 400 (regular) | 1.5 |
| Overflow caption ("and {N} more") | `text-xs` | 12px | 400 (regular) | 1.5 |

**Weight rationale:** The hero right column already establishes a regular/medium pair through `SpecsSublabel` and the LikeButton row; introducing a third weight (e.g., semibold) for this module would conflict with the established rhythm and pull visual focus away from the hero's primary brand+model heading. Two weights total (regular 400, medium 500) ŌĆö within the 2-weight maximum.

**Size rationale:** Three sizes (14px header, 14px primary chip text, 12px secondary chip text + caption). The header and chip primary text share 14px deliberately ŌĆö they are peers in the visual hierarchy (the header labels the list; each chip identifies a person). Tertiary text (displayName, "+N more") drops to 12px to signal lower priority. Within the 3-4 size maximum.

Typography is inherited from the existing system ŌĆö do not add new sizes or weights.

## Component Layout & Composition

This section is the primary deliverable for this phase. It locks the chip row structure, vertical stacking, header treatment, and overflow caption.

### Module Anatomy

```
<section aria-label="People you follow who own this" className="space-y-2">
  <h3 className="text-sm font-medium text-foreground">From your circle</h3>
  <ul className="space-y-2">
    <li className="group relative flex items-center gap-3 min-h-[44px]">
      <Link href="/u/{username}/collection"
            aria-label="{displayName ?? '@'+username}'s collection"
            className="absolute inset-0 rounded-md focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring" />
      <AvatarDisplay avatarUrl={...} displayName={...} username={...} size={40} />
      <div className="min-w-0">
        <p className="text-sm font-medium text-foreground truncate">@{username}</p>
        {displayName && (
          <p className="text-xs text-muted-foreground truncate group-hover:text-foreground transition-colors">{displayName}</p>
        )}
      </div>
    </li>
    {/* ...repeat up to 5 chips ordered by watches.createdAt DESC... */}
  </ul>
  {totalCount > owners.length && (
    <p className="text-xs text-muted-foreground">and {totalCount - owners.length} more</p>
  )}
</section>
```

### Visual Layout (D-04, locked)

- **Direction:** Vertical chip stack ŌĆö one chip per row, top-to-bottom ŌĆö NOT horizontal scroll like `OtherOwnersRoster`. The hero right column is the narrow `2fr` track of the `[3fr_2fr]` grid; a horizontal-scroll roster inside a narrow column reads cramped. Vertical stacking flows naturally into the existing `space-y-6` column rhythm.
- **Chip row structure:** `flex items-center gap-3` ŌĆö avatar on the left (40px), username/displayName text column on the right.
- **Text column:** `min-w-0` to enable `truncate` ellipsis on long usernames or displayNames at narrow widths.
- **Click surface:** Absolute-inset `<Link>` covers the entire chip row (avatar + text). User can tap anywhere on the row. Matches `OtherOwnersRoster.tsx:69-73` pattern exactly.
- **Visible chip count:** Top 5 owners by `watches.createdAt DESC` (recency of THEIR ownership of the catalog ref ŌĆö "who in my circle just got one"). Per D-04b.
- **Overflow:** When `totalCount > 5`, render plain text caption `and {N} more` below the 5 chips (where N = `totalCount - owners.length`). NO see-all link, NO inline expand, NO interactive control. Per D-04c.

### Module Placement in Hero (D-10, locked)

Renders as a child of `WatchDetailHero` inside the right column container `<div className="space-y-6 min-w-0">` (`WatchDetailHero.tsx:230`), positioned between the LikeButton + jump-to-comments row (line ~276) and the Last-Worn line (line ~300).

```
Right column vertical order (top ŌåÆ bottom):
  1. Title + spec block (brand h2 / model / ref / SpecsSublabel)
  2. LikeButton + jump-to-comments anchor row
  3. ŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆ FollowedOwnersModule ŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆ  ŌåÉ NEW (Phase 65)
  4. Last worn line              (owner only ŌĆö owned/grail)
  5. Flag-as-good-deal checkbox  (owner only ŌĆö wishlist/grail)
  6. Owner action buttons        (owner only)
```

For a **non-owner** viewing this watch, the right-column sequence is: title block ŌåÆ like+jump ŌåÆ "From your circle" module. Owner-only blocks (4/5/6) do not render for non-owners; the module sits visually directly above the bottom of the right column on those views.

For an **owner** viewing their own watch, the module appears between the like/comment row and their owner-only rows. Self-exclusion (D-05a) in the DAL ensures the viewer never appears in their own chip list ŌĆö the module renders only when followed collectors also own this catalog ref, which is the intended social-proof framing.

### Hide-If-Empty Contract (FOLL-01, locked)

`if (owners.length === 0) return null` at the top of the `FollowedOwnersModule` component. The module is **entirely absent from the DOM** when zero followed owners exist ŌĆö no header, no skeleton, no "(none)" placeholder, no zero-state card. The right column's `space-y-6` rhythm absorbs the absence cleanly (LikeButton row ŌåÆ Last-Worn line gap collapses to 24px).

This same contract covers the Branch 1 null-catalogId case (D-01a): when `watch.catalogId === null`, the page-level call site short-circuits to `{ owners: [], totalCount: 0 }`, the component returns `null`, no DOM emitted.

### Responsive / Mobile Behavior (success criterion 5, locked)

The hero is `lg:grid-cols-[3fr_2fr]` (Phase 64 D-01). Below the `lg` breakpoint (1024px) the right column stacks below the photo block as a single column. The module is already a vertical stack at every breakpoint ŌĆö **no responsive variants, no separate mobile component, no `lg:hidden`/`hidden lg:block` JSX duplication.** The natural single-column collapse satisfies "Mobile: the module stacks naturally below the hero's right-column content."

Each chip's `min-h-[44px]` and `truncate` ensures mobile tap targets and overflow stay correct at any viewport width inside the right column.

## Interaction & State Contract

| State | Treatment |
|-------|-----------|
| Default | Avatar + `@username` (foreground) + optional displayName (muted-foreground). Whole row is a tap target via absolute-inset `<Link>`. |
| Hover (pointer) | DisplayName line shifts from `text-muted-foreground` to `text-foreground` via `group-hover:` + `transition-colors`. NO background swatch, NO accent fill. Subtle by design. |
| Focus (keyboard) | `focus-visible:ring-2 focus-visible:ring-ring` on the absolute-inset `<Link>` produces a 2px accent-colored focus ring around the chip rounded-md outline. Same focus pattern as `OtherOwnersRoster.tsx:72`. |
| Active (click/tap) | Native `<a>` behavior ŌĆö full-page SPA navigation to `/u/{username}/collection`. No optimistic state, no loading spinner inside the chip (Next.js handles the route transition). |
| Disabled | N/A ŌĆö this module has no disabled state. |
| Loading | Pre-fetch in `Promise.all` per D-09. **No skeleton inside the module.** If the JOIN profiles slow (>100ms p95 hero impact, per D-09 fallback), the planner may wrap the module in a sibling `<Suspense fallback={...}>` at the **page level** (NOT inside `WatchDetailHero`); the fallback would be a single 44px-tall skeleton row with a 40px circular skeleton and a short text-skeleton. This loading shape is documented for the fallback case only ŌĆö default path has no skeleton. |
| Empty (`owners.length === 0`) | `return null` ŌĆö no DOM. See "Hide-If-Empty Contract" above. |
| Error | N/A at the component level ŌĆö the DAL is server-side and any error propagates to the route's error boundary. The module itself accepts `owners: FollowedOwner[]` and trusts the prop; defensive `?? []` at the call site protects against undefined. |

**No client-side state.** The component is pure-presentation RSC; no hooks, no event handlers, no `'use client'` directive. Chips are plain `<Link>` elements. Navigation, scroll restoration, and route prefetch are all handled by Next.js standard behavior.

**No animation.** No fade-in, no stagger, no spring. The module appears at SSR/initial-paint and stays static until route change. This is consistent with `OtherOwnersRoster` and the Phase 64 hero columns.

## Accessibility Contract

- **Semantic structure:** `<section>` wraps the module (carries the literal `aria-label="People you follow who own this"`); an `<h3>` provides the visible heading "From your circle"; chips render inside a semantic `<ul>` with one `<li>` per owner. Screen readers announce the list count ("list, 5 items") and let users navigate by item.
- **Each chip is a single tap target.** The absolute-inset `<Link>` is the only focusable element inside each `<li>`; the avatar `<Image>` and text `<p>` tags are non-interactive and inherit the link's hit area. No nested tab stops, no duplicate focus rings.
- **Accessible link name:** `aria-label="{displayName ?? '@'+username}'s collection"`. The aria-label overrides the visible text content for screen readers and produces an unambiguous link name (e.g., "Jane Doe's collection" or "@janedoe's collection"). Matches `OtherOwnersRoster.tsx:71`.
- **Focus ring:** `focus-visible:ring-2 focus-visible:ring-ring` produces a 2px accent ring around the chip's `rounded-md` outline on keyboard focus. The `focus-visible` pseudo-class ensures pointer clicks don't trigger the ring (preserves clean visual default). Hit-area is the full chip; focus ring traces the chip outline.
- **Tab order = DOM order = visual order.** Chips appear in `watches.createdAt DESC` order; tab moves through them in the same top-to-bottom sequence. No `tabindex` manipulation, no CSS order tricks.
- **Avatar `alt` text:** Handled by `AvatarDisplay` primitive ŌĆö produces `alt="{displayName ?? username} avatar"` on the `<Image>` (`AvatarDisplay.tsx:38`). The chip's `aria-label` provides the dominant accessible name; the avatar alt is supplementary.
- **Color contrast:** All text uses `--foreground` (high contrast against `--background`) or `--muted-foreground` (mid contrast, used only for secondary displayName + "+N more" caption ŌĆö both supportive, not load-bearing for comprehension). The base-nova preset's OKLCH values are WCAG AA contrast-compliant in both light and dark themes (verified by the preset's design intent ŌĆö no per-phase verification required).
- **Reduced motion:** No animations or transitions on the module other than `transition-colors` (200ms default) on the displayName hover state. `transition-colors` respects `prefers-reduced-motion: reduce` via the project's existing `globals.css` accommodation pattern (no explicit override needed ŌĆö `transition-colors` is a non-motion transition and is the project's standard for color shifts).
- **Touch targets:** `min-h-[44px]` on each `<li>` guarantees the full chip is finger-tappable on mobile (40px avatar + 4px ensures the row is Ōēź 44px even when only `@username` shows without a displayName).
