## State Machine & Interaction Contract

Phase 20.1's core interaction is a state-driven single-page flow. This section is the primary executor reference.

### Flow States (D-01)

```
idle
  → extracting        (user clicks Extract Watch or pastes URL + submits)
  → verdict-ready     (extraction + verdict compute complete together — D-07 sync wait)
      → form-prefill  (user clicks "Add to Collection" — D-12)
      → submitted-wishlist (user clicks "Add to Wishlist" — D-13; toast + reset to idle)
      → skipped       (user clicks "Skip" — D-14; reset to idle + add to recently-evaluated)
  → extraction-failed (API error; D-18 unified fallback; preserves partial data; offers manual)
```

Transitions that reset to `idle`: submitted-wishlist, skipped, "start over" after extraction-failed.

State machine implementation is Claude's discretion: bare `useState` + discriminated union OR `useReducer`. No FSM library required at this scale.

### Section Visibility per State

| Section | idle | extracting | verdict-ready | form-prefill | extraction-failed |
|---------|------|------------|---------------|--------------|-------------------|
| Paste input + Extract Watch button | visible | disabled (Working...) | hidden | hidden | visible (re-enabled) |
| "or enter manually" link (D-03) | visible | hidden | hidden | hidden | visible |
| Working... copy (D-07) | hidden | visible | hidden | hidden | hidden |
| Spec preview (brand/model/image) | hidden | hidden | visible | hidden | visible (partial) |
| `<CollectionFitCard>` (or "no collection" notice) | hidden | hidden | visible | hidden | hidden |
| 3-button row (D-11) | hidden | hidden | visible | hidden | hidden |
| Wishlist rationale textarea (D-13) | hidden | hidden | hidden (unless Wishlist clicked) | hidden | hidden |
| `<WatchForm>` prefilled (D-12) | hidden | hidden | hidden | visible | hidden |
| Recently evaluated rail (D-14) | visible if items exist | visible if items exist | visible if items exist | hidden | visible if items exist |
| Extraction-failed message + partial data | hidden | hidden | hidden | hidden | visible |

### Visuals — Focal Point

In `verdict-ready` state, the `<CollectionFitCard>` verdict label is the primary focal point; the 3-button row is the secondary anchor.

### 3-Button Visual Hierarchy (D-11)

| Button | Variant | Copy | Size |
|--------|---------|------|------|
| Add to Wishlist | `default` (filled, accent background) | "Add to Wishlist" | `default` |
| Add to Collection | `outline` | "Add to Collection" | `default` |
| Skip | `ghost` | "Skip" | `default` (or `sm` on mobile to reduce visual weight) |

Buttons render as a horizontal row on tablet/desktop, stacked column on mobile (< 640px). Full-width on mobile. All three maintain minimum 44px touch height.

### Empty-Collection Edge (D-06)

When `viewer.collection.length === 0`: skip the verdict card entirely. Flow advances `extracting` → `verdict-ready` showing only the spec preview + 3 buttons. No `<CollectionFitCard>` renders. Add the copy: "Your collection is empty — fit score not available yet." as a `text-sm text-muted-foreground` note above the 3-button row.

### Working State Copy (D-07)

Single "Working..." state renders a centered inline spinner + text during extraction + verdict compute. Copy: `"Extracting + computing fit..."` — 14px/400 IBM Plex Sans, `text-muted-foreground`. Do NOT show two separate progress states; single reveal when both resolve.

## Accessibility Contract

- All three decision buttons (`Add to Wishlist`, `Add to Collection`, `Skip`) have explicit `aria-label` values matching their visible text
- Working/extracting state: `aria-busy="true"` on the form root; spinner has `aria-hidden="true"`; visible "Extracting + computing fit..." text serves as the live announcement
- `aria-live="polite"` region wraps the flow section below the paste input — announces state changes (verdict ready, extraction failed) without screen-reader interruption
- Wishlist rationale textarea: `aria-describedby` pointing to the pre-filled verdict rationale hint text
- Recently evaluated chips: `role="list"` parent, each chip is `<button>` with `aria-label="{brand} {model} — re-evaluate"`
- Extraction-failed state: `role="alert"` on the error banner so screen readers announce immediately
- Search accordion CTAs (D-04): buttons are inside `Accordion.Panel` — tab order follows base-ui's built-in focus management; no custom tab trap needed
