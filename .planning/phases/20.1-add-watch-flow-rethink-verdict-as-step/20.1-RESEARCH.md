# Phase 20.1: Add-Watch Flow Rethink + Verdict-as-Step — Research

**Researched:** 2026-04-29
**Domain:** Next.js 16 App Router state-driven flow orchestration, verdict consumption, Server Action commit paths
**Confidence:** HIGH

## Summary

Phase 20.1 is **almost entirely orchestration over fully-locked substrate** that landed in Phases 17, 19, 19.1, and 20. The verdict pipeline (`getVerdictForCatalogWatch` Server Action, `<CollectionFitCard>` pure renderer, `useWatchSearchVerdictCache` cache hook, `<VerdictSkeleton>` loading state) is byte-locked and ready to consume. The `addWatch` Server Action accepts `status: 'wishlist' | 'owned'` plus optional `notes` — both Wishlist and Collection commit paths flow through it without server changes. The only server-side mutation in this phase is **D-08: extend `/api/extract-watch` to include `catalogId` in its JSON response** so the client can immediately pivot from extraction → verdict compute on a single round-trip.

The phase's hard work is on the client: a state machine over `idle | extracting | verdict-ready | form-prefill | submitted-wishlist | skipped | extraction-failed`, mounted at `/watch/new`, that reuses Phase 15 WYWT's discriminated-union pattern. The existing `<UrlImport>` component's "Apply to Form" callback contract goes away — `<UrlImport>` either splits into `<PasteSection>` + `<VerdictStep>` + `<DecisionButtons>` or is replaced entirely by a new `<AddWatchFlow>` orchestrator. `<WatchForm>` stays unchanged for the manual-entry path (D-03) and gets a new prefilled-Collection mode (D-12) — the form already accepts a `watch?` prop in `'create'` mode that can hold extracted data.

`/search?tab=watches` accordion (D-04) and `/catalog/[catalogId]` (D-05) sprout 3 inline CTAs that call the same `addWatch` Server Action with prefilled `catalogId`-keyed payloads. The `AddWatchCard` text changes (D-15) and a new `AddToWishlistCard` ships on `WishlistTabContent` (D-16). Recently-evaluated rail (D-14) is `useState` only — no persistence, no DB.

**Primary recommendation:** Implement the state machine as a discriminated union in a single `<AddWatchFlow>` Client Component at `/watch/new`. Mirror Phase 15 WYWT's `step` pattern. Extend `/api/extract-watch` response shape to carry `catalogId`. Reuse `getVerdictForCatalogWatch`, `useWatchSearchVerdictCache`, `<CollectionFitCard>`, `<VerdictSkeleton>`, `addWatch` verbatim. No new Server Actions needed.

### Pattern 1: State Machine via Discriminated Union (Phase 15 WYWT mirror)

**What:** Use a single `useState` whose value is a discriminated union over `kind`. State transitions happen via `setState` calls inside event handlers and inside `useTransition` callbacks for async work. No state machine library needed at this scale.

**When to use:** The entire `<AddWatchFlow>` orchestrator. Phase 15 `WywtPostDialog` uses a 2-step `'picker' | 'compose'` variant `[VERIFIED: src/components/wywt/WywtPostDialog.tsx:60]`. Phase 20.1 needs more states but the pattern scales.

**Example shape:**

```typescript
// Source: derived from Phase 15 WYWT pattern; shape matches CONTEXT.md D-01
type FlowState =
  | { kind: 'idle' }
  | { kind: 'extracting'; url: string }
  | {
      kind: 'verdict-ready'
      catalogId: string
      extracted: ExtractedWatchData
      verdict: VerdictBundle | null  // null when collection.length === 0 (D-06)
    }
  | { kind: 'form-prefill'; extracted: ExtractedWatchData; catalogId: string | null }
  | { kind: 'wishlist-rationale-open'; verdict: VerdictBundle | null; extracted: ExtractedWatchData; catalogId: string | null }
  | { kind: 'submitting-wishlist'; /* same payload */ }
  | { kind: 'submitting-collection' }
  | { kind: 'manual-entry' }
  | { kind: 'extraction-failed'; partial: ExtractedWatchData | null; reason: string }

const [state, setState] = useState<FlowState>({ kind: 'idle' })
const [, startTransition] = useTransition()

// On Extract click:
startTransition(async () => {
  setState({ kind: 'extracting', url })
  try {
    const res = await fetch('/api/extract-watch', { method: 'POST', body: JSON.stringify({ url }) })
    const data = await res.json()
    if (!res.ok) {
      setState({ kind: 'extraction-failed', partial: null, reason: data.error })
      return
    }
    // D-08 — catalogId in response
    if (!data.catalogId) {
      // No catalog upsert (likely missing brand+model) → still go to verdict-ready with no verdict
      setState({ kind: 'verdict-ready', catalogId: '', extracted: data.data, verdict: null })
      return
    }
    // Compute verdict via Server Action (D-09 fallback handled inside composer)
    const v = await getVerdictForCatalogWatch({ catalogId: data.catalogId })
    setState({
      kind: 'verdict-ready',
      catalogId: data.catalogId,
      extracted: data.data,
      verdict: v.success ? v.data : null,
    })
  } catch (err) {
    setState({ kind: 'extraction-failed', partial: null, reason: String(err) })
  }
})
```

### Pattern 3: Cache-Reuse via `useWatchSearchVerdictCache`

**What:** The `useWatchSearchVerdictCache(collectionRevision)` hook returns a per-mount Map of `catalogId → VerdictBundle`. Caller does cache.get(id) before invoking the Server Action; on hit, render instantly. On miss, fire Server Action and `cache.set(id, bundle)` on success.

**For 20.1:** The same hook is reused at `/watch/new`. The collectionRevision must be passed in as a prop from the page Server Component (mirror `/search/page.tsx:38`'s pattern: `viewerCollection.length`). When the user re-pastes the same URL or clicks a "Recently evaluated" chip, the cache hit returns the prior bundle instantly.

### Pattern 4: ADD-06 Inline 3-CTAs Inside Accordion Panel

**What:** D-04 says the 3 buttons render INSIDE `<Accordion.Panel>` below `<CollectionFitCard>`. The base-ui `<Accordion.Trigger>` absorbs whole-row clicks. Pitfall 6 (Phase 20 RESEARCH) — the trigger swallows the click.

**Solution:** Buttons must call `e.stopPropagation()` in their click handlers. The buttons live inside the panel (NOT inside the trigger), so they're safely outside the trigger's click target — but propagation can still bubble to the Accordion.Root's `onValueChange`. Use `onClick={(e) => { e.stopPropagation(); handleAdd(...); }}`.

**Wishlist commit from /search:** Calls `addWatch` directly with `{ status: 'wishlist', brand, model, reference, ...spec }` reconstituted from `SearchCatalogWatchResult` (already on hand). Toast on success. Stay on `/search`.

**Collection commit from /search:** Two flavors per CONTEXT D-04 / D-12:
- (a) **Inline commit** — same `addWatch` call but with `status: 'owned'` and no chance to fill ownership fields (`pricePaid`, `acquisitionDate`). Followed by `router.push('/u/{username}')` or `/`. Simpler; loses the ownership-data-entry affordance.
- (b) **Navigate to prefilled form** — `router.push('/watch/new?catalogId=xxx&intent=owned')`. The flow at `/watch/new` reads the searchParams, fetches catalog data from `getCatalogById(catalogId)`, advances the state machine to `form-prefill`. More clicks but preserves D-12's "ownership-only fields entry" semantic.
- **Recommendation:** Pick (b). It preserves D-03 / D-12 / D-15 coherence — the user always lands in the same prefilled form regardless of entry surface. Path (a) creates a mute commit with no chance to enter a price paid, which is a real loss for collectors.

## Common Pitfalls

### Pitfall 1: Next.js 16 `params` / `searchParams` are Promises

**What goes wrong:** Treating `searchParams` as a synchronous object — destructuring without `await` — produces `undefined` and breaks deep-link prefill.

**Why it happens:** Pre-15 muscle memory; many web examples on the internet show synchronous access.

**How to avoid:** Always `await searchParams` (or use `React.use(searchParams)` from a Client Component if needed). `[CITED: node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/page.md]` `[VERIFIED: src/app/catalog/[catalogId]/page.tsx:39 uses this pattern correctly]`.

**Warning signs:** TypeScript error "Property 'catalogId' does not exist on type 'Promise<...>'". `console.log(searchParams)` shows `Promise { ... }`.

### Pitfall 2: `<Accordion.Trigger>` swallows panel button clicks

**What goes wrong:** When the 3 CTA buttons are placed inside `Accordion.Panel` for D-04, clicking them can bubble up to the Accordion root and toggle the panel collapsed.

**Why it happens:** base-ui's Accordion uses event bubbling for keyboard + click handling.

**How to avoid:** Each button's `onClick` calls `e.stopPropagation()`. Phase 20's WatchSearchRowsAccordion already encountered the inverse (Pitfall 6) — the row's contents have `pointer-events-none` so the trigger absorbs clicks. For our buttons, we want the opposite — they should trap their own clicks. `[VERIFIED: src/components/search/WatchSearchRowsAccordion.tsx:74 uses preventDefault on Escape; same pattern applies]`.

**Warning signs:** Clicking "Add to Wishlist" inside a panel collapses the panel.

### Pitfall 3: Cache invalidation after commit

**What goes wrong:** User pastes URL → verdict caches → clicks "Add to Wishlist" → commits → returns to `/watch/new` and pastes the same URL → cached verdict shows stale collection-fit (the new wishlist entry isn't reflected).

**Why it happens:** `useWatchSearchVerdictCache` keys by `collectionRevision` (collection length). Adding a wishlist row increments collection length, so the cache *should* drop. **BUT** the `<AddWatchFlow>` component must re-mount or receive an updated `collectionRevision` prop for the hook to see the new revision.

**How to avoid:** After `addWatch` succeeds, call `router.refresh()` (NOT `router.push('/watch/new')`). `router.refresh()` re-runs the page Server Component, which re-fetches `getWatchesByUser` and passes the new length down. Phase 9 FollowButton does this exact pattern `[VERIFIED: src/components/profile/FollowButton.tsx:98]`. **OR** simply `router.push('/u/{username}')` to land elsewhere — D-12 says Collection commit lands on `/watch/[id]`, D-13 says Wishlist commit toasts + resets-or-navigates per planner's call.

**Warning signs:** Collection adds don't show updated fit on subsequent paste of same URL.

### Pitfall 4: `useTransition` is required for the Server Action click handler

**What goes wrong:** Calling `await addWatch(...)` directly without `useTransition` wrapping leaves the UI without a pending state during the (potentially multi-second) commit. The button can be double-clicked.

**Why it happens:** Server Actions in Next.js 16 don't auto-wire pending state for click-handler invocations (only `<form action={...}>` invocations get `useFormStatus`).

**How to avoid:** Always `const [pending, startTransition] = useTransition()` and call inside `startTransition(async () => { ... })`. Disable the button on `pending`. **All Phase 13/15/19/20 server actions follow this contract** `[VERIFIED: WatchForm.tsx:127, ComposeStep.tsx:104, FollowButton.tsx]`.

**Warning signs:** Buttons stay enabled during commit; double-clicks duplicate watches.

### Pitfall 5: Stripping the verdict copy in `notes` when blank

**What goes wrong:** D-13 says wishlist rationale textarea is pre-filled with verdict headline AND user can blank it. If we naively send `formData.notes = textarea.value || verdictHeadline`, blanking doesn't actually save empty notes — the verdict copy ghosts back in.

**Why it happens:** Defensive `||` fallback in form handler.

**How to avoid:** Send the textarea value verbatim, including empty string. Pre-fill happens on render only; commit reads what the user actually has.

```typescript
// CORRECT
const notes = textarea.value  // '' is valid

// WRONG
const notes = textarea.value || verdictHeadline
```

**Warning signs:** Wishlist entries always have notes even when user explicitly cleared the field.

### Pitfall 6: `addWatch` photoSourcePath ownership check rejects unauthenticated paths

**What goes wrong:** `addWatch` validates that `photoSourcePath` first segment matches the current user's id `[VERIFIED: src/app/actions/watches.ts:75-80]`. URL-extract path has no `photoSourcePath` (Phase 19.1 D-19 forbids it from URL surface), so this is fine. But if the planner's prefilled-Collection path tries to copy `photoSourcePath` from somewhere, it'll fail validation.

**How to avoid:** URL-extract paths NEVER set `photoSourcePath`. Manual-entry path is the ONLY surface that ships a photo blob. Plan-time check: prefilled-Collection commit (D-12) does NOT pass `photoSourcePath`.

**Warning signs:** Collection commit returns "Photo path does not match authenticated user" error.

### Pitfall 7: `verdict.framing` MUST be 'cross-user' from `getVerdictForCatalogWatch`

**What goes wrong:** The Server Action hardcodes `framing: 'cross-user'` `[VERIFIED: src/app/actions/verdict.ts:64]`. If `<AddWatchFlow>` is computing verdicts for a user evaluating a watch they may eventually own, the same hardcoded framing flows. This is correct: the watch isn't owned yet, so cross-user framing fits.

**How to avoid:** Don't second-guess the framing. The verdict bundle is right as-is for the verdict-step UI.

**Warning signs:** None — this is informational.

### Pitfall 8: Empty-collection edge case (D-06) requires `verdict === null` handling

**What goes wrong:** If `viewer.collection.length === 0`, the Server Action `getVerdictForCatalogWatch` does compute a bundle (composer doesn't error) but the result is structurally empty (no `mostSimilar`, no contextual phrasings beyond the fixed-label fallback). D-06 explicit: skip the verdict card entirely in this case.

**How to avoid:** In `<AddWatchFlow>`, after extraction, check `collection.length === 0` (passed in as `collectionRevision === 0` from Server Component). Skip the `getVerdictForCatalogWatch` call and set `state = { kind: 'verdict-ready', verdict: null, ... }`. `<VerdictStep>` renders the spec preview + 3 buttons + the "Your collection is empty — fit score not available yet." copy from UI-SPEC. `[VERIFIED: UI-SPEC.md line 226]`.

**Warning signs:** Empty-collection users see a confusing "Hard Mismatch" verdict on every paste.

### Pitfall 9: `extracted-watch.styleTags` / `roleTags` should NOT carry forward to the prefilled form

**What goes wrong:** Phase 19.1 D-18 hides `styleTags` / `roleTags` / `designTraits` pickers in `<WatchForm>`. If the URL extraction populates these arrays and we naively spread them into the form, those fields ARE in the watch row but the user can't see/edit them.

**How to avoid:** This is actually correct behavior — the extracted values commit to `watches.styleTags` / `roleTags` / `designTraits` columns. The user just can't edit them in the form (per Phase 19.1 lock). Plan should accept this. The arrays are still useful for `analyzeSimilarity()`. **Don't strip them.**

**Warning signs:** None — this is informational; reminding planner not to "fix" the hidden-pickers behavior.

### Pitfall 10: `addWatch`'s "explore" tag revalidation triggers cross-user cache fan-out

**What goes wrong:** `addWatch` does `revalidateTag('explore', 'max')` `[VERIFIED: src/app/actions/watches.ts:251]`. This invalidates Trending + Gaining Traction rails. After commit + redirect, returning to `/explore` will re-render. This is correct.

**Mind:** Don't re-add `revalidateTag` calls in your phase code — `addWatch` already handles fan-out.

```typescript
// Source: src/components/search/WatchSearchRowsAccordion.tsx:53-69 pattern
import { useWatchSearchVerdictCache } from '@/components/search/useWatchSearchVerdictCache'

function AddWatchFlow({ collectionRevision }: { collectionRevision: number }) {
  const cache = useWatchSearchVerdictCache(collectionRevision)
  const [, startTransition] = useTransition()

  const handleExtract = (url: string) => {
    setState({ kind: 'extracting', url })
    startTransition(async () => {
      const res = await fetch('/api/extract-watch', { method: 'POST', body: JSON.stringify({ url }) })
      const data = await res.json()
      if (!res.ok || !data.catalogId) {
        // D-18 — extraction-failed branch; keep partial data if available
        setState({ kind: 'extraction-failed', partial: data?.data ?? null, reason: data.error ?? 'Unknown' })
        return
      }
      // Cache hit?
      const cached = cache.get(data.catalogId)
      if (cached) {
        setState({ kind: 'verdict-ready', verdict: cached, catalogId: data.catalogId, extracted: data.data })
        return
      }
      // Empty collection → skip verdict (D-06)
      if (collectionRevision === 0) {
        setState({ kind: 'verdict-ready', verdict: null, catalogId: data.catalogId, extracted: data.data })
        return
      }
      const v = await getVerdictForCatalogWatch({ catalogId: data.catalogId })
      const bundle = v.success ? v.data : null
      if (bundle) cache.set(data.catalogId, bundle)
      setState({ kind: 'verdict-ready', verdict: bundle, catalogId: data.catalogId, extracted: data.data })
    })
  }
  // ...
}
```

## Security Domain

> This phase touches user mutations and an authenticated Server Action surface. Security review is in scope.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | All Server Actions (`addWatch`, `getVerdictForCatalogWatch`) gate on `getCurrentUser()`; `/api/extract-watch` route gates the same. UnauthorizedError → return ActionResult `{ success: false, error: 'Not authenticated' }` (no throw across boundary). `[VERIFIED: src/app/actions/watches.ts:62, src/app/actions/verdict.ts:36-40, src/app/api/extract-watch/route.ts:11-17]` |
| V3 Session Management | yes | Supabase SSR cookie-based session; `getCurrentUser()` reads + verifies on every request. No additional change needed in this phase. |
| V4 Access Control | yes | `addWatch` reads `userId` from session — never accepted from input (D-02 cross-phase decision). `photoSourcePath` first segment must equal authenticated user id `[VERIFIED: src/app/actions/watches.ts:75-80]`. `getVerdictForCatalogWatch` reads viewer's collection by session user id only. |
| V5 Input Validation | yes | All Server Actions validate inputs via zod. `addWatch`'s `insertWatchSchema` enforces status enum, brand/model presence, photoSourcePath regex. `getVerdictForCatalogWatch` enforces UUID for catalogId via `.strict().uuid()`. `[VERIFIED]` `/api/extract-watch` validates URL parses + protocol allowlist (http/https only) `[VERIFIED: src/app/api/extract-watch/route.ts:30-45]`. SSRF protected via `safeFetch` from `@/lib/ssrf`. |

### Known Threat Patterns for Next.js + React + Supabase + Server Actions stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Mass-assignment via Server Action body | Tampering | zod `.strict()` on schemas; `addWatch` uses `safeParse` + summary error reporting; `getVerdictForCatalogWatch` uses `.strict()` `[VERIFIED]` |
| User-controlled URL paste → SSRF | Information Disclosure | `safeFetch` from `@/lib/ssrf`; protocol allowlist check; existing tests `tests/ssrf.test.ts` |
| Client-supplied user_id in addWatch payload | Tampering | `addWatch` reads `user.id` from `getCurrentUser()` only; payload `userId` would be silently dropped (not in schema) |
| Cross-user catalog evaluation leaking another user's watches | Information Disclosure | `getVerdictForCatalogWatch` queries viewer's own collection only; never reads other users' `watches` rows. Verified at Phase 20 D-08 layer in `findViewerWatchByCatalogId` (scope by both userId AND catalogId). |
| Verdict cache pollution across users | Information Disclosure | `useWatchSearchVerdictCache` is per-mount (component state); each user has own browser session. Cache cannot leak across users. |
| Sonner toast injection (XSS via verdict copy) | Tampering | All verdict phrasings are template-substituted server-side from a fixed library (`src/lib/verdict/templates.ts` Phase 20 lock). Phrasings never embed user-controlled HTML. Sonner `toast.success(string)` accepts string only — React renders text safely. |
| Wishlist `notes` storing arbitrary user input | Tampering / Storage | `notes` is a TEXT column rendered as React children — no `dangerouslySetInnerHTML` consumer. Existing patterns OK. zod schema on `addWatch` accepts `notes: z.string().optional()` — no length cap currently; planner should consider one. (Existing `WatchForm` uses unbounded `<Textarea>`; pre-existing risk.) |
| Deep-link prefill (`/watch/new?catalogId=xxx`) catalogId tampering | Tampering | Server Component reads `searchParams.catalogId`, passes to `getCatalogById(catalogId)`. If catalogId is malformed UUID or not found, `getCatalogById` returns null and the form-prefill state simply doesn't load (graceful). Validated at the DAL layer. |
| Race: extract returns catalogId, user navigates away before verdict resolves | Threat | `useTransition` already prevents UI updates after unmount in React 19. State updates after unmount are no-ops. |

**Phase-specific security review items:**

1. **`addWatch` notes length cap** — current schema has `notes: z.string().optional()` with no max length. With D-13 verdict-copy pre-fill, users pasting other content into notes could store arbitrary-length strings. Recommend planner adds `.max(2000)` or similar. Pre-existing concern; not strictly Phase 20.1's to fix, but flag.

2. **`/watch/new?intent=owned` deep-link from /search** — Server Component must validate `searchParams.intent === 'owned'` strictly (whitelist) before passing to client. Free-string intent could lead to unexpected state-machine transitions.

3. **D-08 `catalogId` in response** — string already exists in scope; no new disclosure. The catalog row is public-readable (Phase 17 RLS) so exposing the ID does not leak protected data.
