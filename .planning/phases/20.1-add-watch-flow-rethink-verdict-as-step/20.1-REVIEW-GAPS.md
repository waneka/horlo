## Summary

This review covers gap-closure work from Plans 06 (null-verdict UX + observability), 07 (accordion expand affordance + Link/chevron split + All-tab unification), and 08 (manual-entry escape affordance). No Critical issues found. Auth gating, input validation, server-side error sanitization, and SSRF protection in the extract route are correctly placed. The Server Action uses `z.strict()` + UUID validation to block mass-assignment and bad input. Observability around the silent null-verdict paths is well-instrumented with stable error strings.

The four Warnings concern small but real correctness/UX issues: (1) re-entrancy on the chevron-only "Add to Collection" navigation that has no committing-state UX; (2) wishlist-commit error path drops user-typed notes on rollback; (3) response-shape spread fragility in the extract route where `...result` could shadow the `success` flag if `fetchAndExtract` ever returns one; (4) brittle `getElementById` DOM coupling for the auto-focus behavior. The Info findings flag style/maintainability nits — magic numbers, an inline IIFE inside JSX, an over-broad aria-live region, and a couple of minor typing/extraction opportunities.

None of these block phase completion. The Warnings are worth addressing in a follow-up cleanup pass; the Info items can stay as-is.

## Warnings

### WR-01: `handleAddToCollection` navigates without committing-state UX or re-entrancy guard

**File:** `src/components/search/WatchSearchRowsAccordion.tsx:103-105`
**Issue:** When the user clicks the "Add to Collection" button inside an open accordion panel, the handler immediately calls `router.push(...)` with no `committingId`/`committingTarget` set and no disabled state. This diverges from the sibling "Add to Wishlist" button (lines 73-101), which does set both. A user who double-clicks (touch devices, slow network) can fire two navigations back-to-back. While `router.push` is generally idempotent, the button never visually acknowledges the click (no "Saving..." or spinner), so users may click again thinking the first click missed.

It also means `disabled={committingId !== null}` (line 170) is the only guard — and it's only set by the wishlist path. So the wishlist-in-flight state correctly disables Collection, but Collection-in-flight does not disable Wishlist.

**Fix:** Either add a navigation-pending state, or — at minimum — disable the button the moment it's clicked:

```tsx
const handleAddToCollection = (r: SearchCatalogWatchResult) => {
  if (committingId) return
  setCommittingId(r.catalogId)
  setCommittingTarget('collection')
  // committingId persists through the navigation; the unmount cleans it up.
  router.push(`/watch/new?catalogId=${encodeURIComponent(r.catalogId)}&intent=owned`)
}
```

And mirror the wishlist button's pending UX (Loader2 + "Opening...") inside the Collection button.

### WR-02: Wishlist-commit failure rollback discards user-typed notes

**File:** `src/components/watch/AddWatchFlow.tsx:248-269`
**Issue:** When `addWatch` fails inside `handleWishlistConfirm`, the rollback at lines 261-267 transitions back to `wishlist-rationale-open` with the original `captured` snapshot — but `captured` doesn't carry the `notes` the user just typed. Their input is lost; they must retype on retry. Pitfall 5 explicitly notes notes are forwarded verbatim, but the retry path doesn't preserve them.

This is especially painful because the failure mode the user is most likely to encounter (network blip, auth lapse) is exactly the case where they'd want their notes back.

**Fix:** Either thread `notes` into the rolled-back state so `WishlistRationalePanel` can re-prefill, or keep the `submitting-wishlist` shape including notes and have the panel read them:

```tsx
} else {
  toast.error(result.error)
  setState({
    kind: 'wishlist-rationale-open',
    catalogId: captured.catalogId,
    extracted: captured.extracted,
    verdict: captured.verdict,
    initialNotes: notes, // new field; panel reads it once on mount/state change
  })
}
```

This requires a small extension to the `wishlist-rationale-open` `FlowState` variant and a one-line change in `WishlistRationalePanel` to seed its textarea from `initialNotes`.

### WR-03: `...result` spread can shadow `success: true` if `fetchAndExtract` ever returns a `success` field

**File:** `src/app/api/extract-watch/route.ts:134-142`
**Issue:** The response is built as:

```ts
return NextResponse.json({
  success: true,
  catalogId,
  catalogIdError,
  ...result,
})
```

Property-spread order means later keys win. If `fetchAndExtract` ever returns an object that includes a `success` (or `catalogId`, or `catalogIdError`) field — even unintentionally during a refactor — the explicitly-set values above are silently overwritten. Today this works because `result` happens not to carry those keys, but the contract is implicit.

The client at `AddWatchFlow.tsx:126` uses `if (!res.ok)` — which keys off HTTP status, not the body's `success` — so the immediate UX is not broken. But it's a latent footgun for any caller that does check `body.success`, and `catalogId`/`catalogIdError` are read directly by the client (`AddWatchFlow.tsx:135-138`) so a future shadow there would silently break observability.

**Fix:** Pin the explicit keys after the spread (or, preferably, name the inner payload):

```ts
return NextResponse.json({
  ...result,
  success: true,
  catalogId,
  catalogIdError,
})
```

or restructure so the extracted payload is namespaced:

```ts
return NextResponse.json({
  success: true,
  catalogId,
  catalogIdError,
  data: result.data,
  // ... other named fields the client actually reads
})
```

The second form makes the contract explicit and would also let you drop the client-side `data.data ?? {}` ambiguity at `AddWatchFlow.tsx:134`.

### WR-04: Brittle direct-DOM auto-focus via `getElementById('paste-url')`

**File:** `src/components/watch/AddWatchFlow.tsx:87-92`
**Issue:** The `useEffect` reaches into the DOM via `document.getElementById('paste-url')` to focus the URL input on transitions back to idle. This couples `AddWatchFlow` to an implementation detail of `PasteSection` (the `id="paste-url"` attribute) without any compile-time enforcement. If `PasteSection` is ever renamed, restructured, or wrapped in a different component, the focus behavior silently breaks — and there's no test that fails loudly because the missing element returns `null`, the `?.focus()` no-ops, and nothing announces the regression.

**Fix:** Lift the input ref into `AddWatchFlow` and pass it as a prop to `PasteSection`:

```tsx
const inputRef = useRef<HTMLInputElement>(null)

useEffect(() => {
  if (state.kind === 'idle') {
    inputRef.current?.focus()
  }
}, [state.kind])

// then
<PasteSection
  ref={inputRef}
  ...
/>
```

Or expose an imperative `focus()` handle via `useImperativeHandle` if `PasteSection` already wraps the input in additional structure. Either way, the link is type-checked.

**File:** `src/components/watch/VerdictStep.tsx:59`
**Issue:** The polite live region wraps the spec card, the verdict (or fallback), AND the 3-button row. When the verdict transitions from skeleton to ready, screen readers will announce the brand+model heading, the spec headline, the verdict copy, and all three button labels in sequence. ### IN-04: `data.data ?? {}` empty-fallback obscures shape contract

**File:** `src/components/watch/AddWatchFlow.tsx:134`
**Issue:** `const extracted: ExtractedWatchData = data.data ?? {}` casts an empty object to `ExtractedWatchData`. The type system claims every field is present (per the type definition), but `{}` has none. Downstream code at `extractedToPartialWatch` (line 468) and `buildAddWatchPayload` (line 500) handles this defensively with `?? ''` / `?? []` fallbacks, so it works — but it does so by re-checking each field. The cast is technically a lie that the rest of the code carefully papers over.

**Fix:** Either widen the type to `Partial<ExtractedWatchData>` at this boundary (most honest), or drop the fallback and surface a parse error if `data.data` is missing — the API route always returns `data` on the success branch, so a missing `data.data` is a genuine contract violation worth flagging:

```ts
if (!data.data) {
  setState({
    kind: 'extraction-failed',
    partial: null,
    reason: 'Server returned no extracted data',
  })
  return
}
const extracted: ExtractedWatchData = data.data
```

### IN-05: `synth:` rail-id format risks collision with user-content if catalog UUIDs ever change shape

**File:** `src/components/watch/AddWatchFlow.tsx:219-221`
**Issue:** `synth:${brand.trim().toLowerCase()}|${model.trim().toLowerCase()}` produces a stable id from brand+model. The comment correctly notes cache lookups will miss intentionally. Today this is safe because real catalog ids are UUIDs (no `synth:` prefix). It's worth keeping a mental note that any future change to catalog id format must preserve the prefix exclusion — there's no test that pins the contract.

A second tiny concern: brand and model could in theory contain a literal `|` character, which would make two distinct brand/model pairs collide on the rail. Vanishingly unlikely for real brand strings, but worth using a separator that can't appear in either field, or URL-encoding the parts.

**Fix:** No code change required if the team accepts the assumption. If you want belt-and-braces, encode the parts:

```ts
const railId = state.catalogId
  ? state.catalogId
  : `synth:${encodeURIComponent(brand.trim().toLowerCase())}|${encodeURIComponent(model.trim().toLowerCase())}`
```
