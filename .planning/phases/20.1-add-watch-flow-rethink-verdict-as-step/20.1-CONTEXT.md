# Phase 20.1: Add-Watch Flow Rethink + Verdict-as-Step — Context

**Gathered:** 2026-04-29
**Status:** Ready for planning

<domain>
## Phase Boundary

The add-watch flow at `/watch/new` is reorganized so URL-paste → `<CollectionFitCard>` verdict preview → 3-button decision (Wishlist / Collection / Skip) is one coherent gesture on a single state-driven page. The Skip path absorbs the lightweight evaluate-only use case (paste a watch found elsewhere, see fit, bail) — there is NO separate `/evaluate` route. The `/search?tab=watches` accordion (Phase 20 FIT-04) gains the same 3 CTAs inline, prefilled with `catalogId` instead of URL. The `/catalog/[catalogId]` page (Phase 20 D-10) mirrors the same CTAs in cross-user framing. Manual entry is preserved as a quiet secondary affordance via an inline link below the URL input.

**In scope (ADD-01..07):**
- Rework `/watch/new` from "form with UrlImport at top + Apply to Form" into a state-driven 3-section flow: Paste → Verdict → 3-button → optional prefilled Form (Collection path)
- Ship the Wishlist-with-verdict-rationale path (editable textarea pre-filled with verdict copy)
- Ship the Collection-as-prefilled-form path with status='owned' default and ownership-field entry
- Ship the Skip path with reset-and-refocus + in-session "Recently evaluated" rail
- Ship the manual-entry inline-link affordance (skips verdict step)
- Wire `/search` accordion CTAs (ADD-06): inline "Add to Wishlist" / "Add to Collection" / Skip-via-collapse using same Server Actions
- Wire `/catalog/[catalogId]` cross-user CTAs (ADD-06 extension): same 3-button pattern in cross-user framing (D-08 self-via-cross-user keeps existing "You own this" UI)
- Extraction-failure fallback (ADD-07): preserve any partial extracted data and offer manual continuation — no dead-ends
- Profile tab end-of-grid CTA copy: AddWatchCard → "Add to Collection"; new wishlist-tab card → "Add to Wishlist"
- Page heading on `/watch/new` reframes for evaluate-aware copy

**Out of scope (deferred):**
- Empty-state CTA copy variations on the four empty grids (Phase 25 UX-01..04 owns these — Collection / Wishlist / Worn / Notes empty states)
- Categorized URL-extract error copy (Phase 25 UX-05 — `host-403` / `structured-data-missing` / `LLM-timeout` / `quota-exceeded` / `generic-network`); ADD-07 in this phase ships a single unified fallback, not 5 categorized variants
- Persistent "Recently evaluated" history (this phase: in-session only, last 5; persistent surface is v4.x deferred)
- LLM-generated verdict copy (FIT-06, v5+ — substrate uses Phase 19.1 D-14 confidence gate + Phase 20 template composer)
- A/B testing infra for FIT-02 phrasings (v5+)
- `analyzeSimilarity()` engine body changes (v5+ — Phase 20 D-09 byte-lock still in force)
- WatchForm `notesPublic` and `isChronometer` UI surfacing (Phase 23 FEAT-07/08)
- Top-nav CTA rename (Phase 20.1 keeps `+` / "Add watch" — too broad to scope-creep)
- TOS / acceptable-use policy for user-uploaded photos (Phase 19.1 deferred; pre-multi-user-launch concern)

<decisions>
## Implementation Decisions

### Flow Architecture

- **D-01: In-place at `/watch/new`, state-driven sections.** Single page, no multi-route wizard, no formal stepper UI. Sections appear / collapse based on a state machine: `idle` (paste input visible) → `extracting` (single "Working..." spinner) → `verdict-ready` (verdict card + 3-button) → optional `form-prefill` (Collection path only) OR `submitted-wishlist` (toast + reset) OR `skipped` (reset + focus). No URL/route changes; `/watch/new` remains the canonical entry. Honors the ROADMAP "single coherent gesture" framing without adding stepper-shaped friction.

- **D-02: Page heading on `/watch/new` reframes for evaluate-aware copy.** Today: "Add Watch". New copy reframes the three outcomes (e.g. "Add a watch — or just evaluate one"). Final exact copy is Claude's discretion at plan time; constraint is that the heading must hint at the Skip outcome.

- **D-03: Manual entry as inline link below URL input.** Quiet text link "or enter manually" beneath the paste field. Click → toggles to manual-only form (skips verdict step entirely per ADD-05). URL paste is the dominant affordance; manual is the rare flow (users typing watches they already own).

- **D-04: ADD-06 catalog deep-link from `/search` accordion — inline 3 CTAs inside the expanded verdict.** Buttons render directly inside the `WatchSearchRowsAccordion` panel below the verdict card. Wishlist + Collection clicks call the same Server Actions used by `/watch/new`, prefilling `catalogId` (not URL). Skip = collapse (the existing accordion behavior). Wishlist commit lands user with toast on `/search`; Collection commit routes user to a prefilled form path (D-09 logic) or directly to `/watch/[id]` per Collection-path decision below.

- **D-05: `/catalog/[catalogId]` page mirrors the same 3-button CTAs in cross-user framing.** When the page is NOT in `self-via-cross-user` framing (Phase 20 D-08), the same 3 CTAs render below `<CollectionFitCard>`. When the viewer already owns this catalog ref (D-08), the current "You own this" callout stays unchanged. Skip on this surface = navigate back (no paste field to reset).

- **D-06: Empty-collection edge — skip the verdict step entirely.** When `viewer.collection.length === 0` (Phase 20 D-07), the flow advances Paste → extracted-specs preview → 3 buttons (no verdict card). Honors D-07 "fit is meaningless without a collection signal" — first-watch-ever flow stays fast and uncluttered. Skip behavior is identical to non-empty case.

### Verdict Timing & Taste Latency

- **D-07: Sync wait + single reveal.** When user clicks Extract, render a single "Working..." state with copy that hints at the work (e.g., "Extracting watch + computing fit"). Specs preview + verdict card + 3-button section all appear together when the round-trip completes. Acceptable wait: ~5-10s on first paste; faster on cache hits. Single coherent gesture; no two-stage reveal.

- **D-08: `/api/extract-watch` route handler returns `catalogId` in its response.** The route already syncs through catalog upsert + taste enrichment per the existing implementation (despite the "fire-and-forget" comment, the actual code uses `await` for correctness). Extension: include the `catalogId` (string | null) in the JSON response so the client verdict step can call `getVerdictForCatalogWatch({ catalogId })` immediately. No behavioral change to the route; only the response shape grows. The extracted spec payload remains as-is for downstream form prefill.

- **D-09: Enrichment failure → fixed-label fallback per Phase 19.1 D-14.** When taste enrichment times out, hits a 429, or `ANTHROPIC_API_KEY` is missing, the verdict step still renders — using the 6 fixed `SimilarityLabel` values (the `confidence < 0.5` path from Phase 19.1 D-14). The 3 buttons stay functional. **Never block commit on enrichment failure.** This honors the existing fire-and-forget posture's spirit while the route handler's await ensures we wait long enough to know whether enrichment succeeded.

- **D-10: Reuse Phase 20's verdict cache pattern.** Use `useWatchSearchVerdictCache` (or a sibling hook with shared semantics) keyed by `catalogId + collectionRevision`. Re-pasting the same URL or arriving from `/search` inline-expand → instant verdict from cache. Cache invalidates when the viewer mutates their collection (collectionRevision counter bumps). Idle paste-then-skip-then-paste-same-URL cycles pay zero recompute cost.

### 3-Button UX & Commit Paths

- **D-11: Wishlist primary, Collection secondary, Skip tertiary.** Visual hierarchy:
  - "Add to Wishlist" — primary button (filled, accent color)
  - "Add to Collection" — secondary button (outlined)
  - "Skip" — tertiary (ghost button or text link, smaller weight)
  - Rationale: most verdict-driven adds are aspirational ("I want this someday" → wishlist). Collection-as-owned is a less-common path (users typing watches they already own often use manual entry). Skip is a quiet but legitimate outcome.

- **D-12: Add-to-Collection commit path = two-step prefilled form.** Click "Add to Collection" → form section reveals below the verdict on `/watch/new` (or, from `/search` ADD-06, navigates to `/watch/new?catalogId=xxx&intent=owned`), prefilled with extracted spec + `status='owned'`. User fills ownership-only fields (`pricePaid`, `acquisitionDate`, `notes`) and submits. Honors ADD-03 literally; user controls ownership data entry. **Status field is locked to `owned` in this path** (the verdict-step decision IS the status decision; no re-pick).

- **D-13: Add-to-Wishlist commit path = editable textarea with verdict copy pre-filled.** Click "Add to Wishlist" → small inline panel/dialog reveals notes textarea pre-populated with the verdict's headline phrasing (e.g., "Aligns with my heritage-driven taste — overlaps strongly with Submariner"). User can edit the text, blank it entirely, or accept and submit. `status='wishlist'`, extracted spec fields commit alongside notes. **Honors ADD-02 'optionally' literally** — the user controls whether and what verdict copy lands in `notes`.

- **D-14: Skip path = reset + auto-focus URL input + add to in-session "Recently evaluated" rail.** Click "Skip":
  - Verdict + 3-button + form sections collapse
  - URL input clears and gets keyboard focus
  - The just-evaluated `catalogId` (with brand/model/imageUrl thumbnail) is appended to a session-only "Recently evaluated" rail rendered below the input (capped at 5 most recent entries; FIFO eviction)
  - Clicking a chip in the rail re-opens the verdict from cache (D-10) without re-paste
  - **No persistence** — the rail is `useState` only; resets on page nav away or full reload. Persistent history is a deferred v4.x idea.

### Profile Tab CTA Coherence

- **D-15: AddWatchCard on Collection tab text changes from "Add Watch" to "Add to Collection".** The Plus-icon end-of-grid card on `CollectionTabContent.tsx` (currently `<AddWatchCard />`) has its display text + `aria-label` updated. Component file may rename; backwards-compat shim not needed (single callsite owner-only). Still links to `/watch/new` (no `?intent=owned` — user picks status via the in-flow 3-button as today's primary path).

- **D-16: Wishlist tab gets a NEW end-of-grid CTA card matching the AddWatchCard pattern, text "Add to Wishlist".** Currently `WishlistTabContent.tsx` has the comment "'Add to Wishlist' affordances. Not currently used." This phase ships that card. Same Plus-icon dashed-border card UX as `<AddWatchCard />`. Links to `/watch/new` (route entry stays unified — verdict step is shared regardless of which tab the user came from). Phase 25 UX-02 will still own the EMPTY-state Wishlist CTA copy (when the grid is empty); D-16 ships the END-of-grid card on populated wishlists.

- **D-17: Top-nav `+` / `aria-label="Add watch"` stays unchanged.** Already short and intent-broad; doesn't need rework. Per-tab CTAs (D-15, D-16) carry the status-aware semantics where it matters.

### Extraction Failure Recovery (ADD-07)

- **D-18: Single unified fallback this phase; categorized errors defer to Phase 25 UX-05.** When extraction fails, show "Extraction didn't work — fill in manually" with any partial data the extractor recovered (often brand + model from URL slug). User continues into the manual form path (skips verdict). Categorized error copy (`host-403` / `structured-data-missing` / `LLM-timeout` / `quota-exceeded` / `generic-network`) is Phase 25 UX-05 work — Phase 20.1 ships only the unified fallback so the no-dead-end success criterion is met.

### Engine Lock + Type Reuse

- **D-19: `analyzeSimilarity()` and `<CollectionFitCard>` remain byte-locked from Phase 20.** This phase consumes the Phase 20 outputs (`<CollectionFitCard>` pure renderer, `getVerdictForCatalogWatch` Server Action, `useWatchSearchVerdictCache` cache hook) without modification. Engine + composer + viewer-taste-profile + shims are all Phase 20 artifacts; new flow code orchestrates them but does NOT branch their internals.

### Claude's Discretion

The planner can decide:
- Exact page-heading copy on `/watch/new` (D-02 — must hint at evaluate outcome)
- Exact button copy ("Add to Wishlist" vs "Save to Wishlist", "Skip" vs "Just evaluating", etc.) — keep consistent with existing app vocabulary
- Loading-state copy during D-07 sync wait (e.g., "Extracting + computing fit", "Looking up watch...")
- State-machine implementation: bare `useState` + discriminated union, `useReducer`, or a tiny FSM library — whichever stays light
- How the inline manual-entry link is positioned relative to the Extract button (alongside, below, or as a secondary CTA after error)
- Animation/transition for section reveal (instant vs CSS transition vs Framer Motion) — match existing patterns
- Exact "Recently evaluated" rail visual (chips with thumbnail, list with text labels, etc.)
- Whether ADD-06 routing from `/search` Collection path navigates to `/watch/new?catalogId=xxx` or commits inline with a route after — D-04 leaves room for either; pick whichever keeps server-action surface coherent
- Whether D-15 `AddWatchCard` is renamed (e.g. `AddToCollectionCard`) or stays `AddWatchCard` with text props
- Whether D-16's wishlist card is a copy-paste of `AddWatchCard` or shared component with a `variant` prop
- Whether the verdict-rationale textarea (D-13) is a full dialog or inline expand under the Wishlist button
- ADD-07 partial-data carry: which fields the extractor surfaces when extraction "partially fails" (it currently returns whatever was salvageable; planner decides what's useful in the manual fallback)
- Whether `/catalog/[catalogId]` 3 CTAs (D-05) are rendered by the page Server Component or a Client Component child — depends on Server Action call shape

<specifics>
## Specific Ideas

- **"Single coherent gesture"** is the framing anchor — D-01 in-place state-machine, D-07 sync wait + single reveal, and D-11 visual hierarchy all serve this. Two-stage reveals or formal stepper UIs would dilute the gesture.
- **Skip is a first-class outcome.** D-14 makes Skip ergonomic for the evaluate-many use case (browsing watch forums, store pages). The "Recently evaluated" rail is the small productive flourish that makes the evaluate-only path feel intentional, not abandoned.
- **Wishlist verdict-rationale via editable textarea** (D-13) is the user-aligned reading of ADD-02's "optionally captured." Auto-fill silently was rejected as too presumptuous; checkbox was rejected as too ceremonious. Textarea pre-fill that the user can accept, edit, or blank gives them full control with minimum friction.
- **Phase 19.1's fire-and-forget posture is honored in spirit, broken in letter** — the route handler already awaits per the existing implementation. D-08 acknowledges this: extending the response shape is the smallest change that makes verdict-as-step viable, and fixed-label fallback (D-09) covers the failure case so we never block the user on enrichment.
- **Page heading reframe (D-02) without top-nav rename (D-17)** is a deliberate scope choice — the surface that reveals the new flow's outcomes is the page itself; entry CTAs can stay aspirational ("Add watch" as the broad invitation) without misleading.
- **Wishlist tab end-of-grid CTA (D-16)** is a small Phase 25 UX-02 scope-pull; the user explicitly asked for it during discussion. Justification: the wishlist tab is where users most naturally think "I should evaluate this watch I've been eyeing" — having the CTA there is coherent with the new flow's evaluate-then-commit-to-wishlist path. Empty-state copy stays Phase 25 scope.
