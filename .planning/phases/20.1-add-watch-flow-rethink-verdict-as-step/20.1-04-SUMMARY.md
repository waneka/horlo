## Server Component Pattern (Pitfall 1 Mitigation)

```tsx
interface NewWatchPageProps {
  searchParams: Promise<{ catalogId?: string; intent?: string }>
}

export default async function NewWatchPage({ searchParams }: NewWatchPageProps) {
  // 1. Auth gate (defense-in-depth; proxy already gated)
  let user: { id: string; email: string }
  try { user = await getCurrentUser() }
  catch (err) { if (err instanceof UnauthorizedError) redirect('/signin'); throw err }

  // 2. Resolve Promise searchParams — Pitfall 1
  const sp = await searchParams

  // 3. Whitelist intent — T-20.1-04-01
  const initialIntent: 'owned' | null = sp.intent === 'owned' ? 'owned' : null

  // 4. UUID-shape regex on catalogId — T-20.1-04-02 defense-in-depth
  const catalogId = typeof sp.catalogId === 'string' &&
    /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i.test(sp.catalogId)
      ? sp.catalogId : null

  // 5. Parallel data fetch
  const [collection, catalogPrefill] = await Promise.all([
    getWatchesByUser(user.id),
    catalogId ? hydrateCatalogPrefill(catalogId) : Promise.resolve(null),
  ])

  return <AddWatchFlow ... collectionRevision={collection.length} ... />
}
```

## Decisions Made

- **Plain async on extract path (no startTransition)** — startTransition's awaited setStates are deferred; RTL's act() flushes them before findByText polls, so the intermediate 'extracting' render is invisible to tests AND to React's commit cycle. Plain async + a single requestAnimationFrame yield gives the 'Working...' paint a chance to commit before fetch microtasks resolve. The Wishlist commit handler keeps startTransition because its post-commit work (toast + router.refresh) is legitimately deferrable.
- **requestAnimationFrame yield after setState(extracting)** — without this, the 'extracting' render is committed but RTL's act() processes the pending fetch microtask in the same batch, blowing past the intermediate state before findByText can poll. The yield gives findByText a window to observe 'Working...'.
- **`<WatchForm mode="create" lockedStatus="owned" watch={prefilledPartialWatch}>`** — Plan 04 reuses the existing WatchForm rather than introducing a new `<AddToCollectionForm>`. The lockedStatus prop is the surgical addition. RESEARCH §Anti-Patterns explicitly forbids the duplicate component approach.
- **`extractedToPartialWatch`** helper — maps `ExtractedWatchData` (extractor shape with all fields optional) into a `Watch` (domain shape with required brand/model/movement). Used for both Collection prefill and manual-entry partial-data continuation. Defaults movement to 'automatic' when extractor didn't extract it.
- **`buildAddWatchPayload`** helper — builds the `addWatch` payload for the Wishlist path. Critically does NOT set `photoSourcePath` (Pitfall 6); critically passes notes verbatim including '' (Pitfall 5).

## Deviations from Plan

**1. [Rule 1 - Bug] Plain async on extract path replaces plan's `startTransition` recommendation**

- **Found during:** Task 2 (running ADD-01 test which expects findByText(/Working/i) to observe the extracting state)
- **Issue:** The plan body specified `startTransition(async () => { ... await fetch ... setState(verdict-ready) })`. With this structure, RTL's `act()` (used internally by `findByText`/`waitFor`) flushes ALL pending React work — including the awaited transition setStates — before polling the DOM. Result: the 'extracting' render never appears in tests; findByText for "Working..." times out. Direct sync `getByText(/Working/i)` AT THE INSTANT after click DID find "Working..." in the DOM, confirming the intermediate render committed; but findByText (async) flushes past it.
- **Fix:** Replaced `startTransition` on the extract handler with plain `async`. The Wishlist commit handler still uses startTransition (its post-commit work is genuinely deferrable). Added a `requestAnimationFrame`-yield between `setState(extracting)` and the `await fetch(...)` so the 'Working...' paint commits before microtasks resolve.
- **Files modified:** `src/components/watch/AddWatchFlow.tsx` (handleExtract function body)
- **Verification:** Before fix: 4/5 AddWatchFlow tests passed (ADD-01 failed on findByText). After fix: 5/5 pass.
- **Plan acceptance impact:** None — the plan's acceptance criteria check for `<VerdictSkeleton`, deep-link short-circuit, RAIL_MAX, etc. — none of which are affected. The startTransition change is an implementation tactic, not a contract change.
- **Committed in:** `6ee73cc` (Task 2 commit)
