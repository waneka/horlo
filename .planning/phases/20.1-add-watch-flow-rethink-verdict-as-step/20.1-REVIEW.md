# Phase 20.1: Code Review Report

**Reviewed:** 2026-04-30T08:45:00Z
**Depth:** standard

## Summary

The Add-Watch flow refactor lands a clean state-machine over four leaf Client Components, with disciplined separation between Server Component data hydration (`/watch/new/page.tsx`, `/catalog/[catalogId]/page.tsx`) and the Client `AddWatchFlow` orchestrator. Pitfalls 1, 3, 5, and 6 are explicitly mitigated in code with comments.

One **Critical** issue: `WishlistRationalePanel` has a real failing test on this branch (`WishlistRationalePanel.test.tsx > ADD-02 — textarea pre-fills from verdict.contextualPhrasings copy`) — confirmed by running vitest. The contract between the implementation and its test for the empty-string `initialNotes` case is broken; either the implementation should treat `''` as "no initial notes" via `||` instead of `??`, or the test should pass `initialNotes={undefined}`.

Two **Warnings**: (1) the catalog page lacks UUID-format validation on `params.catalogId`, which can cause a 500 error boundary instead of `notFound()` 404 on malformed URLs because `findViewerWatchByCatalogId` runs concurrently with `getCatalogById` inside `Promise.all`; (2) the `extraction-failed` flow state captures a `reason` string that is never rendered.

The remaining six items are quality / dead code issues — unused union variants, dead fallback branches, missing UUID defense-in-depth parity between sibling pages, and an unused prop pattern. None are security or correctness blockers.

## Critical Issues

### CR-01: WishlistRationalePanel — empty-string `initialNotes` swallows verdict prefill (failing test)

**File:** `src/components/watch/WishlistRationalePanel.tsx:56`
**Issue:** The state initializer uses nullish-coalescing:

```ts
const [notes, setNotes] = useState<string>(() => initialNotes ?? defaultRationale(verdict))
```

When the parent passes `initialNotes=""` (an explicit empty string), `??` does NOT fall back to `defaultRationale(verdict)` — it returns `""`. The corresponding test (`WishlistRationalePanel.test.tsx:32-44`) constructs the component with exactly `initialNotes=""` and asserts the textarea contains the verdict's contextual phrasing. Confirmed failing on this branch:

```
× ADD-02 — textarea pre-fills from verdict.contextualPhrasings copy
   AssertionError: expected '' to contain 'Aligns with my heritage-driven taste'
```

Two valid fixes — pick one consistent with the broader Pitfall 5 contract (blank-preserve on re-open):

**Fix A — fix the test (preferred per Pitfall 5 semantics):**
The Pitfall 5 contract is "blank textarea must commit blank verbatim". Re-opening a previously-blanked entry should NOT regenerate verdict copy on top of the user's clear. Therefore `initialNotes=""` should mean "user explicitly chose blank — keep it blank". The test that passes `initialNotes=""` and expects verdict copy contradicts that contract. Update the test:

```tsx
// src/components/watch/WishlistRationalePanel.test.tsx:32-44
render(
  <WishlistRationalePanel
    verdict={verdictWithCopy}
    // omit initialNotes (or pass undefined) so verdict copy fills in
    onConfirm={() => {}}
    onCancel={() => {}}
    pending={false}
  />,
)
```

**Fix B — fix the implementation (if Pitfall 5 is OK with empty re-fill on mount):**

```ts
const [notes, setNotes] = useState<string>(
  () => (initialNotes && initialNotes.length > 0 ? initialNotes : defaultRationale(verdict))
)
```

Phase 20.1 ships with this test failing in Wave 0 / RED tracking; if Wave 1 GREEN was meant to clear it, the implementation deviated from the test fixture's expectation. Reconcile before merge.

## Warnings

### WR-01: Catalog page lacks UUID validation on `params.catalogId` — 500 instead of 404 on malformed URLs

**File:** `src/app/catalog/[catalogId]/page.tsx:39-48`
**Issue:** `params.catalogId` is fed straight to `getCatalogById(catalogId)` and to the inline `findViewerWatchByCatalogId(user.id, catalogId)` inside a `Promise.all`. The latter performs `eq(watchesTable.catalogId, catalogId)` against a Postgres `uuid` column. If a user visits `/catalog/not-a-uuid`, Postgres throws `invalid input syntax for uuid`, which rejects the `Promise.all`, which surfaces as a 500 error boundary — bypassing the `notFound()` path that `getCatalogById` would have taken if invoked alone.

The sibling page `src/app/watch/new/page.tsx:51-55` validates the catalog UUID with a regex precisely for this defense-in-depth reason:

```ts
const catalogId =
  typeof sp.catalogId === 'string' &&
  /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i.test(sp.catalogId)
    ? sp.catalogId
    : null
```

**Fix:** Mirror the regex check at the top of `CatalogPage` and call `notFound()` when the format is invalid, so the page collapses to a clean 404 before any DB query runs:

```ts
export default async function CatalogPage({ params }: CatalogPageProps) {
  const { catalogId } = await params
  if (!/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i.test(catalogId)) {
    notFound()
  }
  const user = await getCurrentUser()
  // ...
}
```

This both prevents the 500 path and avoids issuing two DB queries (`getCatalogById` + `findViewerWatchByCatalogId`) for definitely-bogus inputs.

### WR-02: `extraction-failed.reason` is captured but never rendered

**File:** `src/components/watch/AddWatchFlow.tsx:124-127, 161, 359-391`
**Issue:** The `extraction-failed` flow state carries a `reason` string (line 124, line 161), but the rendered Card body (lines 359-391) never displays it. The user only sees the generic "Extraction didn't work" / "Fill in manually using any details below." copy. When the API surface returned a structured error (e.g., "Unsupported domain", "Rate limited"), or `String(err)` from the catch block produced a debuggable message, that signal is silently dropped.

**Fix:** Either surface `state.reason` (e.g., a small `<p className="text-xs text-destructive">` below the Card description) so users get actionable feedback, or remove `reason` from the union type to make the dead capture explicit. Surfacing is the higher-value option:

```tsx
{state.reason && (
  <p className="text-xs text-muted-foreground">{state.reason}</p>
)}
```

(Caveat: `String(err)` from line 161 may include "Error:" prefix — sanitize before display.)

### IN-01: `submitting-collection` and `manual-entry` partial-flow states partially defined / unused

**File:** `src/components/watch/flowTypes.ts:22, 24`
**Issue:** The `FlowState` union declares `{ kind: 'submitting-collection'; ... }` (line 22), but `AddWatchFlow.tsx` never sets state to that variant — collection is always a synchronous router navigation through `<WatchForm>` after entering `form-prefill`. Similarly, `PendingTarget` (line 47) declares `'collection' | 'skip'`, but only `'wishlist' | null` is ever set in the orchestrator (skip is sync; collection navigates).

**Fix:** Either implement the unused states (e.g., a future "submitting collection" pending UI) or trim the union to match actuality:

```ts
export type FlowState =
  | { kind: 'idle' }
  | { kind: 'extracting'; url: string }
  | { kind: 'verdict-ready'; ... }
  | { kind: 'wishlist-rationale-open'; ... }
  | { kind: 'submitting-wishlist'; ... }
  | { kind: 'form-prefill'; ... }
  | { kind: 'manual-entry'; partial?: ExtractedWatchData | null }
  | { kind: 'extraction-failed'; partial: ExtractedWatchData | null; reason: string }

export type PendingTarget = 'wishlist' | null
```

Trim avoids future readers wondering whether a state machine handler is missing.

### IN-02: `findViewerWatchByCatalogId` declared return type doesn't match runtime — and caller has dead fallback

**File:** `src/app/catalog/[catalogId]/page.tsx:173-195, 59`
**Issue:** The return type annotation says `Promise<{ id: string; acquisitionDate: string | null } | null>` (line 176), but the implementation always assigns `iso` (a non-null string) into `acquisitionDate` (line 191-194):

```ts
const iso = row.acquisitionDate !== null
  ? row.acquisitionDate
  : new Date(row.createdAt).toISOString()
return { id: row.id, acquisitionDate: iso }
```

So the runtime invariant is `acquisitionDate: string` (never null), but the declared type allows null. The caller at line 59 then writes:

```ts
ownedAtIso: viewerOwnedRow.acquisitionDate ?? new Date().toISOString(),
```

The `?? new Date().toISOString()` branch is unreachable — `acquisitionDate` is always truthy.

**Fix:** Tighten the return type and remove the dead fallback:

```ts
async function findViewerWatchByCatalogId(
  userId: string,
  catalogId: string,
): Promise<{ id: string; acquisitionDate: string } | null> { ... }

// caller:
ownedAtIso: viewerOwnedRow.acquisitionDate,
```

Reduces cognitive load and makes the type the truth.

### IN-03: Catalog page issues a wasted DB query on the empty-collection deep-link path

**File:** `src/app/watch/new/page.tsx:57-60`
**Issue:** `Promise.all([getWatchesByUser(...), catalogId ? hydrateCatalogPrefill(catalogId) : ...])` always runs the catalog hydration when `catalogId` is set, even when `intent` was rejected (e.g., `?catalogId=valid&intent=malicious-string`). The hydrated prefill is then ignored because `AddWatchFlow`'s short-circuit requires `initialIntent === 'owned'`.

**Fix:** Only hydrate when both `catalogId` AND `initialIntent === 'owned'`:

```ts
const shouldHydrate = catalogId !== null && initialIntent === 'owned'
const [collection, catalogPrefill] = await Promise.all([
  getWatchesByUser(user.id),
  shouldHydrate ? hydrateCatalogPrefill(catalogId) : Promise.resolve(null),
])
```

Out of v1 perf scope, but worth a one-line tightening since the contract already implies it.

### IN-04: `void framing` workaround for unused-prop lint smell

**File:** `src/components/watch/CatalogPageActions.tsx:115-118`
**Issue:** The `framing` prop is documented as "Reserved for future variant" and silenced with `void framing` to satisfy `noUnusedLocals`. This is a code smell — the prop appears in the interface contract but does nothing.

**Fix:** Either remove the prop until needed (callers stop passing `framing="cross-user"`) or wire it to an actual visual variant. Keep the prop only when the variant lands. Removing is the cleaner option since the page Server Component already gates rendering of this component on `framing === 'cross-user'` (catalog page line 132-142).

### IN-05: AddWatchFlow URL-input auto-focus uses `getElementById` instead of a ref

**File:** `src/components/watch/AddWatchFlow.tsx:82-87`
**Issue:** The auto-focus effect reaches into the DOM with `document.getElementById('paste-url')` and casts:

```ts
const el = document.getElementById('paste-url') as HTMLInputElement | null
el?.focus()
```

This couples the orchestrator to a string id owned by `<PasteSection>`. If `PasteSection` ever renames the input id (e.g., to support multiple instances on a page), this silently no-ops. React's idiomatic mechanism here is a forwarded ref or an imperative handle.

**Fix:** Forward a ref through `<PasteSection>` and call `inputRef.current?.focus()`:

```tsx
// PasteSection.tsx — forward ref to <Input>
export const PasteSection = forwardRef<HTMLInputElement, PasteSectionProps>(...)

// AddWatchFlow.tsx
const urlInputRef = useRef<HTMLInputElement | null>(null)
useEffect(() => {
  if (state.kind === 'idle') urlInputRef.current?.focus()
}, [state.kind])
// ...<PasteSection ref={urlInputRef} ...
```

Lower-priority refactor; current code works because the id is stable.

### IN-06: `WatchForm` edit-path bypasses lockedStatus defense-in-depth

**File:** `src/components/watch/WatchForm.tsx:171-174`
**Issue:** When `mode === 'edit'`, the submit calls `editWatch(watch.id, formData)` — passing raw `formData` instead of the `submitData` object that applies the `finalStatus = lockedStatus ?? formData.status` defense-in-depth. In practice this doesn't matter today because the only Phase 20.1 caller using `lockedStatus` is `mode='create'`, but the asymmetry is a footgun for any future caller passing `lockedStatus` with `mode='edit'`.

**Fix:** Pass `submitData` to both branches:

```ts
const result =
  mode === 'edit' && watch
    ? await editWatch(watch.id, submitData)
    : await addWatch(submitData)
```

The `photoSourcePath` field is still gated behind `mode === 'create'` upstream, so it won't leak into the edit path.
