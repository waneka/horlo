### CR-01: WishlistRationalePanel — empty-string `initialNotes` swallows verdict prefill (failing test)

**Files modified:** `src/components/watch/WishlistRationalePanel.test.tsx`
**Commit:** `313ad6c`
**Applied fix:** Removed `initialNotes=""` from all 4 test render calls so the textarea pre-fills from `defaultRationale(verdict)`.

**Reasoning (Fix A vs Fix B):**

The reviewer flagged this as nuanced — either fix the implementation (`??` → `||`) or fix the test. I chose **Fix A (the test)** based on the following analysis of the Pitfall 5 contract:

1. **RESEARCH.md §Pitfall 5 (lines 471-487)** explicitly defines the contract as a COMMIT-time invariant, not an initial-render invariant:
   > "Send the textarea value verbatim, including empty string. **Pre-fill happens on render only; commit reads what the user actually has.**"
   The pitfall forbids `notes = textarea.value || verdictHeadline` at commit time only. The implementation already does the right thing in `handleConfirm`: `onConfirm(notes)` (verbatim).

2. **The implementation comment at WishlistRationalePanel.tsx:55** matches the established prop contract: "initialNotes wins over verdict-derived default; verdict-derived wins over ''". With `??`, only `null`/`undefined` triggers fallback — `""` is treated as a real user-provided value. This is consistent with the prop docstring at line 29: *"Optional starting notes (e.g., when re-opening from rail). Defaults to verdict headline if absent."* — "if absent" semantically means undefined.

3. **Re-open semantics:** the `initialNotes` prop is reserved for future re-open flows from `RecentlyEvaluatedRail` (where a previously cleared entry must NOT regenerate verdict copy on re-open). Phase 20.1's current `AddWatchFlow.tsx:330-335` orchestrator passes `<WishlistRationalePanel verdict={...}>` WITHOUT `initialNotes` — the deep-link / re-open semantics aren't wired yet, but the prop's nullish-coalescing behavior is the right contract for when they are.

4. **Switching to `||` would break Pitfall 5 itself for the future re-open case:** a user who re-opens a previously blanked entry from the rail would have their `""` silently overwritten with verdict copy on mount. That regression is exactly what Pitfall 5 is named for.

5. The test fixture passing `initialNotes=""` and expecting verdict copy contradicted both (a) the comment on line 55 and (b) the prop docstring. The test was fixed in 4 places: `ADD-02 textarea pre-fill`, `Pitfall 5 blank-preserve`, `edit + confirm`, and `cancel button`. Each test now omits the prop and lets `defaultRationale(verdict)` populate the textarea on render.

**Logic-bug caveat note:** This finding involves nullish-coalescing semantics, which is purely structural — not a logic bug requiring runtime semantic verification. Tier 1 (re-read) + running the test suite confirms behavior. Result: 4/4 tests in `WishlistRationalePanel.test.tsx` pass after the fix.
