## Accomplishments

- Closed UAT gap 1 — VerdictStep no longer shows misleading "Your collection is empty" copy when verdict is null and the viewer has a non-empty collection. New `hasCollection: boolean` prop drives the fallback copy split: `false` → existing D-06 empty-collection notice, `true` → "Couldn't compute fit right now — you can still add this to wishlist or collection."
- Closed UAT gap 3 — AddWatchFlow.handleSkip always pushes a rail entry. Previously guarded with `if (state.catalogId)` which silently no-op'd the push when catalogId was empty (silent catalog upsert failure path); now synthesizes a stable id from `synth:{brand}|{model}` so the chip still renders.
- Closed UAT gap 2 (regression guard) — added a positive-case test asserting WishlistRationalePanel pre-fills its textarea with `verdict.contextualPhrasings[0]` when the verdict bundle is non-null. The original gap-2 symptom (empty textarea) was an upstream null-verdict consequence; the prefill logic itself was correct, so this test guards against future regressions.
- Added `catalogIdError: string | null` field to the `/api/extract-watch` JSON response. Set to a sanitized reason code when the catalog upsert path could not produce a real id (`'brand/model missing from extraction'`, `'catalog upsert returned null id'`, or `'catalog upsert threw: <truncated msg>'`).
- Added 3 `console.warn` diagnostic statements in `AddWatchFlow.handleExtract` — one per `setState({ verdict: null })` branch (`collection-empty`, `catalogId-missing`, `verdict-failed`) — so future occurrences can be diagnosed by tailing the browser console.
- Locked the verdict Server Action observability contract via JSDoc — error strings on the failure path are stable and surfaced via `console.warn`; consumers depend on specific error text.

## Decisions Made

- **`hasCollection` is a required prop, not optional** — Required ensures every call site threads it explicitly; existing tests were updated rather than relying on a default. Eliminates ambiguity at TypeScript boundary.
- **Synthesized rail-entry id with `synth:` prefix** — Prefix prevents cache-lookup collisions with real UUIDs; cache misses on synth ids are intentional. Synthesized ids are NEVER passed to `getVerdictForCatalogWatch` (Zod-validates UUID shape and would reject), so no security boundary crossed.
- **Sanitize `catalogIdError` before sending to client** — Truncate `err.message` to 200 chars; never include stack trace. Static reason codes preferred over exception text. Server-side `console.error` preserves the full err for ops; client only sees the sanitized string.
- **`verdict.ts` code unchanged; JSDoc-only update** — All 4 error paths already return stable strings. Locked the observability contract in JSDoc to prevent future refactors from collapsing specific errors to generic 'failed'.
