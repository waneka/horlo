## Flow architecture + ADD-06 + manual entry

### Q1: Where does the new flow live structurally?

| Option | Description | Selected |
|--------|-------------|----------|
| In-place at /watch/new, state-driven sections (Recommended) | Single page with sections that appear as state advances: Paste → Verdict → 3-button → optional Form. No multi-route wizard. | ✓ |
| Multi-step wizard at /watch/new | Formal stepper UI (Step 1 of 3 / Back / Next). Separate panes per step. | |
| New paste-first route, /watch/new becomes manual-only | Carve a new route (e.g. /watch/add); /watch/new becomes manual entry. | |

**User's choice:** In-place state-driven sections.
**Notes:** Reuses existing route, no nav churn, no breadcrumbs. Honors "single coherent gesture" framing without stepper-shaped friction.

### Q2: How does /search inline-expand route into this flow with catalogId (ADD-06)?

| Option | Description | Selected |
|--------|-------------|----------|
| 'Add to Wishlist'/'Add to Collection' buttons inside the accordion (Recommended) | Add 3 CTAs directly to expanded verdict card; no re-paste. | ✓ |

| Hybrid — inline commit for Wishlist; /watch/new for Collection | Asymmetric: Wishlist inline; Collection routes for ownership fields. | |

**User's choice:** Inline 3 CTAs in accordion.
**Notes:** Same Server Actions used by /watch/new; user lands on /watch/[id] (collection) or /search with toast (wishlist).

### Q3: How does manual entry stay accessible as the secondary affordance (ADD-05)?

### Q4: Should /catalog/[catalogId] also expose the 3-button CTAs?

## Skip / evaluate-loop ergonomic (ADD-04)

| Option | Description | Selected |
