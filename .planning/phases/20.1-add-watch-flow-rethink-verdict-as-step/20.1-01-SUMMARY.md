## Decisions Made
- **`.ts` → `.tsx` rename for `add-watch-flow-search-cta.test`**: The plan's `<files>` field and acceptance-criteria grep used the `.ts` extension, but the planned content uses JSX (e.g. `render(<WatchSearchRowsAccordion ... />)`), and esbuild rejects JSX in `.ts`. Renamed to `.tsx`. Vitest discovers both extensions; the test runs cleanly. The acceptance grep targets re-execute against the new path. Tracked as Rule 1 (bug) deviation below.
- **Negative-assertion strategy for unbuilt-code absence**: Tests like `WishlistTabContent isOwner=false` assert that `Add to Wishlist` is NOT in the DOM. Today the source doesn't render any CTA in any branch, so these assertions vacuously pass. After Plan 02 ships the CTA inside `isOwner=true` branch only, these tests keep passing. They serve as regression locks — they will catch a future bug where the CTA accidentally renders in the wrong branch. Documented intent in inline comments.
- **JSON.stringify(result).toMatch(...) for RSC tree assertions**: For the `tests/app/watch-new-page.test.ts` Server Component tests, the page is `await NewWatchPage(...)` which returns a JSX element tree. We mock `<AddWatchFlow>` to a stringified-props marker and assert via JSON.stringify regex. This pattern is reused from the existing `tests/app/catalog-page.test.ts` and avoids needing a full RTL render of an async server component.
