<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Server Action (toggleLike / addComment) | viewer-supplied `{type, id, body}` crosses into server execution |
| Server Action → cache tag namespace | revalidateTag controls which viewers' cached counts are invalidated |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-63-01 | Tampering (IDOR) | toggleLikeAction / addCommentAction target id | mitigate | Owner/ownerId is resolved from the DB by the action (not trusted from client); Zod `.strict()` schemas (`toggleLikeSchema`, `addCommentSchema`) already reject extra fields. This plan adds no new client input path — unchanged. |
| T-63-02 | Information Disclosure | getBatchedWatchCounts `liked`/`canComment` per-viewer fields via getBatchedWatchCountsCached `'use cache'` | mitigate | `liked` is scoped to `viewerId` (the Q6 query filters `eq(watchLikes.userId, viewerId)`); the cache entry is tagged `viewer:${viewerId}:counts` so viewer A's liked state can never be served to viewer B. `viewerId` is passed as an argument resolved OUTSIDE the `'use cache'` scope (no request-time API inside the cached function). |
| T-63-03 | Elevation of Privilege | `canComment` flag as a gate | accept (defense-in-depth elsewhere) | `canComment` here is derived from the existing authoritative `allowedSet`; it is a read-only flag. The REAL write gate is `createComment` re-checking `canViewerCommentOnTarget` server-side (Plan 03 / existing code) — the flag cannot be used to bypass the write gate. No new privilege surface introduced. |
| T-63-04 | Information Disclosure (stale cache) | viewer:{userId}:counts not busted on mutation | mitigate | This plan ADDS the missing `revalidateTag('viewer:{userId}:counts','max')` to both actions — closing the confirmed D-12 gap so a viewer's own liked/count state re-hydrates fresh, never serving stale cross-navigation state. |

All threats are existing-surface; this plan introduces no new high-severity threat. Standard mitigations (DB-resolved ownerId, Zod `.strict()`, viewer-scoped cache tag, auth-outside-cache) hold.
</threat_model>
