<objective>
Close VERIFICATION gaps #2 (WR-01 — silent duplicate via ConfirmStep) and #3 (WR-02 — silent no-banner advance on transient dupe-resolver failure).

**Gap #2 (WR-01) — one-line fix + regression test:** In the `confirming` render branch (AddWatchFlow.tsx:608-628), the ConfirmStep receives `pending={state.pending}` — meaning the primary CTA stays active even when DupeBanner is mounted next to it. A user who ignores the banner and clicks the ConfirmStep primary creates exactly the silent duplicate the banner exists to prevent. Fix: compute the `pending` prop as `state.pending || state.dupeContext != null`. When dupeContext is present, the ConfirmStep primary disables; the user MUST interact with one of the DupeBanner buttons (View existing / Move to Collection / Add another copy → clears dupeContext → ConfirmStep re-enables) to proceed.

**Gap #3 (WR-02) — branch fix in handleSearchPick + regression test:** In the search-pick path (handleSearchPick lines ~143-167 — the no-redirect branch when `viewerState === 'owned'` falls through to confirming, or the explicit `viewerState === 'wishlist'` confirming construction at lines ~185-206), when `resolveDupeContext` returns null AND the search result advertised a non-null `viewerState`, the orchestrator KNOWS a dupe exists but silently drops it. Fix: detect the known-dupe-but-null-resolver case; show `toast.error('Couldn\\'t check your collection — try again')`; stay on search-idle (do not transition to confirming).
