<objective>
Close VERIFICATION gap #1 half-A (CR-01): the EXIF-cleaned Blob captured by `StructuredEntryPanel`'s `CatalogPhotoUploader` is currently written into a destructured-setter-only useState (`const [, setPhotoBlob] = useState<Blob | null>(null)`) and silently dropped. The user sees the upload succeed visually; the photo dies in dead state.

Purpose: Plumb the captured Blob upward so AddWatchFlow (in the next gap plan, 70-07) can call `uploadCatalogSourcePhoto` and forward `photoSourcePath` into the `addWatch` payload (mirroring `WatchForm.tsx:222-249` precedent). This plan ONLY widens the contract — it does NOT change AddWatchFlow's payload yet (70-07 does that). Splitting the work keeps each plan small + makes the regression visible at each step.

Output:
- `StructuredEntryPanel.tsx` reads the captured `photoBlob` (no longer write-only) and forwards it upward — either by widening `onSubmitStructured` to a 3-arg `(extracted, catalogId, photoBlob?)` shape OR by adding a new `onPhotoReady?: (blob: Blob | null) => void` sibling prop. **The CONTEXT.md and review code excerpt both suggest the 3-arg widen path (matches the existing Wave-0-style additive patches in 70-01 and the planning_context hint).** Choose the 3-arg widen for symmetry with how `onSubmitStructured` was widened from 1 → 2 args during 70-01.
- `SearchEntry.tsx` forwards the widened arg through its pass-through closure (today line 336 `onSubmitStructured={onSubmitStructured}` — typed callback already passes through; only the type signature widens).
- Co-located tests assert the Blob flows through end-to-end (StructuredEntryPanel test asserts the parent receives the blob; SearchEntry test asserts the pass-through preserves arity).
- AddWatchFlow keeps its current 2-arg handler signature in this plan (third arg is `optional` — backward compat). 70-07 changes the handler body to consume it.
