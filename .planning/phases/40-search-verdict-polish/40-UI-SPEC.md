## CSS Chain Assertions (Phase 40 — CSS Blind Spot Compliance)

Per memory `feedback_ui_spec_css_chain_blind_spot` (2026-05-05): the checker validates declared tokens, not whether the CSS chain actually produces the claimed visual contract. Assert chains explicitly.

### Chain 1 — Chip Selected State Contrast

**Claim:** A selected chip renders with visible contrast — amber-warm background + dark foreground.

**Chain:**
```
Chip selected: aria-pressed="true" on <button> element
  → Tailwind class: `bg-accent text-accent-foreground border-accent`
  → CSS custom property: --color-accent: var(--accent)
  → :root --accent: oklch(0.76 0.12 75)  (light)
  → Computed: warm amber background
  → --accent-foreground: oklch(0.18 0.01 75)
  → Computed: near-black text
  → Contrast ratio: oklch(0.18) on oklch(0.76) ≈ 5.8:1 (exceeds WCAG AA 4.5:1)
```

**Unselected chip chain:**
```
  → Tailwind class: `bg-secondary text-secondary-foreground border-border`
  → --secondary: oklch(0.96 0.005 75) (near-white)
  → --secondary-foreground: oklch(0.22 0.005 75) (near-black)
  → --border: oklch(0.9 0.005 75) (light gray border)
  → Contrast ratio: oklch(0.22) on oklch(0.96) ≈ 7.8:1 (exceeds WCAG AA)
```

Executor must implement: `<button aria-pressed={isSelected} className={cn('...', isSelected && 'bg-accent text-accent-foreground border-accent', !isSelected && 'bg-secondary text-secondary-foreground border-border')} ...>`

### Chain 2 — Sheet `side='bottom'` Slide Animation

**Claim:** The bottom-sheet slides up on open and down on close with 200ms ease-in-out.

**Chain (verified against `src/components/ui/sheet.tsx:52-58`):**
```
SheetContent side='bottom'
  → data-side="bottom" set on SheetPrimitive.Popup
  → Tailwind classes applied:
      data-[side=bottom]:inset-x-0          → full-width
      data-[side=bottom]:bottom-0           → anchored to bottom edge
      data-[side=bottom]:h-auto             → height wraps content
      data-[side=bottom]:border-t           → top border (border-border token)
      data-[side=bottom]:data-starting-style:translate-y-[2.5rem]  → enters from below
      data-[side=bottom]:data-ending-style:translate-y-[2.5rem]    → exits downward
  → transition duration-200 ease-in-out    → 200ms smooth
  → data-starting-style/data-ending-style are @base-ui/react Dialog lifecycle hooks
  → opacity-0 on start/end; full opacity on open
```

**Scroll safety for up to ~8 chips per group (Style group):**
```
SheetContent with side='bottom' + h-auto:
  → Sheet will expand to content height
  → With 3 groups × 8 chips max + section headers + footer = ~280px content
  → On short viewports (iPhone SE: 568px), sheet could exceed 50% viewport
  → Safety chain: add max-h-[80vh] overflow-y-auto to SheetContent className
  → Ensures content is scrollable on short viewports without overlapping status bar
```

Executor must add `className="max-h-[80vh] overflow-y-auto"` to the `SheetContent` render.

### Chain 3 — FIT-05 2-Column Grid on Mobile

**Claim:** FIT-05 compare table renders as 2 columns on all viewport sizes (D-13 "max 2 items mobile").

**Chain:**
```
D-13: "max 2 items mobile — still 2 columns side-by-side, no third column ever"
  → Always-2-col decision (auto-resolved): grid-cols-2 at all breakpoints
  → Mobile: grid-cols-2 with compressed cell padding (px-2 py-2)
  → No responsive grid-cols-1 fallback — D-13 explicitly keeps 2-col on mobile
  → CSS: `grid grid-cols-2 gap-px bg-border rounded-md overflow-hidden`
  → Each cell: `bg-background px-2 py-2 text-sm`
  → On 320px viewport (iPhone SE min): each column = (320 - 2*16px margin - 1px gap) / 2 ≈ 143px
  → 143px is sufficient to render "1.0" (scalar) or "Heritage Tool" (archetype) without overflow
```

Executor must use `grid-cols-2` (no `sm:grid-cols-2` responsive variant) per D-13.

verdict.mostSimilar.length > 0 &&

### Delta Row

Single line of plain text below the compare grid. `text-sm text-muted-foreground`.

**Algorithm (D-16):**

1. Scalar deltas: `|candidate.formality - owned.formality|`, `|candidate.sportiness - owned.sportiness|`, `|candidate.heritageScore - owned.heritageScore|`. If either side is null, delta = null (excluded from comparison).
2. Categorical: `primaryArchetype` — delta = 0 if equal, 1 if different. Same for `eraSignal`.
3. `designMotifs` — delta = `1 - jaccard(candidate.designMotifs, owned.designMotifs)`.
4. Fallback test: all non-null scalar deltas < 0.1 AND both categoricals match AND motif jaccard >= 0.8 → render fallback.
5. Otherwise: pick highest delta dimension and render phrase.

**Copy templates:**

| Dimension | Direction condition | Phrase |
|-----------|--------------------|---------| 
| `formality` | candidate > owned | "This is more formal" |
| `formality` | candidate < owned | "This is more casual" |
| `sportiness` | candidate > owned | "This is more sport" |
| `sportiness` | candidate < owned | "This is less sport" |
| `heritageScore` | candidate > owned | "More heritage-leaning" |
| `heritageScore` | candidate < owned | "More modern in character" |
| `primaryArchetype` | different | "Different archetype: {candidate} vs {owned}" |
| `eraSignal` | different | "Different era: {candidate} vs {owned}" |
| `designMotifs` | jaccard < 0.8 | "Different design motifs" |
| Fallback | all deltas small | "Very similar across all taste dimensions" |

Source: CONTEXT.md § D-16 + § Specific Ideas. Archetype and era values in the phrase use the capitalized display form (same transform as column cells).

**Scalar threshold: 0.1** (CONTEXT.md leading suggestion, confirmed by D-16 spec text).
**Motif threshold: 0.8** jaccard similarity.
