## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| verify script -> local DB | Read-only coverage queries against the local catalog |
| generated SQL migration -> prod (supabase db push --linked) | Committed SQL crosses into the production database |
| operator -> review file / prod push | Operator judgment gates which values reach prod |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-12 | Tampering | A live LLM enrichment run accidentally pointed at the prod DB | mitigate | D-14 — scripts run with `--env-file=.env.local` (DATABASE_URL = `127.0.0.1:54322`); the playbook step-0 preflight explicitly verifies the local URL before any live run; prod's DATABASE_URL stays commented out in `.env.local`. Prod only ever ingests the reviewed SQL migration. |
| T-44-13 | Tampering | A migration file with a non-14-digit timestamp is silently skipped by `supabase db push` | mitigate | `generateMigrationFilename` (Plan 03) produces exactly 14 digits; the playbook step-7 instructs verifying `supabase db push` reports the new migration applied (not 0). |
| T-44-14 | Information Disclosure | `SUPABASE_ACCESS_TOKEN` exposed during the prod push | accept | Token is read from the operator's environment for a one-shot CLI push; not logged by the playbook steps. Single-operator, local shell — accepted. |
| T-44-15 | Elevation of Privilege | The prod push applies unreviewed data | mitigate | The push is an explicit operator-gated step (Task 2 checkpoint, `autonomous: false`); the migration files are committed and code-reviewable before the push; verification runs both before (local) and after (prod). |
</threat_model>
