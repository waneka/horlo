<objective>
Build the two new factual-fill scripts (D-10): `factual-propose.ts` runs a
gap-driven web_search LLM pass and writes a hand-editable JSONL review file
(D-01..D-05, D-13); `factual-apply.ts` reads the operator-approved review file
and emits a timestamped SQL data migration (D-03, D-14).

Purpose: ENRH-05 — every `/search` filter dimension (`movement_type`,
`case_size_mm`, `style_tags`) must be populated, and missing cover photos
backfilled where sourceable, WITHOUT the LLM ever writing factual columns
directly. The editable review file IS the human-review gate the requirement
demands; the SQL migration is the only path to prod.

Output: `factual-propose.ts`, `factual-apply.ts`, an exported `sanitizeHttpUrl`,
two new `db:` npm scripts, and a dry-run test.

- `enrichWithWebSearch(client, customTools, initialMessages, customToolName, opts?)`
  -> `{ toolUse: ToolUseBlock | null, sourceUrls: string[], webSearchUnavailable: boolean }`
- `extractSourceUrls(content)` and `WEB_SEARCH_TOOL`

Review file format (JSONL — RESEARCH Pattern 4), one line per field per catalog_id:
  { "catalog_id": "<uuid>", "field": "movement_type"|"case_size_mm"|"style_tags"|"image_source_page_url",
    "current": <value|null>, "proposed": <value>, "source_url": "<url>", "approved": null }
Operator sets approved:true to confirm, approved:false to reject, or deletes the line.

Gap query (gap-driven, D-05):
  WHERE movement_type IS NULL OR case_size_mm IS NULL OR array_length(style_tags, 1) IS NULL

watches_catalog factual columns: movement_type (enum: auto|manual|quartz|spring_drive),
case_size_mm (number), style_tags (text[] notNull default '{}'), image_url, image_source_url.
NOTE: style_tags is the FACTUAL /search filter column — distinct from design_motifs
(the taste column the enricher writes). Never conflate (RESEARCH Pitfall 5).

Migration filename: YYYYMMDDHHMMSS_phase44_factual_data.sql — exactly 14 digits,
timestamp must be > 20260517000000 (RESEARCH Pattern 5, Pitfall 3).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| web_search result -> factual-propose | Untrusted external page content used to propose factual values |
| LLM tool output -> review file | LLM-proposed values written to a local JSONL file |
| review file (operator-edited) -> factual-apply | Operator-approved file is the trust boundary feeding the SQL generator |
| factual-apply -> supabase/migrations/ -> prod | Generated SQL is pushed to the production catalog |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-08 | Tampering | SQL injection via LLM/operator-supplied values in factual-apply's UPDATE statements | mitigate | Every approved entry is validated by `field` before SQL emission — `movement_type` enum allow-list, `case_size_mm` numeric-range check, `style_tags` array-of-string check, URL fields through `sanitizeHttpUrl`. Values are emitted as typed/escaped SQL literals; no raw LLM string is interpolated. Invalid entries are dropped and logged. |
| T-44-09 | Tampering | LLM hallucinating a non-http(s) or javascript: URL into a review-file source/image field | mitigate | Reuse exported `sanitizeHttpUrl` from `src/data/catalog.ts` on every URL-valued field in factual-apply — non-http/https URLs return null and the field is dropped. |
| T-44-10 | Information Disclosure | `ANTHROPIC_API_KEY` leaked via factual-propose verbose logging | mitigate | Per-row logs include only `catalog_id`, `status`, `timestamp`; the API-key guard checks presence without printing the value; no full HTTP headers are logged. |
| T-44-11 | Spoofing | A direct image URL hotlinked from a hallucinated source corrupting catalog imagery | mitigate | D-04 — factual-propose proposes only an `image_source_page_url` (a brand/retailer page); the operator grabs the actual image manually. No LLM-emitted direct image URL is ever committed. |
</threat_model>
