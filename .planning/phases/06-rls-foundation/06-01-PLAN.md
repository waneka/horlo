Write and apply the RLS migration for all three existing tables (users, watches, user_preferences), then verify cross-user isolation works via Supabase User Impersonation.

Purpose: This is the foundational security layer for all v2.0 multi-user features. Without RLS, any user with a valid JWT and the anon key could read/write other users' data via direct PostgREST calls. The DAL WHERE clauses are the primary enforcement for Server Actions, but RLS adds defense-in-depth that cannot be bypassed by direct API callers.

Output: One SQL migration file applied to production Supabase, RLS verified working.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client (anon key + JWT) -> PostgREST | Untrusted: any authenticated user can craft direct API calls to PostgREST with their JWT. RLS policies are the enforcement layer here. |
| Server Actions (DAL) -> Postgres via DATABASE_URL | Trusted: service role connection bypasses RLS. DAL WHERE clauses are the enforcement layer (unchanged). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | PostgREST SELECT on watches/users/user_preferences | mitigate | RLS SELECT policy: `USING (user_id = (SELECT auth.uid()))` on each table |
| T-06-02 | Tampering | PostgREST UPDATE changing user_id | mitigate | UPDATE policy WITH CHECK: `WITH CHECK (user_id = (SELECT auth.uid()))` prevents ownership injection |
| T-06-03 | Denial of Service (indirect) | Per-row auth.uid() evaluation | mitigate | `(SELECT auth.uid())` wrapper hoists to InitPlan (1 call per query, not per row) |
| T-06-04 | Denial of Service | RLS enabled without policies | mitigate | D-08: enable + all policies in same migration file — never separated |
| T-06-05 | Elevation of Privilege | Service role key exposure | accept | Service role key only in server-side DATABASE_URL env var; never exposed to client; no change from current architecture |
</threat_model>
