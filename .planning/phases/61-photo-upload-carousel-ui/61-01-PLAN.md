## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → server action | Untrusted `{watchId, photoId, storagePath, orderedIds}` payloads cross into `watchPhotos.ts` actions |
| server action → DAL | Actions delegate to Phase 60 DAL functions that own the `watches.user_id` ownership gate |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-61-01 | Tampering / Elevation | reorderWatchPhotosAction with another user's watchId | mitigate | `bulkReorderPhotos` begins with `watches.id = watchId AND watches.user_id = user.id` ownership check (user.id from session, never payload). Test (b)/(c) in Task 1 asserts non-owner + unauth rejection. UI `viewerCanEdit` hiding is NOT the control. |
| T-61-02 | Tampering | deleteWatchPhotoAction with another user's photoId/watchId | mitigate | `deleteWatchPhoto` gates on `watches.user_id = user.id`; cross-user delete throws → action returns "Couldn't delete photo." Test asserts unauth + ownership-fail paths. |
| T-61-03 | Tampering (mass assignment) | extra/forged keys (e.g. `userId`) in action payloads | mitigate | zod `.strict()` rejects any undeclared key; `userId` always `user.id` from `getCurrentUser()`. Task 1 test (d) asserts `.strict()` rejection. |
| T-61-04 | DoS | upload beyond 10 photos per watch | mitigate | `addWatchPhoto` DAL cap check throws `PhotoCapExceededError` (MAX_PHOTOS_PER_WATCH=10); action maps it to a cap-reached result. Backstop is server-side, not UI. |
| T-61-05 | Tampering | client-supplied `sort_order` | mitigate | `addWatchPhoto` computes `nextSort` server-side; client never passes sort_order. `bulkReorderPhotos` derives order from array index. |
| T-61-06 | Repudiation | n/a — no audit-log requirement this phase | accept | Personal single-tenant app; no audit-trail requirement in REQUIREMENTS.md §PHOTO. |
</threat_model>
