<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Supabase Storage | Client-direct `uploadWatchPhoto` to the `watch-photos` bucket; RLS enforces `{userId}/` folder scoping |
| RSC → signed URL | Page signs storage paths into time-boxed read URLs before handing to the client |
| client → server action | Upload/delete/reorder payloads cross into Plan 01's actions (gated there) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-61-07 | Information Disclosure | signed URL leaks time-boxed read to a storage object | accept | 60-min TTL matches the established `wear-photos` precedent; folder scoping `{userId}/...` is RLS-enforced at upload; route is dynamic (`getCurrentUser()` → no ISR caching past TTL — RESEARCH Open Q3 / A2). Acceptable for a personal auth-gated app. |
| T-61-08 | Spoofing / SSRF | desktop drop zone receiving spoofed URLs instead of files | mitigate | Drop zone reads `e.dataTransfer.files` (File objects via HTML5 DnD) only — never fetches arbitrary URLs. No server-side fetch of client input. |
| T-61-09 | Elevation of Privilege | non-owner attempting upload/delete/reorder via the UI | mitigate | UI controls gated on `viewerCanEdit` (hide), AND every mutation routes through Plan 01 actions whose DAL re-checks `watches.user_id` (the real control). UI hiding is defense-in-depth only. |
| T-61-10 | Tampering | EXIF metadata / oversized images bypassing the pipeline | mitigate | All files pass through inherited `stripAndResize` (EXIF strip + ≤1080px JPEG @0.85) before upload; `accept="image/*,.heic,.heif"` restricts picker; cap enforced in DAL backstop. |
| T-61-11 | Information Disclosure | raw storage path rendered to non-owner via `getSafeImageUrl` | mitigate | Non-owners never receive `signedPhotos` (owner branches only); catalog-only branch unchanged. `getSafeImageUrl` returns null for raw paths so no path string leaks into markup. |
</threat_model>
