## Deviations

Three deviations from plan, all upstream of the actual apply:

1. **Local-keyed brand decisions file did not apply to prod.** Of 18 distinct UUIDs in the file, 0 existed in prod's `brands` table. Strict gate fired on the `merge:20969364-...` Hamilton target. Recovery: `--force --mode=brands` against prod (loaded prod state into a fresh file). 17 auto-resolved + 38 needs-review.

2. **Canonical Hamilton brand name flipped.** Operator intent was canonical = `Hamilton` (not `Hamilton Watch`). Prod had only `Hamilton Watch`; no `Hamilton` row to merge INTO. Recovery: one-off SQL `UPDATE brands SET name = 'Hamilton' WHERE id = '294591c7-daa3-4c84-8b16-49a031842cc5'` (the `name_normalized` column is generated; auto-recomputed to 'hamilton'). Regenerated brand decisions; Hamilton auto-resolved cleanly, Hamilton Watch resolved as needs-review with candidate `hamilton (1.00)`.

3. **Brand policy: Hamilton Watch → merge into renamed Hamilton; rest → new.** All 37 non-Hamilton-Watch needs-review rows flipped to `new`. IWC + IWC Schaffhausen left as separate `new` rows per CONTEXT.md operator note. Family policy: all 139 needs-review → `new` (no high-confidence typo merges).

The decision-file regeneration was a 30-minute operator detour that wasn't anticipated in Plan 79-05's truths. Documented as part of POST-DEPLOY for milestone retrospective.
