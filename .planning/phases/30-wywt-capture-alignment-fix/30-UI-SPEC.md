## Visual Contract — Phase 30 Specific

This section is the operative contract for Phase 30. It supersedes generic template
fields above in specificity.

### Camera Preview Wrapper

**Before (bug):** `<div class="relative w-full overflow-hidden rounded-md bg-black">`
has no fixed aspect ratio. `<video class="block w-full object-cover">` resolves to
whatever the browser computes from the stream's intrinsic ratio. On iOS this produces
a black bar at the bottom of the preview wrapper because the stream is 1920×1080
(16:9) and the wrapper is a full-width rectangle without a declared height, leaving
a gutter between the visible video rect and the wrapper bottom edge.

**After (fix):** Add `aspect-square` to the wrapper class list.

Final wrapper class string:
```
relative w-full aspect-square overflow-hidden rounded-md bg-black
```

`aspect-square` is confirmed valid in this codebase under Tailwind 4 — used at:
- `src/components/explore/DiscoveryWatchCard.tsx:34`
- `src/components/watch/WatchDetail.tsx:117`

No alternative syntax (`aspect-[1/1]`, inline style) is needed.

### Video Element

`<video>` class string (Phase 30 hotfix — `h-full` is **required**, see below):
```
block h-full w-full object-cover
```

`object-cover` semantics: the stream (e.g., 1920×1080) scales up so its SHORT edge
fills the square wrapper. The long-edge overflow is clipped. This is the visual crop
the user sees, and it is exactly what the capture canvas must reproduce.

**Why `h-full` is mandatory (post-ship hotfix):** `object-cover` only crops content
that overflows the *element's box*. Without `h-full`, the `<video>` element keeps its
intrinsic 16:9 aspect (~360×202 inside a 360×360 wrapper), so the box matches the
stream's shape and `object-cover` has nothing to crop. The video collapses to ~56% of
the wrapper height, leaving the bottom 44% as a black bar (the wrapper's `bg-black`).
The capture math then assumes a crop that never happened — the saved JPEG ends up
showing the centered slice of the stream while the user was actually looking at the
full stream squished into the top half. With `h-full`, the video element matches the
square wrapper, `object-cover` engages, the visible video equals the wrapper rect,
and `computeObjectCoverSourceRect` produces a saved JPEG that matches what the user
saw under the overlay.

### SVG Overlay

`<WristOverlaySvg className="pointer-events-none absolute inset-0" />` — UNCHANGED.
The overlay covers the wrapper via `inset-0`. When the wrapper becomes square, the
overlay naturally covers the square. The SVG's internal geometry (`viewBox=0 0 100
100`, `preserveAspectRatio="xMidYMid meet"`, arm lines y=38/y=62, circles r=22/r=17
at (50,50)) is LOCKED per D-10. Do not edit `WristOverlaySvg.tsx`.

### Capture Canvas

The canvas dimensions and source rect change from full-intrinsic to wrapper-rect-mapped.

**Before:** `captureCanvas.width = video.videoWidth; captureCanvas.height =
video.videoHeight; ctx.drawImage(video, 0, 0)` — captures the full stream including
any letterbox bands.

**After (capture math):** Given `wrapperW` and `wrapperH` from
`wrapperRef.current.getBoundingClientRect()` (both equal when wrapper is square),
and stream intrinsic `streamW = video.videoWidth`, `streamH = video.videoHeight`:

```
videoScale = max(wrapperW / streamW, wrapperH / streamH)
sx = (streamW - wrapperW / videoScale) / 2
sy = (streamH - wrapperH / videoScale) / 2
sw = wrapperW / videoScale
sh = wrapperH / videoScale

captureCanvas.width  = wrapperW   // square
captureCanvas.height = wrapperH   // square
ctx.drawImage(video, sx, sy, sw, sh, 0, 0, wrapperW, wrapperH)
```

This maps the `object-cover` visible crop back to stream coordinates and draws only
that region. The saved JPEG is exactly what the user saw in the wrapper (WYSIWYG).

### Wrapper Ref

A `wrapperRef` (`useRef<HTMLDivElement | null>(null)`) is added to
`CameraCaptureView.tsx` and placed on the wrapper `<div>`. The existing `videoRef`
is unchanged. The capture readiness guard at line 68 is extended to also check
`wrapperRef.current` is non-null.

### Downstream Renderers (read-only, no edits)

The saved JPEG is now 1:1 square (wrapper pixel dimensions, then `stripAndResize`
caps at 1080px). Existing renderers that display the saved photo:

| Component | Expected aspect | Impact |
|-----------|----------------|--------|
| `WywtTile` | roughly square crop | No change needed — square JPEG slots in |
| `WywtSlide` | roughly square crop | No change needed |
| `WearDetailHero` | `aspect-[4/5]` container | Hero clips the 1:1 JPEG — square photo in 4:5 box is fine |
| `WearPhotoClient` | `aspect-[4/5]` container | Same — no regression |

The `aspect-[4/5]` downstream containers will show a small top/bottom strip of the
saved photo cropped. This is acceptable — the watch centered in the 1:1 photo will
still appear centered in a 4:5 crop, which is the fix goal.
