## Defense-in-depth properties verified

1. **RLS enabled on all three tables.** `ALTER TABLE public.users / public.watches / public.user_preferences ENABLE ROW LEVEL SECURITY;` — see `20260420000000_rls_existing_tables.sql` lines 14, 37, 60.
2. **All UPDATE policies have both USING and WITH CHECK clauses.** `users_update_own`, `watches_update_own`, `user_preferences_update_own` — verified by the Phase 11 audit checklist in `20260423000005_phase11_debt02_audit.sql` lines 14-20.
3. **All 12 policies use `(SELECT auth.uid())` InitPlan-cached form.** Verified by the Phase 11 audit checklist in `20260423000005_phase11_debt02_audit.sql` lines 17-18.

## Ongoing regression gate

`tests/integration/debt02-rls-audit.test.ts` provides the ongoing regression gate — any future schema change that drops one of the 12 expected policies will fail both (a) the next `supabase db push` (via the DO-block in `20260423000005_phase11_debt02_audit.sql`) and (b) the integration test suite.

## Why this phase does not add new RLS SQL

The underlying work shipped in Phase 6 (2026-04-19) and the audit shipped in Phase 11 (2026-04-22). MR-03 was folded into the v3.0 roadmap as requirement `DEBT-02` and closed there. Re-adding RLS here would duplicate existing policies. Phase 999.1 exists solely to formalize the paperwork close-out.
