<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client (comment UI, Phase 57) → comment actions | HTTP-callable; the payload is untrusted. A malicious client could try to forge an authorId to edit/delete another user's comment. |
| comment action → DAL (createComment gate) | The wishlist mutual-follow gate is enforced in the DAL (Phase 54). The action surfaces the gate rejection as a structural code, it does not re-implement the gate. |
| comment action → next/cache | Only the owner's profile tag is invalidated; comment threads are uncached (D-06) so there is no shared cache surface to leak gated thread content. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-IDOR | Tampering | edit/delete authorId | mitigate | edit/delete schemas accept ONLY commentId (+body); `.strict()` rejects a forged authorId; authorId passed to the DAL is `getCurrentUser().id`; the DAL's authorship WHERE clause is the second layer. A client cannot edit/delete a comment they did not author. Verified by comments.test.ts SEC-03 case. |
| T-55-GATE | Tampering (gate bypass) | addCommentAction gate handling | mitigate | The DAL `createComment` throws `CommentGateError` for a non-mutual-follower on a wishlist watch (Phase 54 load-bearing gate); the action catches it `instanceof` (not string-match) and returns `code:'gate'` (D-09). The action does NOT re-check the gate — the DAL owns it; the action's job is structural surfacing. |
| T-55-LEAK | Information Disclosure | comment cache | mitigate / accept | SEC-05 comment half is satisfied structurally: D-06 makes comment threads uncached Server Components inside Suspense, so there is no shared cache entry that could serve viewer A's gated thread to viewer B. The only invalidation is the owner's profile grid-count tag (no gated content). Accept the slight profile-tag over-invalidation (<500 watches/user). Verified by comments.test.ts SEC-05 case (no comments:* tag). |
| T-55-SPAM | Denial of Service (inbox) | edit/delete notification | mitigate | editCommentAction and deleteCommentAction never call logNotification; only addCommentAction (INSERT) fires, and only for non-self targets (NOTIF-12). Verified by comments.test.ts edit/delete no-notification cases. |
</threat_model>
