<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client (LikeButton, Phase 56) → toggleLikeAction | Server Actions are HTTP-callable; the client payload is fully untrusted. Only { type, id } is accepted; the actor and owner are derived server-side. |
| toggleLikeAction → next/cache | Cache tags fan out to other viewers (count) and scope to the actor (RYO). A wrong tag scope leaks one viewer's state into another's cache. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-IDOR | Tampering | toggleLikeAction input + owner resolution | mitigate | Zod `.strict()` rejects any extra key (a forged owner/actor id fails the parse); the actor is `getCurrentUser().id`; the owner is resolved from watches.user_id / wear_events.user_id server-side. A crafted request with a mismatched owner cannot redirect the notification. Verified by reactions.test.ts SEC-03 cases. |
| T-55-LEAK | Information Disclosure | cache tag scoping | mitigate | `updateTag('viewer:{userId}:reactions')` scopes the actor's own liked-state cache (RYO, immediate expiry, Server-Action-only); `revalidateTag('reactions:{type}:{id}','max')` is the cross-user count only. Viewer A's liked state lives under A's viewer tag, never B's. Verified by reactions.test.ts SEC-05 case. |
| T-55-SPAM | Denial of Service (inbox) | like-notification fire condition | mitigate | logNotification fires only on `liked===true` (not unlike) and only for non-self targets; combined with the plan-02/03 ON CONFLICT dedup, rapid like/unlike → at most one notification. Verified by reactions.test.ts NOTIF-11 + logger.test.ts NOTIF-14. |
</threat_model>
