## Summary

Phase 57.1 is a surgical polish pass on already-shipped components. The UI-SPEC is the locked visual contract; this research answers the two genuinely open questions and validates the structural assumptions the planner will work from.

**D-01 resolution (highest risk item):** `router.refresh()` (from `next/navigation`) is the correct client-side tool here. The Next.js 16 docs confirm it re-fetches the current route's RSC payload and merges it without losing unaffected client state. Crucially, it operates on the **client cache** — it does NOT interact with the server-side tag cache (`revalidatePath`/`revalidateTag`). This means it cannot re-trigger the Router Cache poisoning pattern that caused the #419/#404 incidents on `/u/[username]/[tab]`. Those incidents were caused by `unstable_instant = { prefetch: 'static' }` (aborted secondary prerender) and stale prefetched RSC payloads — not by `router.refresh()` calls. The `/watch/[id]` page does not use `unstable_instant` and `CommentThread` is an uncached RSC (no `'use cache'` directive, confirmed in source), so `router.refresh()` on the success branch is safe and correct here.

**Prop chain verification:** The UI-SPEC's assumed plumbing (`WearCard → WearCommentHost → CommentList` for `onCountChange`, and `page.tsx → CommentThread → CommentList` for `suppressCompose`) matches the actual code structure **with one meaningful delta** noted below: `CommentList` currently has no `suppressCompose` prop and no `onCountChange` prop — both are new additions planned by the spec, which is expected. However, the SC-6 CMNT-09 count gate has a latent bug risk documented below.

**Primary recommendation:** Use `router.refresh()` (client component, on success branch only) for D-01. Add `revalidatePath` to `addCommentAction` as a belt-and-suspenders measure only if needed — but based on the architecture, `router.refresh()` alone is sufficient and the cleaner approach since `CommentThread` is already uncached.

## D-01: `router.refresh()` — Detailed Findings

### What the Next.js 16 docs say

**Source:** `node_modules/next/dist/docs/01-app/03-api-reference/04-functions/use-router.md` [VERIFIED]

> `router.refresh()`: Refresh the current route. Making a new request to the server, re-fetching data requests, and re-rendering Server Components. **The client will merge the updated React Server Component payload without losing unaffected client-side React (e.g. `useState`) or browser state (e.g. scroll position). This clears the Client Cache for the current route, but does NOT invalidate the server-side cache.**

Key properties confirmed from the docs:
1. `router.refresh()` re-renders the entire RSC tree for the current route — including the `CommentThread` sibling RSC in `/watch/[id]`.
2. It **preserves** unaffected client-side `useState` — so `WatchDetail`'s like state, `isEditing` states, etc. are untouched.
3. It clears the **client cache** only, not the server-side tag cache. This is the correct scoping for this use case.
4. A separate `refresh` function in `next/cache` exists but is **server-only** (can only be called from Server Actions, not Client Components). [VERIFIED from `node_modules/next/dist/docs/01-app/03-api-reference/04-functions/refresh.md`]

**There are TWO different `refresh` APIs in Next.js 16:**
- `router.refresh()` from `next/navigation` — client component, clears client cache, re-fetches RSC payload
- `refresh()` from `next/cache` — server-only, works inside Server Actions

For D-01 the client-component seam (`CommentList` is `'use client'`) requires `router.refresh()` from `next/navigation`. **This is already imported in `CommentCompose.tsx` (line 4), confirming the import pattern is established in the codebase.** [VERIFIED: source read]

### Why the cache-poisoning incidents do NOT apply here

The Router Cache poisoning incidents (Phases 51, 52, 56A) shared these characteristics:
- They involved `unstable_instant = { prefetch: 'static' }` aborting secondary prerenders (React #419)
- They involved stale prefetched RSC payloads being served on navigation to dynamic routes
- The fix was `unstable_instant = false` on `/u/[username]/[tab]` and `Cache-Control: no-store` on 307 redirects

`router.refresh()` is categorically different: it is an **explicit, user-triggered refresh** of the current route — not a prefetch, not a static prerender. It makes a fresh server request and replaces the client cache entry. [VERIFIED from docs]

`/watch/[id]/page.tsx` does NOT export `unstable_instant` — confirmed by source read. The route is dynamically rendered (calls `getCurrentUser()`, `getWatchByIdForViewer()`, etc. at request time). `router.refresh()` on a fully dynamic route is the standard pattern and carries no cache-poisoning risk. [VERIFIED: source read + docs]

### Where the post-success seam is

**The seam is in `CommentList.tsx`**, not in a separate client island. The flow:

```
/watch/[id]/page.tsx (Server Component)
  └─ <CommentThread viewerId target canComment ...> (RSC — no 'use cache')
       └─ <CommentList initialComments ...> ('use client')
            └─ handleSubmit() → calls addCommentAction → on success: currently setComposeKey(k+1)
```

`CommentList.handleSubmit()` already has a `startTransition` block that runs after `addCommentAction` resolves. The `router.refresh()` call goes **inside this transition, on the `result.success` branch, after the reconcile**. `CommentList` already imports nothing from `next/navigation` — the `useRouter()` hook must be added there (not in `CommentCompose` which already imports it but only uses `router.push` for anon bounce). [VERIFIED: source reads]

**The planner must add `useRouter()` to `CommentList.tsx` and call `router.refresh()` in the success branch of `handleSubmit`.** The call goes after `setComposeKey((k) => k + 1)` so the compose is cleared first.

### D-01 Recommendation (planner instruction)

Add to `CommentList.tsx`:
```ts
const router = useRouter()
// inside handleSubmit, after setComposeKey:
router.refresh()
```

Scope guard: only on the `/watch/[id]` surface. Since `CommentList` is shared across all surfaces (wears-lane, wear detail, watch detail), the `router.refresh()` call would fire on all three. This is **acceptable** — `router.refresh()` on the wears pages simply re-fetches the RSC data for those pages too, which is harmless (the wears-lane optimistic count is already local state and survives the merge per the docs). It is not incorrect behavior.

However, if the planner wants to scope it strictly to `/watch/[id]` only (per CONTEXT D-01 "scope guard"), the cleanest approach is to add an optional `onPostSuccess?: () => void` prop to `CommentList` and call it from the success branch. The `/watch/[id]` page threads `onPostSuccess={() => router.refresh()}` down through `CommentThread → CommentList`. The wears surfaces leave it undefined. This avoids importing `useRouter` into `CommentList` entirely. **Either approach is valid; the planner may choose.**

**Fallback:** If `router.refresh()` causes unexpected issues in prod testing, fall back to leaving the count static on `/watch/[id]` as the UI-SPEC default (no change to the component). The count updates correctly on next page load/navigation.

## Common Pitfalls

### Pitfall 1: Using `refresh()` from `next/cache` instead of `router.refresh()`
**What goes wrong:** `refresh()` from `next/cache` is server-only; calling it in a client component throws a runtime error.
**Why it happens:** Both APIs are called "refresh" — different imports, different contexts.
**How to avoid:** In `CommentList` (`'use client'`), import `useRouter` from `next/navigation` and call `router.refresh()`.

### Pitfall 2: Gating the count read on `canCommentDisplay` (breaks CMNT-09 for owner)
**What goes wrong:** After SC-6, `canCommentDisplay=false` for the owner. If the count read on page.tsx line 69 uses `canCommentDisplay`, the owner's `commentCount` prop becomes 0 and the badge disappears.
**Why it happens:** UI-SPEC SC-6 contract says "use `canCommentDisplay`" for the count read — that wording assumes `canCommentDisplay` equals `canComment` for non-owners (true) but is false for owners where GATE-04 means `canComment=true`. The owner needs the count.
**How to avoid:** Keep line 69 as `canComment ? ... : 0`. The `canComment` variable already returns `true` for the owner via GATE-04.

### Pitfall 3: `router.refresh()` clears the compose box state
**What goes wrong:** `router.refresh()` re-renders RSC siblings but merges client state. However, `CommentList` uses `composeKey` to re-mount `CommentCompose` on success — this happens before `router.refresh()` runs. The compose box is correctly cleared by `setComposeKey` first.
**Why it happens:** Non-issue if sequenced correctly (`setComposeKey` then `router.refresh()`).
**How to avoid:** Call `setComposeKey` before `router.refresh()` in the success branch — order matters.

### Pitfall 4: SC-3 + D-04 — the `mt-1` delete confirm block below the body
**What goes wrong:** After SC-3 moves the icons to the meta row, the `isDeleting` confirm block (currently below the body) must also move to be inline on the meta row per D-04. If only the icons are moved but the confirm remains below, the confirm still hangs below the body — exactly the problem SC-3 was fixing.
**Why it happens:** Two separate JSX blocks need to move; easy to miss the second one.
**How to avoid:** The meta row flex container must have two mutually exclusive states: `isAuthor && !isEditing && !isDeleting → icons`; `isDeleting → confirm text`. Both render inside the same `flex items-center gap-1 flex-wrap` wrapper.

### Pitfall 5: `onCountChange` on rollback direction
**What goes wrong:** On rollback-delete (delete fails, comment re-inserted), `onCountChange(+1)` must be called to restore the badge. On optimistic-delete (before server confirms), `onCountChange(-1)` must be called. On optimistic-insert rollback (post fails), `onCountChange(-1)` reverses the earlier `+1`. Easy to flip signs or miss a branch.
**Why it happens:** Four branches in `CommentList` (insert success path, insert rollback, optimistic delete, delete rollback).
**How to avoid:** Map each branch explicitly in the plan task. The UI-SPEC interaction table (Section: Interaction Contracts) is the canonical reference.
