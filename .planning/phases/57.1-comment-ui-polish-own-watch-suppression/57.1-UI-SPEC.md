### SC-1: Wears-Lane Comment Drawer — Desktop Max-Width Centering

**Affected file:** `src/components/wear/WearCommentHost.tsx`

**Current state (lines 86-88):**
```
<SheetContent side="bottom" className="bg-background max-h-[60vh] overflow-y-auto z-50">
```

**Required change:**
The `SheetContent` inner content area must be centered with a max-width on desktop.
The Sheet panel itself spans full viewport width (that is the shadcn Sheet behavior and
must not change). The centering applies to the content INSIDE the Sheet.

**Contract:**
- Wrap the `SheetHeader` + `CommentList` in a new `<div>` inside `SheetContent`.
- That wrapper div: `className="mx-auto w-full max-w-[640px]"`
- Mobile behavior: unchanged (the 640px cap is wider than any mobile viewport so
  it never constrains on mobile — behavior is identical).
- Desktop behavior: content is horizontally centered at 640px maximum width.
- The 640px cap matches the WearsLane photo column width (600px) plus 40px of lateral
  breathing room. It is NOT 100% of the Sheet width on large displays.
- Do NOT change `SheetContent` className itself; do NOT add `max-w-` to SheetContent.

**Precise diff shape:**
```tsx
// Before
```

### SC-2: CommentCompose — Inline Post Button

**Affected file:** `src/components/comment/CommentCompose.tsx`

**Current state (lines 50-81):**
The current layout is `flex flex-col gap-2` — textarea on top, then a second
flex row containing the char counter and the Post button stacked below.

**Required change:**
The Post button must sit inline (to the right) of the textarea on the same visual
row — a side-by-side layout. The char counter stays above or below the textarea
as a subordinate element; it does not share a row with the Post button.

**Contract:**
- Outer container: `flex items-end gap-2` (replaces `flex flex-col gap-2`)
- Textarea: `flex-1 min-w-0` — takes all available horizontal space
- Post button: `flex-shrink-0 self-end h-8 px-4 ...` (retains existing button classes;
  `self-end` aligns it to the bottom of the textarea when the textarea grows)
- Char counter: moves ABOVE the textarea as a `<div>` with `text-right` before the
  main flex row, OR remains as a trailing element inside the textarea wrapper.
  Recommended approach: char counter stays below the textarea-flex row as a separate
  `<p>` with `text-right text-xs` — it is only visible near the 450-char threshold.
- The Post button MUST NOT move to a new line below the textarea on any viewport.
- The char counter appears at `body.length >= 450` (unchanged CMNT-05 threshold).
- The char counter turns `text-destructive` at `body.length >= 480` (unchanged).
- Post button disabled states (pending / blank / over-limit) are unchanged.
- `min-h-[44px] min-w-[44px]` on the Post button is preserved for touch targets.

**Precise layout classes:**
```tsx
// Outer wrapper
<div className="flex items-end gap-2">
  {/* Textarea wrapper — grows to fill */}
  <div className="flex-1 min-w-0 flex flex-col gap-1">
    <textarea ... className="w-full rounded-md border border-input bg-background px-3 py-2 text-sm text-foreground placeholder:text-muted-foreground focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring disabled:cursor-not-allowed disabled:opacity-50 resize-none" />
    {body.length >= 450 && (
      <span className={cn('text-xs tabular-nums text-right', body.length >= 480 ? 'text-destructive' : 'text-muted-foreground')}>
        {body.length}/500
      </span>
    )}
  </div>
  {/* Post button — right-aligned, inline */}
  <button type="button" disabled={isDisabled} className="flex-shrink-0 self-end h-8 px-4 bg-primary text-primary-foreground text-sm font-semibold rounded-md hover:opacity-90 focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring disabled:cursor-wait disabled:opacity-50 min-h-[44px] min-w-[44px]" onClick={handleSubmit}>
    Post
  </button>
</div>
```

**Inheritance note:** `CommentCompose` is shared — this change automatically applies
to the wears-lane bottom-sheet drawer, `/wear/[wearEventId]`, and `/watch/[id]`.
No per-host changes needed for SC-2.

### SC-3: CommentItem — Inline Edit/Delete Controls

**Affected file:** `src/components/comment/CommentItem.tsx`

**Current state (lines 155-179):**
The Pencil and Trash2 buttons are in a `flex items-center gap-2 mt-1` row rendered
BELOW the comment body — this is stacked below the text, not inline with it.

**Required change:**
Edit/delete controls must render inline with the comment's meta line (username ·
timestamp row), not stacked below the body.

**Contract:**
- The meta line (`<p>` with username + timeAgo + edited indicator) gains a
  `flex items-center gap-2` wrapper so the author controls sit on the same line.
- Pencil and Trash2 buttons remain `min-h-[44px] min-w-[44px]` for touch targets.
- The `gap-2 mt-1` block that currently renders controls below the body is removed.
- The delete confirm (`flex items-center gap-1 mt-1` with "Delete? · Cancel") retains
  its current position below the body — this is acceptable as a confirmation state.
- Edit mode (textarea swap) retains its current layout below the meta line — acceptable.
- The controls are ONLY visible when `isAuthor && !isEditing && !isDeleting` (unchanged).
- Icon size: `size-3.5` (14px) — reduced from `size-4` so controls sit proportionally
  on the meta line (12px text). This is the only size change in this component.

**Precise layout for the meta + controls row:**
```tsx
{/* Meta line with inline author controls */}
<div className="flex items-center gap-1 flex-wrap">
  <p className="text-xs text-muted-foreground flex-1 min-w-0">
    <Link href={...} className="font-semibold text-foreground hover:underline focus-visible:underline focus-visible:outline-none">
      {comment.author.username}

    {timeAgo(comment.createdAt)}
    {comment.editedAt && (
      <> · <span className="text-xs text-muted-foreground">edited</span></>
    )}
  </p>
  {isAuthor && !isEditing && !isDeleting && (
    <div className="flex items-center gap-1 flex-shrink-0">
      <button aria-label="Edit comment" type="button" disabled={pending} onClick={...}
        className="min-h-[44px] min-w-[44px] inline-flex items-center justify-center focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring rounded-md">
        <Pencil className="size-3.5 text-muted-foreground hover:text-foreground" aria-hidden />
      </button>
      <button aria-label="Delete comment" type="button" disabled={pending} onClick={...}
        className="min-h-[44px] min-w-[44px] inline-flex items-center justify-center focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring rounded-md">
        <Trash2 className="size-3.5 text-muted-foreground hover:text-destructive" aria-hidden />
      </button>
    </div>
```

**The standalone `<div className="flex items-center gap-2 mt-1">` block (lines 156-179)

**Inheritance note:** `CommentItem` is shared — this change automatically applies to

### SC-4: Wears-Lane Comment Drawer — Bottom Spacing

**Affected file:** `src/components/wear/WearCommentHost.tsx`

**Current state (line 86):**
`<SheetContent side="bottom" className="bg-background max-h-[60vh] overflow-y-auto z-50">`
The SheetContent has no explicit bottom padding. The shadcn Sheet default adds `p-6`
on the content area, but safe-area inset coverage for mobile is not guaranteed.

**Required change:**
The inner content wrapper introduced in SC-1 (the `max-w-[640px]` div) must include
bottom padding that pushes the last comment and the compose box clear of the device
bottom bar on iPhone (notch/home-bar area).

**Contract:**
- The `max-w-[640px]` wrapper div from SC-1 gets: `className="mx-auto w-full max-w-[640px] pb-[calc(1.5rem+env(safe-area-inset-bottom))]"`
- This is `24px` (1.5rem) baseline padding plus the device's safe-area-inset-bottom.
- On desktop (no notch), `env(safe-area-inset-bottom)` resolves to 0px, so this is
  equivalent to `pb-6`.
- On iOS Safari with home indicator, the inset adds ~34px, giving ~58px total bottom
  clearance — enough for the compose box to clear the home indicator.
- This replaces any simple `pb-6` that may already be present on the inner content.

**The combined final className for the inner wrapper (SC-1 + SC-4 merged):**
```tsx
<div className="mx-auto w-full max-w-[640px] pb-[calc(1.5rem+env(safe-area-inset-bottom))]">
```

### SC-5: Optimistic Comment Count Badge

**Affected files:**
1. `src/components/wear/WearCard.tsx` — hosts the `commentCount` badge in the engagement row
2. `src/components/comment/CommentList.tsx` — the component that owns optimistic inserts

**Context:** `commentCount` is currently a static prop passed from the server page.
When the user posts a comment, the badge does not increment until the page is refreshed
because `commentCount` is a server-resolved number, not a client-tracked counter.

**Required change: WearCard (bottom-sheet variant)**
`WearCard` currently renders `commentCount` directly from its prop in the engagement row.
It must instead maintain a local `commentCount` state that starts from the prop and is
incremented/decremented as comments are added or deleted.

However, `WearCard` is not the comment-owner — `WearCommentHost` owns the `CommentList`.
The cleanest seam: lift the count adjustment into `WearCommentHost` via a callback,
or expose a ref. The preferred approach (minimal blast radius) is:

**Preferred approach — local state + callback in WearCard:**
- `WearCard` manages `const [localCommentCount, setLocalCommentCount] = useState(commentCount)`
- `WearCard` passes `onCountChange: (delta: number) => void` to `WearCommentHost`
- `WearCommentHost` passes `onCountChange` to `CommentList`
- `CommentList` calls `onCountChange(+1)` on optimistic insert and `onCountChange(-1)` on
  optimistic delete; on rollback it calls the inverse.

**Contract for `WearCard.tsx`:**
- Add `const [localCommentCount, setLocalCommentCount] = useState(commentCount)` (initialized from prop)
- Replace every `{commentCount > 0 && ...}` with `{localCommentCount > 0 && ...}`
- Add `onCountChange={(delta) => setLocalCommentCount((n) => Math.max(0, n + delta))}` prop on both `WearCommentHost` usages
- Guard `Math.max(0, ...)` to prevent negative badge display

**Contract for `WearCommentHost.tsx`:**
- Add `onCountChange?: (delta: number) => void` to both union variants of `WearCommentHostProps`
- Thread it through to `CommentList` as the same prop

**Contract for `CommentList.tsx`:**
- Add `onCountChange?: (delta: number) => void` to `CommentListProps`
- Call `onCountChange?.(+1)` immediately after the optimistic insert in `handleSubmit`
- Call `onCountChange?.(-1)` on rollback (inside the `!result.success` branch, after `prev.filter`)
- Call `onCountChange?.(+1)` on rollback-delete (inside `handleRollbackDelete`)
- Call `onCountChange?.(-1)` on optimistic delete (inside `handleDeleteOptimistic`)
- Note: on successful reconcile, no additional call is needed — the count is already +1 from the optimistic insert

**Contract for `/watch/[id]` (SC-5 architectural note):**
The `commentCount` badge in `WatchDetail.tsx` (line 169-175) is a static span in the
`'use client'` island, sourced from `commentCount` prop. The `CommentThread` RSC is a
separate sibling subtree — it cannot call back into `WatchDetail`. This is the
"architecturally feasible" caveat in ROADMAP SC-5. The `/watch/[id]` badge is NOT made
optimistic in this phase. The count updates only on next render/refresh. Document
this explicitly in code via a comment; do not attempt to bridge the RSC/client boundary.

**No wiring change is needed for `/wear/[wearEventId]` inline variant:**
The inline `WearCard` at `/wear/[wearEventId]` already renders `commentCount` from
its prop. The same `WearCard` changes above apply — `WearCard` manages local state
regardless of variant. The inline variant also benefits from `onCountChange` because
`WearCommentHost variant="inline"` renders `CommentList` the same way.

### SC-6: Own-Watch Compose Suppression on /watch/[id]

**Affected file:** `src/app/watch/[id]/page.tsx`

**Current state (lines 47-49):**
```ts
const [likeState, canComment] = await Promise.all([
  getLikesForTargetCached(user.id, target),
  canViewerCommentOnTarget(user.id, target),
])
```

`canViewerCommentOnTarget` in the DAL returns `true` for the owner (GATE-04). This
means the owner currently sees a compose box on their own watch. Phase 57.1 reverses
this for the `/watch/[id]` route only — the compose box must be suppressed for the
watch owner. All other viewers' access is unchanged.

**Contract:**
After `canViewerCommentOnTarget` resolves, apply an owner override:
```ts
// SC-6 (Phase 57.1): suppress compose on own watches — owner uses Notes for private context.
// GATE-04 still allows the DAL to pass through comments (existing comments remain readable).
// This is a display-only suppression at the route level; the DAL gate is not changed.
const canCommentDisplay = isOwner ? false : canComment
```

Then pass `canCommentDisplay` (not `canComment`) to `CommentThread`:
```tsx
<CommentThread
  viewerId={user.id}
  target={target}
  canComment={canCommentDisplay}  // SC-6: false for owner
  ...
/>
```

**Constraints:**
- `canComment` (the raw DAL result) is still used for `ownerFollowsViewer` and
  `viewerIsFollowing` resolution below (lines 56-63) — do NOT rename that variable;
  introduce `canCommentDisplay` as an additional derived value.
- The `commentCount` read (line 69: `canComment ? ...length : 0`) must use
  `canCommentDisplay` so the owner's badge still shows the count (CMNT-09 — the count
  badge is display-only; it must remain visible to the owner).
- No DAL changes. No Server Action changes. The DAL `canViewerCommentOnTarget` continues
  to return `true` for the owner — the override is a route-level display gate only.
- The `CommentGateLocked` component is NOT shown for the owner. When `canComment=false`,
  `CommentList` renders `CommentGateLocked`. That is wrong for the owner case. The
  correct behavior: when the viewer is the owner AND `canCommentDisplay=false`, render
  nothing in the compose slot — no gate copy, no locked state.

**Therefore, `CommentList` must distinguish "owner suppressed" from "gate locked".**

**Contract for `CommentList.tsx`:**
Add an optional prop `suppressCompose?: boolean` (default `false`).
- When `suppressCompose=true`: render neither `CommentCompose` nor `CommentGateLocked` in
  the compose slot — the list renders with no compose area above it.
- When `suppressCompose=false` (default): existing behavior (canComment → compose or gate).

**Contract for `CommentThread.tsx`:**
Add `suppressCompose?: boolean` to `CommentThreadProps`. Pass it through to `CommentList`.

**Contract for `/watch/[id]/page.tsx`:**
Pass `suppressCompose={isOwner}` to `CommentThread`.

**The full call chain:**
```
page.tsx → CommentThread suppressCompose={isOwner} → CommentList suppressCompose={isOwner}
```

When `suppressCompose=true` in `CommentList`, the compose slot is `null` — no
`<CommentCompose>`, no `<CommentGateLocked>`. The comment list still renders below.
Existing comments (posted by others before the owner suppressed) remain readable.

**Wears hosts are unaffected:** `WearCommentHost` never passes `suppressCompose`.
Only `/watch/[id]` sets `suppressCompose={isOwner}`.

## Interaction Contracts

### Optimistic Comment Insert (SC-5)

| Event | State change | Visible effect |
|-------|-------------|----------------|
| User submits comment (Post clicked) | `localCommentCount += 1` in WearCard | Badge increments immediately |
| Server action succeeds | No count change (already incremented) | Count stays correct |
| Server action fails (rollback) | `localCommentCount -= 1` in WearCard via `onCountChange(-1)` | Badge decrements |
| Gate rejection (`code==='gate'`) | `canComment` flips to false; `onCountChange(-1)` | Compose hides; badge decrements |
| Optimistic delete | `localCommentCount -= 1` immediately | Badge decrements |
| Delete rollback | `localCommentCount += 1` | Badge increments |

### Own-Watch Compose Suppression (SC-6)

| Viewer | Watch status | Compose slot |
|--------|-------------|--------------|
| Owner (isOwner=true) | any | null (no compose, no gate copy) |
| Mutual follower | wishlist | CommentCompose rendered |
| Non-mutual follower | wishlist | CommentGateLocked (gate copy + FollowButton) |
| Any authenticated viewer | owned/sold/grail | CommentCompose rendered |

### CommentCompose Layout (SC-2)

| State | Layout |
|-------|--------|
| Default | Textarea (flex-1) + Post button (flex-shrink-0) side by side, items-end aligned |
| Typing (< 450 chars) | No char counter visible |
| Nearing limit (>= 450 chars) | Char counter appears below textarea in muted-foreground; layout unchanged |
| Over warning (>= 480 chars) | Char counter turns text-destructive; Post button disabled |
| Pending (action in flight) | Textarea disabled; Post button disabled + cursor-wait |

### CommentItem Controls (SC-3)

| State | Controls location |
|-------|------------------|
| Default (author viewing) | Pencil + Trash2 icons inline with meta line (username · timestamp row) |
| Editing | Save + Cancel buttons below the textarea (unchanged) |
