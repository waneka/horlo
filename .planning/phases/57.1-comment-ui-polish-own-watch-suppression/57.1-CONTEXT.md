# Phase 57.1: Comment UI Polish + Own-Watch Suppression - Context

**Gathered:** 2026-05-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Polish the Phase 57 comment surfaces from prod UAT, and reverse the owner-can-comment
case for **watches only**. This is a POLISH phase that refines already-shipped shared
components (`CommentThread`, `CommentList`, `CommentItem`, `CommentCompose`,
`CommentGateLocked`, `WearCommentHost`, `WearCard`) — **no new components**.

The six ROADMAP success criteria are the fixed scope:
1. Desktop: `/wears/[username]` comment drawer content centered with a max-width (mobile unchanged).
2. Post button **inline** with the compose input (shared `CommentCompose`).
3. Comment edit/delete controls **inline** with the comment row (shared `CommentItem`).
4. `/wears/[username]` drawer has adequate bottom spacing.
5. Posting increments the comment-count badge optimistically (no refresh) — **required** on
   wears-lane + `/wear/[wearEventId]`; on `/watch/[id]` where architecturally feasible.
6. A viewer on their **OWN** watch at `/watch/[id]` sees **no compose box** (owner case of
   CMNT-01 reversed); every other viewer's access unchanged.

**The approved `57.1-UI-SPEC.md` is the locked visual + interaction contract.** It specifies
precise class names, prop chains, and file-level diffs for all six criteria. This CONTEXT
captures only the product/UX judgment calls resolved in discussion that **refine or confirm**
the spec's defaults — downstream agents follow the UI-SPEC for everything else.

**Out of scope (later):** SEED-015 (inline grid like/comment engagement), SEED-016
(`/watch/[id]` redesign). Do NOT expand into these. No DAL or Server Action changes — the
DAL `canViewerCommentOnTarget` still returns `true` for owners (GATE-04 unchanged); SC-6 is a
**route-level display gate only**.

</domain>

<decisions>
## Implementation Decisions

These four decisions came out of discussion. Each is annotated with how it relates to the
approved UI-SPEC. Where a decision **refines** the spec, the spec's default is superseded by
the decision below and the planner should plan to the decision (and may update the UI-SPEC
note to match).

### SC-5 — Comment count update on /watch/[id]
- **D-01 (REFINES UI-SPEC SC-5 architectural note):** On `/watch/[id]`, after a **successful**
  comment post, call `router.refresh()` to re-fetch the RSC so the count badge updates without a
  manual refresh. The UI-SPEC default was "leave it static, document the RSC/client boundary."
  That default is superseded: we accept the boundary (no true optimistic increment, no
  restructuring of `WatchDetail`) but trigger a server re-fetch so the count is not visibly stale.
  - **Planner constraint:** verify `router.refresh()` plays well with the Next 16 Router Cache
    behavior recorded in project memory (stale client-component instance on revisited dynamic
    URLs; proxy/Router-Cache poisoning mitigations). Do not introduce a refresh path that
    re-triggers the cache fragility Phases 51/52/56A fought. If `router.refresh()` proves unsafe
    here, fall back to the spec's static default rather than restructuring — flag it.
  - **Scope guard:** `router.refresh()` fires only on the success branch, only on `/watch/[id]`.
    The wears-lane and `/wear/[wearEventId]` keep pure optimistic local state (see D-03).

### SC-6 — Owner suppression presentation
- **D-02 (CONFIRMS UI-SPEC SC-6 / Copywriting Contract):** When `suppressCompose=true` (viewer is
  the watch owner), the compose slot renders **`null`** — **fully silent**, no helper copy, no gate
  state. No new copy string is introduced. The comment list still renders below (existing comments
  from others remain readable); the count badge stays visible to the owner (`canCommentDisplay`
  feeds the count read so CMNT-09 holds). This matches the ROADMAP intent ("notes cover that").

### SC-5 — Optimistic count consistency (wears-lane + /wear)
- **D-03 (CONFIRMS UI-SPEC SC-5):** The wears-lane / `/wear/[wearEventId]` optimistic count is
  **local state only** and self-corrects on the next full page load/navigation. **No reconcile
  logic** (no drawer-reopen re-sync, no post-success refetch on these surfaces). Drift requires a
  concurrent commenter on the same target while the drawer is open — an accepted rare edge for a
  personal-collector app. Eventual consistency on reload is sufficient for MVP.

### SC-3 — Delete-confirm placement
- **D-04 (REFINES UI-SPEC SC-3):** The inline "Delete? · Cancel" confirm renders **inline on the
  meta row**, replacing the edit/delete icons in place (the icons and the confirm are
  mutually-exclusive states of the author controls: icons when
  `isAuthor && !isEditing && !isDeleting`; confirm when `isDeleting`). The UI-SPEC kept the confirm
  below the body and called it "acceptable" — that is superseded: keeping a control element below
  the body re-introduces the exact "controls hanging below" feel the inline move (SC-3) was fixing.
  - **Planner note:** "Delete? · Cancel" replaces equal-width icons on the meta row, so it should
    fit; watch for wrap on very narrow widths (bottom-sheet on small phones) — allow the meta row
    to wrap gracefully (`flex-wrap` already in the SC-3 layout) rather than truncating.

### Carried forward from Phase 57 (LOCKED — do not re-litigate)
- **GATE-03 two-state gate copy** for non-owners (Phase 57 D-01..D-04) is unchanged: pre-follow
  "Follow {username} to comment" + FollowButton; followed-but-not-mutual "{username} needs to
  follow you back before you can comment". SC-6 suppression is **owner-only** and does NOT touch
  the non-owner gate.
- **Edit/delete UX** (Phase 57 D-05..D-07): always-visible pencil + trash on the author's own
  comments; edit-in-place (Save/Cancel textarea swap, 500-char enforcement); `[edited]` suffix on
  the meta line. SC-3 moves these inline; D-04 moves the delete confirm inline too. Nothing else
  about edit/delete changes.
- **Count surfacing** (Phase 57 D-08): `MessageCircle` icon + bare inline number, **hidden at
  zero**, in the footer action row next to the like control. SC-5 makes this number update
  optimistically; the visual form is unchanged.
- **Newest-first comments + compose-above** (operator override 2026-05-22, reconciled in Phase 57):
  unchanged. The SC-2 inline Post button sits with the compose input which remains above the list.

### Claude's Discretion
- Exact char-counter placement in `CommentCompose` (above vs. below the textarea-flex row): the
  UI-SPEC recommends below; planner may follow the spec's recommended shape.
- The precise SC-5 callback plumbing (`onCountChange` chain `WearCard → WearCommentHost →
  CommentList`) is the UI-SPEC's recommended seam; planner may keep it or choose an equivalent
  minimal-blast-radius approach, provided the optimistic increment/rollback semantics in the
  UI-SPEC interaction table hold.

</decisions>

## Existing Code Insights

### Reusable Assets
- All five comment components and the two wear hosts already exist and shipped in Phase 57/56A —
  this phase only edits them. No scaffolding.
- `cn()` (`src/lib/utils.ts`) for conditional class composition; existing OKLCH token system in
  `src/app/globals.css`; shadcn `Sheet` family already installed.
- `FollowButton` (`src/components/profile/FollowButton.tsx`) powers the GATE-03 locked state — not
  touched here, but it's the non-owner path that SC-6 must leave intact.

### Established Patterns
- Optimistic mutation pattern already lives in `CommentList` (optimistic insert at top → reconcile
  on success / rollback on failure). SC-5 hooks `onCountChange` into the existing insert/delete/
  rollback branches — no new mutation machinery.
- 44px touch-target minimums (`min-h-[44px] min-w-[44px]`) on all comment buttons — preserve.
- Owner identity is already resolved as `isOwner` in `src/app/watch/[id]/page.tsx`; SC-6 reuses it.

### Integration Points
- `router.refresh()` (D-01) attaches to the **success branch** of the post action on `/watch/[id]`
  only — the client island that renders compose/thread. Must not affect wears surfaces.
- SC-6 keeps the DAL gate (`canViewerCommentOnTarget`) untouched; introduces `canCommentDisplay`
  as an additional derived value at the route level so the count read (CMNT-09) still works for the
  owner.

### Verification note (from project memory)
- **Mobile/visual behavior verifies on PROD**, not locally: structural code passes locally, then
  push origin main → Vercel and the user confirms the visual/mobile behavior. Bundle this phase's
  UI changes for a single deploy; build-gate before push. Local e2e skips (empty test DB).
