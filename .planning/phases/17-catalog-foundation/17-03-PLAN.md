## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated user → addWatch Server Action | Auth-gated by `getCurrentUser()` at line 54; user.id flows into linkWatchToCatalog |
| Authenticated user → POST /api/extract-watch | Auth-gated at line 10; SSRF + protocol allowlist at lines 39-44 |
| URL extractor LLM output → catalog write | Sanitized in Plan 02 DAL (`sanitizeHttpUrl`, `sanitizeTagArray`) |
| Catalog DAL failure → user-visible error | MUST be silenced (Pitfall 9) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-03-01 | Denial of service | Catalog wiring throws → addWatch fails → user can't add watch | mitigate | Inner try/catch around catalog wiring; catalog failure logged to console.error and swallowed. The watch row is committed regardless. Asserted by `tests/actions/addwatch-catalog-resilience.test.ts`. |
| T-17-03-02 | Tampering | linkWatchToCatalog called with another user's watchId | mitigate | Plan 02's `linkWatchToCatalog` UPDATE WHERE includes `eq(watches.userId, userId)` — cross-user is a no-op (zero rows). The Server Action's `user.id` comes from `getCurrentUser()` (auth-gated). Defense-in-depth at DAL + action layers. |
| T-17-03-03 | Information disclosure | URL-extract path leaks SSRF target via image_source_url | accept | image_source_url is the URL the user submitted (not the resolved IP). The SSRF check at route.ts line 39-44 is upstream of the catalog write. Already hardened in v1.0 (SEC-01). |
| T-17-03-04 | Tampering | Image extractor LLM emits malicious image_url that flows into catalog | mitigate | Plan 02's `sanitizeHttpUrl()` rejects non-http/https. Asserted by phase17-upsert-coalesce.test.ts "image_source_url rejects non-http". This plan does not duplicate that check. |
| T-17-03-05 | Repudiation | Failed catalog wiring leaves no audit trail | accept | console.error log on failure is acceptable for v4.0 (mirrors logActivity / logNotification fire-and-forget). v5+ admin tooling can add an audit table if needed. |
</threat_model>
