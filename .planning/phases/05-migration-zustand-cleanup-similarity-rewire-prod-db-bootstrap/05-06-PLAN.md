## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator's terminal → prod Supabase DB | Direct Postgres connection with service password; can modify or drop schema |
| operator's terminal → Vercel production env | Sets secrets that horlo.app runtime reads; wrong values break the live site |
| horlo.app → Supabase Auth | Real user signup flow; test user briefly exists in prod auth table |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-06-01 | I (Information Disclosure) | DB password typed into terminal | mitigate | Password is never committed; the runbook uses a placeholder. Operator types it interactively into supabase link and into the DATABASE_URL shell variable. After the run, the operator should clear shell history (history -c) if the password was typed in a command rather than entered at a prompt. |
| T-05-06-02 | T (Tampering) | drizzle-kit migrate with wrong URL (port 6543) | mitigate | Runbook Section 2 explicitly calls out the direct-vs-pooler distinction and forbids port 6543 for migrations. Rollback section documents recovery via psql and drizzle.__drizzle_migrations table cleanup. This checkpoint task explicitly repeats the "do NOT use the port 6543 pooler URL here" warning. |
| T-05-06-03 | I (Information Disclosure) | Vercel env var leak via wrong scope | mitigate | Runbook and this checkpoint task both mandate the production argument on every vercel env add invocation. If operator omits it, horlo.app simply fails to load data — the smoke test catches this before the checkpoint resolves. |
| T-05-06-04 | R (Repudiation) | test user lifecycle | mitigate | Smoke test Step 6 and the parity test Step 7 both explicitly require deleting the test user from the Supabase dashboard after verification. Prevents leftover test identities in prod auth logs. |
| T-05-06-05 | D (Denial of Service) | bad migration applied to prod | mitigate | Runbook rollback section documents recovery. Migration is small (initial snapshot + shadow-user trigger). Worst case: drop and recreate tables via psql. Residual risk accepted because no prod data exists yet (the app starts empty per D-01). |
| T-05-06-06 | S (Spoofing) | operator improvises against prod without runbook coverage | mitigate | Checkpoint instructions explicitly say: "If a step is ambiguous or a command fails in a way the runbook doesn't cover, STOP and describe the situation — do NOT improvise against prod." Escalates ambiguity back to the planning loop rather than risking an undocumented prod change. |
