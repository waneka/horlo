<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Next.js server (dynamic route) | Dynamic watch ID is user-supplied; must be scoped by session userId |
| client component → Server Action | Client passes watchId; Server Action re-verifies userId ownership via getCurrentUser() + DAL check |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04-01 | E (Elevation of Privilege) / IDOR | watch/[id]/page.tsx Server Component | mitigate | The page calls getWatchById(user.id, id) where user.id comes from getCurrentUser() (session) and id comes from the URL param. The DAL function (Phase 3 + Phase 4) enforces userId scoping at the SQL WHERE clause level — a user requesting another user's watch ID returns null, not data. notFound() is called on null. No client-supplied userId is ever trusted. |
| T-05-04-02 | E (Elevation of Privilege) / IDOR | Server Action calls from WatchDetail / WatchForm | mitigate | Server Actions in src/app/actions/watches.ts read userId from the session (Phase 4 D-02), not from client input. Client only passes watchId; the action re-verifies ownership before mutating. Phase 4 already added a real IDOR integration test against local Postgres (Phase 4 plan 04). No new test needed here. |
| T-05-04-03 | T (Tampering) | router.refresh() timing race | accept | If a user clicks markAsWorn twice rapidly, useTransition keeps the button disabled (isPending) between calls. Residual risk: the second click could land before the first refresh completes, but the second action re-reads the server truth anyway. Low-impact race, accepted. |
| T-05-04-04 | S (Spoofing) | Pitfall 2 (use(params) in Server Component) | mitigate | Acceptance criteria include grep "use(params)" returning no output on both dynamic pages. If executor copies the old client pattern, the grep gate fails. |
| T-05-04-05 | I (Information Disclosure) | WatchDetail collection prop | accept | WatchDetail receives the full user collection to feed SimilarityBadge's analyzeSimilarity. This is the same data the home page shows the user — no new exposure. Alternative (pre-compute similarity server-side) would require running analyzeSimilarity in a Server Component, which is outside DATA-05 scope (the requirement says the similarity engine REMAINS client-side with props). |
</threat_model>
