**Layout proportions:**
- Top Cards use shadcn `<Card>` primitive (NOT `<SettingsSection>`). [Claude's Discretion]: `<Card>` is the right primitive here because each top knob has a `<CardTitle>` + `<CardDescription>` semantic header — the description is content, not a label, and `<SettingsSection>` only has a single `<h2>` slot. Using `<Card>` matches the visual rhythm of the existing `<PreferencesClient>` Cards below; using `<SettingsSection>` would create a visual mismatch where the lifted controls look subordinate to the taste-tag pickers below.
- Select trigger width: `w-full sm:max-w-md` — full width on mobile, capped at 28rem (448px) on `≥sm` so the Select doesn't span the entire panel awkwardly. Matches existing `<PreferencesClient>:393` `w-full sm:max-w-xs` pattern but slightly wider to accommodate the 4 new option labels (longest: "Brand Loyalist — Same maker, different models" at ~46 chars).
- Card-to-Card gap: `space-y-6` (24px) on the wrapper. Matches the Phase 22 `<SettingsSection>` `mb-6` rhythm, so the new top Cards read as siblings to Privacy/Notifications cards on adjacent tabs.

**Divider treatment between top Cards and embedded PreferencesClient:** [Claude's Discretion — D-CHOICE-DIVIDER]
- Use a `border-t border-border pt-6` divider with a "Taste preferences" `text-xs uppercase` label above the embedded PreferencesClient.
- Rationale: Without the divider, the embedded PreferencesClient's first Card (`<Card><CardTitle>Style Preferences</CardTitle>...`) would visually collide with the lifted `<Card>`s above — three Cards in a row with no semantic break. The label "Taste preferences" frames the taste-tag pickers as a distinct fine-tuning surface and mirrors the `<SettingsSection>` heading style (`text-xs font-normal uppercase tracking-wide text-muted-foreground` from `SettingsSection.tsx:16`) so the visual language stays consistent.
- The divider is a 1px `border-border` line (existing token), 24px above the next-section heading.

### WatchForm — `isChronometer` Checkbox Row (D-09 / D-10)

Inserted at the bottom of the Specifications `<Card>`, BELOW the existing 9-cell grid (Movement, Case Size, Lug-to-Lug, Production year, Water Resistance, Strap Type, Crystal, Dial Color):

```html
<Card>
  <CardHeader>
    <CardTitle>Specifications</CardTitle>
  </CardHeader>
  <CardContent className="grid gap-6 sm:grid-cols-2 lg:grid-cols-3">
    {/* ... existing 9 grid cells unchanged ... */}

    {/* NEW: Chronometer certification row (full-width, spans all columns) */}

```

**Layout details:**
- `sm:col-span-2 lg:col-span-3` — the Checkbox row spans all columns of the responsive grid so it sits as a deliberate full-width row beneath the 9-cell spec grid (NOT crammed into a 4th grid cell that would be left-aligned with "Movement"). The mobile single-column grid renders the row inline; on `≥sm` (2 cols) and `≥lg` (3 cols) the row uses `col-span` to break out of the column flow.
- `space-y-2` matches the existing grid cells' vertical rhythm.
- The "(COSC or equivalent)" qualifier is rendered in `text-muted-foreground` to subordinate it to the primary label "Chronometer-certified" — pattern matches `<PreferencesClient>:386` `<p className="text-xs text-muted-foreground mt-2">Range: 20mm – 55mm</p>`.
- Checkbox + label use the existing pattern from `<WatchForm>`'s Complications block (lines 519-531) — `flex items-center space-x-2 cursor-pointer` over a `<label>` element wrapping a shadcn `<Checkbox>` and a `<span>`. **Use `gap-2` instead of `space-x-2`** because `gap-2` is the modern flex-gap utility and matches the rest of Phase 22/23 spacing tokens; `space-x-2` is the older sibling-margin approach. Both render identically in this 2-element layout.

### WatchForm — `notesPublic` Pill (D-13 / D-14 / D-16)

Inserted BELOW the Notes Textarea in the existing Notes `<Card>`:

```html
<Card>
  <CardHeader>
    <CardTitle>Notes</CardTitle>
  </CardHeader>
  <CardContent className="space-y-3">
    <Textarea
      value={formData.notes}
      onChange={(e) => setFormData((prev) => ({ ...prev, notes: e.target.value }))}
      placeholder="Any additional notes about this watch..."
      rows={4}
    />
    <div className="flex items-center gap-2">
      <span className="text-xs text-muted-foreground">Visibility:</span>
      <button
        type="button"
        role="switch"
        aria-checked={formData.notesPublic === true}
        aria-label={
          formData.notesPublic === true
            ? 'Note is public, click to make private'
            : 'Note is private, click to make public'
        }
        onClick={() =>
          setFormData((prev) => ({ ...prev, notesPublic: !(prev.notesPublic === true) }))
        }
        className={cn(
          'rounded-full px-2.5 py-0.5 text-xs font-normal transition-colors',
          'focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring/50',
          formData.notesPublic === true
            ? 'bg-accent text-accent-foreground hover:bg-accent/90'
            : 'bg-muted text-muted-foreground hover:bg-muted/70',
        )}
      >
        {formData.notesPublic === true ? 'Public' : 'Private'}
      </button>
    </div>
  </CardContent>
</Card>
```

**Layout details:**
- `<CardContent className="space-y-3">` — replaces the existing default content padding with explicit `space-y-3` (12px) between the Textarea and the visibility row. This is tighter than `space-y-4` (which would be the form-default rhythm) because the pill is subordinate metadata to the textarea content, not a separate field.
- The "Visibility:" label is rendered as a `<span>` (NOT a `<Label htmlFor>`) because the pill is a `<button role="switch">`, not a form input. The `aria-label` on the button itself carries the accessible name; the on-screen "Visibility:" caption is purely visual context.
- The pill matches `<NoteVisibilityPill>` exactly (D-14 contract): `rounded-full px-2.5 py-0.5 text-xs font-normal transition-colors`, `bg-accent text-accent-foreground` when public, `bg-muted text-muted-foreground` when private.
- **Implementation choice (D-14):** This UI-SPEC selects **option (b) — inline pill** rather than extracting a shared `<VisibilityPill>` primitive. Rationale: the WatchForm pill uses local React state (`formData.notesPublic`), while `<NoteVisibilityPill>` uses `useOptimistic` + Server Action. Extracting a presentational primitive that both wrap would force either (i) lifting state into both call sites with a shared color/copy contract — adding indirection without saving lines — or (ii) parameterizing the primitive over its state machine, which is over-engineering for a 4-line Tailwind chain. Option (b) accepts the styling duplication; the colors and shape are locked here so any future drift is a checker violation, not a primitive change.
- Hover/focus states (NEW in this phase):
  - Hover (public): `hover:bg-accent/90` — slight darken on the accent fill
  - Hover (private): `hover:bg-muted/70` — slight lighten on the muted fill (matches the pattern from `<InlineThemeSegmented>:57` `hover:bg-accent/10`)
  - Focus-visible: `focus-visible:ring-2 focus-visible:ring-ring/50` — same accent-as-focus-color signal that the rest of the design system uses (`button.tsx`, `input.tsx`)
  - Disabled: not specified — the pill has no disabled state in the WatchForm context (the form's submit button gates persistence; the pill itself is always interactive while the form is open).

### WatchDetail — `isChronometer` Display Row (D-11)

Inserted into the existing Specifications `<dl>` (`WatchDetail.tsx:240-287`), AFTER the `productionYear` row (line 281-286), as an only-if-true entry:

```html
<dl className="grid grid-cols-2 gap-4 text-sm">
  {/* ... existing rows up through productionYear ... */}

  {watch.isChronometer === true && (
    <div>
      <dt className="text-muted-foreground">Certification</dt>
      <dd className="font-semibold flex items-center gap-1">
        <Check className="size-4 text-foreground" aria-hidden />
        <span>Chronometer</span>
      </dd>
    </div>
  )}
</dl>
```

**Layout details:**
- The conditional `watch.isChronometer === true` (NOT `watch.isChronometer`) is deliberate — the column is `boolean | null` per `src/db/schema.ts` (line 95: `is_chronometer boolean DEFAULT false`); legacy rows may surface as `null` and must NOT render a "Certification: ✓ Chronometer" entry. Mirrors the existing `caseSizeMm`, `lugToLugMm`, etc. only-if-truthy pattern.
- Icon: `lucide-react` `Check` at `size-4` (16px) — matches the icon density elsewhere in `<WatchDetail>` (the `<WatchIcon className="h-16 w-16">` placeholder is the only larger icon; smaller surface chrome stays at 16px).
- Icon color: `text-foreground` (NOT `text-accent`). Rationale: the certification row is an informational fact, not an action or affordance; using `text-accent` would conflict with the locked color contract (accent reserved for active tab strip, toggle-ON fill, focus ring, pending banner left-border — see Color section below). `text-foreground` keeps the icon visually equal-weight to the "Chronometer" word.
- `gap-1` (4px) between icon and label — tighter than the rest of the `<dl>` because the icon-label pair reads as a single "✓ Chronometer" token, not two separate elements.
- `<dd className="font-semibold flex items-center gap-1">` — `font-semibold` matches the existing `<dd>` weight throughout the spec list (line 243, 248, 254, etc.), so the new row reads as a sibling to the existing rows.

## Anti-Patterns Flagged

These are explicitly forbidden by this contract. The checker should fail any deviation.

1. **DO NOT use `<SettingsSection>` for the two new top Preferences Cards.** Use `<Card>` + `<CardHeader>` + `<CardTitle>` + `<CardDescription>` + `<CardContent>`. The Cards need the title+description+content semantic; SettingsSection provides only a single h2 slot.
2. **DO NOT extract a shared `<VisibilityPill>` primitive in Phase 23 (D-14 option a).** Inline the pill in WatchForm with the locked colors and shape. The duplication is intentional; if a third pill surface appears later, extract then.
3. **DO NOT add explanatory copy under the Theme heading in `<AppearanceSection>` (D-05).** No `<CardDescription>`, no helper text. The Light/Dark/System icon labels carry the meaning.
4. **DO NOT modify `<InlineThemeSegmented>` in any way (D-07).** Reuse byte-identical. The Floating UI dismissal workaround stays harmless and inert in the new context.
5. **DO NOT modify `<NoteVisibilityPill>` (D-15).** It is the canonical per-row toggle; needs no changes in Phase 23.
6. **DO NOT modify `<PrivacySection>` or `<NotificationsSection>` (D-08).** Phase 22 was the restyle. Phase 23 verifies via smoke test, no diff.
7. **DO NOT keep the `collectionGoal` / `overlapTolerance` Selects inside `<PreferencesClient>`'s "Collection Settings" Card (D-02).** Lift them to the top Cards; delete the Collection Settings Card from PreferencesClient.
8. **DO NOT add a `Cancel` / `Save` button to the new top Preferences Cards.** The Selects use the existing per-change `savePreferences` transition pattern (auto-save on change). Adding a Save button would diverge from PreferencesClient's existing behavior.
9. **DO NOT use `text-accent` for the WatchDetail Chronometer Check icon.** Use `text-foreground`. Accent is reserved per the Color contract; the certification row is informational, not an active affordance.
10. **DO NOT default `notesPublic` to `false` (D-16).** It MUST default to `true` to match the DB column default (`notes_public NOT NULL DEFAULT true`) and the read-side fallback (`watch.notesPublic !== false` in `NoteRow.tsx:40`). Defaulting to `false` would silently hide existing notes from `/u/{username}/notes`.
11. **DO NOT show the WatchDetail Chronometer row when `watch.isChronometer === false` or `null`.** Only render when strict-equal to `true`. Mirrors the existing only-if-truthy pattern for `caseSizeMm`, `lugToLugMm`, etc.
12. **DO NOT introduce custom Tailwind tokens or CSS variables.** Use only existing `bg-card`, `bg-muted`, `bg-accent`, `text-foreground`, `text-muted-foreground`, `text-accent-foreground`, `border-border`, `ring-ring`. The Tailwind 4 config is locked.
13. **DO NOT alter the `/preferences` redirect (D-15 from Phase 22).** Already shipped; verify-only.
14. **DO NOT introduce new icons.** The Chronometer Check uses `lucide-react`'s `Check` (existing import path); the Theme card icons (Sun/Moon/Monitor) are inherited from `<InlineThemeSegmented>`. No other new lucide icons needed.
