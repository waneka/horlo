# Phase 49.1: Remove Genre Surface - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 49.1 executes the Phase 49 spike's primary recommendation (`remove-genre`, §7 + §9 Ship-Now: YES, ROADMAP SC#4 escape hatch). The genre/archetype *taxonomy surface* is removed: `watches_catalog.primary_archetype` is dropped, `GenreChips` + `ArchetypeChips` + `/explore/genres` are deleted, the catalog DAL stops accepting `genre`/`archetype` filters, the LLM enricher stops writing the column, and the similarity engine's 0.04 `archetypeMatch` sub-weight is redistributed across the surviving taste sub-budget.

**Critical scope correction (this discussion):** The Phase 49 spike under-counted consumers — its 8-9 file estimate covered only the `/search` surface. Scouting found ~12 additional consumers of `primary_archetype` / `ARCHETYPE_CONFIG` / `PrimaryArchetype` (verdict templates, viewerTasteProfile, fit-delta, `/explore` CollectorArchetypes rail, ReferenceIdentityCard headline, CollectionFitCompareTable row, SearchPageClient editorial header, catalog page projection, search action Zod, types, enricher chain, tests). Two of these (`fills-a-hole` and `collection-skews-contrast` in `src/lib/verdict/templates.ts`) are Phase 20 D-01 "ARE NOT optional" templates. The decisions below assign a verdict to each surface.

**In scope (per ROADMAP SC#1-7 + scope corrections below):**
1. Drop `primary_archetype` from `watches_catalog` (1 drizzle migration + 1 supabase migration via `supabase db push --linked`).
2. Delete `src/components/search/GenreChips.tsx`, `src/components/search/ArchetypeChips.tsx`, `src/app/explore/genres/page.tsx`.
3. **Carve-out from SC#2:** `src/lib/archetype-config.ts` is NOT fully deleted — the `ARCHETYPE_CONFIG` constant and `PrimaryArchetype` type alias are retained for display-only use by the `/explore` CollectorArchetypes rail (D-EXPLORE-01). Tests covering the kept constant remain.
4. Update `FilterDrawer.tsx` to stop importing/rendering both chip groups; remove the `/explore/genres` deep-link from `BrowseModule.tsx`.
5. Remove `filters.genre` and `filters.archetype` from `CatalogSearchFilters` (`src/data/catalog.ts`); remove the archetype-wins tiebreaker; delete `getBrowseGenreCounts()` from `src/data/browse.ts`.
6. Rewire `getBrowseArchetypeCounts()` to derive from `style_tags` (D-EXPLORE-01); CollectorArchetypes deep-links repoint to `/search?style=X`.
7. Rebalance `TASTE_SUB_WEIGHTS` in `src/lib/similarity.ts` via proportional rescale (D-SIM-01); remove the categorical archetype-match block.
8. Pivot the 3 archetype-using verdict templates in `src/lib/verdict/templates.ts` to use `eraSignal` / `dominantEraSignal` (D-VERDICT-01); keep `era-echo` as a distinct template; update `composer.test.ts` fixtures.
9. Drop archetype from 3 UI/engine surfaces (D-SCOPE-01): ReferenceIdentityCard → era-only headline; CollectionFitCompareTable → delete Row 4 "Archetype" (6 → 5 rows); `fit-delta` → drop `primaryArchetype` dimension (6 → 5 ranked dimensions).
10. Strip `primaryArchetype` field from `CatalogTasteAttributes` in `src/lib/types.ts`; strip the field from the catalog page projection in `src/app/catalog/[catalogId]/page.tsx`; strip `genre`/`archetype` from the Zod schema in `src/app/actions/search.ts`; strip the archetype editorial header + inline removable archetype/genre facet chips from `src/components/search/SearchPageClient.tsx`.
11. Update the enricher (`src/lib/taste/enricher.ts`, `vocab.ts`, `prompt.ts`) to stop writing `primary_archetype` — drop from LLM tool schema, prompt instruction, Zod wire schema, and `validateAndCleanTaste()`.
12. Delete or update tests: `archetype-config.test.ts` (trim to ARCHETYPE_CONFIG-only assertions), `browse.test.ts` (remove genre count tests), `viewerTasteProfile.test.ts` (drop `dominantArchetype` assertions), `composer.test.ts` (repoint archetype-template fixtures to era), `shims.test.ts` (drop `primaryArchetype`), `enricher.test.ts` + `vocab.test.ts` (drop archetype assertions), `catalog-facets.test.ts` + `watches-leftjoin.test.ts` (drop archetype facet rows), `CollectorArchetypes.test.tsx` (repoint DAL mock to style-tags shape), `fit-delta` tests (drop archetype dimension).
13. Add new tests per SC#7 covering the simplified search filter surface and the rebalanced similarity weights.

**Out of scope:**
- `style_tags` vocab governance (open → closed enum). Deferred; covered in 49-CONTEXT.md deferred list.
- Watch-detail architecture (Phase 50 / ARCH-01).

- Re-deriving an "archetype" surrogate from style_tags for ANYTHING other than the `/explore` CollectorArchetypes rail (verdict templates pivot to era; ReferenceIdentityCard / CompareTable / fit-delta lose the dimension entirely).

## Implementation Decisions

### D-VERDICT — Verdict engine template strategy

- **D-VERDICT-01:** The 3 archetype-using templates in `src/lib/verdict/templates.ts` pivot to `eraSignal` / `dominantEraSignal`:
  - `fills-a-hole` (Phase 20 D-01 mandated): "Fills a hole — your first ${era} piece" (or analogous era-axis copy; exact wording is planner discretion)
  - `collection-skews-contrast` (Phase 20 D-01 mandated): "Your collection skews ${dominant_era} — this is a ${contrast_era} piece"
  - `archetype-echo` (supporting): repointed to era, but kept narratively distinct from `era-echo`
- **D-VERDICT-02:** The reserved `era-echo` template (`{ id: 'era-echo', predicate: () => null, … }`) stays as its own template (`"Echoes the ${era} era of your collection"`). The 3 rewritten templates must remain narratively distinct from `era-echo` — exact angle is planner discretion (e.g. fills-a-hole = "first of ${era}", contrast = "your collection skews ${era_A}, this is ${era_B}", echo-renamed = "another ${era} piece, leaning further in").
- **D-VERDICT-03:** `viewerTasteProfile.ts` — the `dominantArchetype` field is dropped from `ViewerTasteProfile`; `dominantEraSignal` is retained (already computed via the same `mode()` helper). The `EMPTY_PROFILE` literal loses `dominantArchetype: null`.
- **D-VERDICT-04:** `composer.test.ts` archetype-template fixtures are repointed to era. The Phase 20 D-01 "ARE NOT optional" status is preserved by the pivot (the templates still fire — just on a different axis); no explicit D-01 override is recorded.

### D-EXPLORE — /explore CollectorArchetypes rail rewire

- **D-EXPLORE-01:** The v5.1 EXPL-05 `CollectorArchetypes.tsx` rail is preserved. `getBrowseArchetypeCounts()` in `src/data/browse.ts` is rewired to derive from `style_tags`:
  ```sql
  SELECT tag AS archetype, COUNT(*)::int AS count
  FROM watches_catalog, unnest(style_tags) AS tag
  WHERE tag = ANY(ARRAY['dress','dive','field','pilot','chrono','gmt','racing','sport','tool','hybrid'])
  GROUP BY tag ORDER BY count DESC
  ```
  Per spike Q1 (49-SPIKE.md §4), 99% of catalog rows have the archetype value verbatim in `style_tags`, so post-rewire counts will be near-identical to today's.
- **D-EXPLORE-02:** `ARCHETYPE_CONFIG` (10-entry display-name + description lookup) is retained in `src/lib/archetype-config.ts` as a display-only constant; the `PrimaryArchetype` type alias is retained as the union of the 10 archetype keys used by `ARCHETYPE_CONFIG`. This is an explicit carve-out from ROADMAP SC#2's literal "delete archetype-config.ts" — documented here so downstream verifiers do not flag it as a regression.
- **D-EXPLORE-03:** CollectorArchetypes deep-links change from `/search?tab=watches&archetype=${value}` to `/search?tab=watches&style=${value}`. StyleChips already supports multi-select on `style_tags`. No old `/search?archetype=…` redirect needed (URL is internal to the rail; no external bookmarks expected for a v5.1 chip rail launched <1 week before this removal).

### D-SCOPE — Direct-UI archetype-display surfaces

All three "what's the archetype?" UI surfaces drop the archetype hook rather than substituting `style_tags[0]`. Rationale: `style_tags[0]` is an open-vocab, non-normalized field where 'sport' appears top-2 across every archetype (spike Q3) — a surrogate would produce worse copy than just removing the question.

- **D-SCOPE-01a:** `ReferenceIdentityCard.tsx` headline drops the archetype side. The existing `hasHeadline = Boolean(eraLabel || archetypeLabel)` guard already handles the era-only case cleanly; render reduces to "Modern era" (or whatever era value is set). `ARCHETYPE_LABELS` constant is deleted. `taste.primaryArchetype` references are removed.
- **D-SCOPE-01b:** `CollectionFitCompareTable.tsx` Row 4 "Archetype" is deleted entirely. Table goes from 6 rows to 5 (Formality, Sportiness, Heritage, Era, Design Motifs). The remaining row set spans the same taste sub-budget the engine actually scores (D-SIM-01), keeping table and engine in sync.
- **D-SCOPE-01c:** `src/lib/verdict/fit-delta.ts` drops the `primaryArchetype` dimension. The ranked dimension list goes from 6 (formality > sportiness > heritageScore > primaryArchetype > eraSignal > designMotifs) to 5 (formality > sportiness > heritageScore > eraSignal > designMotifs). The `"Different archetype: X vs Y"` delta phrase branch is removed.
- **D-SCOPE-01d:** `SearchPageClient.tsx` archetype editorial header (Phase 46 D-13, lines 392-414 and 481-487) and inline removable archetype/genre facet chips (lines 418-454 and 493-526) are deleted along with `genre`/`archetype` URL params and `ARCHETYPE_CONFIG` references at this surface. `ARCHETYPE_CONFIG` import is removed from SearchPageClient even though the constant is retained for `/explore`.
- **D-SCOPE-01e:** `src/app/catalog/[catalogId]/page.tsx:94` removes `primaryArchetype: catalogEntry.primaryArchetype` from the `catalogTaste` literal. The `CatalogEntry` type's `primaryArchetype` field is dropped upstream when `watches_catalog` schema drops the column.
- **D-SCOPE-01f:** `src/app/actions/search.ts` Zod schema drops `.genre` and `.archetype` fields; the filters passthrough at lines 125-126 drops both.

### D-SIM — Similarity sub-weight redistribution

- **D-SIM-01:** Proportional rescale (×1.25). Post-removal `TASTE_SUB_WEIGHTS`:
  - `numericTrioCosine`: 0.40 → **0.50** (effective 0.10)
  - `eraMatch`: 0.20 → **0.25** (effective 0.05)
  - `motifsJaccard`: 0.20 → **0.25** (effective 0.05)
  - Sub-weights sum to 1.00 (SC#4 invariant preserved). Trio-vs-categorical 2:1:1 ratio preserved per Phase 38 D-03.
- **D-SIM-02:** Implementation must follow the file's existing anti-pattern comment (lines 11-12: "hardcoding rescaled values as magic numbers" forbidden). Define a base sub-weight map (`TASTE_SUB_WEIGHTS_BASE = { numericTrioCosine: 0.40, eraMatch: 0.20, motifsJaccard: 0.20 }`) and an algorithmic rescale (`RESCALE = 1 / (1 - REMOVED_SUB_WEIGHT)` where `REMOVED_SUB_WEIGHT = 0.20`). The Object.fromEntries(...map(× RESCALE)) idiom already used for `WEIGHTS` at lines 27-32 is the precedent.
- **D-SIM-03:** Remove the categorical archetype-match block at `src/lib/similarity.ts:124-127` (`if (t1.primaryArchetype !== null && t2.primaryArchetype !== null && t1.primaryArchetype === t2.primaryArchetype) { contrib += TASTE_SUB_WEIGHTS.archetypeMatch * 1.0 }`).
- **D-SIM-04:** The `TASTE_WEIGHT = 0.20` constant at line 24 stays unchanged — only the internal taste sub-weights are redistributed; the outer 0.20 reservation for taste vs the 8-dim existing-weights group is preserved. Verdict label thresholds in `THRESHOLDS` and `GOAL_THRESHOLDS` are unchanged.

### Migration & rollback

- **D-MIG-01:** One drizzle migration (`ALTER TABLE watches_catalog DROP COLUMN primary_archetype`); one supabase migration pushed via `supabase db push --linked` per project memory `project_drizzle_supabase_db_mismatch`. No data backfill (the column is destroyed by design — spike §8 Cost row B). Per memory `project_db_wipeable_2026_05_09`, `watches_catalog` is carved out of the wipeable rule; this option is unaffected because we are destroying the column's data by design, not migrating it.
- **D-MIG-02:** Sequencing — **column drop ships LAST**, after all reader code has been removed and deployed. Order: (1) land all code changes that stop reading `primary_archetype` (the ~20 files in D-SCOPE / D-VERDICT / D-EXPLORE / D-SIM) and deploy to prod; (2) THEN run the migration on prod via `supabase db push --linked`. This prevents the "code still reads dropped column" failure mode. The drizzle local migration can be authored first and `drizzle-kit push`-ed locally to keep dev environments in sync during code work, but prod stays on the old schema until step 2.
- **D-MIG-03:** Rollback path (if prod breaks post-drop): re-add the column via a forward migration, then re-run the LLM enrichment pipeline against all ~100 catalog rows. Spike §9 notes this is feasible-but-costly (Anthropic API cost + LLM non-determinism risk). The enricher code path is being removed in this phase; a rollback would need to revert that removal first. Not a blocker — just documented so a future incident response knows the path.
- **D-MIG-04:** Per project memory `project-catalog-id-divergence`, catalog data migrations are keyed by `(brand, model, reference)` not `id` — but this migration is a schema DROP, not a data migration, so id-keying is not in play.

### Claude's Discretion

- Exact narrative wording for the 3 rewritten verdict templates (D-VERDICT-01). The decision locks the *axis* (era) and the *distinctness from era-echo* (D-VERDICT-02); the exact copy and predicate gates are planner discretion.
- Whether `archetype-config.ts` is renamed (e.g. to `explore-archetype-display.ts`) to reflect its narrower post-49.1 role, or left at its current path with a comment explaining the carve-out. Planner discretion.
- Whether the `getBrowseArchetypeCounts()` rewire keeps the function name (semantically misleading — it now reads style_tags) or renames it (e.g. `getBrowseStyleArchetypeCounts()`). Planner discretion.
- Whether the 11 archetype-touching test files are deleted, trimmed, or repointed individually — choice is per-file based on which assertions still apply after the column drop. Planner discretion.
- Plan wave structure (1 wave vs multi-wave). The work is mostly orthogonal across the 4 decision areas, so it could plausibly run as 4 parallel sub-plans (verdict / explore-rail / scope-uis / similarity) plus a final wave for schema + enricher + migration. Planner call.
