# Phase 49.1: Remove Genre Surface - Research

**Researched:** 2026-05-19
**Domain:** Schema-and-surface removal across DAL + DB + similarity engine + verdict templates + LLM enricher
**Confidence:** HIGH

## Summary

CONTEXT.md is already exhaustive — every locked decision (D-VERDICT, D-EXPLORE, D-SCOPE, D-SIM, D-MIG) is grounded in file:line citations that this research has spot-verified. The implementation has three structural risks the planner must absorb:

1. **Test surface is ~2× the CONTEXT.md estimate.** CONTEXT.md listed 11 archetype-touching test files under `src/`. There are an additional **~15 test files under `/tests/`** (the root-level test tree per `vitest.config.ts` `include`) that also reference `primaryArchetype` / `genre` / `archetype`. Total: **~26 test files** [VERIFIED: ripgrep]. Two of them (`tests/components/search/DrawerChips.test.tsx` and `tests/components/search/SearchPageClientChips.test.tsx`) carry full describe blocks for the components being deleted — those describes go away with the components.

2. **A standalone CHECK constraint blocks naive DROP COLUMN ordering.** Phase 19.1 added `watches_catalog_primary_archetype_check` as a named CHECK constraint at the table level (`supabase/migrations/20260430000000_phase19_1_taste_constraints.sql:14-22`). Postgres drops the constraint implicitly when the column is dropped, so the forward migration is still one statement — but the supabase prod-push must use `IF EXISTS` guards and the local drizzle migration must not assume the constraint stays after the column goes. **Action:** the planner's migration template should be `ALTER TABLE watches_catalog DROP COLUMN IF EXISTS primary_archetype;` — the `IF EXISTS` is idempotent against re-runs (gotcha #1 territory from `project_drizzle_supabase_db_mismatch`).

3. **Phase 39b D-39b-04 "identical-rendering lock" is intact and exposed.** The /catalog and /watch routes both render `ReferenceIdentityCard` with the same `CatalogTasteAttributes` shape. The catalog page builds that shape via a manual projection at `src/app/catalog/[catalogId]/page.tsx:90-99`; the /watch page receives it pre-built via `getWatchesByUser`'s LEFT JOIN at `src/data/watches.ts:130-162`. **Both projection sites must drop `primaryArchetype` in the same atomic change** or the card renders differently across the two routes and breaks D-39b-04.

**Primary recommendation:** Plan as 4 parallel waves (verdict / explore-rail / scope-uis / similarity) gated by a final wave for type-system unification (`CatalogTasteAttributes` field removal) + enricher + drizzle migration locally + supabase migration prod. The schema DROP runs last, after deploy.

### Pattern 1: `'use cache'` + `cacheTag` (preserved on rewired DAL)

**What:** Server-only DAL functions wrap their body in `'use cache'` + `cacheTag(...)` + `cacheLife('hours')` so Next.js Cache Components caches the result globally and revalidateTag can invalidate it.
**When to use:** Viewer-independent reads. Already used by all 5 functions in `src/data/browse.ts` and `getTopStyleTags` in `src/data/catalog.ts`.
**Example:**
```typescript
// Source: src/data/browse.ts:44-56 (verified live; pattern documented at
// node_modules/next/dist/docs/01-app/03-api-reference/01-directives/use-cache.md)
export async function getBrowseArchetypeCounts(): Promise<Array<{ archetype: string; count: number }>> {
  'use cache'
  cacheTag('explore', 'explore:browse')
  cacheLife('hours')
  const rows = await db.execute(
    sql`SELECT primary_archetype AS archetype, COUNT(*)::int AS count
        FROM watches_catalog
        WHERE primary_archetype IS NOT NULL
        GROUP BY primary_archetype
        ORDER BY count DESC`,
  )
  return rows as unknown as Array<{ archetype: string; count: number }>
}
```

The Phase 49.1 rewire keeps the function signature, cache tag scope (`'explore', 'explore:browse'`), and TTL (`'hours'`) unchanged. Only the SQL body changes. **No client of this function needs to be updated** — the return type `Array<{ archetype: string; count: number }>` is identical.

### Pattern 2: `sql\`unnest(style_tags)\`` already proven in this codebase

**What:** Drizzle's `sql\`\`` template tag passes the `unnest(...)` lateral join through to Postgres unchanged.
**When to use:** Any GROUP BY over array-column values.
**Example (live, identical shape to the D-EXPLORE-01 SQL):**
```typescript
// Source: src/data/catalog.ts:533-544 — getTopStyleTags
export async function getTopStyleTags(limit = 8): Promise<string[]> {
  'use cache'
  cacheLife('hours')
  const rows = await db.execute(
    sql`SELECT tag, COUNT(*) AS freq
        FROM watches_catalog, unnest(style_tags) AS tag
        GROUP BY tag
        ORDER BY freq DESC
        LIMIT ${limit}`,
  )
  return (rows as unknown as Array<{ tag: string }>).map((r) => r.tag)
}
```

The D-EXPLORE-01 rewire adds a `WHERE tag = ANY(ARRAY[...])` predicate and reads to a different return shape — but the lateral-join idiom is byte-identical. **Implementation risk: LOW.** The 10-value archetype list can be passed via parameter binding using `sql.join`:

```typescript
import { PRIMARY_ARCHETYPES } from '@/lib/taste/vocab'  // 10 values; const-asserted
// ...inside getBrowseArchetypeCounts (rewired):
const archetypeList = sql.join(PRIMARY_ARCHETYPES.map(v => sql`${v}`), sql`, `)
const rows = await db.execute(
  sql`SELECT tag AS archetype, COUNT(*)::int AS count
      FROM watches_catalog, unnest(style_tags) AS tag
      WHERE tag = ANY(ARRAY[${archetypeList}])
      GROUP BY tag
      ORDER BY count DESC`,
)
```

Alternative (CONTEXT.md Claude's Discretion — `archetype-config.ts` ownership): if the planner extracts `PRIMARY_ARCHETYPES` into `archetype-config.ts` to consolidate carve-out state, this query continues to work — only the import path changes.

### Pattern 3: Algorithmic weight rescale (anti-magic-number)

**What:** `similarity.ts` lines 10-12 explicitly forbid hardcoded rescaled values. Use a `_BASE` constant + algorithmic transform.
**When to use:** D-SIM-02 implementation.
**Example:**
```typescript
// Source: src/lib/similarity.ts:27-32 (existing precedent for D-SIM-02)
const WEIGHTS = {
  ...Object.fromEntries(
    Object.entries(EXISTING_WEIGHTS_BASE).map(([k, v]) => [k, v * EXISTING_SCALE]),
  ),
  taste: TASTE_WEIGHT,
} as { [K in keyof typeof EXISTING_WEIGHTS_BASE]: number } & { taste: number }
```

The D-SIM-02 rewire:
```typescript
const TASTE_SUB_WEIGHTS_BASE = {
  numericTrioCosine: 0.40,
  eraMatch:          0.20,
  motifsJaccard:     0.20,
} as const  // sums to 0.80 (after removing 0.20 archetypeMatch)
const REMOVED_SUB_WEIGHT = 0.20
const RESCALE = 1 / (1 - REMOVED_SUB_WEIGHT)  // 1.25
const TASTE_SUB_WEIGHTS = Object.fromEntries(
  Object.entries(TASTE_SUB_WEIGHTS_BASE).map(([k, v]) => [k, v * RESCALE]),
) as { [K in keyof typeof TASTE_SUB_WEIGHTS_BASE]: number }
```

The exact ratio (0.40/0.20/0.20) → (0.50/0.25/0.25) preserves Phase 38 D-03's 2:1:1 trio-vs-categorical ratio per D-SIM-01.

### Anti-Patterns to Avoid

- **Hardcoded 0.50/0.25/0.25 in `TASTE_SUB_WEIGHTS`** — explicitly forbidden by the comment at `similarity.ts:10-12`. Use the `RESCALE` algorithm.
- **Deleting `archetype-config.ts` outright** — violates D-EXPLORE-02. The /explore CollectorArchetypes rail still uses `ARCHETYPE_CONFIG[value].displayName` for editorial copy.
- **Deleting `PrimaryArchetype` type** — violates D-EXPLORE-02. `ARCHETYPE_CONFIG: Record<PrimaryArchetype, ArchetypeConfig>` needs the union.

## Common Pitfalls

### Pitfall 1: Dropping the column before code deploys
**What goes wrong:** Prod runtime crashes on every read of `watches_catalog` (e.g., `getWatchesByUser` LEFT JOIN, `mapRowToCatalogEntry`, the catalog page SELECT) because drizzle expects `primary_archetype` to exist in the SQL projection.
**Why it happens:** The drizzle schema declares the column at `src/db/schema.ts:390`. If schema.ts is updated and merged BUT the drizzle-kit migration runs on prod before the code deploys, every drizzle query for `watches_catalog` will try to project a column that no longer exists.
**How to avoid:** Land + deploy all code changes first (waves 1-N including the schema.ts update — drizzle is happy to project a column the DB has, just not to project one the DB doesn't have). Run the prod supabase migration LAST. D-MIG-02 mandates this ordering.
**Warning signs:** Prod 500s on `/explore`, `/search`, `/catalog/[id]`, `/watch/[id]`, `/collection` — anything touching `watches_catalog`.

### Pitfall 2: Forgetting that `schema.ts` AND the DB column must change together
**What goes wrong:** If schema.ts retains `primaryArchetype: text('primary_archetype')` but the DB column is dropped, drizzle TypeScript types claim the column exists but every SELECT crashes at runtime.
**Why it happens:** The schema is the single source of truth in the typed DAL; dropping the column without updating the schema leaves the type system out of sync with reality.
**How to avoid:** Wave order — schema.ts update + drizzle local migration ship together to dev (so `drizzle-kit push` updates both sides atomically locally). Prod gets the migration LAST and the schema.ts update is already in code (via the deploy).
**Warning signs:** TypeScript compile succeeds but `db.select({ primaryArchetype: watchesCatalog.primaryArchetype })` 500s.

### Pitfall 3: D-39b-04 identical-rendering lock broken by asymmetric projection updates
**What goes wrong:** `/catalog/[catalogId]` page manually projects `CatalogEntry` fields into a `CatalogTasteAttributes` literal (page.tsx:90-99). The /watch/[id] page receives the same shape via `getWatchesByUser`'s LEFT JOIN (data/watches.ts:130-162). If one of these projections drops `primaryArchetype` but the other doesn't, the TypeScript compiler fails — but if both drop AND the `CatalogTasteAttributes` interface still has the field, the compiler also fails.
**Why it happens:** Three sites must change in lockstep: (1) `src/lib/types.ts:208,224` (interface field), (2) `src/data/watches.ts:134,157` (SELECT + mapper), (3) `src/app/catalog/[catalogId]/page.tsx:94` (manual projection).
**How to avoid:** Bundle the type-system change with the two projection-site changes in the same wave (the "type-system unification" wave per CONTEXT.md Claude's Discretion). Treat them as a single atomic task.
**Warning signs:** `tsc` errors of the form "Type '{ ... }' is missing the following properties from type 'CatalogTasteAttributes': primaryArchetype" or "Property 'primaryArchetype' does not exist on type 'CatalogTasteAttributes'".

### Pitfall 4: Missing one of the 4 enricher touch sites
**What goes wrong:** The LLM still writes `primary_archetype` but the column is gone → silent enrichment failure or a Zod parse error on every enrichment.
**Why it happens:** The enricher has FOUR sites touching this field that must all change:
1. `src/lib/taste/enricher.ts:38` — `primary_archetype` in `TASTE_TOOL.input_schema.properties`
2. `src/lib/taste/enricher.ts:51` — `'primary_archetype'` in the `required` array
3. `src/lib/taste/prompt.ts:26` — the "primary_archetype (pick exactly one):" line in `SYSTEM_FRAMING`
4. `src/lib/taste/vocab.ts:60` — `primary_archetype: z.string()` in `TasteSchema`
PLUS `src/lib/taste/vocab.ts:98-128` — the `validateAndCleanTaste()` archetype block.
**How to avoid:** Plan a single "enricher" task that touches all 5 sites atomically. Use `validateAndCleanTaste` return-type narrowing (`Omit<CatalogTasteAttributes, 'extractedFromPhoto'>`) to force TypeScript to flag the field-removal mismatch.
**Warning signs:** `taste_enrichment_failed:zod_parse_failed` events in production, or `taste_vocab_drift` warnings on every enrichment.

### Pitfall 5: `PRIMARY_ARCHETYPES` const is load-bearing post-removal
**What goes wrong:** The planner deletes `PRIMARY_ARCHETYPES` from `vocab.ts` because the column is gone, but the rewired `getBrowseArchetypeCounts()` SQL needs the 10-value list to filter (`WHERE tag = ANY(ARRAY[...])`), AND `PrimaryArchetype` type (retained per D-EXPLORE-02) needs it.
**Why it happens:** Surface-level read of "remove genre" suggests removing the vocab too. CONTEXT.md flags this in `canonical_refs:115` as planner discretion.
**How to avoid:** Keep `PRIMARY_ARCHETYPES` (and the derived `PrimaryArchetype` type union — already duplicated at `src/lib/types.ts:234-236`). Remove only the parts of `vocab.ts` that wrote the column. If the planner wants to consolidate, move `PRIMARY_ARCHETYPES` from `vocab.ts` to `archetype-config.ts` alongside `ARCHETYPE_CONFIG` (CONTEXT.md Claude's Discretion — planner call).
**Warning signs:** TypeScript errors like "Cannot find name 'PRIMARY_ARCHETYPES'" in `getBrowseArchetypeCounts` or in `archetype-config.ts`'s type signature.

### Pitfall 6: `next/cache` `revalidateTag` is NOT needed but `cacheTag` MUST be preserved
**What goes wrong:** The planner thinks "the SQL changed → must call `revalidateTag('explore')` to bust the cache" → adds an unnecessary mutation hook.
**Why it happens:** `cacheTag` is invalidated by callers via `revalidateTag`, not by the cache producer. Changing the function body doesn't change the cache key (which is computed from the function inputs — there are none — and the build ID). On the next deploy, the build ID changes and the cache is invalidated automatically.
**How to avoid:** Do not add `revalidateTag` calls. The existing `revalidateTag('explore', 'max')` in catalog mutation paths (per `src/data/browse.ts:18-27` documentation) continues to bust the cache on the rare catalog write.
**Warning signs:** Unnecessary code complexity; potentially redundant cache busts.

### Pitfall 7: Drizzle migration filename ordinal collision
**What goes wrong:** The planner generates `drizzle/migrations/0012_phase49_1_drop_primary_archetype.sql` but `drizzle-kit push` against the meta journal expects ordinal sequencing. Picking `0011` (the last existing migration) would collide; picking `0013` would skip.
**Why it happens:** Drizzle ordinals are sequential, not date-based. The latest existing migration is `0011_phase38_catalog_id_notnull.sql`; the next one MUST be `0012`. Supabase, by contrast, is date-prefixed (latest is `20260519000000_phase47_published_at.sql`).
**How to avoid:** Use `ls drizzle/migrations/ | tail -1` to confirm the highest ordinal before authoring. The new drizzle file MUST be `0012_phase49_1_drop_primary_archetype.sql`. The new supabase file MUST be timestamped >`20260519000000` (e.g., `20260519010000_phase49_1_drop_primary_archetype.sql` or `20260520000000_…`).
**Warning signs:** `drizzle-kit push` errors about ordinal mismatch; the meta journal showing a gap or duplicate.

### Pitfall 8: Vitest test file count under-estimate (CONTEXT.md missed half)
**What goes wrong:** Planner allocates effort for 11 test files (CONTEXT.md count) but discovers ~15 more in `/tests/` at execution time, blowing the wave estimate.
**Why it happens:** CONTEXT.md only enumerated `src/`-local tests. The vitest config (`vitest.config.ts`) globs both `tests/**/*.test.{ts,tsx}` AND `src/**/*.test.{ts,tsx}`. The /tests directory contains additional test files at `tests/unit/`, `tests/app/`, `tests/integration/`, `tests/components/`, `tests/actions/`, `tests/static/`, plus `tests/fixtures/` and `tests/setup/`.
**How to avoid:** Use the full test inventory in §"Validation Architecture" below (26 files total). Plan tasks per-file, not in aggregate.
**Warning signs:** Vitest fails on `npm test` post-edit with errors not anticipated in the wave plan.
