**Accent reserved for (explicit list):**
- Active/selected state of chip-group items (if chip pattern chosen for condition / box_papers / paid_currency selectors)
- Badge `variant="default"` fill (if sold badge uses `variant="default"` — see Badge section below)
- Focus ring on form inputs and accordion trigger

**Sold badge color decision** (RESEARCH §2 — Option A chosen):
- `status === 'sold'` → `variant="secondary"` (muted background, visually distinct from outline; not alarming)
- All other statuses → `variant="outline"` (existing behavior)
- Rationale: secondary is less disruptive than destructive (red) for a sold state that is a neutral lifecycle event, not an error.

### Components Modified

#### WatchCard.tsx — Sold Badge Treatment

**Change:** One-line conditional variant swap on the existing status Badge.

```tsx
// Before (all statuses):
<Badge variant="outline">{watch.status}</Badge>

// After (Phase 37):
<Badge variant={watch.status === 'sold' ? 'secondary' : 'outline'}>
  {watch.status}
</Badge>
```

- Badge position: unchanged — `absolute top-2 right-2 flex flex-col items-end gap-1` (RESEARCH §2, line 51–72)
- Badge text: `watch.status` verbatim (capitalize via existing CSS; no copy change)
- `variant="secondary"` renders with `--muted` background and `--secondary-foreground` text
- No additional overlay, strikethrough, or opacity change (Option B/C deferred)

#### WatchForm.tsx — "Collector's Record" Accordion Disclosure

**Change:** Add a new Card section after the existing "Notes" Card, before the Save/Cancel Actions row (per RESEARCH §1 — "after Notes, before Actions"). Gated on `mode === 'edit'`.

**Primitive:** `@base-ui/react/accordion` — import path `from '@base-ui/react/accordion'`. NOT from `@/components/ui/accordion` (that file does not exist — RESEARCH §10 landmine L-07).

**Pattern (mirror WatchSearchRowsAccordion.tsx):**

```tsx
import { Accordion } from '@base-ui/react/accordion'
import { ChevronDown } from 'lucide-react'

{mode === 'edit' && (
  <Card>
    <Accordion.Root>
      <Accordion.Item value="collectors-record">
        <Accordion.Header>
          <Accordion.Trigger
            className="flex w-full items-center justify-between px-6 py-4 text-sm font-semibold text-foreground data-[panel-open]:[&>svg]:rotate-180 [&>svg]:transition-transform"
          >
            <span>Collector's Record</span>
            <ChevronDown className="h-4 w-4" aria-hidden />
          </Accordion.Trigger>
        </Accordion.Header>
        <Accordion.Panel
          className="overflow-hidden px-6 pb-4 data-[open]:animate-in data-[open]:fade-in-0 data-[ending-style]:animate-out data-[ending-style]:fade-out-0 duration-150"
        >
          {/* 7 provenance field inputs — see Field Layout below */}
        </Accordion.Panel>
      </Accordion.Item>
    </Accordion.Root>
  </Card>
)}
```

**Collapsed-by-default contract:** `Accordion.Root` has NO `defaultValue` prop and no controlled `value` prop set to a non-empty array. The base-ui Accordion renders collapsed when `value` is absent or `[]` (uncontrolled with no initial open item). The accordion MUST be uncontrolled for this component — WatchForm's state does not need to track accordion open state.

**Edit-page-only gate:** The `{mode === 'edit' && (...)}` guard mirrors the existing `{mode === 'create' && <CatalogPhotoUploader />}` pattern at WatchForm.tsx line 615. The accordion is NOT rendered (not hidden, not collapsed behind a display:none — literally absent from the DOM) on the create page.

**Chevron animation:** `data-[panel-open]:[&>svg]:rotate-180` on the Trigger — same data attribute as WatchSearchRowsAccordion. The animation uses `[&>svg]:transition-transform` for smooth 180° rotate.

**Panel animation:** `data-[open]:animate-in data-[open]:fade-in-0 data-[ending-style]:animate-out data-[ending-style]:fade-out-0 duration-150` — tw-animate-css classes already imported via `globals.css line 2 @import "tw-animate-css"`.

### Field Layout Inside Accordion Panel

7 provenance fields organized in 3 logical groups. Each group uses `space-y-4` vertical rhythm. Groups separated by `mt-6` (24px).

**Group 1 — Acquisition**

| Field | Input Primitive | Type | Tailwind Classes |
|-------|----------------|------|-----------------|
| `purchase_date` | `<Input type="date">` | date (YYYY-MM-DD string) | Standard form input styling — same as existing inputs |
| `year_of_acquisition` | `<Input type="number" min="1900" max="2100">` | integer | Same |
| `serial` | `<Input type="text" placeholder="e.g. F12345">` | text | Same |

Note: `acquisition_date` (legacy text field) is NOT added to the form in Phase 37. It is already in `formData` initial state but not currently rendered in WatchForm JSX (RESEARCH §1 landmine L-01). Phase 37 does not add it.

**Group 2 — Condition & Documentation**

| Field | Input Primitive | Allowed Values | Notes |
|-------|----------------|----------------|-------|
| `condition` | `<Select>` (matches WatchForm aesthetic) | mint, near_mint, excellent, good, fair, poor | Display: capitalize + underscore→space |
| `box_papers` | `<Select>` | none, box_only, papers_only, full_set | Display: "None", "Box only", "Papers only", "Full set" |

Chip-group alternative (RESEARCH §8 Option B): allowed by plan discretion but `<Select>` is the default recommendation to maintain form aesthetic consistency. If planner chooses chip-group, use the FilterBar Badge chip pattern (RESEARCH §8) with `variant={active ? 'default' : 'outline'}` and `cursor-pointer transition-colors`.

**Group 3 — Financials & Service**

| Field | Input Primitive | Type | Notes |
|-------|----------------|------|-------|
| `paid_currency` | `<Select>` | USD, EUR, GBP, JPY, CHF, AUD, CAD, HKD, SGD, CNY | Companion to existing `pricePaid` field; do NOT add a second price input |
| `service_history` | `<Textarea>` | text | Mirrors existing `notes` Textarea — same row count (3 rows) and resize behavior |

**pricePaid cross-reference (landmine L-03):** The existing `pricePaid` field is in the main form body (WatchForm.tsx lines 373–387), NOT in the Accordion. The Accordion should NOT add a second price input. The `paid_currency` Select sits conceptually alongside `pricePaid` but physically inside the Accordion (since it's a new Phase 37 field). This is acceptable UX — the collector understands currency as a property of their purchase price, even if the inputs are in different sections.

## Interaction Contracts

### Accordion — Collapsed Default

- Renders collapsed on every page load (uncontrolled, no `defaultValue`).
- User opens with click or keyboard (Enter/Space on trigger) — base-ui Accordion handles focus management natively.
- Chevron icon rotates 180° on open (`data-[panel-open]:[&>svg]:rotate-180`).
- Panel fades in (`data-[open]:animate-in data-[open]:fade-in-0`) over 150ms.
- Panel fades out (`data-[ending-style]:animate-out data-[ending-style]:fade-out-0`) over 150ms.
- No visual regression on collapsed state — when closed, the Card shows only the trigger row with "Collector's Record" + chevron.

### Accordion — Edit-Only Gate

- `mode === 'create'` → accordion Card is NOT in the DOM (no `display:none`, no hidden attribute — literally not rendered).
- `mode === 'edit'` → accordion Card renders after the Notes Card, before the Save/Cancel button row.

### Sold Badge — WatchCard

- `watch.status === 'sold'` → Badge `variant="secondary"` (muted background, stands out from outline).
- All other statuses → Badge `variant="outline"` (existing behavior, no change).
- No additional opacity, overlay, strikethrough, or grayscale on the sold WatchCard — the card remains fully interactive (sold watches stay editable per CONTEXT.md D-14 / deferred section "Sold-watch read-only edit page").

### Status Transition — recordDivestment Wire-up

- The `owned → sold` transition is detected server-side inside `editWatch` (RESEARCH §4 — Option A: server-side detection).
- WatchForm client code does NOT call `recordDivestment` separately. The dual-write (INSERT divestments + UPDATE watches.status) happens atomically inside a single Server Action execution, wrapped in `db.transaction()`.
- No confirmation dialog in Phase 37 (CONTEXT.md D-12 — dialog deferred to v5.x).
- Error surface: if the Server Action returns `{ ok: false }`, the existing `useFormFeedback` hook surfaces the error via the existing `FormStatusBanner` component — no new error UI.

### Provenance Field Inputs — Behavior

| Field | Behavior |
|-------|----------|
| `purchase_date` | `<input type="date">` — native date picker; value is YYYY-MM-DD string; empty string → `undefined` in formData |
| `year_of_acquisition` | `<input type="number">` — min=1900, max=2100; integer only; empty → `undefined` |
| `serial` | `<input type="text">` — any characters accepted; no max-length UI constraint (CONTEXT.md D-07) |
| `condition` | `<Select>` — single value or unset (placeholder); clearing sets field to `undefined` |
| `box_papers` | `<Select>` — single value or unset; clearing sets field to `undefined` |
| `paid_currency` | `<Select>` — single value or unset; clearing sets field to `undefined` |
| `service_history` | `<Textarea>` — multiline free text; no character limit shown; autoresize off (matches `notes` Textarea) |

All 7 fields are nullable. No validation added beyond existing WatchForm required-field checks (brand + model only). Provenance fields have no required constraints.
