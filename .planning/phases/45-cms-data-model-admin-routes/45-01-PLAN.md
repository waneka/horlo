<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anon/authenticated DB role → Postgres RLS | Untrusted reads/writes cross here; RLS is the gate |
| non-owner session → CMS tables | A logged-in non-owner must never read drafts or write any CMS row |
| catalog mutation → CMS FK graph | A catalog delete must not orphan a CMS reference |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-45-01 | Information Disclosure | curated_lists / collection_paths draft rows | mitigate | Public SELECT policy `USING (status = 'published')` for authenticated+anon; owner-only SELECT policy gated on `is_admin` EXISTS predicate |
| T-45-02 | Elevation of Privilege | INSERT/UPDATE/DELETE on all 5 CMS tables | mitigate | Write policies gated on `EXISTS (SELECT 1 FROM profiles WHERE id = (SELECT auth.uid()) AND is_admin)` |
| T-45-03 | Tampering / DoS | watches_catalog row referenced by a CMS table | mitigate | FK `ON DELETE RESTRICT` on catalog-referencing columns — DB refuses the delete |
| T-45-04 | Elevation of Privilege | profiles.is_admin column | accept | Single-owner project; `is_admin` set only by migration via email key. RLS on profiles already restricts UPDATE to `id = auth.uid()`; a future second user cannot self-grant admin without a separate migration — re-check when a 2nd user signs up |
| T-45-05 | Tampering | cms_settings multi-row insert | mitigate | `CHECK (id = 1)` single-row constraint blocks any second settings row |
</threat_model>
