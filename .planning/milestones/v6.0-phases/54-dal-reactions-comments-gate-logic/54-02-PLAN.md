<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server Action (Phase 55) → DAL | All callers pass an explicit `userId`/`viewerId` resolved from `getCurrentUser()`; the DAL never reads the session. |
| DAL → Postgres (Drizzle `db`) | Drizzle bypasses RLS; the DAL functions run as the DB owner. Likes are intentionally open to any authenticated user on all statuses (GATE-02, Phase 53). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-03 | Tampering (IDOR) | deleteLike | mitigate | `deleteLike` WHERE includes `userId = caller` AND the target FK — a user cannot delete another user's like row. |
| T-54-04 | Tampering | createLike duplicate spam | mitigate | `onConflictDoNothing()` + `*_unique_pair` UNIQUE (Phase 53) makes likes idempotent (LIKE-05); no duplicate rows. |
| T-54-05 | Information disclosure | getLikesForTarget viewer state | accept | `viewerHasLiked` reflects only the passed `viewerId`'s own like; per-viewer cache scoping (SEC-05) is enforced in Phase 55, not here. Count is public by design (count-only display, "not Instagram"). |
| T-54-06 | Spoofing | mutual-follow direction confusion | mitigate | `isMutualFollow` checks BOTH directions in one query and requires both rows; a one-way follow returns false (GATE-05) — prevents a one-way follower passing the wishlist gate in Plan 03. |
</threat_model>
