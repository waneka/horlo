the bottom-sheet drawer.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → RSC (router.refresh on /watch/[id]) | The post-success `router.refresh()` re-fetches the current route's RSC payload. This crosses the client→server boundary via the Next.js runtime. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-57.1-02 | Information Disclosure | router.refresh() RSC re-fetch | accept | `router.refresh()` re-runs the SAME server tree under the SAME viewer session — it cannot return data the viewer was not already authorized to see (the DAL gate `getCommentsForTarget` / `canViewerCommentOnTarget` runs server-side per-viewer, unchanged). No client-supplied parameter influences what is fetched. RESEARCH-verified: clears client cache only, does not touch the server tag cache. |
| T-57.1-03 | Denial of Service | router.refresh() fired on every successful post (all surfaces) | accept | One refresh per successful comment post is human-paced and bounded by the existing post action's rate (no loop, no auto-retry). Wears surfaces tolerate it (local optimistic state survives the merge per Next.js docs). Not a meaningful amplification vector for a personal-collector app. |
| T-57.1-04 | Tampering | WearCard local count state | accept | `localCommentCount` is a display-only optimistic counter guarded by `Math.max(0, ...)`; it never feeds back into any server write and self-corrects to the authoritative server count on reload (D-03). No integrity impact. |
</threat_model>
