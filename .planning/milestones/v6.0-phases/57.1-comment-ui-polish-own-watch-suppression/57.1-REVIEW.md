# Phase 57.1: Code Review Report

**Depth:** standard
**Files Reviewed:** 7

## Summary

Reviewed the Phase 57.1 delta (diff base `14762b5^`) across the seven comment/wear UI files. The phase is a polish layer over the shipped Phase 57 comment stack: SC-5 optimistic count chain, D-01 `router.refresh()` on post-success, SC-6 `suppressCompose` own-watch suppression, plus layout reshuffles in `CommentCompose` and `CommentItem`.

The deliberate invariant on `src/app/watch/[id]/page.tsx` — count read on raw `canComment`, compose gated on `canCommentDisplay` — is implemented correctly (lines 59, 78, 172, 177) and is NOT flagged. The SC-6 conditional precedence in `CommentList` (`!suppressCompose && (canComment ? … : …)`) is also correct, and the owner gets defense-in-depth (both `canCommentDisplay=false` and `suppressCompose=true`).

No BLOCKER-tier issues: no security gaps, no data loss, no crashes. The optimistic add/rollback and delete/rollback count deltas are balanced. However, the D-01 `router.refresh()` implementation is **asymmetric** — it fires only on add, not on delete — which leaves the `/watch/[id]` RSC count badge (the stated D-01 target) stale after a delete. Two further count-accuracy and cross-surface side-effect concerns are documented below.

## Warnings

### WR-01: D-01 `router.refresh()` is add-only — delete leaves the `/watch/[id]` count badge stale

**File:** `src/components/comment/CommentList.tsx:129` (refresh placement) and `137-141` (delete path with no refresh)
**Issue:** D-01's own comment states its purpose is to "re-fetch RSC payload so /watch/[id] count badge updates." But `router.refresh()` is only called in the success branch of `handleSubmit` (add). The delete path (`handleDeleteOptimistic`) removes the comment from the client list and calls `onCountChange?.(-1)` — which is a no-op on `/watch/[id]` because `CommentThread`/`page.tsx` never pass `onCountChange`. It does NOT call `router.refresh()`. The WatchDetail badge is a Server Component prop (`commentCount`, resolved in `page.tsx:78-80`, rendered in `WatchDetail.tsx:169-174`). `deleteCommentAction` only `revalidateTag('profile:…')` (see `src/app/actions/comments.ts:306-333`) and `CommentThread` is an uncached RSC, so nothing updates the badge. Net effect: after an owner or comment-author deletes a comment on `/watch/[id]`, the thread shows one fewer comment while the badge above keeps the higher count until a full reload. This is the exact count-drift class D-01 was meant to prevent, but only the add direction was wired.
**Fix:** Mirror the add-success refresh on the delete-success path so the RSC count badge stays consistent. Add `router.refresh()` after the server confirms the delete (not in the optimistic step, and not on rollback):
```tsx
// In CommentItem.handleDeleteConfirm, after a successful delete the parent
// already removed the row. Add a success-only refresh callback, OR move the
// refresh into CommentList by giving onDeleteOptimistic a paired success hook.
// Simplest: in CommentList, add a handleDeleteSuccess passed down to CommentItem:
function handleDeleteConfirm() {
  onDeleteOptimistic(comment.id)
  startTransition(async () => {
    const result = await deleteCommentAction({ commentId: comment.id })
    if (!result.success) {
      onRollbackDelete(comment)
      console.error('[CommentItem] delete failed:', result.error)
      return
    }
    onDeleteSuccess?.() // CommentList calls router.refresh() here
  })
}
```
Ensure the refresh fires ONLY on delete success (not on the optimistic step and not on rollback), matching the add-path discipline.

### WR-02: `router.refresh()` fires on every add, including the bottom-sheet `/wears/[username]` lane — D-01 "harmless on wears" claim is incomplete

**File:** `src/components/comment/CommentList.tsx:125-129`
**Issue:** `router.refresh()` is unconditional on add-success across every surface that mounts `CommentList`, including the bottom-sheet comment host inside `WearsLane` (`src/components/wears/WearsLane.tsx:398`). The D-01 comment dismisses this as "Harmless on wears surfaces — local optimistic count survives the RSC merge." That claim under-accounts for the side effects: `router.refresh()` busts the Router Cache and re-fetches the *entire* `/wears/[username]` RSC payload while the comment sheet is open. WearsLane's `select` effect depends on `slides` (`WearsLane.tsx:176`); a refreshed `slides` reference re-runs that effect, which immediately calls `handleSelect()` → `markViewed()` (`WearsLane.tsx:162-176`). The optimistic local count does survive (WearCard `localCommentCount` is `useState`-seeded once and never re-synced), and `markViewed` is idempotent (`useViewedWears.ts:65-71`), so there is no functional corruption — but a full embla-lane re-fetch + effect re-run on every comment post is an unintended, heavier-than-documented side effect on a known-fragile surface (see the project's documented Router Cache stale-instance and embla swipe-pause concerns).
**Fix:** Scope the refresh to the surface that needs it. Either (a) only call `router.refresh()` when `onCountChange` is undefined (i.e. the `/watch/[id]` RSC-badge surface, not the wears surfaces which already have local-count state), or (b) pass an explicit `refreshOnSuccess?: boolean` prop set true only from `CommentThread`/`page.tsx`:
```tsx
// Option A — refresh only where there is no local-count owner:
if (!onCountChange) router.refresh()
```
Then update the D-01 comment to reflect that wears surfaces are intentionally excluded.

### WR-03: `Math.max(0, n + delta)` count clamp can permanently desync the WearCard badge

**File:** `src/components/wear/WearCard.tsx:235` and `249`
**Issue:** Both `onCountChange` handlers clamp with `setLocalCommentCount((n) => Math.max(0, n + delta))`. The clamp prevents a visible negative count, but it silently swallows underflow and can leave `localCommentCount` permanently off-by-one from the true comment total for the life of the mounted instance (D-03 only "self-corrects on next page load"). Concrete path: the seed prop `commentCount` (`= initialComments.length`) and the actual rendered list can diverge if the gate/visibility of `initialComments` differs from the count source, or under interleaved optimistic add/delete where a `-1` lands at 0 and is clamped, then a later `+1` rollback over-counts. Because there is no `useEffect` re-sync of `localCommentCount` from the `commentCount` prop (confirmed: no effect in WearCard), even the `router.refresh()` from WR-02 that updates the `commentCount` prop will NOT correct the badge — the stale local state wins until remount.
**Fix:** This is an accepted-tradeoff per D-03, but at minimum add a prop-sync so a successful `router.refresh()` (which updates `commentCount`) reconciles the badge instead of being ignored:
```tsx
useEffect(() => {
  setLocalCommentCount(commentCount)
}, [commentCount])
```
Note the tradeoff: this would clobber an in-flight optimistic delta if the prop updates mid-transition. If that risk is unacceptable, keep the local-only model but remove the misleading "self-corrects on next page load" framing and document that `router.refresh()` does NOT correct this badge.

## Info

### IN-01: Duplicated inline count-change handler across both WearCommentHost branches

**File:** `src/components/wear/WearCard.tsx:235` and `249`
**Issue:** The identical arrow `(delta) => setLocalCommentCount((n) => Math.max(0, n + delta))` is repeated verbatim in the bottom-sheet and inline `WearCommentHost` branches. Duplication invites divergence if the clamp logic changes (see WR-03).
**Fix:** Hoist to a single named handler and pass the reference to both branches:
```tsx
const handleCountChange = (delta: number) =>
  setLocalCommentCount((n) => Math.max(0, n + delta))
// …onCountChange={handleCountChange} in both branches
```

### IN-02: `composeKey` comment is stale relative to its implementation

**File:** `src/components/comment/CommentList.tsx:69`

The comment describes a different design than what shipped and will mislead future readers.
**Fix:** Update the comment to: `// composeKey: bump to remount (and thus clear) CommentCompose on successful post`.
