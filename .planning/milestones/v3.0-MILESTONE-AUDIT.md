## Cross-Phase Integration

### Verified Flows

1. **Three-tier visibility** — Phase 11 schema → Phase 12 DAL ripple → Phase 15 photo flow → `/wear/[id]` gating. Zero functional `worn_public` references in `src/`.
2. **Notifications** — Phase 11 schema/RLS → Phase 13 logger/DAL/UI → wired into `followUser` + `addWatch` actions → `NotificationBell` rendered once in Phase 14 `Header` and passed by reference to both nav surfaces.
3. **People search** — Phase 14 nav search input → Phase 16 `/search` → Server Action → `searchProfiles` DAL → Phase 11 pg_trgm GIN indexes.
4. **Public-paths gate** — `src/lib/constants/public-paths.ts` is the single source consumed by `proxy.ts` + 3 nav components. No forks.
5. **Shared NavWearButton** — Same component used in `BottomNav` and `DesktopTopNav` via `appearance` prop; opens the same lazy-loaded `WywtPostDialog`. NAV-09 satisfied.
6. **Nav route links** — All routes referenced by nav components exist on disk (`/`, `/explore`, `/watch/new`, `/search`, `/notifications`, `/wear/[id]`).

### Partial Wiring

**WYWT-17/18 post-submit navigation** — `logWearWithPhoto` returns `{ success, data: { wearEventId } }` but `ComposeStep.handleSubmit` discards `wearEventId`; `WywtPostDialog.onSubmitted` only closes the dialog. After a successful wear post the user lands back on `/` with a toast — they never auto-navigate to the new `/wear/[wearEventId]`.

**Note:** Phase 15 VERIFICATION.md marks both WYWT-17 (route exists with gating + uniform 404) and WYWT-18 (rail overlay preserved; other entry points navigate to detail) as **SATISFIED**. The requirement text does not explicitly mandate post-submit navigation. This is a UX enhancement question, not a requirement gap.
