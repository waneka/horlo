# Phase 999.1: Code Review Report

**Reviewed:** 2026-04-22
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

The phase shipped its three stated objectives cleanly: (1) PreferencesClient now inspects `ActionResult` from `savePreferences` and renders a `role="alert"` banner on failure with a `Saving…` aria-live indicator while a save is in flight, (2) the unused `UnauthorizedError` imports are gone from both Server Action files, and (3) the MR-03 closure note is in tree with verbatim migration filenames. The MR-01 fix faithfully follows the canonical `ProfileEditForm` pattern documented in the plan, and the `StringArrayKeys` mapped type added to PreferencesClient is a nice incremental hardening over the original raw `keyof` cast.

That said, the standard-depth pass surfaced several issues in surrounding code that this phase had to interact with but did not change. The most significant is **server-side input validation gaps in both Server Action Zod schemas** — `pricePaid`/`targetPrice`/`marketPrice`/`caseSizeMm`/`lugToLugMm`/`waterResistanceM` accept negative numbers, `imageUrl` accepts arbitrary strings (no `.url()`), `acquisitionDate` accepts arbitrary strings (no ISO date check), `productionYear` accepts any integer (no realistic range), `notes` accepts unbounded length, `preferredCaseSizeRange` accepts unbounded `min`/`max` and does not enforce `min <= max`. None of these are critical (no SQL injection vector, no auth bypass), but they collectively let bad data into the row store where the client is the only validator. The other meaningful finding is a **stale-closure race in `toggleArrayItem`**: the function reads `preferences[field]` from a closure captured at render time, so two rapid clicks on different checkboxes (before React commits the in-between state) can drop one of the two toggles. This is the same general pattern as the rest of the file, so it predates this phase, but it's worth flagging because the new `setSaveError(null)` happens-before semantics in `updatePreferences` interact with it.

No critical security or correctness issues. The phase fixes are sound; the warnings below are mostly latent issues in the surrounding code that the standard-depth lens picked up.

## Warnings

### WR-01: Stale closure in `toggleArrayItem` can drop rapid sequential toggles

**File:** `src/components/preferences/PreferencesClient.tsx:65-71`
**Issue:** `toggleArrayItem` reads `preferences[field]` from the render-time closure of `preferences` and computes `newArray` from it, then calls `updatePreferences(patch)`, which itself reads `preferences` from its own closure to build `next = { ...preferences, ...patch }`. If a user clicks two different checkboxes in the same React tick (before re-render commits), both handlers see the same `preferences` snapshot. The second toggle's `newArray` is built from a `currentArray` that does not include the first toggle's change, so the patch sent to the server reflects only the second toggle for its field — which is fine — but `setLocalPreferences(next)` in `updatePreferences` overwrites local state with `{ ...stalePrefs, ...secondPatch }`, dropping the first toggle's local-state effect entirely. The user sees their first click "uncheck itself" until the next navigation reconciles.

This is the same pattern that pre-existed in the file, so it isn't a regression — but the new `setSaveError(null)` clear at the top of `updatePreferences` interacts with it in a way that could mask the bug from the user (they see no error, just a checkbox flipping back).

**Fix:**
```tsx
const updatePreferences = (patch: Partial<UserPreferences>) => {
  setLocalPreferences((prev) => ({ ...prev, ...patch }))
  setSaveError(null)
  startTransition(async () => {
    const result = await savePreferences(patch)
    if (!result.success) {
      setSaveError(result.error)
    }
  })
}

const toggleArrayItem = (field: StringArrayKeys, item: string) => {
  // Read latest state inside the updater to avoid stale-closure drops.
  setLocalPreferences((prev) => {
    const currentArray = prev[field]
    const newArray = currentArray.includes(item)
      ? currentArray.filter((i) => i !== item)
      : [...currentArray, item]
    const next = { ...prev, [field]: newArray }
    setSaveError(null)
    startTransition(async () => {
      const result = await savePreferences({ [field]: newArray } as Partial<UserPreferences>)
      if (!result.success) setSaveError(result.error)
    })
    return next
  })
}
```

### WR-02: Server-side numeric validation accepts negative values for prices and dimensions

**File:** `src/app/actions/watches.ts:18-25`
**Issue:** `pricePaid`, `targetPrice`, `marketPrice`, `caseSizeMm`, `lugToLugMm`, `waterResistanceM` are all declared as `z.number().optional()` with no `.nonnegative()` (or `.positive()` / `.min(...)`) constraint. A malicious or misbehaving client can send `pricePaid: -50000` and the action will accept it, write it to the DB, and surface negative values back to the UI (which then shows `$−50,000`). Similar concern for `caseSizeMm: -42` rendering as `−42mm` everywhere it appears. The client UI only enforces these via `<input type="number" min="…">`, which is trivially bypassable.

**Fix:**
```ts
pricePaid: z.number().nonnegative().optional(),
targetPrice: z.number().nonnegative().optional(),
marketPrice: z.number().nonnegative().optional(),
caseSizeMm: z.number().positive().max(100).optional(),
lugToLugMm: z.number().positive().max(100).optional(),
waterResistanceM: z.number().int().nonnegative().max(12000).optional(),
```

### WR-03: `imageUrl` field accepts arbitrary strings — no URL validation

**File:** `src/app/actions/watches.ts:37`
**Issue:** `imageUrl: z.string().optional()` does not constrain the string to a valid URL or a safe scheme. Anything goes — `imageUrl: "javascript:alert(1)"`, `imageUrl: "<img src=x onerror=...>"`, or arbitrary garbage. Any downstream `<img src={watch.imageUrl}>` that does not itself validate is exposed. Because Next/Image typically validates URLs, the immediate XSS risk is low, but plain `<img>` consumers (`logActivity` writes `imageUrl` into the activity log payload, which various feed renderers may consume) are not guaranteed to.

**Fix:**
```ts
imageUrl: z.string().url().optional(),
// Optionally restrict to known-safe schemes:
// imageUrl: z.string().url().refine(
//   (u) => /^https?:/.test(u), 'Only http/https URLs allowed'
// ).optional(),
```

### WR-04: `preferredCaseSizeRange` server schema does not enforce bounds or `min <= max`

**File:** `src/app/actions/preferences.ts:21-26`
**Issue:** The server accepts any `{ min: number, max: number }` pair, including `{ min: -1000, max: 1e9 }` or `{ min: 50, max: 20 }` (inverted). The client UI enforces both constraints (`clampCaseSize` plus the `if (nextMin > currentMax) return` guard), but the server should not trust the client. Inverted ranges will silently break similarity-engine reasoning that assumes `min <= max`.

**Fix:**
```ts
preferredCaseSizeRange: z
  .object({
    min: z.number().min(20).max(55),
    max: z.number().min(20).max(55),
  })
  .refine((r) => r.min <= r.max, {
    message: 'min must be <= max',
    path: ['min'],
  })
  .optional(),
```

### WR-05: `notes` field accepts unbounded string length

**File:** `src/app/actions/watches.ts:36` and `src/app/actions/preferences.ts:31`
**Issue:** Both `notes` schemas are `z.string().optional()` with no `.max(...)` ceiling. A misbehaving client can POST a multi-megabyte string. Postgres `text` columns accept arbitrary length, so this becomes a denial-of-service / storage-amplification vector even without authentication bypass (the user can attack their own account, but the bigger problem is API gateway / DB write load if a script does it in a loop). It also makes the `revalidatePath` payload heavy.

**Fix:**
```ts
// In both files:
notes: z.string().max(10_000).optional(),
```

## Info

### IN-01: `Textarea` for `notes` calls `updatePreferences` on every keystroke — no debounce

**File:** `src/components/preferences/PreferencesClient.tsx:458`
**Issue:** `onChange={(e) => updatePreferences({ notes: e.target.value })}` fires `savePreferences` for every single character typed. For a 200-character note that's 200 server roundtrips, and because each one is wrapped in `startTransition` the user can have a backlog of overlapping in-flight saves. The last-write-wins ordering is not guaranteed across the network — out-of-order completions can briefly show stale state in the banner / aria-live region. Other text/textarea fields in similar codebases typically debounce by 300-500ms.

**Fix:** Introduce a debounced wrapper specifically for free-form text fields (notes), or commit on `onBlur` instead of `onChange`. Example using `onBlur`:
```tsx
<Textarea
  defaultValue={preferences.notes ?? ''}
  onBlur={(e) => {
    if (e.target.value !== preferences.notes) {
      updatePreferences({ notes: e.target.value })
    }
  }}
  placeholder="Describe your collecting philosophy..."
  rows={4}
/>
```

### IN-02: Magic number defaults `46` and `34` in case-size handlers

**File:** `src/components/preferences/PreferencesClient.tsx:342, 370`
**Issue:** When `preferredCaseSizeRange` is undefined, the min handler defaults `currentMax` to `46` (line 342) and the max handler defaults `currentMin` to `34` (line 370). These two magic numbers don't appear anywhere else as named constants and are inconsistent with the `CASE_SIZE_MIN`/`CASE_SIZE_MAX` named constants declared two screens up (lines 73-74). A reader has to chase the meaning of `46` separately from the meaning of `34`.

**Fix:**
```tsx
const CASE_SIZE_DEFAULT_MIN = 34
const CASE_SIZE_DEFAULT_MAX = 46
// ... then:
const currentMax = preferences.preferredCaseSizeRange?.max ?? CASE_SIZE_DEFAULT_MAX
// ...
const currentMin = preferences.preferredCaseSizeRange?.min ?? CASE_SIZE_DEFAULT_MIN
```

### IN-03: `collectionGoal` schema enum includes a value the UI never offers

**File:** `src/app/actions/preferences.ts:29` vs `src/components/preferences/PreferencesClient.tsx:432-441`
**Issue:** The Zod schema accepts `'brand-loyalist'` as a valid `collectionGoal`, but the `<Select>` in PreferencesClient only renders three options (`balanced`, `specialist`, `variety-within-theme`). This is a documented gap — `'brand-loyalist'` exists in the type definition (`src/lib/types.ts:11-15`) but is unreachable from the UI. Either the UI is missing an option or the schema/type is too permissive. **Fix:** Add the missing `<SelectItem value="brand-loyalist">` if it's a real product option, or remove `'brand-loyalist'` from the schema/type if it's vestigial.

### IN-04: Repeated `try { user = await getCurrentUser() } catch { ... }` pattern across 4 actions

**File:** `src/app/actions/watches.ts:50-51, 102-103, 134-135` and `src/app/actions/preferences.ts:41-42`
**Issue:** Four near-identical guards repeat the same pattern. The plan explicitly defers migrating to `err instanceof UnauthorizedError`, which is fine, but a small `requireUser()` helper would still tighten the surface area and make the future migration a single-site edit.

**Fix:**
```ts
// In src/lib/auth.ts:
export async function requireUser(): Promise<
  { ok: true; user: { id: string; email: string } } | { ok: false; error: string }
> {
  try {
    return { ok: true, user: await getCurrentUser() }
  } catch {
    return { ok: false, error: 'Not authenticated' }
  }
}

// Then in each action:
const auth = await requireUser()
if (!auth.ok) return { success: false, error: auth.error }
const user = auth.user
```

### IN-05: Brittle DAL error-message string matching for 404 mapping

**File:** `src/app/actions/watches.ts:88, 120, 143`
**Issue:** All three watch actions detect the "not found" case via `err.message.includes('not found or access denied')`. This couples the action layer to the exact wording of an error message in the DAL. A typo, refactor, or i18n in the DAL silently breaks the 404 mapping — clients get a generic "Failed to …" instead of "Not found". The DAL should ideally throw a typed error (e.g., `NotFoundError`) so the action layer can `instanceof`-check it.

**Fix:**
```ts
// In src/data/watches.ts (or src/lib/errors.ts):
export class NotFoundError extends Error {
  constructor(message = 'Not found or access denied') {
    super(message); this.name = 'NotFoundError'
  }
}

// In actions/watches.ts:
} catch (err) {
  console.error('[addWatch] unexpected error:', err)
  if (err instanceof NotFoundError) {
    return { success: false, error: 'Not found' }
  }
  return { success: false, error: 'Failed to create watch' }
}
```
