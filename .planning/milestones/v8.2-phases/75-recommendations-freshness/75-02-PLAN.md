<objective>
DISC-RECS-VARIATION — Add deterministic-per-6h-window sampling + sparse-pool catalog-popularity top-up to `getRecommendationsForViewer` so the home rail visibly rotates across sessions and never renders <8 cards on sparse data.

Purpose: `getRecommendationsForViewer` is fully deterministic per inputs — at the small horlo.app user base, after viewer-owned/wishlist/grail exclusion the candidate pool collapses to 1-2 entries and the same 2 cards surface every refresh. Fix is two surgical changes to the existing pure-server DAL: (1) bump SEED_POOL_SIZE 15→30 + sample 15 from the top-30 using a viewerId+6h-window-seeded mulberry32 PRNG (same window = same recs = cache-stable; next window = different recs = rail rotates 4× daily); (2) when post-exclusion candidateMap.size<8, append synthetic catalog-popularity recs (ordered by `watches_catalog.ownersCount DESC, brand ASC`) so the rail always renders ≥8 cards. Rationale templates unchanged — synthetic rows fall through to the existing `community-fallback` ("Popular in the community") path.

Output:
- src/data/recommendations.ts with SEED_POOL_SIZE=30, ROTATION_WINDOW_MS=6h, inline mulberry32 PRNG, seedFor(viewerId, windowBucket) hash, Fisher-Yates shuffle of top-30 → first 15, topUpFromCatalogPopularity helper
- src/lib/discoveryTypes.ts with `representativeOwnerId: string | null` (D-12)
- src/components/home/RecommendationCard.tsx verified to handle representativeOwnerId:null (inspect-only — current source already does NOT render an owner link; no change required if verification confirms)
- src/data/__tests__/recommendations.test.ts (NEW file) with 4 pure-function unit cases
</objective>
