<objective>
Implement the `logWearWithVideo` Server Action and its `logWearEventWithVideo` DAL helper as a direct structural parallel to `logWearWithPhoto` (Phase 15) ŌĆö same auth-first ordering, same Zod-validate-then-IDOR-check pipeline, same generic-error-on-Storage-failure shape, same fire-and-forget activity log, same cache invalidation. Three documented divergences from the Phase 15 analog: (a) TWO `.list()` probes instead of one (VID-08, one per Storage object); (b) explicit 5 MB byte-length gate on `videoBytes` before Storage calls (VID-09 ŌĆö no Phase 15 analog because photos were small); (c) compensating `.remove([videoPath, posterPath])` operating on a 2-element array on insert failure (VID-10).

Purpose: this is the security pipeline that makes the server "safely accept a wear-event video upload" ŌĆö the phase goal. Plan 02's client-side path builders are NOT a trust path; this plan establishes the actual gate by (a) recomputing the Storage paths from `getCurrentUser().id` + a server-validated `wearEventId` UUID inside the Server Action, (b) probing both Storage objects exist before any DB write, and (c) cleaning up orphan Storage objects if the DB write fails. Mirrors Phase 15 threat mitigations T-15-04 / T-15-17 / T-15-18 verbatim, with the documented divergences above.

- `tests/actions/wearEventsVideo.test.ts` ŌĆö 9 unit cases covering VID-07/VID-08/VID-09/VID-10/VID-16 with mocked Supabase storage
