## Phase Details

### Phase 17: Catalog Foundation
**Goal**: A canonical `watches_catalog` table is laid silently underneath per-user `watches`, populated from both manual entry and URL extraction, and refreshed daily — unblocking /search Watches, /explore Trending, and /evaluate deep-link without modifying `analyzeSimilarity()`.
**Depends on**: Nothing (first phase of v4.0; v3.0 production frame is the substrate)
**Requirements**: CAT-01, CAT-02, CAT-03, CAT-04, CAT-05, CAT-06, CAT-07, CAT-08, CAT-09, CAT-10, CAT-11, CAT-12
**Success Criteria** (what must be TRUE):
  1. The DB has a `watches_catalog` table that an anonymous Supabase client can read but cannot write to (public-read RLS, service-role-only writes).
  2. Adding a watch via the existing form OR via URL extraction populates `watches_catalog` and links the per-user `watches` row by `catalog_id` (URL extraction enriches missing fields; typed input never overwrites).
  3. Existing `watches` rows from before the migration have non-NULL `catalog_id` after the backfill script runs once — and re-running the script is a no-op.
  4. `owners_count` and `wishlist_count` on `watches_catalog` refresh daily in production via `pg_cron`, and a developer can refresh them locally with `npm run db:refresh-counts`.
  5. A `watches_catalog_daily_snapshots` row is recorded per catalog row per day, providing the data substrate for the Phase 18 Gaining Traction rail.
**Plans**: 6 plans (all complete)

## Milestone Summary

### Decisions

- **Catalog identity is silent infrastructure in v4.0** — `watches_catalog` lays underneath but `analyzeSimilarity()` does NOT read from it; engine rewire deferred to v5.0+ to preserve the current scoring model and avoid regression risk during the discovery push.
- **Catalog FK nullable indefinitely** — `watches.catalog_id` will NEVER `SET NOT NULL` in v4.0; backfill safety wins over normalization purity until two consecutive prod deploys verify 100% backfill (CAT-14, deferred to v5.0+).
- **Catalog `analyzeSimilarity()` byte-lock** — preserved across Phase 17, 19.1, and 20 (verifier shasum checks against pre-phase HEAD).
- **Phase 19.1 inserted mid-milestone** for LLM-derived taste enrichment — feeds Phase 20 verdict copy quality without engine changes.
- **`/evaluate` route eliminated** — URL-paste capability moved into the Add-Watch Flow as `verdict-as-step`. The 3-button gesture (wishlist / owned / skip) is the canonical evaluation surface; `tests/no-evaluate-route.test.ts` static guard prevents reintroduction.
- **BottomNav reshape to 5 slots** (Home / Search / Wear / Explore / Profile) supersedes original DISC-08 / NAV-14 wording; Phase 18 D-03 / D-04 explicitly amend "Notifications stays in TopNav bell, Profile permanent in BottomNav."
- **Custom Resend SMTP `mail.horlo.app`** — DKIM/SPF/DMARC verified before any code change; D-07 round-trip gate (Invite + real Gmail signup) had to pass Inbox (not Spam) BEFORE flipping Confirm-email toggle ON.
- **Auth email templates standardized** — All three Supabase Auth templates (Confirm signup / Reset Password / Change Email) now route through `/auth/callback?token_hash={{ .TokenHash }}&type=...&next=...` PKCE+SSR pattern. Phase 21 over-delivered by realigning all three (preempting SET-04 in Phase 22).
- **Settings tab state is hash-driven via `window.history.pushState`** (not `router.push`) — switching tabs doesn't re-run the page Server Component loader; URL fragment shareable and back-button friendly.
- **Email change UI does NOT display new email as current pre-confirmation** — T-22-S4 mitigation; pending banner says "Confirmation sent to BOTH old@ AND new@", and the disabled input still shows currentEmail.
- **24-hour session staleness threshold for password change re-auth** — `isSessionStale(lastSignInAt)` opens a re-auth dialog with chained `signInWithPassword` → `updateUser` (D-08 Option C with 401 catch as defense-in-depth).
- **Notification enum cleanup via rename+recreate** — Postgres has no `ALTER TYPE … DROP VALUE`; the rename+recreate pattern (with T-24-PARTIDX partial-index surgery for enum-bound dependents) is the costly but safe path.
- **Hybrid form feedback pattern: Sonner toast + inline `aria-live="polite"` banner** — Toast for transient success cue, banner for accessible persistence/error state. `useFormFeedback` hook + `FormStatusBanner` component standardize across 7 forms.
- **5-category URL-extract error taxonomy** — `host-403`, `structured-data-missing`, `LLM-timeout`, `quota-exceeded`, `generic-network`. Locked D-15 copy + D-14 lucide icons; AddWatchFlow's `extraction-failed` branch mounts `<ExtractErrorCard>` with retry/manual recovery CTAs.
