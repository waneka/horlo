<domain>
## Phase Boundary

Phase 79 EXECUTES v8.4 canonicalization ‚Äî it is the data-write phase.

**Delivers:**
1. Brand backfill `--apply` ‚Äî every `watches_catalog` row resolves to a canonical `brand_id`. New `brands` rows are created for the 33 operator-marked `new` entries; the single `Hamilton Watch ‚Üí Hamilton` merge collapses correctly.
2. Family decisions artifact (`.planning/v8.4-family-merge-decisions.md`) emitted by the same script in `--mode=families` dry-run, operator-edited offline, then consumed by `--apply`.
3. Family backfill `--apply` ‚Äî every `watches_catalog` row resolves to a canonical `family_id`. Operator `merge:<uuid>` decisions on typo cases (`Brut Date` ‚Üí `Brut Datejust`) auto-append the source string to the canonical family's `aliases` array.
4. `watches` display hydration (DISP-03) ‚Äî every user's `watches` row has its `brand` + `model` text columns overwritten from the canonical `brands.name` / `watch_families.name` resolved through `catalogId`. All catalogId-non-NULL rows (which is all rows, since catalogId is NOT NULL since Phase 38) regardless of status. Notes / serial / all other text untouched.
5. Post-flight assertion (MIG-04) using a DIFFERENT predicate from the UPDATE's WHERE-clause verifies zero unresolved rows on `watches_catalog`.
6. Operator-facing `79-POST-DEPLOY.md` (auto-generated by the script after a successful apply) with counts, post-flight assertion result, and sign-off checklist ‚Äî mirrors Phase 78 pattern.

**Explicitly NOT in this phase:**
- NOT NULL flip on `brand_id` / `family_id` (Phase 80 ‚Äî CANON-01/02). Phase 79 leaves the columns nullable but verifies zero NULLs via post-flight assertion.
- Ingest path changes ‚Äî `/api/extract-watch` (Phase 80 ‚Äî INGEST-01..04).
- Recommender JOIN-through swap and Server Action display auto-overwrite on add/edit (Phase 81).
- Admin UI surfaces ‚Äî `/admin/brands` + `/admin/families` (Phase 82 ‚Äî OPS-01/02).
- New schema columns. All Phase 79 work runs against the schema Phase 78 already shipped.

Phase 79 succeeds when: a single `--apply` run against prod completes atomically, the post-flight assertion returns zero unresolved rows, every user's collection grid + detail page + profile rail renders the canonical brand+model string, and `79-POST-DEPLOY.md` is committed with operator sign-off.

<decisions>
## Implementation Decisions

### Apply Gating + Transaction Boundary

- **D-79-01: Strictest pre-flight gate.** Before `--apply` issues any SQL write, the script verifies the decisions file(s) are fully resolved AND match the current catalog state. Refuses to run if ANY of: a row's status is still `needs-review`, a status cell contains an unknown token (per D-78-02 grammar), a `merge:<uuid>` cell points to a non-existent `brands.id` / `watch_families.id`, OR the live catalog has `(brand)` / `(brand, model)` rows that aren't represented in the decisions file. Operator must re-run `--regenerate` to merge-forward and re-review. No `--allow-skips`-style escape hatches. Why: matches Tyler's "Phase 79 is the high-risk phase" framing ‚Äî every safety net cheap; one extra `--regenerate` cycle is a small price to surface drift before SQL hits prod.

- **D-79-02: Confirmation pattern ‚Äî silent local, interactive prod.** The script inspects the active DATABASE_URL connection: when it points at local Supabase (`127.0.0.1:54322`), `--apply` runs silently for fast iteration. When it points at prod, `--apply` prints a summary block (`N brands to create, N catalog rows to resolve, N families to create, N user watches to hydrate, N aliases to append`) and waits for the operator to type `yes` at stdin before issuing the first write. Why: local runs happen many times during dev ‚Äî interactive prompts add friction without buying safety; prod runs are rare and high-stakes ‚Äî the summary forces eyeballing the magnitude before commit.

- **D-79-03: Single atomic transaction wrapping all four writes.** One `BEGIN` opens the apply; the script in order issues (1) INSERT new `brands` rows, (2) UPDATE `watches_catalog.brand_id`, (3) INSERT new `watch_families` rows + UPDATE `watch_families.aliases` (alias appends from merge decisions), (4) UPDATE `watches_catalog.family_id`, (5) UPDATE `watches.brand` + `.model` (hydration), (6) run the MIG-04 post-flight assertion in the same transaction. If any step fails (network blip, Ctrl-C, assertion fails), the entire transaction rolls back ‚Äî the database looks like the script never ran. Why: cleanest rollback story for the highest-risk write in the milestone; the alternative (staged transactions) leaves the DB in a partially-resolved state that's harder to reason about and recover from.

- **D-79-04: Re-run safety via "already applied" detection.** Before the strict pre-flight gate even runs, the script first queries `SELECT count(*) FROM watches_catalog WHERE brand_id IS NULL OR family_id IS NULL`. If the count is zero, print `Already applied ‚Äî nothing to do.` and exit zero. Idempotent on re-run per ROADMAP MIG-02 success criterion #1. Why: avoids accidental double-apply (which would either fail with unique constraint violations on `brands.name_normalized` or no-op silently and confuse the operator); makes the script CI/CD-friendly without sacrificing the strict gate.

### Family Decisions Artifact + Script Topology

- **D-79-05: Same script, `--mode=brands|families|both` switch.** All v8.4 canonicalization logic lives in `scripts/v8.4-brand-canonicalization.ts` (existing file from Phase 78 MIG-01). Phase 79 extends it with three new mode values + `--apply`. Default mode preserves Phase 78's brand-only dry-run behavior. `--mode=families` reads `.planning/v8.4-brand-merge-decisions.md` in-memory (resolves the brand decisions WITHOUT writing them to the DB) and emits `.planning/v8.4-family-merge-decisions.md`. `--mode=both` (the apply path) requires both files to exist + pass the strict gate; runs the full atomic transaction. Why: one connection bootstrap, one parse routine, one gate implementation ‚Äî fewer drift surfaces between brand and family code paths; the args matrix is contained.

- **D-79-06: Merge decisions auto-append the source string to the canonical family's aliases.** When the operator marks a family row as `merge:<datejust-uuid>`, the apply not only updates the catalog row's `family_id` but ALSO appends `lower(trim('Brut Date'))` to that canonical family's `aliases` text[] (idempotent ‚Äî checks for containment first to avoid duplicates if `--apply` is re-run on a partially-resolved state). This is what makes Phase 80's INGEST-04 work: a future `/api/extract-watch` URL with `Brut Date` resolves via `aliases @>` containment to the canonical `Brut Datejust` family. Why: aliases are the entire point of the column; this is the only phase that has the operator-decided merge data needed to populate them ‚Äî putting it anywhere else duplicates the operator-decision parsing.

- **D-79-07: Family dry-run reads brand decisions in-memory (Option 2 path).** The family dry-run does NOT require brand `--apply` to have run first. Instead, it parses `brand-merge-decisions.md`, builds an in-memory map of `brand_raw ‚Üí resolved canonical brand identity` (auto-resolved ‚Üí existing uuid; merge:<uuid> ‚Üí that uuid; new ‚Üí synthetic placeholder keyed by brand_raw), and uses that map to scope `(brand_raw, model_raw)` ‚Üí canonical brand. Result: the family file is already deduped (`Hamilton Watch + Khaki Field` and `Hamilton + Khaki Field` collapse to one canonical row in the file) BEFORE the operator reads it. Why: preserves the single atomic `--apply` (one prod operation, not two); the in-memory chain is a contained complexity inside the script; strict gate catches the "operator edited brand file after generating family file" stale case via `--regenerate`.

### Watches Display Hydration (DISP-03)

- **D-79-08: Write through every catalog-linked watch, no exceptions.** Step (5) of the atomic transaction is `UPDATE watches SET brand = b.name, model = f.name FROM brands b, watch_families f, watches_catalog c WHERE watches.catalog_id = c.id AND c.brand_id = b.id AND c.family_id = f.id`. Every catalog-linked row (effectively every row, since `watches.catalog_id` is NOT NULL since Phase 38; the column is only nullable post-hoc if a catalog row is later deleted via Phase 17 ON DELETE SET NULL) gets its brand+model overwritten unconditionally ‚Äî even if the stored text already matches the canonical (no-op write). Only `brand` + `model` columns are touched; `notes`, `serial`, `reference`, and every other text column stay untouched. Why: simplest SQL, easiest to reason about, and the `updated_at` bump is acceptable for a one-shot milestone-level migration (the alternative "skip no-op writes" optimization saves nothing operationally and adds a WHERE-clause that has to match the post-flight predicate divergence rule).

### `new` Row Defaults + Audit Artifact

- **D-79-09: Operator-marked `new` brand and family rows default to `needs_review = false`.** Every `INSERT INTO brands (name, needs_review) VALUES (...)` and `INSERT INTO watch_families (...)` in Phase 79's apply sets `needs_review = false`. The operator marked the row `new` ‚Äî that IS the approval signal. Phase 82's `/admin/brands` + `/admin/families` queues will be empty by default after Phase 79 ships; rows only appear there when Phase 80's auto-create path flags them on ingest. The IWC + IWC Schaffhausen case (Tyler's per-row operator note in the artifact) is handled by manual /admin/brands cleanup in Phase 82, NOT by a Phase 79 flag. Why: keeps Phase 79 in the "execute operator decisions" lane; punts ambiguity to the Phase 82 UI where the operator already has merge/rename/confirm actions; avoids extending the D-78-02 grammar with a `new!` token.

- **D-79-10: Full `79-POST-DEPLOY.md` auto-generated by the script after successful apply.** After the atomic transaction commits and the post-flight assertion returns zero unresolved rows, the script writes `.planning/phases/79-backfill-migration-display-hydration/79-POST-DEPLOY.md` containing: (a) summary counts (N brands created, N catalog rows resolved per status type ‚Äî auto-resolved/merge/new, N families created, N aliases appended, N user watches hydrated), (b) the post-flight assertion query + result, (c) operator sign-off checklist (verification queries to paste into Supabase SQL editor, parallel to Phase 78's 5-query checklist), (d) "what this push does NOT do" forward-armor section listing Phase 80/81/82 deferred work. Operator commits the file post-verification. Why: mirrors Phase 78's successful POST-DEPLOY pattern (high-information artifact, low-effort to read in a milestone retrospective); the auto-generation prevents the "wrote the script, forgot to write the audit" gap.

### Claude's Discretion

- **Post-flight assertion phrasing:** per MIG-04 + `[[post-flight-assertion-predicate-divergence]]`, the assertion must use a DIFFERENT predicate from the UPDATE's WHERE-clause. The UPDATE writes `WHERE brand_id IS NULL` (and family_id). The assertion uses `SELECT COUNT(*) FROM watches_catalog WHERE brand_id IS DISTINCT FROM NULL` (positive predicate; should equal `(SELECT COUNT(*) FROM watches_catalog)`) AND analogously for family_id. Fails the transaction if any row is unresolved. Planner picks the precise SQL phrasing; the divergence is the contract.
- **Migration filename:** no NEW `.sql` migration is required for Phase 79 ‚Äî all writes happen via the `tsx --apply` script against the schema Phase 78 already shipped. Phase 79 ships scripts + decision artifacts + POST-DEPLOY record, not new SQL migration files. If the planner discovers a missing schema piece (none expected), it'll add a hand-written portable `.sql` migration per `[[drizzle-supabase-db-mismatch]]` rules.
- **In-memory brand-decision map data structure:** planner picks; a `Map<brand_raw_normalized, { kind: 'existing' | 'merge' | 'new', uuid?: string, synthetic_key?: string }>` is the obvious shape.
- **Local-first verification gate:** Phase 79 MUST verify on local Supabase against the seeded local catalog (~205 rows + the 4 seeded users' watches) BEFORE the prod push, per CLAUDE.md "## Local-First Development" and `[[local-first-dev]]`. The local seed includes the SEED-021 canonical drift cases (Hamilton/Hamilton Watch); the local apply MUST produce zero unresolved rows AND hydrate the seeded users' watch text from `Hamilton Watch Khaki` ‚Üí `Hamilton Khaki` before any `--apply` runs against prod.
- **Connection model:** reuses Phase 78 D-78-06 ‚Äî `tsx` script, `DATABASE_URL` env var, service-role bypass on RLS. No change.

## Specific Ideas

- **The single brand merge case is the canonical end-to-end correctness check.** `Hamilton Watch ‚Üí 20969364-f3b1-4b1d-ab2f-e5d22e9ffabc` (Hamilton). After `--apply`:
  - `brands.id = '20969364...'` row's `needs_review` stays whatever it was (untouched).
  - Every `watches_catalog` row with `brand_raw = 'Hamilton Watch'` has `brand_id = '20969364...'`.
  - Every `watches_catalog` row with `brand_raw = 'Hamilton'` ALSO has `brand_id = '20969364...'` (the auto-resolved case).
  - Both groups of catalog rows now JOIN to the SAME brand ‚Üí recommender exclusion key works (RECO-01 precondition).
  - Every user's `watches` row pointing at any of those catalog rows has `watches.brand = 'Hamilton'` (canonical name from `brands.name`, NOT the user's free-text).
- **The Brut Date ‚Üí Brut Datejust case is the canonical FAMILY merge/alias test.** After `--apply`:
  - `watch_families` row for canonical `Brut Datejust` has `aliases @> ARRAY['brut date']`.
  - Every `watches_catalog` row with `model_raw = 'Brut Date'` resolves to the canonical `Brut Datejust` family id.
  - No duplicate `Brut Date` family row is created (gate enforced by `watch_families_brand_name_unique` constraint, but the script's resolution logic prevents the INSERT attempt entirely).
- **IWC + IWC Schaffhausen is NOT a Phase 79 concern.** Operator marked both `new` deliberately (artifact line 12 comment). Phase 79 creates two `brands` rows with `needs_review = false`. The follow-up merge happens in Phase 82's `/admin/brands` UI. Don't auto-flag this case in Phase 79.
- **POST-DEPLOY counts:** the script's summary block + the `79-POST-DEPLOY.md` artifact must include (a) brand counts by decision type (19 auto-resolved + 1 merge + 33 new = 53), (b) catalog rows resolved (should equal `SELECT count(*) FROM watches_catalog`), (c) family counts by decision type (TBD when family dry-run runs), (d) aliases appended (one per family `merge:` decision), (e) watches hydrated (should equal `SELECT count(*) FROM watches`). The post-flight assertion query result lands in the artifact too.
- **The `tsx --apply` mental model:** Phase 79 does NOT push a new SQL migration via `supabase db push --linked`. It pushes a script. Operator runs `tsx scripts/v8.4-brand-canonicalization.ts --apply --mode=both` against prod after local-first verification. This is a different mental model from Phase 78 (which DID push a SQL migration). Plan-phase MUST distinguish these.

<deferred>
## Deferred Ideas

- **Two-step prod sequencing (brand `--apply` ‚Üí verify ‚Üí family `--apply`).** Considered as a safety checkpoint. Rejected: strict gate + single atomic transaction + post-flight assertion provide equivalent safety with one prod operation. Two-step risks catalog drift between pushes (ingest could hit prod between brand apply and family dry-run).
- **Combined single review file (brands + families interleaved).** Considered for the dry-run output. Rejected: a single file mixing both domains is harder to scan and edit; two separate files (`brand-merge-decisions.md` + `family-merge-decisions.md`) map cleaner to operator workflow.
- **`new!` / `new:review` extended grammar for per-row `needs_review = true` flagging.** Considered for the IWC + IWC Schaffhausen case. Rejected (D-79-09): keeps grammar small; punts the IWC follow-up cleanup to Phase 82's `/admin/brands` UI which already has merge/rename actions.
- **Pre-apply rollback table (`watches_pre_v8_4_strings` for old brand/model values).** Considered as a belt-and-suspenders post-deploy safety net. Rejected: D-08 says reversibility is dry-run only; a one-shot rollback table adds cleanup debt for a low-probability scenario already covered by the atomic transaction + post-flight assertion. Snapshot-the-DB-before-apply (via Supabase backup) is the right belt for the suspenders case.
- **Skip no-op writes during hydration** (only touch rows where canonical differs from stored). Considered for efficiency. Rejected (D-79-08): simpler SQL is worth the trivial DB churn for a one-shot milestone-level migration.
- **Filtering hydration by status** (e.g., skip `sold` watches). Considered. Rejected: the success criterion + DISP-03 both say "every catalog-linked watches row" ‚Äî no status filtering. A user's sold watch should still display canonical strings on their profile rail / collection grid.
- **Send a user-facing notification when their watch's display text changes.** Considered as v8.4 polish. Rejected: scope creep ‚Äî belongs in its own phase (or, more likely, is just unnecessary since the change is purely cosmetic and corrects drift the user didn't intend).
- **Pre-flight count-match verification** (decisions file row count must equal distinct catalog brand count). Already covered implicitly by the strict gate's "catalog has rows not in decisions file" check (D-79-01).
- **Operator dry-run preview of the actual SQL** (`--apply --dry-sql` that prints the INSERT/UPDATE statements without executing). Considered as added safety. Rejected: the summary counts + interactive `yes` prompt + atomic transaction provide equivalent safety; printing 200+ SQL statements adds noise without buying real verification.

</deferred>
