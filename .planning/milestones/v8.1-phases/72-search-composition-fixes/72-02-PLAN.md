<objective>
Close SRCH-02 (combobox keyboard navigation broken) and SRCH-03 ("Not finding it?" footer click no-op) with two surgical edits in `SearchEntry.tsx`, plus regression tests in `SearchEntry.test.tsx`.

Purpose:
- SRCH-02 — base-ui's default `isItemEqualToValue = Object.is` fails for our object-valued `<Combobox.Item value={r}>` (fresh result objects never reference-equal across renders). `selectedIndex` stays `-1`, `activeIndex` never gets initialized, and Enter falls into the v1.3.0 `#4235` "preserve input on Enter when nothing highlighted" path → popup closes without firing `onPick`. Adding `isItemEqualToValue={(a, b) => a.catalogId === b.catalogId}` to `<Combobox.Root>` restores correct active-index tracking and keyboard navigation.
- SRCH-03 — the footer `<button>` lives inside `<Combobox.List>` (which renders as `role="listbox"`); base-ui only delegates click routing to `<Combobox.Item>` children of List, so the plain button never receives clicks in real browsers (jsdom doesn't enforce this so test 12 "passes" today despite the prod bug). Moving the button OUTSIDE List but INSIDE `<Combobox.Popup>` as a sibling restores native click semantics.

Output:
- `src/components/watch/SearchEntry.tsx` — `isItemEqualToValue` prop added (single addition per D-07); footer button relocated as a sibling of `<Combobox.List>` within `<Combobox.Popup>`.
- `src/components/watch/SearchEntry.test.tsx` — extended with a new describe block for SRCH-02 keyboard nav (ArrowDown/ArrowUp/Enter/Escape) and a structural assertion for SRCH-03 (footer is NOT a descendant of the listbox element).
</objective>
